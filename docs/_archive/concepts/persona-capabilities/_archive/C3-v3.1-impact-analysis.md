## 2. File touch list for "execute now"

- **`src-tauri/src/engine/template_v3.rs`**
  - `migrate_question_use_case_id()` — read `use_case_id` (string), emit
    `use_case_ids` (array of one). Called during flatten.
  - `preserve_connector_required()` — default missing `required` to
    `true`; pass field through to flat IR connector list.
  - `preserve_composition()` — pass `trigger_composition` and
    `message_composition` from persona to flat IR top level.
  - `strip_use_case_flow_for_prompt()` — if a flat IR consumer asks for
    the prompt-building view, use_case_flow is excluded. Prompt builder

    code. **No-op to implement, but added as a comment near the flat
    output struct.**

## 3. Deferred work (captured as follow-up scope)

Each item below is a self-contained ticket for a later session.

### D1 — Trigger/message composition adoption step
- New components: `TriggerCompositionStep.tsx`, `MessageCompositionStep.tsx`.
- Slots between `UseCasePickerStep` and `QuestionnaireFormGrid` in the
  adoption flow.
- Respects `persona.trigger_composition` and `message_composition`.
- Quick-setup presets + custom-cron escape hatch.
- Event-listen triggers skip the composition step for their UC.
- Depends on: nothing in v3.1 normalizer work.

### D2 — Connector gate (required + optional)
- New component: `ConnectorGateStep.tsx`.
- Renders before the questionnaire for each connector referenced by an
  enabled UC.
- Required + missing → empty-state with vault CTA; blocks progression.
- Optional + missing → pick-or-skip; sets a session flag consumed by
  promote.
- Promote pipeline respects skip flag — strips connector + tolerates
  fallback.
- Depends on: `persona.connectors[].required` (done this pass) and
  `fallback_note` (done this pass).

### D3 — `ui_component` registry
- New file: `src/features/templates/inputSchemaComponents.ts`.
- Exports `COMPONENT_REGISTRY: Record<string, ComponentType<InputSchemaFieldProps>>`.
- First entry: `CodebaseSelector` for the Idea Harvester.
- Questionnaire rendering looks up `input_schema[i].ui_component` and
  falls back to the `type`-based renderer when the key is unknown.
- Depends on: nothing — `ui_component` is already a free-form string
  field that authors set (null for most fields).

### D4 — Event-name linter in checksum script
- `scripts/generate-template-checksums.mjs` gains a pass that walks
  every event_type across every template and warns on any that doesn't
  match `/^[a-z][a-z0-9_]*\.[a-z][a-z0-9_]*\.[a-z][a-z0-9_]*$/`.
- Non-blocking (warn only) — not enforced at build time.

### D5 — Retroactive light edits on existing 3 templates
- `email-morning-digest` — flow simplification (P6), ensure `use_case_ids`.
- `youtube-content-pipeline` — same.
- `autonomous-issue-resolver` — same.
- The 13 overlays for autonomous-issue-resolver stay valid since
  overlays don't touch structural fields.

## 4. Migration risk

- **Existing personas promoted from v3 templates** — DB-stored
  `design_context` already serialized the flat IR with singular
  `use_case_id`. Reading path handles both shapes. No migration
  needed.
- **Templates with flow diagrams that already entered prompts** —
  unlikely (prompt builder reads `operating_instructions`), but
  verify with a grep.
- **Overlay merge** — P5's switch from singular to plural:
  `mergeTemplateOverlay()` handles both because its merge is schema-
  agnostic. Overlays don't set `use_case_ids` (structural field).
