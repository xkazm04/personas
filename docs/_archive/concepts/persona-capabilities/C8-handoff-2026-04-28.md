## Architecture — what's true now

### Build wizard pipeline (post-C7)

```
UnifiedMatrixEntry
  → createPersona({system_prompt: "You are…"})    [ persona row, enabled=false ]
  → start_build_session(persona_id, intent)
  → phases:  initializing → analyzing → resolving ↔ awaiting_input → draft_ready
  → test_build_draft  [ run each tool against live API ]      → testing → test_complete
                                                                          ↓
                                                                  ┌───────┴───────┐
                                                                  │               │
                                                          simulate_build_draft   promote_build_draft  ← C7 dry-run path
                                                          (NEW C7)
                                                                                  ↓
                                                                          design_context written,
                                                                          triggers + subs + smee_relays
                                                                          materialised
                                                                                  ↓
                                                                          Promoted
```

Phase machine is **strict** (`build_session.rs::validate_transition`):
`DraftReady → Testing | Resolving | Promoted`,
`Testing → TestComplete | DraftReady | Testing`,
`TestComplete → Testing | Promoted`. There is **no SimulationPhase** — dry-run is a side action callable from `draft_ready` / `testing` / `test_complete` / `promoted`.

### IR shape (post-C7) — what's actually persisted where

Three storage planes, each with different semantics:

| Field | Where written | What it carries | Who reads it |
|---|---|---|---|
| `build_sessions.agent_ir` | LLM emits `agent_ir` event → runner stores raw JSON | Full v3 IR, LLM-emitted UC ids (`uc_generate_invoice`), `persona` block | `simulate_build_draft`, `test_build_draft`, dry-run snapshot |
| `personas.design_context` | `promote_build_draft_inner::update_persona_in_tx` | `useCases[]` (camelCase, UUID-rekeyed: `uc-b98e…`) + `summary` + `builderMeta`. **C7**: now also `review_policy`, `generation_settings`, `memory_policy` per UC | `pick_use_cases_array` helper (tries snake/camel both), `pick_generation_policy`, `cascade_use_case_toggle` |
| `personas.last_design_result` | `promote_build_draft_inner` | Flat v2 shape (`suggested_*`, `use_case_flows`, `structured_prompt`, `full_prompt_markdown`). **C7**: now also `persona` block (mission/principles/constraints/decision_principles) | `auto_triage::extract_principles_from_design_result`, design preview UI |

**Critical mental model**: the LLM emits one IR shape (v3 nested with `persona` block + `use_cases` array using LLM names), the promote pipeline transforms it into TWO outputs (camelCase design_context with UUID-rekeyed ids, flat v2 last_design_result with persona block preserved). The two outputs serve different consumers and have different key conventions — use `engine::design_context::pick_use_cases_array` whenever you need to read `use_cases` from any source.

### Trigger config (post-C7)

`TriggerConfig::Webhook` extended with:
- `smee_channel_url: Option<String>` — set by the LLM from the user's typed-payload answer per build prompt rule 24
- `smee_event_filter: Option<String>` — comma-separated event_type allowlist forwarded to `smee_relays.event_filter`

Validator (`validation/trigger.rs`) rejects malformed smee URLs at build/promote time.

`auto_create_smee_relays` (post-tx, best-effort) iterates webhook triggers carrying `smee_channel_url`, calls `smee_repo::create` with `target_persona_id = persona_id`. Idempotent on re-promote (same URL+persona = skip), refuses to repoint cross-persona.

### Build prompt rules (post-C7)

| Rule | What it does |
|---|---|
| 21 | `mode: "auto_triage"` for "let agent decide" / "decision-principle-judged" intents |
| 22 | Verbatim event names — never rewrite backtick-quoted user-supplied event_types |
| **23 (new)** | When the LLM needs a sample/template/example to design correctly, emit clarifying_question with `accepts_reference: true`. User attaches file/URL/inline; backend fences contents into the answer text |
| **24 (new)** | When the LLM picks `webhook` trigger, MUST also emit clarifying_question with `accepts_webhook_source: true` so the user can attach a smee.io URL. Promote auto-creates the smee_relays binding |

### Auth tier system

Three tiers in `ipc_auth.rs`:
- **Public** — no token, no OAuth (read-only local DB queries: `list_personas`, `list_credentials`, `vault_status`, **C7** `smee_relay_list`, `smee_relay_delete`)
- **Privileged** — IPC session token required (writes/deletes for credentials, vault, etc.)
- **Cloud** — token + Google OAuth (cloud forwarding, GitLab deploy)

**C7 lesson**: a command's tier should follow what it actually DOES, not what it might LOOK like. `smee_relay_list` and `smee_relay_delete` are local-DB-only — they don't talk to smee.io (which is account-less anyway). Originally classified as Cloud, relaxed to Public+`require_auth`-inner so the test bridge could call them and offline mode users could use SmeeRelayTab.

### auto_triage runtime (post-C7)

When a capability declares `review_policy.mode = "auto_triage"`:
1. `dispatch.rs::ProtocolMessage::ManualReview` creates the review row with status `Pending`
2. Spawns a tokio task running `engine::auto_triage::evaluate()` — that calls `claude -p -` in single-turn mode against an evaluator prompt built from `decision_principles[]`, `principles[]`, `constraints[]` (extracted from `last_design_result.persona`)
3. Verdict (`approve` / `reject`) → `update_status(Approved|Rejected, reasoning)` + audit `review.auto_triage.approved` or `.rejected`
4. On any failure (CLI spawn / timeout / parse) → `update_status(Resolved, …)` + audit `review.auto_triage.fallback`

Dispatch loop is never blocked — the evaluator runs out-of-band.

## Lessons learned — patterns to repeat / pitfalls to dodge

### 1. The build session is a stateful conversation, not a one-shot

The LLM emits ~5-30 events per build (clarifying_question / capability_resolution / persona_resolution / agent_ir). Phase G builds in 6 rounds; Phase D run 1 wedged at 22 rounds without ever emitting agent_ir. **The intent matters enormously** — verbose intents with multi-step chains can confuse the LLM into never finalising. When a Phase driver hangs without producing agent_ir, simplify the intent before debugging the driver.

### 2. The agent_ir-landing race is real

The build session can transition phase to `test_complete` BEFORE `session.agent_ir` is persisted to the row. C6 added a 2s retry inside `promote_build_draft_inner` (`engine/build_session/runner.rs`'s loop variant). **C7 lesson**: add a client-side `wait_for_agent_ir(persona_id, max_seconds=60)` poll in every Phase driver between `answer_dimensions` and `test_and_promote`. It probes `getActiveBuildSession.agentIr` until non-null. Without it, drivers fail with "Build session has no agent_ir (field is null after retry)" intermittently.

### 3. The LLM places connectors at THREE possible levels

Found during Phase I:
- `last_design_result.persona.connectors[]` — persona-wide (v3 build prompt asks for this)
- `last_design_result.use_case_flows[i].tool_hints[]` — per-UC (v3 reference style)
- `design_context.useCases[i].connectors[]` — per-UC structured (legacy / promote-time)

Acceptance gates need to probe **all three**. The Phase I driver's `_all_persona_connectors` + `_connector_names(uc, including tool_hints)` helpers are the canonical pattern.

### 4. Promote re-keys UC ids; session.agent_ir keeps the LLM names

Post-promote: `design_context.useCases[i].id == "uc-b98e7b9a-3617-49e1-a587-b5bc6886eb35"` (UUID).
Pre-promote: `session.agent_ir.use_cases[i].id == "uc_generate_invoice"` (LLM-emitted snake_case).

`simulate_build_draft` reads `session.agent_ir` so it expects the LLM names. If you fetch UC ids from `getPersonaDetail` (which surfaces design_context) and feed them to `simulateBuildDraft`, you get a "use_case_id 'uc-b98e…' not found" error. **Workaround**: feed the LLM-emitted ids from `getActiveBuildSession.agentIr.use_cases[i].id` instead.

This is the open #1 follow-up — `simulate_build_draft` should accept either id form via a small lookup adapter.

### 5. Windows console encoding (cp1252) crashes on non-ASCII

Czech `č`, German `ä`, etc. crash Python `print()` with `UnicodeEncodeError`. Add at the top of every driver:

```python
try:
    sys.stdout.reconfigure(encoding="utf-8", errors="replace")
    sys.stderr.reconfigure(encoding="utf-8", errors="replace")
except (AttributeError, ValueError):
    pass
```

The JSON report file is unaffected (Path.write_text defaults to UTF-8 on Python 3.9+).

### 6. The test bridge can't call Privileged or Cloud commands

`window.__TAURI_INTERNALS__.invoke` is monkey-patched at app startup to attach the IPC session token, but the patch races with the first invoke on Windows WebView2 (per the comment at `ipc_auth.rs:498-510`). The frontend `invokeWithTimeout` wrapper injects the token explicitly as a fallback — but the test bridge uses raw `invoke` which doesn't.

**Practical implication**: any IPC the bridge needs to call must be in the **Public** tier (no token required). Local-DB read-only commands and unsensitive deletes can safely move there. Two commands moved this wave: `smee_relay_list` and `smee_relay_delete`. Their inner handlers still call `require_auth` so they're not entirely ungated.

If you need bridge access to a Privileged or Cloud command, either (a) move it to Public if its blast radius is OK, or (b) add a test-only `bridge_<cmd>` shim in `test_automation.rs` that bypasses the gate.

### 7. Tauri-cli rebuild detection: 30-90 seconds

Touching a Rust file under `src-tauri/src/**` triggers tauri-cli to recompile + restart the binary. The webview is killed too. The /health probe goes red for ~40-90 seconds (varies by which file changed — touching a leaf file is faster than touching `lib.rs`).

Pattern in driver-fix iterations:

```bash
echo "// touch $(date +%s)" >> src-tauri/src/<file>.rs       # force rebuild
START=$(date +%s)
until ! curl -s --max-time 1 http://127.0.0.1:17320/health 2>/dev/null | grep -q '"status":"ok"'; do
  if [ $(($(date +%s) - START)) -gt 30 ]; then echo "rebuild not detected"; break; fi
  sleep 2
done
START=$(date +%s)
until curl -s --max-time 2 http://127.0.0.1:17320/health 2>/dev/null | grep -q '"status":"ok"'; do
  sleep 2
done
echo "Backend back (waited $(($(date +%s) - START))s)"
```

**Cleanup**: strip the `// touch <ts>` comment lines from each touched file before committing — they're rebuild-forcing noise, not real changes.

### 8. Background python with `&` double-forks and orphans

`python script.py 2>&1 &` from a Bash run-in-background tool wraps the python in a subshell that backgrounds — and the wrapper exits 0 immediately, but the python keeps running uncaptured. **Pattern**: always use `run_in_background: true` directly, never combine with shell `&`. If you accidentally orphan a build, the process keeps running but produces no further log output to the captured file.

### 9. The auto_triage decision_principles need to be a STRUCTURED ARRAY

The matrix-builder used to embed `decision_principles` only in `structured_prompt.instructions` prose. The auto_triage second-pass evaluator's `extract_principles_from_design_result` reads `persona.decision_principles[]` as a string array. Without #3's hoist fix (now landed), the evaluator ran with empty principles. Verified working in Phase I run 2 — LLM emitted 2 principles, both visible at `last_design_result.persona.decision_principles`.

### 10. Build-prompt rule changes propagate IMMEDIATELY at next CLI spawn

Editing `session_prompt.rs` doesn't need any cache invalidation — the prompt is composed fresh per build session. Touch the file, wait for tauri rebuild, next `startBuildFromIntent` uses the new prompt.
