## C7 â€” 2026-04-27 EOD increment: dry-run preview

Carries forward from the C6 wishlist of new build-flow features (alongside reference-input questionnaires and webhook triggers, both still pending). The dry-run feature lets a user preview a capability's runtime behaviour from the build wizard *before* promoting â€” they get the actual agent output (manual reviews + memories) without sending real notifications.

### Backend â€” `src-tauri/src/commands/design/build_simulate.rs` (new module, 8 unit tests)

- **Pure helper** `build_simulation_design_context(ir: &AgentIr) -> String` â€” produces a snake-case `{"use_cases": [...], "summary": ..., "_simulation_snapshot": true}` JSON snapshot from an `AgentIr`. Each UC entry preserves `id`, `sample_input`, `time_filter`, `model_override`. When the IR omits an `id`, one is fabricated as `uc_<idx>_<title-slug>` so dry-run can still address the UC.
- **`#[tauri::command] simulate_build_draft(session_id, use_case_id, input_override?)`** â€” loads the build session, applies adoption_answers (mirrors `test_build_draft`), persists the snapshot to `personas.design_context`, then delegates to `execute_persona_inner` with `is_simulation=true`. The phase machine is *not* mutated â€” simulation is a side action callable from `draft_ready` / `testing` / `test_complete` / `promoted` (refused from earlier phases with a clear validation error).
- **`#[tauri::command] get_simulation_artefacts(execution_id) -> SimulationArtefacts`** â€” `{ executionId, reviews, memories }`. Uses existing `get_by_execution` repo accessors. **Messages and events are deliberately deferred** â€” `persona_messages` has the `execution_id` column but no scoped accessor today, and `persona_events` uses `source_id` for execution linkage which needs a small new query. The Execution Detail tab (post-promote) covers those views; v1 dry-run focuses on the user-visible artefacts.
- **Risk-grep** confirmed it is safe to write `design_context` to a draft (`enabled=false`) persona: schedulers (`background.rs:760`, `director.rs:548`) gate on `enabled=1`; the bus reads `persona_triggers` / `persona_event_subscriptions` rows (empty for drafts); the frontend persona list filters via `isDraft()` so the row stays in the "building" tab; `simulate_use_case` and `execute_persona_inner` already bypass the persona-enabled gate intentionally.
- **Side-discovery (latent bug, NOT fixed in this slice):** `commands/design/build_sessions.rs::build_design_json` writes the design_context as `{"useCases": [...]}` (camelCase), but `commands/execution/executions.rs:174`, `commands/core/use_cases.rs:568`, and `commands/core/use_cases.rs:60` all read `dc.get("use_cases")` (snake_case). My dry-run snapshot writes snake_case `use_cases` to match the readers â€” and `promote_build_draft` overwrites it with camelCase `useCases`, restoring the latent bug. Worth a follow-up to make the readers tolerant of both shapes (one-line change with a `or_else`). **Fixed later this session â€” see "design_context case-key bug fix" below.**

## C7 â€” 2026-04-27 EOD increment (continued): design_context case-key bug fix + reference-input questionnaire

### Bug fix â€” `design_context` case-key inconsistency

The matrix-builder's `commands/design/build_sessions.rs::build_design_json` writes the design_context envelope as `{"useCases": [...]}` (camelCase via `DesignContextData`'s `rename_all`), but several runtime readers were hardcoded against `dc.get("use_cases")` (snake_case): `simulate_use_case`, `cascade_use_case_toggle`, `execute_persona_inner`, `engine::dispatch::testable::{pick_capability_channels, pick_generation_policy}`, `engine::prompt::advisory`, `engine::prompt::capabilities::{render_active_capabilities, active_capabilities_fingerprint}`. Result: those readers silently saw "no use_cases" on every matrix-built persona â€” `simulate_use_case` and `cascade_use_case_toggle` were effectively broken; the prompt assembler skipped the Active Capabilities section; the dispatcher used permissive defaults instead of capability-scoped policy.

**Fix:** new `src-tauri/src/engine/design_context.rs` module (7 unit tests):
- `pick_use_cases_array(&Value) -> Option<&Vec<Value>>` â€” tries `use_cases` first (snake_case), falls back to `useCases` (camelCase).
- `pick_use_cases_array_mut(&mut Value) -> Option<&mut Vec<Value>>` â€” mutable variant for `cascade_use_case_toggle` / `patch_generation_settings`.

Migrated every design_context reader to use the helper. Template / agent_ir / IR-shape readers (`engine::template_v3`, `commands/design/n8n_transform/streaming`) intentionally stayed snake-case-only â€” those payloads have a well-defined IR schema separate from the design_context envelope.

Two pre-existing camelCase-only readers also migrated (`engine::runner::credentials`, `engine::runner::mod`) so the runner now also accepts both shapes â€” meaning C7 dry-run snapshots (snake_case) are interoperable with the runner that previously only saw promoted personas (camelCase).

**Tests pinned all touched paths:** 7 design_context + 21 dispatch + 20 use_cases + 8 build_simulate + 14 auto_triage all green; no regression.

### Reference-input questionnaire (the next C6 wishlist item)

**Goal:** when the build LLM needs a sample / template / example to design correctly (cues: "match this format", "use this template", `sample_input` / `output_format` fields), it asks for one â€” and the user can attach a file, a URL, or pasted text. Contents are fetched server-side, fenced, and prepended to the answer the CLI sees on its next `--continue` turn.

#### Backend â€” `src-tauri/src/engine/build_session/reference.rs` (new module, 13 unit tests)

- **Pure injector** `inject_reference_into_answer(answer, name, content) -> String` â€” wraps content in `--- ATTACHED REFERENCE: <name> ---\n<content>\n--- END REFERENCE ---\n`, separates from the user's freetext with two newlines (collapses trailing whitespace). Empty answer â†’ no leading separator.
- **File reader** `read_file_reference(path)` â€” extension allowlist (txt/md/json/yaml/csv/html/xml/source-code/ini/log/etc.), 256 KB cap, UTF-8 decode, returns `(basename, content)` so the LLM sees just the filename.
- **URL fetcher** `fetch_url_reference(url)` â€” pre-flight `validate_url_safety` (rejects loopback, private IPs, non-http schemes, blocked metadata hostnames), `build_ssrf_safe_client` for DNS-rebinding-safe transport (10s timeout), content-type allowlist (`text/*`, `application/{json,xml,yaml,toml}`), 256 KB body cap with truncation marker.
- **Materialise dispatcher** `materialise_reference(ReferenceSource::{File|Url|Inline})` â€” single entry routing to the right fetcher.

`UserAnswer` extended with optional `reference: Option<UserAnswerReference>` (`{path, url, inline_content, name}`); `answer_build_question` IPC accepts the new field, resolves it server-side via `materialise_reference`, prepends fenced contents to the answer text before piping to the CLI subprocess.

`BuildEvent::{Question, ClarifyingQuestionV3}` and the parser gain an `accepts_reference: bool` field (defaults false; mirrors the LLM-emitted JSON flag with the same name).

#### Build prompt rule 23 â€” when to ask for a reference

Added to `src-tauri/src/engine/build_session/session_prompt.rs`. Concrete trigger patterns: `sample_input` field design when intent mentions a real format, `sample_output` / template matching, external API target where the user references a fixture/spec/schema, persona tone/style matching an existing artefact. Lists the JSON shape (`accepts_reference: true` flag), the fence format the LLM should expect on the next turn, and a "when NOT to ask" anti-pattern list (intent already inlines example, well-known enums, vault picker scope, confirmation-only questions).

#### Frontend â€” `src/features/agents/components/matrix/ReferenceAttachmentPicker.tsx` (new, 9 Vitest tests)

Three-mode picker: file (via `@tauri-apps/plugin-dialog::open()` with text-extension filters), URL (HTTPS input with submit-on-Enter), inline (textarea paste). Once a reference is attached it renders as a chip with a clear button. Submit button in `SpatialQuestionPopover` is enabled when EITHER text OR a reference is present (reference alone is a valid answer).

`BuildQuestion` extended with `acceptsReference?: boolean`; `BuildReference` interface added. `SpatialQuestionPopover` exposes the picker block when the question carries the flag and widens its `onAnswer` callback to forward the reference. `useBuildSession::answerQuestion` short-circuits batching when a reference is supplied (sends straight to `answer_build_question` with the reference) â€” non-reference answers continue using the existing collect+submit batch path.

Test bridge helper `answerBuildQuestionWithReference(cellKey, answer, reference)` added for future Phase scenarios.

## C7 â€” 2026-04-28 increment: webhook triggers in the build flow

**Goal:** when the build LLM picks `trigger_type: "webhook"` for a capability, the questionnaire asks the user for a smee.io channel URL up front. At promote time the smee_relays binding is auto-created so the persona starts receiving forwarded events immediately â€” no manual SmeeRelayTab dance afterwards.

### Backend

#### Trigger config extension â€” `src-tauri/src/db/models/trigger.rs`

`TriggerConfig::Webhook` gains two optional fields:

```rust
Webhook {
    webhook_secret: Option<String>,
    event_type: Option<String>,
    payload: Option<serde_json::Value>,
    smee_channel_url: Option<String>,    // NEW â€” set by the LLM from the answer
    smee_event_filter: Option<String>,   // NEW â€” comma-separated event_type allowlist
}
```

Both fields are `#[serde(default, skip_serializing_if = "Option::is_none")]` â€” backward-compatible with existing webhook trigger rows. `parse_config` round-trips them; the runtime webhook listener (`engine/webhook.rs`) ignores them (HMAC verification is unchanged).

#### Validator â€” `src-tauri/src/validation/trigger.rs`

When `smee_channel_url` is present and non-blank, it must start with `https://smee.io/`. Rejects malformed URLs at validate time so a bad answer fails fast rather than silently being skipped at smee-relay-create.

#### `UserAnswer.webhook_source` â€” `src-tauri/src/db/models/build_session.rs`

```rust
pub struct UserAnswer {
    pub cell_key: String,
    pub answer: String,
    pub reference: Option<UserAnswerReference>,
    pub webhook_source: Option<UserWebhookSource>,  // NEW
}

pub struct UserWebhookSource {
    pub channel_url: String,
    pub event_filter: Option<String>,
}
```

`BuildEvent::{Question, ClarifyingQuestionV3}` both gain `accepts_webhook_source: bool` (defaults false). The parser reads `accepts_webhook_source` from the LLM's clarifying_question JSON.

#### `answer_build_question` IPC â€” `src-tauri/src/commands/design/build_sessions.rs`

The command now accepts an optional `webhook_source` param. When supplied:
1. Validates `channel_url` starts with `https://smee.io/`.
2. Calls new helper `engine::build_session::reference::append_webhook_source_fence(answer, channel_url, event_filter)` â€” appends a `--- WEBHOOK SOURCE ---\nchannel_url: ...\nevent_filter: ...\n--- END WEBHOOK SOURCE ---` block to the answer text.
3. Pipes the augmented answer to the CLI via the existing send_answer path.

#### Build prompt rule 24 â€” `src-tauri/src/engine/build_session/session_prompt.rs`

New rule explains: when the LLM picks webhook trigger, it MUST emit a `clarifying_question` with `accepts_webhook_source: true`. Documents the exact fence shape the user's answer will arrive in on the next `--continue` turn, and gives a concrete IR example showing where to place the URL on the trigger config (`config.smee_channel_url` + `config.smee_event_filter`). Lists when to ASK (default for webhook triggers) and when to SKIP (intent already inlines a smee URL, non-webhook trigger types, user said "I'll set up smee later").

#### Promote-time auto-bind â€” `src-tauri/src/commands/design/build_sessions.rs::auto_create_smee_relays`

Runs AFTER the promote transaction commits (best-effort â€” failures here don't unwind the persona promote; the user can attach manually via SmeeRelayTab later). Iterates `ir.triggers`, finds webhook triggers carrying `smee_channel_url`, and:

- **Idempotent on same `(persona_id, channel_url)`** â€” skips silently when a binding already exists.
- **Refuses to repoint** â€” if a smee_relay row exists with the same URL pointing at a DIFFERENT persona, logs a warning and skips (the user owns that binding; we don't steal it).
- Calls `smee_repo::create` with `target_persona_id = persona_id`, `event_filter = smee_event_filter`, `label = "Build Â· trigger <description-or-index>"`.
- After all webhook triggers processed, calls `state.smee_relay_notifier.notify()` so the relay manager picks up the new rows immediately.

Returns the count of relays created; surfaced in the promote response as `smee_relays_created`.

7 unit tests in `commands::design::build_sessions::tests` cover: skip when no webhook triggers, skip when no smee URL, basic create, idempotent same-persona, refuse cross-persona repoint, mixed trigger types, blank URL skipped.

### Frontend â€” `src/features/agents/components/matrix/WebhookSourcePicker.tsx` (new, 9 Vitest tests)

Self-contained form: smee URL input with format validation (`https://smee.io/...`) + optional comma-separated event-filter input + "+ Don't have one? Create at smee.io/new" button (`openExternalUrl`). Selected source renders as a chip with channel URL + filter brackets + clear button.

`BuildQuestion` extended with `acceptsWebhookSource?: boolean`; `BuildWebhookSource` interface added (`{channelUrl, eventFilter?}`). `SpatialQuestionPopover` mounts the picker block when the flag is set, alongside (mutually exclusive in practice with) the existing reference picker. Submit is enabled when EITHER text OR a reference OR a webhook source is present.

`useBuildSession::answerQuestion` now short-circuits the batch path for both `reference` AND `webhookSource` payloads â€” the typed payload IS the answer for that question. `useMatrixBuild::handleAnswer`, `PersonaMatrix.onAnswerBuildQuestion` prop, and `SpatialQuestionPopover.onAnswer` callback all widened to forward the new payload.

Test bridge helper `answerBuildQuestionWithWebhookSource(cellKey, answer, source)` added for future Phase scenarios.
