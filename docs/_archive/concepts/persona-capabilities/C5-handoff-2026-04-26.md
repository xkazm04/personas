## What landed this session (changes uncommitted on `master`)

### A ŌĆö Cascade guard scoped to (persona, use_case)
`src-tauri/src/engine/background.rs::dispatch_loop`
`src-tauri/src/db/repos/execution/executions.rs::get_running_count_for_persona_use_case`

The cascade guard at the dispatch loop was per-persona only. UC1's emit landed
while UC1 was still completing (~0.4s race) ŌåÆ guard blocked UC2's match because
"persona already has a running execution". Fixed: when a match has a
`use_case_id`, scope the running-count check to that pair. Verified live ŌĆö
Sentry persona's UC1ŌåÆUC2 chain now fires (4 executions captured: manual UC1 +
2 chained UC2 fires from the two `devops.issue.created` emits).

### B ŌĆö listen_event_type probing covers 5 shapes
`src-tauri/src/commands/design/build_sessions.rs::create_triggers_in_tx`

Original code only probed `event_type` (singular string). Extended to also
handle `event` and the array-form keys `event_types[0]`, `events[0]`,
`subscribe_to[0]`. Pre-empts the "UC2 doesn't auto-fire because chain event
used an unprobed shape" follow-up flagged in the prior handoff ┬¦Phase 3 step 6.

### C ŌĆö Registry-aware connector matching
`src-tauri/src/engine/api_proxy.rs` (added `cached_connector_keywords`,
`refresh_connector_keyword_snapshot`, `connector_keyword_snapshot`)
`src-tauri/src/engine/build_session/gates.rs::intent_implies_connectors`
`src-tauri/src/engine/build_session/templates.rs::extract_keywords`
`src-tauri/src/lib.rs` (seeds the snapshot at startup)
`src-tauri/src/commands/credentials/connectors.rs` (refreshes after CRUD)
`src-tauri/src/commands/credentials/openapi_autopilot.rs` (refreshes after CRUD)

Process-wide `RwLock<Vec<String>>` snapshot of lowercase connector names from
the registry. Two intent-analysis heuristics now consult it as a supplement to
their built-in seed lists. Result: connectors added via the catalog UI
participate in keyword matching without a code edit. Live evidence: the 5
scenarios picked up `notion_search`, `google_calendar`, `linear_search`,
`sentry`, `github` ŌĆö none of which are in the original hardcoded fuzzy-aliases
list.

### D ŌĆö Defensive gate auto-open for already-answered cells
`src-tauri/src/engine/build_session/runner.rs::CapabilityResolutionUpdate`

Before the gate-state check, look up the field's legacy cell name (via
`map_capability_field_to_legacy_dimension`) and check `last_answered_cells`
(still populated until the post-loop clear at line 642). If yes, mark gate
`Open` proactively. Without this defense, an LLM that re-emits a `CapRes` for a
field the user already answered gets suppressed ŌåÆ re-synthesised ŌåÆ user sees
the same question again. The Sentry scenario that previously looped now reaches
`test_complete` in 3 rounds.

### E ŌĆö Bridge error unpacking
`src/test/automation/bridge.ts::unpackError`

13 catch blocks were doing `e instanceof Error ? e.message : String(e)`. Tauri
rejects with structured `AppError` objects that stringify to `"[object Object]"`.
Added a helper that probes `.message` / `.error` / `.reason` / nested
`{error,kind}` shapes before JSON-stringifying. The "[object Object]" promote
failure user reported is now a real error message.

### F ŌĆö `parseIntent` strips `Task: Task: Task: ŌĆ”` accumulation
`src/features/agents/components/matrix/commandPanel/CommandPanelComposer.tsx::parseIntent`

The composer's `useState` initializer dumped the entire parent `intentText`
into `draft.task` verbatim. The propagation `useEffect` then prepended `"Task: "`
via `composeIntent`. Each remount/iteration accumulated another `"Task: "`
prefix. Replaced the initializer with `parseIntent(intentText)` that splits a
composed `Label: value` multi-line text back into structured rows. Idempotent
across remounts.

### G ŌĆö Auto-submit collected answers when queue empties
`src/features/agents/components/matrix/UnifiedMatrixEntry.tsx`

`onSubmitAllAnswers` was wired only to `PersonaMatrix` (legacy 8-dim layout).
The default `GlyphFullLayout` had no Submit-All affordance ŌĆö each `Send` only
called `collectAnswer` (local store) and the CLI never received the batch, so
the LLM re-emitted the same questions on its next turn. Added a debounced
`useEffect` in `UnifiedMatrixEntry` that auto-fires `handleSubmitAnswers` 250ms
after `pendingQuestions` empties and at least one answer is buffered.

### H ŌĆö Header Active/Off toggle now load-bearing
`src-tauri/src/db/repos/resources/triggers.rs::get_due` (added INNER JOIN
personas + `p.enabled = 1`)
`src-tauri/src/engine/background.rs` (added persona.enabled guard at top of
dispatch loop)

The header switch wrote `personas.enabled` to the DB but no dispatch path
ever read it. Live evidence at the time of the fix: 4 of 5 schedule-triggered
personas had `enabled=0` yet their `last_triggered_at` showed fires today on
cron schedule (Sentry Issue Monitor fired at 14:12:39 with `enabled=0`). Both
the schedule path (SQL JOIN) and the event-bus path (post-fetch guard) now
honour the toggle.

## Open architectural items

### 1. Cron timezone interpretation
The "AI Agent Market Intelligence" trigger has cron `"0 7 * * *"` and
timezone `"America/New_York"`. Its `next_trigger_at` resolved to
`2026-04-27T05:00:00+00:00` ŌĆö that's 01:00 EDT, not 07:00 ET. The timezone
isn't being honored by the cron-to-UTC compute. Likely lives in
`engine::cron` or `engine::scheduler`. Worth fixing before users build many
schedule-driven personas.

### 2. agent_ir-null-at-promote race (mitigated, not eradicated)
The build-pipeline reports `phase: test_complete` slightly before
`agent_ir` is persisted to the session row in some cases. If the test
script promotes the moment `test_complete` is reached, promote can fail
with `"Session agent_ir is null"`. Mitigated by the gate-defense (fewer
spurious turns) and bridge error unpacking (real error visible). Needs a
`get_active_build_session().agent_ir IS NOT NULL` precondition in the
promote handler with a brief retry.

### 3. `auto_triage` review_policy runtime side
Build prompt rule 21 produces `mode: "auto_triage"` in the IR. The runtime
self-review pass that interprets it (instead of falling through to
`manual_review`) is unwritten. Carried over from prior handoff.

### 4. Capability-granularity rule strengthening for chained pipelines
Sentry's IR did produce 2 UCs this session, but earlier sessions showed the
LLM occasionally bundling UC1+UC2 into one. If that comes back, add a
build-prompt rule explicitly forbidding the merge when a
`<persona>.<task>.accepted` chain is implied.

### 5. WebView2 dynamic-import staleness after Rust rebuild
When tauri-cli restarts the Rust binary, the WebView2 sometimes loses its
HMR socket and subsequent lazy imports fail with
`Failed to fetch dynamically imported module`. Workaround for users:
hard-refresh inside the persona window. A real fix would either disable
the lazy imports in dev, or have the WebView2 force-reconnect on Rust
binary restart.

### 6. Quick-add connector modal (carried over)
`src/features/shared/components/picker/VaultConnectorPicker.tsx` deep-links
to Catalog instead of an inline create flow. Tracked but not blocking.
