# Credential Resource Scoping (2nd-Level Selection)

Status: **Phase 1 foundation landed. UI picker + HTTP list-endpoint dispatcher + per-connector resource defs are remaining.** See "Remaining work" below.

## Concept

After a user saves a credential and it passes healthcheck, the app can optionally
call service APIs to list "sub-resources" the credential grants access to —
GitHub repos, Supabase projects, Google Drive folders, Figma files, ElevenLabs
voices, etc. The user picks a subset; those picks are persisted alongside the
credential as `scoped_resources`. Templates and agents then see a scoped
credential ("GitHub PAT → repos `xkazm04/personas`, `xkazm04/personas-web`")
instead of a broad one.

Benefits:
- Vault picker can filter credentials by scoped-resource presence.
- Template adoption auto-fills scoped picks into `sample_input`, killing
  clarifying questions.
- Runtime safety: an agent that tries to hit an un-scoped repo can be blocked
  at the tauri-invoke layer (Phase 3).

Hard rule: **the identifiers in `scoped_resources` are not secrets**. They
control blast radius, not auth. Auth fields remain in `credential_fields`,
encrypted.

## Data model

### Schema (landed)

- `persona_credentials.scoped_resources TEXT NULL` — JSON blob, `NULL` = broad scope.
- `connector_definitions.resources TEXT NULL` — JSON array of `ResourceSpec` (spec below), seeded from builtin JSON.

Shape of `scoped_resources`:

```jsonc
{
  "repositories": [
    {
      "id": "xkazm04/personas",
      "label": "xkazm04/personas",
      "meta": { "private": true, "default_branch": "master" }
    }
  ],
  "organizations": [
    { "id": "xkazm04", "label": "Mike Kaz" }
  ]
}
```

- Empty object `{}` = picker was opened and skipped (distinct from NULL = never prompted).
- Keys are `ResourceSpec.id`. Each value is an array — even for `selection: "single"`, where it's a one-element array for shape consistency.

### `ResourceSpec` (landed as TS + Rust JSON passthrough)

```jsonc
{
  "id": "repositories",
  "label": "Repositories",
  "description": "Repos the token can see. Scope narrows blast radius.",
  "selection": "multi",              // "single" | "multi" | "single_or_all"
  "required": false,                 // if true, credential is "needs scope" until picked
  "depends_on": [],                  // e.g. ["organization"] — those picks come first
  "list_endpoint": {
    "method": "GET",
    "url": "https://api.github.com/user/repos?per_page=100&sort=updated",
    "headers": {
      "Authorization": "Bearer {{personal_access_token}}",
      "Accept": "application/vnd.github+json",
      "User-Agent": "personas-desktop"
    },
    "pagination": { "type": "link_header", "max_pages": 5 }
  },
  "response_mapping": {
    "items_path": "$",               // JSONPath-lite; $ = response root is the array
    "id": "full_name",
    "label": "full_name",
    "sublabel": "description",
    "meta": {
      "private": "private",
      "pushed_at": "pushed_at",
      "default_branch": "default_branch"
    }
  },
  "search": { "supported": true, "mode": "client" },
  "cache_ttl_seconds": 600
}
```

**Pagination modes:**
- `{ "type": "none" }` — single page.
- `{ "type": "link_header", "max_pages": N }` — follow RFC 5988 `Link: <…>; rel="next"`.
- `{ "type": "page_param", "page_param": "page", "per_page": 100, "max_pages": N }`.
- `{ "type": "cursor", "cursor_param": "after", "cursor_path": "meta.next_cursor", "max_pages": N }`.

**Template substitution in URL/headers/body:**
- `{{<field_key>}}` — from decrypted `credential_fields`.
- `{{selected.<resource_id>.<item_prop>}}` — from previously-picked resources (requires `depends_on`).

**Search modes:**
- `client` — small lists, filter in memory on the picker input.
- `server` — pass `query_template` like `?q={{query}}` and refetch on each keystroke (debounced).

## Template adoption: `requires_resource`

Templates can pin a connector slot to credentials that have completed the scope picker for a specific resource type. Add the field to the slot's adoption question:

```jsonc
{
  "id": "aq_choose_repo_credential",
  "scope": "connector",
  "connector_names": ["github_repo"],
  "category": "configuration",
  "type": "select",
  "dynamic_source": {
    "source": "vault",
    "operation": "list_credentials",
    "service_type": "source_control",
    "requires_resource": "repositories"
  }
}
```

When the user opens the adoption questionnaire, the option list filters to:
1. Credentials in the `source_control` category (existing behaviour), AND
2. Credentials whose `scoped_resources.repositories` array has at least one pick.

A broad-scope credential (no scope picker completed) and a skipped-scope credential (`{}`) both fail filter (2). The error message reads "Connect a `source_control` credential and pick at least one `repositories`" so users know they need to (re)open the credential edit form and select a repo.

This makes templates that ask "Which repo?" automatically respect the user's scope choices — the agent will only see credentials that have already narrowed their blast radius.

## Smoke test (once picker lands)

1. Add GitHub PAT in vault → healthcheck passes.
2. Picker opens, lists repos + orgs.
3. Select 2 repos → save.
4. `SELECT scoped_resources FROM persona_credentials WHERE service_type='github'` shows the JSON blob.
5. Reopen the credential → Scope tab shows 2 chips → Refresh → still 2 chips.
6. Delete a repo on GitHub → Refresh → shows stale warning on the deleted one.
