# Fleet ‚Äî Claude Code session aggregator

> **Status:** Experimental. DEV builds only. Not yet shipped in any tier.

## What it is

Fleet is a Personas plugin that observes and controls multiple Claude Code (CC) CLI sessions from one Tauri window. It:

- Spawns `claude` in a PTY so we own stdin/stdout (xterm.js renders the live terminal).
- Receives Claude Code's lifecycle **hooks** (`SessionStart`, `Notification`, `Stop`, `PreToolUse`, `SessionEnd`, `UserPromptSubmit`) via an in-app HTTP receiver, turning them into a state machine: *Spawning ‚Üí Running ‚Üí AwaitingInput ‚Üí Idle ‚Üí Stale ‚Üí Exited*.
- Watches `~/.claude/projects/*.jsonl` so transcripts produced by *external* `claude` runs (not spawned from Fleet) also flow through the state machine.
- Groups every tracked session by project (cwd-derived label) in a single grid UI, with status badges that update in real time.
- Lets the user **broadcast** a prompt to any subset of sessions at once ‚Äî selecting "every session currently awaiting input" is the canonical workflow.

The plugin only shows up in `import.meta.env.DEV` builds. The Rust module always compiles so ts-rs bindings and command-name codegen stay stable across build profiles.

## Usage

1. **Open in DEV:** `npm run tauri:dev:lite`. Plugins ‚Üí Fleet appears in the sidebar with a DEV badge.
2. **Install hooks:** Fleet ‚Üí Settings ‚Üí "Install hooks". Patches `~/.claude/settings.json` to POST every lifecycle event to `http://127.0.0.1:<port>/fleet/hooks/*`. Re-install whenever the local_http port changes (Personas detects mismatch on startup and prompts).
3. **Spawn a session:** Fleet ‚Üí Sessions ‚Üí "Spawn session" ‚Üí enter a project directory ‚Üí click Spawn. `claude` boots inside the Fleet-owned PTY. **Spawn with task** (next to Spawn) opens a small composer instead ‚Äî the task you type rides as a positional argv (`claude "<task>"`), so the fresh session starts working on it the moment it boots rather than landing at an empty prompt.
4. **Run external sessions:** Run `claude` from any terminal once hooks are installed ‚Äî it'll register in the grid via hooks alone (no PTY ownership on those rows, but state badges still work).
5. **Broadcast a prompt:** Fleet ‚Üí Sessions ‚Üí "Broadcast" ‚Üí write your prompt ‚Üí click "Waiting (N)" / "All" to pick targets ‚Üí Send. Each session receives the bytes via its PTY's stdin (or via the hook stdin path for external sessions in a future enhancement).
6. **Apply a skill to the focused session (skill drawer):** Click **Skills** ‚Äî above the terminal in the single-pane view, or in the fullscreen grid header. A library drawer slides in from the left over the sidebar region. It lists the **shared library** (`~/.claude/skills`, with a This-project / Global-library toggle), searchable. Click any skill to write its slash command (`/skill-name‚èé`) into the **focused** session's terminal, exactly as if you'd typed it. The drawer header shows which session it'll apply to; pick a session first (click its row/tile) if none is focused. The skill must exist in that session's project ‚Äî install it there first (next).
7. **Install a skill into a repo:** In the skill drawer, hover a skill and click its **Install to repo** (download) icon ‚Üí pick a target project ‚Üí Install. Fleet copies the skill's `SKILL.md` + reference files into that project's `.claude/skills/<name>/`. Enable **Overwrite** to replace an existing copy. This is what makes cross-repo apply (step 6) actually work ‚Äî install once, then apply.
8. **Inspect a session's transcript:** In the single-pane view, flip the right-column toggle to **Insights**; in the fullscreen grid, each tile has its own **Terminal/Insights** toggle in its header. Fleet reads that session's `~/.claude/projects/**/<id>.jsonl` and shows a glanceable rollup ‚Äî token totals (input/output/cache), turns + prompts, tools used (with counts), and files touched. Works for exited sessions too, so it doubles as a "what did this run do" review. Refresh to re-read a live session. The right-column header also carries a **context-size pill** (`ctx`) ‚Äî the size of the context the focused session re-sends each turn (`last_context_tokens`), colored green‚Üíamber‚Üíred as it grows ‚Äî a glanceable efficiency signal.
9. **Search across sessions (Activity tab):** The **Activity** tab lists the most recently-active Claude Code sessions across *all* projects (transcripts modified in the last 7 days, newest first), each with its token/turn/tool/files rollup. Type in the search box to filter across project name, files touched, tool names, and models ‚Äî e.g. "which sessions touched `auth.rs`?" surfaces every matching run with the matching files shown.
10. **Hibernate & wake a session (F3):** Select a session ‚Üí **Hibernate** (moon, in the CLI header). The `claude` process is killed to free it; the row stays as `Hibernated` (indigo) so it stays resumable. To bring it back, select the hibernated row and click **Wake** ‚Äî Fleet runs `claude --resume <id>` in the original cwd and the resumed process restores the conversation. Hibernate needs a bound `claude_session_id` (it's how we resume), so a session must have started before it can sleep.
11. **Auto-hibernate idle sessions (F3/P3.2):** Fleet ‚Üí Settings ‚Üí **Auto-hibernate idle sessions** ‚Üí enable + set the idle threshold (minutes). The always-on staleness ticker then hibernates any Idle/Stale session inactive past the threshold ‚Äî freeing the process even when Fleet isn't focused (it stays resumable via Wake). `AwaitingInput` sessions are never auto-slept (you may be mid-response).
12. **Recover orphaned processes:** Fleet ‚Üí Settings ‚Üí **Running Claude processes**. Fleet's session registry is in-memory, so an app restart loses the session list while the underlying `claude` processes can keep running ‚Äî orphans otherwise reachable only via Task Manager. The panel scans the OS process table (`fleet_detect_processes`) for *interactive* Claude sessions (the app's own transient `claude -p` calls are filtered out so they don't read as orphans), lists each with PID / memory / cwd, and marks which are still Fleet-tracked vs. orphaned. For an orphan you get two actions: **Resume** (`fleet_resume_orphan`) ‚Äî derives the conversation recorded for that cwd, kills the orphan, and spawns a fresh *Fleet-tracked* `claude --resume <id>` in the same cwd, re-adopting the conversation into Fleet ‚Äî and **Kill** (`fleet_kill_pid`, confirm-gated, targeted) to just end it. It scans on open + on **Scan**. The **Settings tab carries an orange badge** with the live orphan count (a FleetPage poll keeps it fresh, ~60s), so a restart's orphans are visible without opening the panel.
13. **Footer quick-toggle (DEV):** a Fleet (grid) icon sits in the desktop footer, just left of the system-load gauge. Outside the grid it jumps straight to the Fleet page; while the fullscreen grid overlay is open it closes it. The footer itself is lifted above the grid overlay (z) while in grid mode, so it stays visible and usable there. DEV-only (matches Fleet's availability). The grid-open state lives in the store (`fleetGridOpen`) so the footer can drive it from outside the Fleet page.
14. **Triage hotkeys:** on the Sessions tab, **N** jumps to the next awaiting-input session (same cycle as the banner's skip button), **‚Üë/‚Üì** move session focus through the visible grouped list, **/** focuses the session search, **G** opens/closes the fullscreen terminal grid, and **?** (or the keyboard icon in the page header) opens the shortcuts reference. Hotkeys ignore typing contexts ‚Äî keys typed into a focused terminal, the search box, or any input never trigger navigation ‚Äî and pause entirely while a modal or the skills drawer is open. While the grid overlay is up only **G** and **N** stay active (`useFleetHotkeys`).

## State machine

| State | How we enter it | Visual |
|---|---|---|
| `Spawning` | PTY just opened, no PTY output yet. First byte of PTY output (claude's banner/prompt) promotes it to `Idle` via `registry::mark_alive` ‚Äî so a fresh interactive `claude` sitting at its prompt (no transcript, no hook yet) reads as **alive + ready**, never as never-attached. | spinner |
| `Running` | SessionStart / PreToolUse / UserPromptSubmit hook fired | blue spinner |
| `AwaitingInput` | Notification hook fired (CC is blocked, waiting for the user) | pulsing amber |
| `Idle` | Stop hook fired (turn ended cleanly) | emerald check |
| `Stale` | No activity for 6 minutes (ticker) ‚Äî the cutoff allows for slow console composition before flagging | orange clock |
| `Hibernated` | Operator clicked **Hibernate** (`fleet_hibernate_session`) ‚Äî the `claude` process was killed to free it, but `claude_session_id` + `cwd` are kept so it can be resumed. NOT terminal. | indigo moon |
| `Exited` | PTY child reaped OR SessionEnd hook | grey ban |

Priority of signals: **process exit > hooks > JSONL mtime > inactivity ticker**. An Exited session never gets re-animated; a Stale session bounces back to Idle on any transcript append.

**Incremental metadata rollup (`transcript_read.rs`, the metadata-abstraction lane).** For scale (Athena managing 10+ parallel CLIs) Fleet must surface per-session *metadata* ‚Äî tokens, tool counts, message counts, context size, files touched ‚Äî without each tile re-parsing a multi-MB transcript or holding raw conversation output in memory. The parser is a foldable accumulator (`RollupAcc::fold_line`) shared by the full-file `summarize_lines` and an **incremental `ingest_delta`**: on each transcript append the watcher folds only the newly-written bytes (`[offset, EOF)`, leaving a half-written trailing line for next time) into a compact per-`claude_session_id` rollup, then discards the raw text. `fleet_session_metadata(claudeSessionId)` returns that live rollup (catching up any delta first ‚Äî full read only on the very first call for a session); the Insights panel prefers it over the whole-file `fleet_read_transcript` and falls back to it on a miss. The raw JSONL stays "on the table" on disk, read in full only on explicit demand. This is the subscription-safe (CLI, not SDK) path to cheap fleet-wide observability; OpenTelemetry (`CLAUDE_CODE_ENABLE_TELEMETRY=1`, every span/metric tagged `session.id`) is the documented complement for aggregate cost/health if a collector is added later.

**Deterministic binding (`pty.rs`, primary for Fleet spawns).** A *fresh* Fleet spawn (not `--resume`) now assigns claude's session id itself ‚Äî `--session-id <uuid>` on the argv ‚Äî and pre-binds `claude_session_id = <uuid>` on the registry row at spawn. So the session is bound from the very first tick (no `cc:-` window, no never-attached race), its transcript is the known `<uuid>.jsonl`, hooks match by the known id (not cwd), and **N concurrent sessions in one cwd never cross-bind** ‚Äî verified with 4 parallel Sonnet sessions in one repo, each tracking its own distinct transcript. `--model <id>` and other flags pass through the same argv (e.g. spawn a fleet on Sonnet for speed). The cwd-reconciliation paths below remain the fallback for externally-launched / resumed sessions that Fleet didn't assign an id to.

**Binding `claude_session_id` ‚Äî fallback paths (`transcript.rs`).** An externally-launched or resumed session starts `cc:-` (unbound) and needs Claude Code's own session id to read its transcript / resume / show Insights. (1) **Primary:** the `SessionStart` hook POSTs `{session_id, cwd}`; `resolve_session_id` matches an unbound Fleet session by cwd and binds it. (2) **Fallback:** if that hook never lands (observed: a session runs + writes a transcript carrying Fleet's injected `mcp__athena__*` tools, yet stays `cc:-`), the JSONL **watcher** now reconciles by cwd ‚Äî when a new transcript appears whose recorded `cwd` matches an unbound, non-Exited Fleet session, it binds `claude_session_id` from the transcript filename and flips the session to `Running` ("Bound to transcript (SessionStart hook missed)"). Without this fallback such a session was mislabeled stale/never-attached and Insights had no id to read. cwd matching is separator/case-insensitive (`normalize_cwd`). **Freshness guard:** the watcher only binds a transcript whose **file-creation time ‚â• the session's spawn time** (‚àí5 s grace) ‚Äî otherwise, in a cwd with *other* concurrent claude sessions (the user's own Claude Code session, or a manual `claude` run in the same repo), it would grab a pre-existing/unrelated transcript and the Fleet session would falsely read `Running` off someone else's activity.

**Test knobs (prod-safe, env-gated):** `PERSONAS_FLEET_STALE_SECS` / `PERSONAS_FLEET_NEVER_ATTACHED_SECS` shorten the cutoffs for fast observation (default 6 min / 2 min when unset); `PERSONAS_FLEET_DEBUG=1` makes the ticker log each session's decision inputs (`state ¬∑ cc ¬∑ idle ¬∑ grewAgo ¬∑ size`) every tick to the dev console ‚Äî the fastest way to see *why* a session is (mis)classified. Drive the whole loop via the test-automation server (`npm run tauri:dev:test`, HTTP `:17320`; `fleet-spawn` / `footer-fleet-toggle` / `fleet-session-row-*` test-ids).

**Growth-hardened staleness (`stale.rs`).** Hooks fire at discrete points, so a session's state could drift ‚Äî a hung session stuck `Running` ("in progress" when it's not), or a working session marked `Stale` mid-long-operation. The ticker polls each session's transcript **size** every tick. Freshness is the **most recent of (a) real transcript growth and (b) hook-driven `last_activity`** (`idle_since = max(last_grew, last_activity)`): a session whose log **grew** is bounced `Stale`/`Idle` ‚Üí `Running`; a `Running`/`Idle`/`Spawning` session that's had **neither growth nor a hook past the cutoff** is marked `Stale`. (Growth alone marked a working session `Stale` during a long tool op ‚Äî hooks firing, transcript not yet flushed; hooks alone let a hung session look alive. The max of both keeps active sessions fresh without reviving genuinely-hung ones, which have neither.) `AwaitingInput` is never staled by flat logs (it's correctly waiting on the user). The decision is a pure `staleness_transition()` fn with unit tests.

**False-`AwaitingInput` correction (two revive paths).** Claude Code fires its idle *"waiting for input"* `Notification` during long tool waits / model-latency gaps, which wrongly parks an in-progress session at `AwaitingInput`. Two correctors pull it back to `Running` ‚Äî but only on *real* evidence of work, so a session genuinely blocked on a question (e.g. AskUserQuestion) is left alone: (1) **Immediate ‚Äî tool activity.** Tool hooks (`PreToolUse`/`PostToolUse`) no longer flow through the lifecycle state machine (they feed operative memory), so a spurious `AwaitingInput` used to get stuck. `receive_hook` now calls `registry::revive_to_running_on_activity` on every tool event ‚Äî a running tool is proof the session is working, so it flips `AwaitingInput`/`Idle`/`Stale` ‚Üí `Running` (emitting only on the actual transition, to keep per-tool volume low). (2) **Backstop ‚Äî growth after the await.** For no-tool sessions (pure text streaming), the ticker revives `AwaitingInput` on transcript growth that happens *strictly after* the await began: the first tick that sees `AwaitingInput` snapshots a size baseline (`AWAITING_BASELINE`) ‚Äî by then the assistant message that may justify a *legitimate* await is already flushed and folded into the baseline ‚Äî and any later tick whose transcript exceeds it means the session kept working, so it's revived. A genuinely-waiting session never grows past the baseline, so it stays `AwaitingInput`.

The flat-log cutoff is `STALE_AFTER_SECS` = **6 min** ‚Äî deliberately generous so a session doesn't flip to `Stale` while the user is still composing a long prompt in the console (the transcript only grows on submit).

**Frozen-process fast path (`is_frozen_mid_run`).** A hung session used to wear the blue `Running` spinner for the full 6-minute cutoff. The PTY reader now stamps `last_pty_output_ms` (throttled to 1/s) on every chunk ‚Äî and since claude redraws its status line continuously *even when idle*, **total PTY silence is a high-confidence "the process is frozen" signal** (while output *presence* proves nothing about work, which is why this field never feeds freshness). A `Running` session with zero PTY bytes AND no transcript growth/hook activity for `STALLED_AFTER_SECS` = **2 min** is flagged `Stale` with a distinct verdict ("No console output for 2 min ‚Äî claude looks frozen mid-run. Safe to kill, or wake it with a prompt."). Sessions that never produced a byte are the never-attached check's case, not this one. Test knob: `PERSONAS_FLEET_STALLED_SECS`; the debug ticker line now includes `outAgo`.

**Never-attached spawns (`is_never_attached`).** A session that *never produces a byte of PTY output* ‚Äî so it's still `Spawning` (never promoted by `mark_alive`) with no bound `claude_session_id` and no activity for `NEVER_ATTACHED_SECS` (2 min) ‚Äî is flagged `Stale` with the distinct reason *"Claude never attached ‚Äî the folder may need trust approval, or claude failed to start. Safe to kill."* This catches the case where `claude` is spawned into a folder it hasn't been trusted in (the first-run trust prompt blocks startup) or otherwise fails to come up ‚Äî previously these were mislabeled as generic "stale" three minutes later, hiding the real problem. It's safe because the transcript watcher bumps `last_activity_ms` (matched by cwd, even before a cc id binds) for any session that's actually running, so a frozen timestamp here genuinely means nothing started. If the same project keeps producing never-attached sessions, set its folder up for Claude Code (trust it once / add `.claude`) or stop spawning there. `Hibernated` is operator-driven (not a signal-derived state): the reaper records the hibernation-triggered child exit as `Hibernated`, not `Exited`, and the staleness ticker leaves it alone. **Wake** (`fleet_wake_session`) spawns a fresh PTY running `claude --resume <claude_session_id>` in the original cwd and drops the placeholder ‚Äî the resumed process restores the conversation itself.

## Session overview surfaces

Above the session grid, the Sessions tab carries a set of glanceable read affordances ‚Äî the desktop groundwork for a future paired mobile companion that surfaces fleet status remotely:

- **Summary pills** (`FleetSummaryPills`) ‚Äî one colored count pill per active lifecycle state, reusing the per-session dot palette. Each pill is a filter toggle that narrows the grid to that state; clicking the active pill clears the filter.
- **Fleet token bar** (`FleetTokenSummaryBar`) ‚Äî a fleet-wide efficiency glance above the grid for when you're running many CLIs at once: combined billable tokens this run, the **cache-hit rate** (share of input served from cache ‚Äî higher is cheaper), and a count of **heavy** sessions whose context is large enough to be worth compacting. Backed by `fleet_token_summary`, which aggregates the same per-session transcript rollups `FleetContextPill`/`FleetSessionInsights` already compute (folding only newly-appended bytes per session, so it stays cheap with a large fleet). The aggregate companion to the per-session context pill ‚Äî it tells you *which* sessions to act on; the pill is where you act.
- **Context pill + inline Compact** (`FleetContextPill`) ‚Äî the active session's CLI header shows `last_context_tokens` (the context it re-sends each turn) as a green‚Üíamber‚Üíred pill. When a session goes **red** (>150k tokens ‚Äî re-sending a heavy conversation every turn), the pill grows an inline **Compact** button that writes `/compact` into that session's PTY (claude's native compaction) so the rest of the run is far cheaper. It's enabled only when the session is idle/awaiting/stale so the command lands between turns. The threshold mirrors the Rust `CONTEXT_BLOAT_TOKENS` constant that drives the fleet bar's "heavy" count.
- **"Needs you" banner** (`FleetNeedsYouBanner`) ‚Äî a pulsing violet strip that appears whenever one or more sessions are `awaiting_input`, listing each as a click-to-focus chip. The at-a-glance "something needs a human" signal.
- **Status legend** (`FleetStatusLegend`) ‚Äî a hover/focus disclosure in the header decoding the two-axis dots (process: spawning/alive/exited ¬∑ activity: working/awaiting/idle/stale). Reuses the exact `CONSOLE_DOT` / `BUSINESS_DOT` maps exported from `FleetStatusDots` so palette and labels can't drift.
- **Mobile companion preview** (`FleetMobilePreview`, in Settings) ‚Äî a read-only render of the glance view (state count chips + the awaiting list) inside a phone frame, fed by live session data. Non-interactive by design: it mirrors what a phone would show, letting the remote surface be validated locally before any mobile client ships.

The "Needs you" banner does more than list ‚Äî it's the desktop stand-in for the companion's remote-approve surface:

- **Inline quick-reply** ‚Äî each awaiting chip carries a reply affordance that opens an inline input writing straight to that session's PTY (`writeInput`, trailing `\r`), so you can unblock a session without opening its terminal. The chip name still jumps to the terminal.
- **Relative "Xs ago" + escalating urgency** ‚Äî every chip shows how long the session has been blocked (from `lastActivityMs`, refreshed every 30s via the shared `useNowTick` hook), alongside the hook message when one was captured. The chip accent escalates with the wait ‚Äî violet while fresh, amber past 5 minutes, red past 15 (`waitTier` in `relativeAgo.ts`) ‚Äî so the oldest, costliest wait pops without any per-second ticking.
- **Desktop alert on awaiting_input** (`notifyFleetAwaiting`) ‚Äî entering `awaiting_input` raises an OS notification once per entry; a bell toggle in the Sessions header (persisted as `fleetNotifyAwaiting`) mutes it. This is the desktop form of the companion's "push when something needs a human".
- **What it needs (F4/P4.1)** ‚Äî the Claude `Notification` hook carries a message ("Claude needs your permission to use Bash"). Fleet captures it into `state_reason` (`hooks.rs`), shows it on the awaiting chip (so you see *what* each session wants without opening the terminal), and includes it in the desktop alert body.
- **Companion approvals** ‚Äî pending companion (Athena) approvals are folded into the same banner with inline Approve/Reject (wired to `companion_approve_action` / `companion_reject_action`), unifying "a session needs input" and "an action needs sign-off".
- **Jump-to-next cycler + "All clear"** ‚Äî when more than one session is awaiting, a skip-forward control cycles terminal focus through them; when sessions exist but nothing's pending, a small emerald "All clear" chip shows instead of the surface vanishing.
- **Per-session sparkline** (`FleetStateSparkline`) ‚Äî each session card carries a tiny inline timeline of recent lifecycle transitions (colored ticks, oldest‚Üínewest), backed by the in-memory `fleetTransitions` ring-buffer in `fleetSlice` (cap 24/session). Spot a flapping or long-stuck session at a glance.
- **Plugin-rail badges** ‚Äî the "needs you" counts also surface on the L2 plugin list: a pulsing violet badge on the Dev Tools row (awaiting sessions) and an amber badge on the Companion row (pending approvals), so you see them without opening either plugin (`PluginsSidebarNav`).
- **Session filter** ‚Äî once more than one session is tracked, a search field above the list narrows the grouped grid by project label or custom name (composes with the state-pill filter); the focused session is persisted across reloads (`fleetActiveSessionId`). The sparkline ticks carry "&lt;state&gt; ¬∑ &lt;Xs ago&gt;" tooltips on hover.

### Pair a device (stage 1)

Fleet Settings also carries `FleetPairDevice` ‚Äî a **stage-1 scaffold** for pairing a phone. It mints an ephemeral local pairing code, shows the endpoint a phone would dial, and renders a QR placeholder with explainer copy (credentials never leave the desktop). It's UI only: no new dependency, no backend call. The secure handshake (relay/P2P), live QR encoding, and the mobile client are architect-scale and tracked for a later stage.

## Terminal experience

The session terminals are rendered with **xterm.js**, but the xterm instances are no longer owned by the React pane. They live in a singleton **terminal manager** (`fleetTerminalManager.ts`, parked on `globalThis` so HMR doesn't reset them), keyed by session id. `FleetTerminalPane` is a thin **mount point** that *attaches* a session's terminal into its container on mount and *detaches* (never disposes) on unmount.

### Output capture is decoupled from rendering (scaling to 16 CLIs)

The single biggest cost at scale is streaming + parsing PTY output for sessions nobody is looking at. So **capture and rendering are decoupled**:

- **Backend ring buffer + subscription gate** (`pty.rs` reader_loop + `registry.rs::OutputRing`). The per-session reader thread ALWAYS drains its PTY into a bounded ring (`OUTPUT_RING_CAP`, 512 KiB) ‚Äî so the pipe never fills and `claude` never blocks, and the app's memory per session stays flat no matter how much a 1M-token run prints. It only emits `FLEET_SESSION_OUTPUT` over IPC **while the session is subscribed**. The reader holds the ring `Arc` directly, so the per-read hot path never touches the registry-map lock.
- **Subscribe on attach, unsubscribe on detach** (`fleet_subscribe_terminal` / `fleet_unsubscribe_terminal`). Attaching a pane subscribes (and gets the ring snapshot back to hydrate the terminal); detaching unsubscribes. So **only the watched session(s) cross IPC and pay the xterm parse/render cost** ‚Äî a 16-CLI fleet with one focused pane streams one session, not sixteen. This is what lets Fleet out-perform sixteen `cmd.exe`/conhost windows each rendering in realtime: the app's work tracks *watched* sessions, not *running* ones.
- **Hydration ordering.** Between issuing the subscribe and writing its snapshot, live chunks are queued (`hydrating` / `pendingLive` in the manager) and flushed right after the snapshot, so a re-focus never interleaves or duplicates. The flip-subscribed-and-snapshot happens under the ring lock, so each byte is either in the snapshot or live-emitted, never both, never neither.
- **Switch-back replays the recent tail, not full history.** Because a parked terminal unsubscribes, switching back replays the ring (‚âà last 512 KiB of scrollback) rather than a fully live-updated buffer. This is the deliberate trade ‚Äî *we don't render background rows in realtime* ‚Äî and it's what keeps a large fleet cheap. The terminal is `reset()` before the snapshot is written so the replay can't duplicate the prior buffer.

- **Grid preview tier (watch-many without rendering-many).** The fullscreen grid overlay no longer mounts a live terminal per tile. Only the **active** tile mounts a real (subscribed) xterm; every other tile renders `FleetTilePreview` ‚Äî a cheap, non-interactive last-N-lines snapshot cooked from the backend ring (`OutputRing::preview_lines` ‚Üí `fleet_terminal_previews`, batched). `useFleetTilePreviews` polls all unwatched tiles in **one** IPC call at a low cadence (~1.2 s); clicking a tile promotes it to the live terminal (and demotes the previous one back to a preview). So a 16-tile grid streams a single session, not sixteen. The cook (`cook_lines`) strips ANSI/CSI/OSC, applies `\n`/`\r`, and resets on erase-display / alt-screen toggles so an interactive TUI's in-place redraws don't accumulate as junk ‚Äî approximate by design, since the focused tile is the source of truth.

Consequences:

- **Near-instant session switching.** Switching the active session ‚Äî or leaving and returning to the Fleet page ‚Äî re-subscribes and replays the ring snapshot in a single write; the durable xterm instance (renderer, keyboard, addons) is reused, never disposed.
- **Fullscreen grid overlay.** A **Grid** button in the action row (left of Spawn, disabled when no live sessions) maximizes a fullscreen, app-wide terminal grid (`FleetTerminalOverlay`, portaled to `document.body` so it sits above the framer-motion transform ancestors). It starts at `top-12` (below the 48px titlebar) rather than `inset-0` so the always-on-top titlebar ‚Äî its window controls **and the global Back button** ‚Äî stay visible and usable above it. The grid is **density-adaptive** ‚Äî columns = `min(4, ceil(‚àöN))`, i.e. 1√ó1 ‚Üí 2√ó2 ‚Üí 3√ó3 ‚Üí 4√ó4 as sessions spawn (rows auto-fill and scroll past 16). While the overlay is open the single pane is unmounted so the two don't contend for the same managed terminal's holder; every tile attaches a durable terminal, so maximizing/minimizing is lossless. Terminals for removed sessions are reaped via `gcTerminals` keyed on the live session list. The overlay is self-sufficient: its header carries **Spawn** (new session in the active project) + **Skills** (drawer), and each tile (`FleetOverlayTile`) has a Terminal/Insights toggle and a **Kill** control ‚Äî so you can grow and prune the fleet without leaving the grid.
- **Three ways back.** The overlay's own header Back button, **Escape**, and the **titlebar Back button** all minimize to the single pane (showing the last-selected session). The titlebar wiring uses a generic `backInterceptor` on `uiSlice.navigateBack`: a fullscreen surface registers a Back handler on mount (and the titlebar surfaces its Back button while one is set), so Back dismisses the overlay instead of navigating the underlying page out from under it. If every session exits while the grid is open it auto-minimizes.
- **Density-scaled font.** While the grid is open the terminal font is scaled to the grid density via a transient override (`setFleetFontOverride`): 1√ó1‚Üí15px, 2√ó2‚Üí14px, 3√ó3‚Üí13px, 4√ó4‚Üí12px (floored at 12px for legibility). The override never touches the persisted single-view `fleetTerminalFontSize`; it's cleared on minimize.
- **Renderer + addons.** WebGL rendering (`@xterm/addon-webgl`) is loaded **per attach** and disposed on detach, so N background terminals don't each pin a GL context (it falls back to the DOM renderer if WebGL is unavailable). `@xterm/addon-unicode11` fixes emoji/CJK/box-drawing widths; `@xterm/addon-web-links` makes URLs clickable, opened via `openExternalUrl` after `sanitizeExternalUrl`. The panes are otherwise chrome-free ‚Äî paste is right-click / Ctrl+Shift+V / Cmd+V; font size, copy-on-select and theme live in Fleet Settings (no per-pane buttons).

### Athena copilot on the grid (experimental)

A UI-fusion layer (`fleetAttention.ts`, `FleetTileAthenaBar.tsx`) that surfaces Athena's existing fleet reasoning *on the terminal tiles*, rather than only in the Needs-You banner. Pure frontend over existing backend (`companion_send_message`, the approval pipeline, `fleet_send_input` / `fleet_intervene`).

- **Attention borders.** Tiles (and the single pane) get a pulsing border by state ‚Äî violet for `awaiting_input`, amber for `stale`, red for a non-zero `exited` ‚Äî via the `fleet-attn-*` classes in `globals.css`, chosen by `sessionAttention()` / `attentionClass()`. Healthy tiles stay plain.
- **On-tile suggestions.** When a pending companion approval is a `fleet_send_input` / `fleet_intervene` targeting a tile's session (matched by parsing `paramsJson.session_id`), the tile shows Athena's proposed text with **Approve** (‚Üí `companion_approve_action` ‚Üí writes it into the PTY) / **Dismiss**. The existing approval pipeline is the write gate ‚Äî nothing auto-types.
- **Ask Athena.** A *genuinely* stale tile (it attached, then went quiet) with nothing pending shows an "Ask Athena" button that fires a session-scoped turn (`companionSendMessage(craftStalePrompt(session))`) asking her to decide the next step and, if there's a clear winner, propose writing it ‚Äî which returns as an on-tile suggestion. The tile shows a "thinking" affordance until the turn resolves. **Never-attached sessions don't get the button** ‚Äî instead the strip shows a non-actionable *"Claude never attached ‚Äî kill & retry"* note, because there's no live agent for Athena to nudge. This stops the wasteful loop where every inert spawn invites a futile investigation. `isNeverAttached` keys on the **Rust ticker's `state_reason` verdict** (the confident 2-min check), not a broad "stale without a cc id" guess ‚Äî a genuinely-working session whose cc id simply hasn't bound yet is normal `stale` and still shows real state + the Ask-Athena button. The tile stays amber `stale` either way (never painted red `failed` ‚Äî that misread as a crash and hid the real state). As a backstop, `craftStalePrompt` also returns a never-attached-specific prompt (telling Athena to confirm the kill, not propose `fleet_send_input`) if she's asked anyway. **Provenance:** the Ask-Athena prompt is *synthetic* (Fleet's words, not the user's), so it's forwarded with `companionSendMessage(prompt, ‚Ä¶, 'Fleet')` ‚Üí the backend records it as a `TurnOrigin::External { source: "Fleet" }` turn (persists as `EpisodeRole::System`, the model gets an explicit "automated request from Fleet ‚Äî not the user" prefix). In the chat it renders as a slim **Fleet divider** (`companion-fleet-marker`, grid icon), never an impersonated user bubble ‚Äî so user / system-Fleet / Athena turns read apart at a glance.

- **Athena visible in the grid.** The fullscreen overlay is `z-[200]`, above the orb's normal `z-50` ‚Äî so while it's open the orb would be hidden. The overlay sets `fleetGridOpen` (fleet slice) on mount and `AthenaOrbLayer` raises itself to `z-[210]` while that's true, so Athena's orb (her thinking/speaking state + decision bubbles) floats above the grid and you can see + react to her there. **The chat panel rides the same lift:** tapping the orb opens the full chat at `z-[220]` (above the overlay) ‚Äî otherwise the panel opened *behind* the grid and read as "orb tapped, then nothing" (the chat, her messages, and any decision/approval UI were all buried under the overlay). Ask-Athena ‚Üí propose ‚Üí approve-on-tile ‚Üí `write_input` is wired end-to-end (verified at `approvals.rs::execute_fleet_send_input`). Note: if a target session never attached (see [State machine](#state-machine) ‚Üí never-attached), Athena correctly declines to propose `fleet_send_input` ‚Äî there's no live agent to type into; the fix is to kill/recover that session, not nudge it.

**Active fleet orchestration (autonomous mode).** When a session enters `AwaitingInput` ‚Äî it finished its turn / is paused for the next instruction ‚Äî and autonomous mode is on, Fleet wakes Athena to manage the fleet: `companion_record_fleet_event(state_changed)` ‚Üí `maybe_orchestrate` ‚Üí `spawn_proactive_turn` with the live **operative-memory digest** as context. Athena assesses the whole fleet from the compact digest (not raw transcripts) and either proposes a `fleet_send_input` for the session that needs it (auto-applied via the autonomous allowlist **only when she spawned that session herself** ‚Äî see [Auto-apply](#) below) or surfaces a concise decision to the user via the orb. Throttled per session (‚â§1 wake/min) so it can't spam or loop. This is the endgame loop ‚Äî Athena keeping 10+ CLIs moving and only escalating key decisions. **Fires Rust-direct:** `apply_hook` calls `orchestrate_on_awaiting` the moment a hook sets `AwaitingInput`, so it works headlessly ‚Äî no dependency on the frontend store or the JS bridge (the frontend bridge also calls it; the per-session throttle dedups the two paths). Verified live: an invoke-spawned session not in the frontend store still auto-woke Athena on task completion.

**Orb UX during orchestration.** The orb is the operator's window into a running fleet ‚Äî it reuses the existing communication surfaces: the **thinking** avatar + **perimeter status dots** (`taskDots`, one per in-flight job) show Athena working, and a glanceable **"Managing the fleet‚Ä¶" caption** appears beside the orb whenever she's working with the grid open. Key decisions she escalates surface in the existing **`OrbDecisionBubble`** (numbered options, `0` = explain/recommend, answerable by click or the `;`-leader key). Both the orb and the decision bubble are **lifted to `z-[210]`/`z-[220]` while the grid overlay (`z-[200]`) is open**, so the cue + the decision render *over* the grid instead of being buried behind it.

**Auto-apply under autonomous mode.** By default Athena's `fleet_send_input` stays approval-gated (a tile click). When **autonomous mode** is on, `fleet_send_input` is on the `auto_resolve_if_allowed` allowlist (`approvals.rs`), so the autonomous chain (`session.rs`) auto-approves + executes it ‚Äî Ask Athena ‚Üí she types into the session, no click. This is a deliberate higher-blast-radius exception (gated on the toggle); it matters that Fleet spawns run with `--dangerously-skip-permissions`, so an auto-applied write executes its tools unprompted. The on-tile suggestion strip shows on the grid tiles; the single pane relies on the Needs-You banner.

> **Confidence gate ‚Äî auto-decide vs. consult (Cautious policy).** Auto-apply is further gated on Athena's *self-reported confidence*. The orchestration directive requires every `fleet_send_input` she proposes to carry a `confidence` param (`"high"`/`"medium"`/`"low"`) plus the exact `text` and a one-line `rationale`. `auto_resolve_if_allowed` only auto-fires when confidence is explicitly **`"high"`** (and the Athena-owned guard still passes); **`"medium"`/`"low"`/missing fail safe to a consult** ‚Äî the approval stays pending and surfaces on the orb as an `OrbDecisionBubble` (`approve`/`reject`, `0` = explain) for the user to decide. So Athena acts alone only on the genuinely-unambiguous next step, and asks you the moment there's real doubt. Consults surface whenever **hands-free decisions OR autonomous mode** is on (`useDecisionQueue`) ‚Äî autonomous mode implies "act for me, and put what you can't auto-do on the orb."
>
> **Notify-only on auto-decide.** A successful high-confidence auto-fire emits `athena://fleet/auto-decided`; the always-mounted `useFleetCompanionBridge` flashes a brief, self-clearing **"Athena ‚Üí \<project\>: \<text\>"** pill beside the orb (`fleetAutoNotice` ‚Üí `AthenaOrbLayer`). It's informational with **no undo** (the user's chosen safety-net level) ‚Äî it never blocks, just makes the hands-off action visible without watching the grid.

> **Athena-owned PTY guard.** Auto-apply is scoped to sessions **Athena spawned herself**. `auto_resolve_if_allowed` checks the target session's visible-name sentinel (`ATHENA_SESSION_NAME_SENTINEL` ‚Äî `fleet_spawn` tags `"athena"`, `fleet_dispatch` tags `"athena-<role>"`; `registry::is_athena_owned`) before auto-firing. If the `session_id` doesn't resolve to an Athena-owned session ‚Äî a hallucinated or stale id, or one of *your own* terminals that drifted into `AwaitingInput` and woke the orchestrator ‚Äî the autoapprove is **declined** and the action is left **pending** for a deliberate tile click instead. So autonomous mode can keep Athena's own dispatched workers moving, but it can never type `{text}\r` into a terminal you started without a human in the loop. (Manual approval is unaffected: an explicit click can still drive any live session.)

## Hook installer details

Each entry is tagged `_fleet: true` so uninstall is surgical:

```json
{
  "hooks": {
    "Stop": [
      {
        "matcher": "*",
        "_fleet": true,
        "hooks": [
          {
            "type": "command",
            "command": "curl -s -m 2 --connect-timeout 1 -X POST --data-binary @- -H \"Content-Type: application/json\" http://127.0.0.1:17400/fleet/hooks/stop",
            "_fleet": true
          }
        ]
      }
    ]
  }
}
```

`curl` is the universal HTTP client (Windows 10+ ships it). Payload is streamed via stdin (`--data-binary @-`) so we never have to JSON-escape on the command line.

## Sharp edges to know about

- **One PTY session per cwd.** The spawn enforces this to keep hook routing unambiguous (we match hooks by claude_session_id once SessionStart fires, but the bootstrap window uses cwd).
- **Soft kill only (for now).** `fleet_kill_session` drops the writer + master, which makes CC see stdin EOF. Phase 6's cancellation token will land a hard kill path that does `Child::kill()`.
- **Port mismatch ‚Äî now self-healing (critical).** local_http binds the first free port in 17400-17415, so a restart can land a *different* port than the installed Claude Code hooks target. When that happens every hook POSTs into the void ‚Äî no SessionStart/Stop/Notification reaches Fleet, so spawned sessions silently lose all hook-driven state and fall back to transcript-growth alone (which reads "Stale" where the session is really idle/awaiting). This was a real, hard-to-spot break. **Startup now self-heals it:** after `local_http::start()` returns the live port, `setup()` calls `check_hooks(port)` and, if hooks are installed but `!port_matches`, immediately `install_hooks(port)` to re-point them. The Settings banner + manual re-install remain as a fallback. Re-install is also now **idempotent across ports**: `is_fleet_tagged` reclaims any entry whose command targets a `/fleet/hooks/` URL (any port, even older untagged ones), so a re-point cleans stale-port stragglers instead of duplicating.
- **No translation of hook payloads.** We only look at the URL path for the event type and `session_id` / `cwd` for routing ‚Äî Claude Code's payload shape varies across versions, so we don't depend on field shapes that may change.
