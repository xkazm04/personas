# Fleet ŌĆö Claude Code session aggregator

> **Status:** Experimental. DEV builds only. Not yet shipped in any tier.

## What it is

Fleet is a Personas plugin that observes and controls multiple Claude Code (CC) CLI sessions from one Tauri window. It:

- Spawns `claude` in a PTY so we own stdin/stdout (xterm.js renders the live terminal).
- Receives Claude Code's lifecycle **hooks** (`SessionStart`, `Notification`, `Stop`, `PreToolUse`, `SessionEnd`, `UserPromptSubmit`) via an in-app HTTP receiver, turning them into a state machine: *Spawning ŌåÆ Idle (launched) ŌåÆ Running (real work) ŌåÆ AwaitingInput ŌåÆ Idle ŌåÆ Stale ŌåÆ Exited*.
- Watches `~/.claude/projects/*.jsonl` so transcripts produced by *external* `claude` runs (not spawned from Fleet) also flow through the state machine.
- Groups every tracked session by project (cwd-derived label) in a single grid UI, with status badges that update in real time.
- Lets the user **broadcast** a prompt to any subset of sessions at once ŌĆö selecting "every session currently awaiting input" is the canonical workflow.

The plugin only shows up in `import.meta.env.DEV` builds. The Rust module always compiles so ts-rs bindings and command-name codegen stay stable across build profiles.

## Usage

1. **Open in DEV:** `npm run tauri:dev:lite`. Plugins ŌåÆ Fleet appears in the sidebar with a DEV badge.
2. **Install hooks:** Fleet ŌåÆ Settings ŌåÆ "Install hooks". Patches `~/.claude/settings.json` to POST every lifecycle event to `http://127.0.0.1:<port>/fleet/hooks/*`. Re-install whenever the local_http port changes (Personas detects mismatch on startup and prompts).
3. **Spawn a session:** Fleet ŌåÆ Sessions ŌåÆ "Spawn session" ŌåÆ enter a project directory ŌåÆ click Spawn. `claude` boots inside the Fleet-owned PTY. **Spawn with task** (next to Spawn) opens a small composer instead ŌĆö the task you type rides as a positional argv (`claude "<task>"`), so the fresh session starts working on it the moment it boots rather than landing at an empty prompt.
4. **Run external sessions:** Run `claude` from any terminal once hooks are installed ŌĆö it'll register in the grid via hooks alone (no PTY ownership on those rows, but state badges still work).
5. **Broadcast a prompt:** Fleet ŌåÆ Sessions ŌåÆ "Broadcast" ŌåÆ write your prompt ŌåÆ click "Waiting (N)" / "All" to pick targets ŌåÆ Send. Each session receives the bytes via its PTY's stdin (or via the hook stdin path for external sessions in a future enhancement).
6. **Apply a skill to the focused session (skill drawer):** Click **Skills** ŌĆö above the terminal in the single-pane view, or in the fullscreen grid header. A library drawer slides in from the left over the sidebar region. It lists the **shared library** (`~/.claude/skills`, with a This-project / Global-library toggle), searchable. Click any skill to write its slash command (`/skill-nameŌÅÄ`) into the **focused** session's terminal, exactly as if you'd typed it. The drawer header shows which session it'll apply to; pick a session first (click its row/tile) if none is focused. The skill must exist in that session's project ŌĆö install it there first (next).
7. **Install a skill into a repo:** In the skill drawer, hover a skill and click its **Install to repo** (download) icon ŌåÆ pick a target project ŌåÆ Install. Fleet copies the skill's `SKILL.md` + reference files into that project's `.claude/skills/<name>/`. Enable **Overwrite** to replace an existing copy. This is what makes cross-repo apply (step 6) actually work ŌĆö install once, then apply.
   - **Provenance & drift tracking:** each install stamps a hidden `.personas-skill-meta.json` sidecar recording the source and a content hash. A row then shows a **badge**: *In sync* (the installed copy still matches its source) or *Diverged* (the copy or its source changed since install). Hand-authored skills with no provenance show no badge (local-only) and are never touched. Before an **Overwrite**, the install modal previews a **diff summary** (files changed / added / removed) so you see exactly what a re-install replaces.
8. **Inspect a session's transcript:** In the single-pane view, flip the right-column toggle to **Insights**; in the fullscreen grid, each tile has its own **Terminal/Insights** toggle in its header. Fleet reads that session's `~/.claude/projects/**/<id>.jsonl` and shows a glanceable rollup ŌĆö token totals (input/output/cache), turns + prompts, tools used (with counts), and files touched. Works for exited sessions too, so it doubles as a "what did this run do" review. Refresh to re-read a live session. The right-column header also carries a **context-size pill** (`ctx`) ŌĆö the size of the context the focused session re-sends each turn (`last_context_tokens`), colored greenŌåÆamberŌåÆred as it grows ŌĆö a glanceable efficiency signal.
9. **Search across sessions (Activity tab):** The **Activity** tab lists the most recently-active Claude Code sessions across *all* projects (transcripts modified in the last 7 days, newest first), each with its token/turn/tool/files rollup. Type in the search box to filter across project name, files touched, tool names, and models ŌĆö e.g. "which sessions touched `auth.rs`?" surfaces every matching run with the matching files shown.
10. **Hibernate & wake a session (F3):** Select a session ŌåÆ **Hibernate** (moon, in the CLI header). The `claude` process is killed to free it; the row stays as `Hibernated` (indigo) so it stays resumable. To bring it back, select the hibernated row and click **Wake** ŌĆö Fleet runs `claude --resume <id>` in the original cwd and the resumed process restores the conversation. Hibernate needs a bound `claude_session_id` (it's how we resume), so a session must have started before it can sleep.
11. **Auto-hibernate idle sessions (F3/P3.2):** Fleet ŌåÆ Settings ŌåÆ **Auto-hibernate idle sessions** ŌåÆ enable + set the idle threshold (minutes). The always-on staleness ticker then hibernates any Idle/Stale session inactive past the threshold ŌĆö freeing the process even when Fleet isn't focused (it stays resumable via Wake). `AwaitingInput` sessions are never auto-slept (you may be mid-response).
    - **Limit live sessions (live-slot scheduler, fleet-scale Tier A):** Fleet ŌåÆ Settings ŌåÆ **Limit live sessions** ŌåÆ enable + set the max (1ŌĆō64, default 10). The fleet becomes *N tracked conversations, Ōēżmax live processes*: the ticker hibernates overflow Idle/Stale sessions (oldest inactivity first; resumable only ŌĆö a bound `claude_session_id` is required), and a spawn/wake at the cap first hibernates the best idle candidate so the new session starts inside the budget (`stale.rs::free_slot_for_spawn`). It's a **soft cap** ŌĆö Running/AwaitingInput/Spawning sessions are never evicted, so a burst of genuinely-working sessions can exceed the limit until some go idle. This is what makes a large (40-session) fleet cost what ~10 processes cost: RAM/CPU tracks *active* work, not tracked conversations. Pushed to Rust via `fleet_set_live_slots` on change + on every Fleet refresh (0 = off), like auto-hibernate. External hooks-only rows (no Fleet-owned process) neither count against nor free a slot; the eviction choice is the pure, unit-tested `live_slot_evictions()`.
12. **Spawn a headless (background) session ŌĆö fleet-scale Tier B:** the *Spawn with task* composer carries a **"Run in background (no terminal)"** toggle. It spawns `claude -p --input-format stream-json --output-format stream-json` with plain piped stdio (`headless.rs`, `fleet_spawn_headless_session`) instead of a PTY ŌĆö **no ConPTY/conhost and no TUI redraw loop**, so an idle headless session costs ~zero CPU. The stdout reader parses one JSON event per line and drives the state machine directly (`system/init` ŌåÆ alive, `assistant` ŌåÆ Running, `result` ŌåÆ Idle) ŌĆö structured events instead of vt100 screen-scraping ŌĆö and feeds *cooked display lines* into the same output ring, so peek/previews/Athena's orchestration context all keep working with cleaner content. Conversation semantics are identical: pinned `--session-id`, persisted transcript, hooks fire, Insights work, and `write_input` wraps any reply (broadcast, quick-reply, Athena's `fleet_send_input`) into a stream-json user message. The UI never mounts an xterm for these (`mode: "headless"` on the session DTO ŌĆö the tile shows a **Background** badge + status block; the single pane shows Insights). Hibernate works (PID-based killer); **Wake resumes the conversation interactively**. If Personas exits, the child's stdin pipe closes and `claude -p` exits after the in-flight turn ŌĆö headless sessions can't linger as invisible orphans. This is the recommended lane for Athena-dispatched batch work at scale.
13. **Recover orphaned processes:** Fleet ŌåÆ Settings ŌåÆ **Running Claude processes**. Fleet's session registry is in-memory, so an app restart loses the session list while the underlying `claude` processes can keep running ŌĆö orphans otherwise reachable only via Task Manager. The panel scans the OS process table (`fleet_detect_processes`) for *interactive* Claude sessions (the app's own transient `claude -p` calls are filtered out so they don't read as orphans), lists each with PID / memory / cwd, and marks which are still Fleet-tracked vs. orphaned. For an orphan you get two actions: **Resume** (`fleet_resume_orphan`) ŌĆö derives the conversation recorded for that cwd, kills the orphan, and spawns a fresh *Fleet-tracked* `claude --resume <id>` in the same cwd, re-adopting the conversation into Fleet ŌĆö and **Kill** (`fleet_kill_pid`, confirm-gated, targeted) to just end it. It scans on open + on **Scan**. The **Settings tab carries an orange badge** with the live orphan count (a FleetPage poll keeps it fresh, ~60s), so a restart's orphans are visible without opening the panel.
14. **Footer status cluster (DEV):** a fleet mark sits in the desktop footer, just left of the system-load gauge, followed by a **colored count chip per live lifecycle state** (`FleetFooterIcon`). It is the whole-fleet glance from anywhere in the app ŌĆö see [Footer status cluster](#footer-status-cluster) below for the counts, the violet needs-you escalation, and the three-way click contract (close grid / raise grid / open page).
15. **Triage hotkeys:** on the Sessions tab, **N** jumps to the next awaiting-input session (same cycle as the banner's skip button), **Ōåæ/Ōåō** move session focus through the visible grouped list, **/** focuses the session search, **G** opens/closes the fullscreen terminal grid, and **?** (or the keyboard icon in the page header) opens the shortcuts reference. Hotkeys ignore typing contexts ŌĆö keys typed into a focused terminal, the search box, or any input never trigger navigation ŌĆö and pause entirely while a modal or the skills drawer is open. While the grid overlay is up only **G** and **N** stay active (`useFleetHotkeys`).

## State machine

| State | How we enter it | Visual |
|---|---|---|
| `Spawning` | PTY just opened, no PTY output yet. First byte of PTY output (claude's banner/prompt) promotes it to `Idle` via `registry::mark_alive` ŌĆö so a fresh interactive `claude` sitting at its prompt (no transcript, no hook yet) reads as **alive + ready**, never as never-attached. | spinner |
| `Running` | **Real progress only**: `UserPromptSubmit` hook, a tool firing (`PreToolUse` ŌåÆ `revive_to_running_on_activity`), or transcript **growth** (the stale ticker). NOT `SessionStart` ŌĆö launching ŌēĀ working. | blue spinner |
| `AwaitingInput` | Notification hook fired (CC is blocked, waiting for the user) | pulsing amber |
| `Idle` | `Stop` hook (turn ended cleanly), **or the session just launched / bound** ŌĆö `SessionStart` and the transcript-bind fallback both land here: Claude is up and ready, not yet working. A bare spawn correctly sits at `Idle` until real work begins. | emerald check |
| `Stale` | No activity for 6 minutes (ticker) ŌĆö the cutoff allows for slow console composition before flagging | orange clock |
| `Hibernated` | Operator clicked **Hibernate** (`fleet_hibernate_session`) ŌĆö the `claude` process was killed to free it, but `claude_session_id` + `cwd` are kept so it can be resumed. NOT terminal. | indigo moon |
| `Exited` | PTY child reaped OR SessionEnd hook | grey ban |

Priority of signals: **process exit > hooks > JSONL mtime > inactivity ticker**. An Exited session never gets re-animated; a Stale session bounces back to Idle on any transcript append.

**Incremental metadata rollup (`transcript_read.rs`, the metadata-abstraction lane).** For scale (Athena managing 10+ parallel CLIs) Fleet must surface per-session *metadata* ŌĆö tokens, tool counts, message counts, context size, files touched ŌĆö without each tile re-parsing a multi-MB transcript or holding raw conversation output in memory. The parser is a foldable accumulator (`RollupAcc::fold_line`) shared by the full-file `summarize_lines` and an **incremental `ingest_delta`**: on each transcript append the watcher folds only the newly-written bytes (`[offset, EOF)`, leaving a half-written trailing line for next time) into a compact per-`claude_session_id` rollup, then discards the raw text. `fleet_session_metadata(claudeSessionId)` returns that live rollup (catching up any delta first ŌĆö full read only on the very first call for a session); the Insights panel prefers it over the whole-file `fleet_read_transcript` and falls back to it on a miss. The raw JSONL stays "on the table" on disk, read in full only on explicit demand. This is the subscription-safe (CLI, not SDK) path to cheap fleet-wide observability; OpenTelemetry (`CLAUDE_CODE_ENABLE_TELEMETRY=1`, every span/metric tagged `session.id`) is the documented complement for aggregate cost/health if a collector is added later.

**Deterministic binding (`pty.rs`, primary for Fleet spawns).** A *fresh* Fleet spawn (not `--resume`) now assigns claude's session id itself ŌĆö `--session-id <uuid>` on the argv ŌĆö and pre-binds `claude_session_id = <uuid>` on the registry row at spawn. So the session is bound from the very first tick (no `cc:-` window, no never-attached race), its transcript is the known `<uuid>.jsonl`, hooks match by the known id (not cwd), and **N concurrent sessions in one cwd never cross-bind** ŌĆö verified with 4 parallel Sonnet sessions in one repo, each tracking its own distinct transcript. `--model <id>` and other flags pass through the same argv (e.g. spawn a fleet on Sonnet for speed). The cwd-reconciliation paths below remain the fallback for externally-launched / resumed sessions that Fleet didn't assign an id to.

**Binding `claude_session_id` ŌĆö fallback paths (`transcript.rs`).** An externally-launched or resumed session starts `cc:-` (unbound) and needs Claude Code's own session id to read its transcript / resume / show Insights. (1) **Primary:** the `SessionStart` hook POSTs `{session_id, cwd}`; `resolve_session_id` matches an unbound Fleet session by cwd and binds it. (2) **Fallback:** if that hook never lands (observed: a session runs + writes a transcript carrying Fleet's injected `mcp__athena__*` tools, yet stays `cc:-`), the JSONL **watcher** now reconciles by cwd ŌĆö when a new transcript appears whose recorded `cwd` matches an unbound, non-Exited Fleet session, it binds `claude_session_id` from the transcript filename and lands the session at `Idle` ("Bound to transcript ŌĆö ready (SessionStart hook missed)") ŌĆö binding ŌēĀ working, so real progress (a tool, transcript growth) is what drives `Running`. Without this fallback such a session was mislabeled stale/never-attached and Insights had no id to read. cwd matching is separator/case-insensitive (`normalize_cwd`). **Freshness guard:** the watcher only binds a transcript whose **file-creation time Ōēź the session's spawn time** (ŌłÆ5 s grace) ŌĆö otherwise, in a cwd with *other* concurrent claude sessions (the user's own Claude Code session, or a manual `claude` run in the same repo), it would grab a pre-existing/unrelated transcript and the Fleet session would falsely read `Running` off someone else's activity.

**Per-session title (`OscTitleScanner` + LLM naming).** Each session carries a `title` field surfaced on the tile/card as `name ?? title ?? projectLabel`, so sessions read distinctly instead of all showing the same project label. Two sources feed it, both via `registry::set_title` ŌĆö which **ignores the literal "Claude Code"** (and empties) so the generic value never clobbers a real one:
> - **OSC window-title (`OscTitleScanner`).** Claude Code emits OSC `0`/`1`/`2` window-title escapes; the PTY `reader_loop` parses them (BEL- or ST-terminated, reassembled across reads) into `title`. In practice CC's *headless* OSC title resolves to the generic **"Claude Code"** for every session ŌĆö which `set_title` drops ŌĆö so this rarely yields a distinct label on its own; it's kept for sessions/versions that do emit a real summary. Captured in the reader (not xterm), so it works even for autonomous status-block tiles, with no UI subscription.
> - **LLM-assigned name (`naming.rs`, primary in practice).** On a spawn **with a task** (`task_from_args` finds the first positional arg, skipping flags + their values), Fleet fires a **fire-and-forget cheap-model (Haiku) call** ŌĆö `spawn_claude_and_collect`, subscription auth ŌĆö that returns a terse 3ŌĆō5-word Title-Case label and stores it via `set_title`. UI-independent, so it labels the session whether or not anyone's watching; a user rename still wins (`name` precedes `title`); a bare interactive spawn (no task) keeps its project label.

**State provenance in the UI.** Hovering a session row's status dots now shows *why* the row is in its state: the last state-change reason plus the three deriving signals with ages ŌĆö `hook activity Xs ago ┬Ę console output Xs ago ┬Ę transcript growth Xs ago` (`lastActivityMs` / `lastPtyOutputMs` / `lastGrewMs` on `FleetSession`, refreshed by the shared 30s tick). When idle/working ever look mixed again, the misclassifying signal is readable in-app instead of needing `PERSONAS_FLEET_DEBUG` and a dev console.

**Tunable cutoffs (Fleet ŌåÆ Settings ŌåÆ State detection cutoffs).** The stale (default 6 min) and frozen (default 2 min) cutoffs are user-tunable from a Settings card ŌĆö persisted in the fleet slice and pushed to the Rust ticker on change + on every Fleet refresh (`fleet_set_state_cutoffs`, clamped server-side), the same plumbing as auto-hibernate. Power users running slow models or huge fleets calibrate state accuracy here instead of via env knobs.

### Debug recorder (DEV) ŌĆö "why did the fleet do that?"

`Record` in the grid overlay header, next to **New session** (`FleetDebugLogButton`, DEV-only). It arms `debug_log.rs`, which appends a high-level, human-readable log to `<app-data>/fleet-debug/fleet-<timestamp>.log` until you stop it. The stop toast carries the path (30 s, because that path is the artifact).

**Why it's Rust-side and not a frontend event tap.** The three things worth debugging are all decided where the operator can't see them: state is decided by four independent signal sources, hibernation by a background ticker that runs whether or not the UI is open, and Athena's orchestration reasons on a path that is *deliberately silent* in the chat (see the orb-only note above). A frontend recorder would only ever see the flattened `{session_id, state, reason}` that crosses IPC ŌĆö never her confidence, her decision class, or the reason she skipped a session entirely.

**What it records** ŌĆö one line per event, `[elapsed] KIND  session  headline | detail`:

| Kind | Covers |
|---|---|
| `LIFE` | spawned (interactive/headless, bare vs with-task), killed, exited (with claude's own final line via `mark_exited`) |
| `STATE` | every lifecycle transition, `from ŌåÆ to` plus the deciding reason ŌĆö tapped at `pty::emit_session_state`, the single emit point all four lanes now funnel through |
| `SLEEP` | hibernations split by cause (manual ┬Ę auto-idle ┬Ę live-slot eviction ┬Ę slot freed for a spawn) and wakes, logged against the new session id with the old one in the detail so a hibernate ŌåÆ wake chain stays followable across the id change |
| `ATHENA` | every orchestration **wake** (situation, screen size, objective), every **skip** with its reason (autonomous mode off ┬Ę throttled ┬Ę screen unchanged ┬Ę already auto-fired on this screen), and every **verdict** (`AUTO_FIRED` / `AUTO_FAILED` / `DEFERRED` + class + confidence + defer reason) with the text she sent and her rationale |

The Athena taps sit at the two natural choke points ŌĆö `fleet_bridge::orchestrate_session` for wakes/skips and `approvals::record_fleet_decision` for verdicts ŌĆö so every situation flavour (`awaiting_input` / `idle_needs_next` / `stuck`) and every action (`fleet_send_input`, `fleet_intervene`, `fleet_wake`, `fleet_resume`) is covered by construction.

**What it deliberately does NOT record:** terminal contents. An Athena wake logs the *size* of the screen she read (`screen=1841ch`, or `EMPTY (reasoning blind)` ŌĆö itself a finding), never the frame. Those carry the user's code and paths, and the file is meant to be shareable. Also absent by design: PTY bytes, per-tool hooks, transcript deltas ŌĆö this is a causal spine, not a trace.

**Cost when off:** one relaxed atomic load per fleet event (`debug_log::is_armed`). Nothing is opened, allocated or written until armed.

**Sharp edges.** The recorder is global Rust state, so it survives the overlay unmounting and keeps recording after you minimize the grid ŌĆö the button re-adopts the running recording when the overlay reopens (that's what the `status` poll is for), and re-arming while active is a no-op rather than a second file. Label resolution uses the registry's non-blocking `try_lookup_*` accessors: `record` is called from paths that already hold the registry lock and `std::sync::Mutex` is not reentrant, so a blocking lookup would deadlock the app inside a debugging tool. Keep that rule for any field added later. Each line is flushed immediately ŌĆö the point is to survive whatever is being debugged, including a hang or a hard exit.

**Test knobs (prod-safe, env-gated):** `PERSONAS_FLEET_STALE_SECS` / `PERSONAS_FLEET_NEVER_ATTACHED_SECS` shorten the cutoffs for fast observation (default 6 min / 2 min when unset; env knobs take precedence over the Settings-tuned values); `PERSONAS_FLEET_DEBUG=1` makes the ticker log each session's decision inputs (`state ┬Ę cc ┬Ę idle ┬Ę grewAgo ┬Ę size`) every tick to the dev console ŌĆö the fastest way to see *why* a session is (mis)classified. Drive the whole loop via the test-automation server (`npm run tauri:dev:test`, HTTP `:17320`; `fleet-spawn` / `footer-fleet-toggle` / `fleet-session-row-*` test-ids).

**Growth-hardened staleness (`stale.rs`).** Hooks fire at discrete points, so a session's state could drift ŌĆö a hung session stuck `Running` ("in progress" when it's not), or a working session marked `Stale` mid-long-operation. The ticker polls each session's transcript **size** every tick. Freshness is the **most recent of (a) real transcript growth and (b) hook-driven `last_activity`** (`idle_since = max(last_grew, last_activity)`): a session whose log **grew** is bounced `Stale`/`Idle` ŌåÆ `Running`; a `Running`/`Idle`/`Spawning` session that's had **neither growth nor a hook past the cutoff** is marked `Stale`. (Growth alone marked a working session `Stale` during a long tool op ŌĆö hooks firing, transcript not yet flushed; hooks alone let a hung session look alive. The max of both keeps active sessions fresh without reviving genuinely-hung ones, which have neither.) `AwaitingInput` is never staled by flat logs (it's correctly waiting on the user). The decision is a pure `staleness_transition()` fn with unit tests.

**False-`AwaitingInput` correction (two revive paths).** Claude Code fires its idle *"waiting for input"* `Notification` during long tool waits / model-latency gaps, which wrongly parks an in-progress session at `AwaitingInput`. Two correctors pull it back to `Running` ŌĆö but only on *real* evidence of work, so a session genuinely blocked on a question (e.g. AskUserQuestion) is left alone: (1) **Immediate ŌĆö tool activity.** Tool hooks (`PreToolUse`/`PostToolUse`) no longer flow through the lifecycle state machine (they feed operative memory), so a spurious `AwaitingInput` used to get stuck. `receive_hook` now calls `registry::revive_to_running_on_activity` on every tool event ŌĆö a running tool is proof the session is working, so it flips `AwaitingInput`/`Idle`/`Stale` ŌåÆ `Running` (emitting only on the actual transition, to keep per-tool volume low). (2) **Backstop ŌĆö growth after the await.** For no-tool sessions (pure text streaming), the ticker revives `AwaitingInput` on transcript growth that happens *strictly after* the await began: the first tick that sees `AwaitingInput` snapshots a size baseline (`AWAITING_BASELINE`) ŌĆö by then the assistant message that may justify a *legitimate* await is already flushed and folded into the baseline ŌĆö and any later tick whose transcript exceeds it means the session kept working, so it's revived. A genuinely-waiting session never grows past the baseline, so it stays `AwaitingInput`.

The flat-log cutoff is `STALE_AFTER_SECS` = **6 min** ŌĆö deliberately generous so a session doesn't flip to `Stale` while the user is still composing a long prompt in the console (the transcript only grows on submit).

**Frozen-process fast path (`is_frozen_mid_run`).** A hung session used to wear the blue `Running` spinner for the full 6-minute cutoff. The PTY reader now stamps `last_pty_output_ms` (throttled to 1/s) on every chunk ŌĆö and since claude redraws its status line continuously *even when idle*, **total PTY silence is a high-confidence "the process is frozen" signal** (while output *presence* proves nothing about work, which is why this field never feeds freshness). A `Running` session with zero PTY bytes AND no transcript growth/hook activity for `STALLED_AFTER_SECS` = **2 min** is flagged `Stale` with a distinct verdict ("No console output for 2 min ŌĆö claude looks frozen mid-run. Safe to kill, or wake it with a prompt."). Sessions that never produced a byte are the never-attached check's case, not this one. Test knob: `PERSONAS_FLEET_STALLED_SECS`; the debug ticker line now includes `outAgo`.

**Never-attached spawns (`is_never_attached`).** A session that *never produces a byte of PTY output* ŌĆö so it's still `Spawning` (never promoted by `mark_alive`) with no bound `claude_session_id` and no activity for `NEVER_ATTACHED_SECS` (2 min) ŌĆö is flagged `Stale` with the distinct reason *"Claude never attached ŌĆö the folder may need trust approval, or claude failed to start. Safe to kill."* This catches the case where `claude` is spawned into a folder it hasn't been trusted in (the first-run trust prompt blocks startup) or otherwise fails to come up ŌĆö previously these were mislabeled as generic "stale" three minutes later, hiding the real problem. It's safe because the transcript watcher bumps `last_activity_ms` (matched by cwd, even before a cc id binds) for any session that's actually running, so a frozen timestamp here genuinely means nothing started. If the same project keeps producing never-attached sessions, set its folder up for Claude Code (trust it once / add `.claude`) or stop spawning there. `Hibernated` is operator-driven (not a signal-derived state): the reaper records the hibernation-triggered child exit as `Hibernated`, not `Exited`, and the staleness ticker leaves it alone. **Wake** (`fleet_wake_session`) spawns a fresh PTY running `claude --resume <claude_session_id>` in the original cwd and drops the placeholder ŌĆö the resumed process restores the conversation itself.

## Terminal experience

The session terminals are rendered with **xterm.js**, but the xterm instances are no longer owned by the React pane. They live in a singleton **terminal manager** (`fleetTerminalManager.ts`, parked on `globalThis` so HMR doesn't reset them), keyed by session id. `FleetTerminalPane` is a thin **mount point** that *attaches* a session's terminal into its container on mount and *detaches* (never disposes) on unmount.

### Output capture is decoupled from rendering (scaling to 16 CLIs)

The single biggest cost at scale is streaming + parsing PTY output for sessions nobody is looking at. So **capture and rendering are decoupled**:

- **Backend ring buffer + subscription gate** (`pty.rs` reader_loop + `registry.rs::OutputRing`). The per-session reader thread ALWAYS drains its PTY into a bounded ring (`OUTPUT_RING_CAP`, 512 KiB) ŌĆö so the pipe never fills and `claude` never blocks, and the app's memory per session stays flat no matter how much a 1M-token run prints. It only emits `FLEET_SESSION_OUTPUT` over IPC **while the session is subscribed**. The reader holds the ring `Arc` directly, so the per-read hot path never touches the registry-map lock.
- **Persistent per-session screen model (fleet-scale Tier C).** `OutputRing::render_screen` no longer re-parses up to 512 KiB through a fresh `vt100::Parser` per call. The ring holds a **persistent parser**, lazily materialized on the first screen read (one catch-up feed of the ring) and then fed **incrementally by every `push`** ŌĆö so steady-state screen reads (orchestration wakes, screen-hash dedupe, execution-time re-checks, previews, exit-reason tails) are O(screen) instead of O(ring). A PTY resize rebuilds it once at the new dims. Memory: one rows├Ścols cell grid per session.
- **Subscribe on attach, unsubscribe on detach** (`fleet_subscribe_terminal` / `fleet_unsubscribe_terminal`). Attaching a pane subscribes (and gets the ring snapshot back to hydrate the terminal); detaching unsubscribes. So **only the watched session(s) cross IPC and pay the xterm parse/render cost** ŌĆö a 16-CLI fleet with one focused pane streams one session, not sixteen. This is what lets Fleet out-perform sixteen `cmd.exe`/conhost windows each rendering in realtime: the app's work tracks *watched* sessions, not *running* ones.
- **Hydration ordering.** Between issuing the subscribe and writing its snapshot, live chunks are queued (`hydrating` / `pendingLive` in the manager) and flushed right after the snapshot, so a re-focus never interleaves or duplicates. The flip-subscribed-and-snapshot happens under the ring lock, so each byte is either in the snapshot or live-emitted, never both, never neither.
- **Switch-back replays the recent tail, not full history.** Because a parked terminal unsubscribes, switching back replays the ring (Ōēł last 512 KiB of scrollback) rather than a fully live-updated buffer. This is the deliberate trade ŌĆö *we don't render background rows in realtime* ŌĆö and it's what keeps a large fleet cheap. The terminal is `reset()` before the snapshot is written so the replay can't duplicate the prior buffer.

- **Attention-driven grid rendering (watch what needs *you*).** The fullscreen grid renders neither a live terminal nor a cooked preview per tile. A tile mounts a real (subscribed) xterm **only** when the session needs the operator (`needsLiveAttention` in `fleetAttention.ts` ŌĆö today `awaiting_input`) or it's the focused tile (the manual peek). Every other session is autonomous (`running`/`spawning`/`idle`/`stale`) and renders a lightweight `FleetTileStatusBlock` ŌĆö state affordance + heartbeat (`formatAgo`) + the conversation-size pill ŌĆö with no xterm, subscription, or ring poll. This is safe because Athena's decision access is **fully decoupled from the UI**: she reads each session via operative memory + the always-populated ring + the transcript (see [Active fleet orchestration](#) below), so dropping a tile's terminal blinds nobody. She triages the autonomous/stale tiles in the background and *escalates by raising the tile* ŌĆö an on-tile proposal, or the session flipping to `awaiting_input`. The live-terminal count therefore tracks how many sessions need you at once (usually 0ŌĆō3), so a 9-tile (3├Ś3) grid stays calm and extends cleanly. *(The earlier cooked-preview tier ŌĆö `FleetTilePreview` / `useFleetTilePreviews` / `fleet_terminal_previews` + `cook_lines`, change-gated by a per-ring `rev` and visibility-gated via `useDocumentVisibility` ŌĆö is retained as infra for a possible future large-grid mode but is currently unused by the grid.)* This is **Increment 1** of the Athena-triage redesign; later increments surface the live `current_tool` and a per-tile Athena watching/reasoning/escalation signal.

Consequences:

- **Near-instant session switching.** Switching the active session ŌĆö or leaving and returning to the Fleet page ŌĆö re-subscribes and replays the ring snapshot in a single write; the durable xterm instance (renderer, keyboard, addons) is reused, never disposed.
- **Fullscreen grid overlay ŌĆö an app-wide LAYER, not a page surface.** The overlay is mounted at app root (`FleetGridLayer`, rendered in `App.tsx`'s global-overlays island, DEV-only), **not** by `FleetGridPage`. That's what lets the footer raise grid mode over Overview, an agent editor, or anything else without a navigation: check the fleet, reply to whatever needs you, minimize, and the page underneath never moved. Two consequences worth knowing:
  - `FleetGridLayer` itself is deliberately **cheap** ŌĆö no xterm in its import graph. It owns the fleet's app-wide *bootstrap* (attaching the session listeners + the first `fleet_list_sessions` snapshot at launch, which is also what keeps the footer counts live and what finally pushes the persisted auto-hibernate / live-slot / state-cutoff policy to the Rust ticker at startup instead of on first Fleet visit) and lazy-loads `FleetGridOverlayHost` ŌĆö the xterm-bearing half ŌĆö only on first open. `@xterm/*` is heavy and WebView2-sensitive (see the frozen-`Object.prototype` note below); it must not sit in the app-boot chunk.
  - **`FleetTerminalOverlay` must never write `fleetGridOpen`.** It used to set the flag on mount and clear it on unmount (so the Athena orb could lift above it). Now that the layer *renders from* that flag, the write is a cycle ŌĆö under StrictMode's mount ŌåÆ cleanup ŌåÆ mount effect double-invoke the cleanup's `setGridOpen(false)` re-enters the layer's render mid-flush and leaves the overlay **wedged open**: mounted, but no longer closeable from the footer. Observed live, then removed; `onClose` is the single close path and the orb reads the same store flag, so nothing was lost. Pinned by `__tests__/FleetTerminalOverlay.test.tsx`.

  - **It leaves the footer alone ŌĆö and the footer becomes the navigation.** The overlay stops at `DESKTOP_FOOTER_HEIGHT_PX` above the bottom rather than running to `bottom-0`, so the desktop footer stays fully visible and clickable in grid mode. That gap is *reserved in layout*, not left to z-order, because z-order provably could not solve it: the footer lives under `PersonasPage`'s `contain: layout style`, and layout containment makes that element both a stacking context and the containing block for fixed descendants ŌĆö so the footer's `z-[210]` was scoped to a subtree that itself sits at `z-auto`, and this `z-[200]` body portal painted straight over it regardless. `DesktopFooter` is therefore **portaled to `document.body`** too, which is what finally makes its z-index mean what it says (and fixes its upward-opening popovers ŌĆö theme / account / project picker / fleet ŌĆö which open *into* the overlay's region). Since the grid also covers the sidebar, the footer's centre slot swaps radio for **`FooterSectionNav`**: a compact rail of every section the navigation registry declares reachable, gated by the same `passesGates` tier/dev filter the sidebar uses, where picking one minimizes the grid and navigates in a single click. Outside grid mode the rail is hidden ŌĆö the sidebar owns navigation then, and two permanent nav systems would just disagree about which is authoritative.

  A **Grid** button in the Sessions action row (left of Spawn, disabled when no live sessions) opens the same overlay by setting the same flag. It is portaled to `document.body` so it sits above the framer-motion transform ancestors, and starts at `top-12` (below the 48px titlebar) rather than `inset-0` so the always-on-top titlebar ŌĆö its window controls **and the global Back button** ŌĆö stay visible and usable above it. The grid is **density-adaptive** ŌĆö columns = `min(4, ceil(ŌłÜN))`, i.e. 1├Ś1 ŌåÆ 2├Ś2 ŌåÆ 3├Ś3 ŌåÆ 4├Ś4 as sessions spawn (rows auto-fill and scroll past 16). While the overlay is open the single pane is unmounted so the two don't contend for the same managed terminal's holder; every tile attaches a durable terminal, so maximizing/minimizing is lossless. Terminals for removed sessions are reaped via `gcTerminals` keyed on the live session list. The overlay is self-sufficient: its header carries **Spawn** (new session in the active project) + **Skills** (drawer), and each tile (`FleetOverlayTile`) has a Terminal/Insights toggle and a **Kill** control ŌĆö so you can grow and prune the fleet without leaving the grid.
- **Four ways back.** The footer's fleet toggle, the overlay's own header Back button, **Escape**, and the **titlebar Back button** all minimize to the single pane (showing the last-selected session). The titlebar wiring uses a generic `backInterceptor` on `uiSlice.navigateBack`: a fullscreen surface registers a Back handler on mount (and the titlebar surfaces its Back button while one is set), so Back dismisses the overlay instead of navigating the underlying page out from under it. If every session exits while the grid is open it auto-minimizes.
- **Density-scaled font.** While the grid is open the terminal font is scaled to the grid density via a transient override (`setFleetFontOverride`): 1├Ś1ŌåÆ15px, 2├Ś2ŌåÆ14px, 3├Ś3ŌåÆ13px, 4├Ś4ŌåÆ12px (floored at 12px for legibility). The override never touches the persisted single-view `fleetTerminalFontSize`; it's cleared on minimize.
- **Renderer + addons.** WebGL rendering (`@xterm/addon-webgl`) is loaded **per attach** and disposed on detach, so N background terminals don't each pin a GL context (it falls back to the DOM renderer if WebGL is unavailable). `@xterm/addon-unicode11` fixes emoji/CJK/box-drawing widths; `@xterm/addon-web-links` makes URLs clickable, opened via `openExternalUrl` after `sanitizeExternalUrl`. The panes are otherwise chrome-free ŌĆö paste is right-click / Ctrl+Shift+V / Cmd+V; font size, copy-on-select and theme live in Fleet Settings (no per-pane buttons).

**Active fleet orchestration (autonomous mode).** When a session enters `AwaitingInput` ŌĆö it finished its turn / is paused for the next instruction ŌĆö and autonomous mode is on, Fleet wakes Athena to manage the fleet: `companion_record_fleet_event(state_changed)` ŌåÆ `maybe_orchestrate` ŌåÆ `spawn_proactive_turn` with the live **operative-memory digest** AND the session's **reconstructed screen** as context. Throttled per session (Ōēż1 wake/min) so it can't spam or loop. This is the endgame loop ŌĆö Athena keeping 10+ CLIs moving and only escalating key decisions. **Fires Rust-direct:** `apply_hook` calls `orchestrate_on_awaiting` the moment a hook sets `AwaitingInput`, so it works headlessly ŌĆö no dependency on the frontend store or the JS bridge (the frontend bridge also calls it; the per-session throttle dedups the two paths). **More-than-AwaitingInput triggers + recovery + durable memory (2026-07-04, Phases 3ŌĆō5).** The event-driven `AwaitingInput` path is now one of three orchestration flavors ŌĆö `orchestrate_on_awaiting` was generalized to `orchestrate_session(situation)`, sharing the gate/throttle/screen-dedupe/spawn machinery, with a per-situation directive:
> - **Stuck (Phase 3b).** `reassess_stuck_sessions` (proactive tick) detects a `dispatched_by_athena` session that failed and stalled (a `recent_failure` stamped + event-silent Ōēź 4 min + not yet intervened) and wakes Athena on its real screen + failure to propose a **`fleet_intervene`** (a one-line unblock typed into the session). > - **Idle-needs-next (Phase 3a).** A session that finished its turn and idles at the prompt (`Stop ŌåÆ Idle`) had no trigger ŌĆö orchestration only fired on `AwaitingInput`. `reassess_idle_needs_next` (tick) wakes Athena on a `dispatched_by_athena` Idle session (**never** the user's own ad-hoc CLIs) to judge **done-vs-needs-next** against the session's objective and send the next step via a gated `fleet_send_input`, or leave a finished session alone. The 60 s throttle + screen-hash dedupe bound the loop to *real progress* (a done session is woken at most once; it only re-triggers after work moves its screen).
> - **Per-session objective (Phase 5b).** Every orchestration directive now carries the session's objective ŌĆö its self-reported `intent` (via the MCP `athena.report_intent` path) or `role`, paired with the owning operation's `user_intent` ŌĆö pulled from operative memory, so she judges the next step against what the session is *for*, not just its screen.
> - **Recovery actions (Phase 4).** Two new confidence-gated dispatcher actions ŌĆö **`fleet_wake`** (revive a hibernated session via `--resume`) and **`fleet_resume`** (adopt an orphaned CLI process after a restart) ŌĆö wrap the existing `fleet_wake_session` / `fleet_resume_orphan` commands, wired through all five gate lists (same boldness bar, no screen re-check since the target is asleep/gone). Taught in the constitution's action menu (v42) for chat-driven revival ("wake the X session I slept"). Deliberately **no timer auto-reviver**: with idle-needs-next driving live sessions to completion, a hibernated dispatched session is one she already judged done, and orphan recovery needs a process scan ŌĆö a timer would fight the resource-saving auto-hibernator.
> - **Durable decision ledger (Phase 5a).** A `fleet_decisions` table persists every fleet verdict ŌĆö each defer (with a reason: `below_confidence_bar` / `screen_changed`) and each `auto_fired` / `auto_failed` ŌĆö so you can see **why** she stopped or acted on a session. `orchestrate_session` also skips re-waking on a screen she already **auto-fired** on, keyed on the STABLE `claude_session_id` + `screen_hash` (the in-memory dedupe is per-run; the registry id is regenerated each launch), so the suppression survives a restart.

**Orb UX during orchestration.** The orb is the operator's window into a running fleet ŌĆö it reuses the existing communication surfaces: the **thinking** avatar + **perimeter status dots** (`taskDots`, one per in-flight job) show Athena working, and a glanceable **"Managing the fleetŌĆ”" caption** appears beside the orb whenever she's working with the grid open. > **Confidence gate ŌĆö the boldness dial (Phase 2, 2026-07-04).** Auto-apply is gated on a **confidence ├Ś decision_class ├Ś boldness** matrix, not a flat high-only rule. >
> | dial | drive_forward | choice |
> |---|---|---|
> | `cautious` | high | high |
> | `balanced` | high\|medium | high |
> | **`bold`** (default) | high\|medium | high\|medium |
>

>

Runs before the pendingŌåÆrunning transition, so a deferred decision is still a clean consult.
>

> **Athena-owned PTY guard ŌĆö RELAXED (2026-06-25, user policy).** Auto-apply used to be scoped to sessions **Athena spawned herself** (`registry::is_athena_owned` via the `ATHENA_SESSION_NAME_SENTINEL`), so even a high-confidence answer to one of *your own* CLIs was left pending ŌĆö which read as "Athena can't do anything autonomously." ### Driving multi-select menus

A Claude Code `AskUserQuestion` **multi-select** is a checkbox TUI, not a text prompt. Writing a typed answer like `"1,2,3,4"` into it only lands the string in the first row and toggles nothing ŌĆö the menu is keyboard-driven (Ōåæ/Ōåō navigate, **space** toggles, **Enter** submits). The early na├»ve `fleet_send_input` did exactly that, so an approved 4-of-4 recommendation checked one box and never submitted. Detecting and correctly driving this is now done in `approvals.rs::execute_fleet_send_input`:

1. **Detect.** Before sending, it reads the session's **reconstructed screen** (`registry::render_screen_for` ŌåÆ the `vt100` ring snapshot ŌĆö the same path orchestration uses to read the prompt) and calls `multiselect_keystrokes(lines, text)`. A screen counts as a multi-select menu only if it has checkbox rows (`[ ]`/`[Ō£ö]`), a `Submit` row, and a `navigate` hint ŌĆö **and** the answer names option numbers. Anything else (single-select, free-text, non-menu) returns `None` and falls through to the plain typed `{text}\r`.
2. **Plan.** It parses the options in display order (`N. [state] Label`, skipping the `Type something` / `Submit` pseudo-rows) and builds a keystroke list: for each option top-down, **space** to toggle the requested-and-unchecked ones (already-checked wanted options are left alone ŌĆö no double-toggle), **Ōåō** between options, then **Ōåō** past the last option (and once more past `Type something` when present) onto `Submit`, then **Enter** (ŌåÆ "Ready to submit?") and **Enter** again (the confirm screen defaults its cursor to "Submit answers").
3. **Fire timed, not in a burst.** The keys are written one at a time on a spawned task with a **120 ms gap** ŌĆö the TUI silently drops a rapid burst. The call returns immediately with `"Driving multi-select on `<id>` (N keystrokes)."` while the task plays out.

**Gotcha ŌĆö the list WRAPS, so never Ōåæ-clamp.** An earlier version prepended an Ōåæ to "clamp" the cursor to the top before toggling. Claude Code's option list wraps on Ōåæ at the top, so the clamp jumped the cursor to the *bottom* (Cancel) and the whole plan misfired as "User declined". The menu is always freshly rendered when orchestration fires (cursor already on option 1), so the driver starts from option 1 with no leading Ōåæ (`3692d130d`). Three unit tests pin the sequences (select-all-four, skip-already-checked, no-op for non-menus). **Verified live 2026-06-25** both manually and end-to-end through the autonomous path (Athena proposed `1,2,3,4` ŌåÆ approve ŌåÆ driver ŌåÆ Claude acknowledged all four selections and proceeded, *not* "declined").

> **Fleet action results don't reach the companion chat.** `fleet_send_input` / `_broadcast` / `_kill` / `_intervene` are operational keystrokes into a CLI the operator is already watching on the grid ŌĆö their "approved & executed / failed" outcomes are noise in the Athena chat, not conversational turns. `log_action_episode` (`approvals.rs`) now early-returns for any `action.starts_with("fleet_")` (traces for debugging instead of persisting a `System` episode), so approving/rejecting fleet actions no longer floods the transcript with `[Athena action approved & executed] fleet_send_input ŌĆ”` lines (`fc79bba34`, from a user report of an overflowing chat). Non-fleet actions still persist as before.

## Hook installer details

Each entry is tagged `_fleet: true` so uninstall is surgical:

```json
{
  "hooks": {
    "Stop": [
      {
        "matcher": "*",
        "_fleet": true,
        "hooks": [
          {
            "type": "command",
            "command": "curl -s -m 2 --connect-timeout 1 -X POST --data-binary @- -H \"Content-Type: application/json\" http://127.0.0.1:17400/fleet/hooks/stop",
            "_fleet": true
          }
        ]
      }
    ]
  }
}
```

`curl` is the universal HTTP client (Windows 10+ ships it). Payload is streamed via stdin (`--data-binary @-`) so we never have to JSON-escape on the command line.

## Sharp edges to know about

- **`--mcp-config` is VARIADIC ŌĆö emit it LAST, after the positional prompt.** Claude Code's `--mcp-config <configs...>` greedily consumes every following non-flag token. If it sits before the spawn's positional prompt it eats the prompt as a bogus config file and claude exits 1 (`MCP config file not found: <cwd>/<prompt>`) ŌĆö the session dies the instant it spawns, which also silently kills Athena's orchestration (a dead session never reaches `AwaitingInput`). `pty.rs::spawn_session` therefore appends `--mcp-config` *after* the caller's args in both the Windows direct-exe and the fallback branch. If you add any new trailing argv, keep `--mcp-config` last. (Found via a live 4-CLI pass: every spawn-with-task exited on spawn.)
- **`claude --resume <id>` needs a prompt or it exits 1.** Claude Code Ōēź 2.1 refuses a bare `claude --resume <id>` with no continuation: it hunts for a deferred-tool marker to auto-continue and, finding none or a stale one, exits **code 1** with *"No deferred tool marker found in the resumed session ŌĆ” Provide a prompt to continue the conversation."* This silently killed every **Wake** (`fleet_wake_session`) and orphan **Resume** (`fleet_resume_orphan`) ŌĆö the session came back and died within seconds ("/resume failed quickly after recover"). Fix (verified live end-to-end 2026-07-02, together with the nesting-strip below: codeword ŌåÆ hibernate ŌåÆ wake ŌåÆ recalled): both resume paths pass `pty::RESUME_CONTINUATION_PROMPT` as the positional after `--resume <id>`, and `spawn_session` **skips LLM naming when `--resume` is present** so the nudge can't become the session title. **Detection half (high value):** `registry::mark_exited` folds the session's final rendered-screen line into the exit reason for any non-zero exit (`last_meaningful_line`), so resume failures self-explain ŌĆö *"Exited with code 1 ŌĆö claude said: No conversation found with session ID ŌĆ”"* ŌĆö instead of a bare, mysterious code 1.
- **Claude Code NESTING env markers disable session persistence ŌĆö STRIP them (the root cause of "resume fails after recover").** When the Personas app is launched from a Claude Code session (the normal dev workflow), every fleet PTY child inherits the parent's session markers (`CLAUDECODE`, `CLAUDE_CODE_CHILD_SESSION`, `CLAUDE_CODE_SESSION_ID`, ŌĆ”). **claude Ōēź 2.1.19x sees them and silently runs as a nested/child session: it never registers in `~/.claude/sessions/` and never persists a `<session-id>.jsonl` transcript** ŌĆö the session works live but leaves *nothing to resume*, so Wake/orphan-Resume died with *"No conversation found with session ID ŌĆ”"* and Insights had no transcript to read. Root-caused via A/B console spawns (identical argv; markers present ŌåÆ no storage; stripped ŌåÆ registers `kind=interactive` + persists) after per-axis in-app toggles exonerated the `--session-id` pin, `--mcp-config`, `CLAUDE_CODE_DISABLE_NONESSENTIAL_TRAFFIC`, and `--dangerously-skip-permissions`. Fix: `pty.rs::spawn_session` strips `CLAUDE_NESTING_ENV` (7 markers) next to the subscription-auth env strip ŌĆö fleet sessions are first-class top-level sessions, never children. **Verified end-to-end after the strip:** transcript persists under the pinned id, the session registers a PID file, and hibernate ŌåÆ wake ŌåÆ *"What was the codeword?"* ŌåÆ correct recall. (If a new claude version adds more `CLAUDE_CODE_*` markers, extend the list ŌĆö symptoms will be the same silent no-transcript behavior.)
- **In-session `/resume` is a known limitation (conflicts with the `--session-id` pin).** Fleet pins `--session-id <uuid>` on every fresh spawn for deterministic binding. When that session runs the **in-session `/resume` slash command** and picks a conversation, claude re-execs *in the same PTY* with `--session-id <uuid> --resume <picked>` and dies instantly: *"Error: --session-id can only be used with --resume if --fork-session is also specified"* ŌĆö the CLI "drops immediately without loading the content." Adding `--fork-session` to the fresh spawn makes `/resume` **load and stay interactive** (verified live end-to-end: picker ŌåÆ select ŌåÆ conversation loads, tile idleŌåÆrunningŌåÆidle, answers normally) ŌĆö so the crash *is* solvable. It was **not** kept: it forks the fresh session's claude id, diverging it from the pinned `--session-id` (a binding/hook risk not fully characterized). Prefer **Wake/Recover** to resume a conversation ŌĆö verified working end-to-end since the nesting-strip fix (see the previous bullet); the transcript-persistence bug that once made *both* paths fail with "No conversation found" is resolved. **Kept from this arc (both correct):** `apply_hook` **ignores a SessionEnd hook while the PTY `child_pid` is still alive** (an in-place re-exec such as `/resume`/`/clear` is not a death ŌĆö the reaper stays authoritative), and `registry::mark_exited` surfaces claude's real final line in the exit reason so these failures self-explain.
- **WebView2 can freeze `Object.prototype` ŌåÆ xterm dies at import (patched dependency).** Since ~2026-06-28 this machine's WebView2 creates every realm with `Object.isFrozen(Object.prototype) === true` (verified: a virgin `about:blank` iframe is born frozen; independent of runtime version 149.0.4022.80/.98, user-data profile, and app code ŌĆö an Edge machine-level experiment). In strict-mode ESM, assigning a member that shadows a frozen inherited data property throws ŌĆö `@xterm/xterm` and `@xterm/addon-webgl` both vendor VS Code's `KeyCodeUtils` namespace which does `o.toString = fn` at module scope, so the Fleet page died at lazy-import with *"Cannot assign to read only property 'toString'"*. The old `public/webview2-compat.js` accessor shim can no longer apply (the frozen prototype is also non-configurable). **Fix ŌĆö two layers:** (1) the `optimizeDeps.rolldownOptions.plugins` "webview2-compat" transform in `vite.config.ts` rewrites `X.toString = fn` ŌåÆ `Object.defineProperty(...)` during dep pre-bundling (covers xterm + decimal.js/d3/es-toolkit in **dev**); (2) `scripts/patches/fix-frozen-intrinsics.mjs` (wired to `postinstall`) mutates node_modules xterm so the two assignments ship patched in **production** too ŌĆö the optimizeDeps transform is dev-only (there is no production webview2 build plugin), so without the postinstall a `vite build` would ship an unpatched xterm. If a future xterm upgrade reintroduces the pattern (or another dep starts throwing this), the tell is the exact "read only property" TypeError at a dep chunk's module scope.
>
>   **2026-07-03 ŌĆö the recurrence was a cache trap; and don't "fix" the dev transform:**
>   - *Patching node_modules under a running dev server can't reach the browser.* Vite serves optimized deps `Cache-Control: max-age=31536000,immutable` under `?v=<hash>` derived from lockfile+config, **not** file content ŌĆö so a content patch keeps the same URL and WebView2's disk HTTP cache serves the pre-patch bundle forever (symptom: `curl http://localhost:1420/node_modules/.vite/deps/@xterm_xterm.js?v=<h>` shows `defineProperty` at the failing line, yet the app still throws there). This is why the committed postinstall fix "un-applied" itself the next morning. To recover: kill the app, then delete `node_modules/.vite` **and** `%LOCALAPPDATA%\com.personas.desktop\EBWebView\Default\{Cache,Code Cache,GPUCache}`, then relaunch. Clearing `.vite` alone is not enough (re-optimize is deterministic ŌåÆ identical hash ŌåÆ browser still cache-hits the stale immutable copy).
>   - *The `optimizeDeps` "webview2-compat" transform is intentionally inert on minified deps ŌĆö leave it that way.* Its `needsTransform` gate requires a space before `=` and its dedup guard is newline-bounded, which together stop it ever touching single-line minified bundles. That looks like a bug (it's why the plugin never patched xterm ŌĆö the postinstall does) but it's load-bearing: the transform's value-extent parser assumes `;`/function-body termination, so a minified sequence-expression assignment like `p.constructor=A,p.render=q$1)` (Preact inside `@sentry-internal/feedback`) makes it swallow past the comma and emit invalid JS ŌåÆ the optimizer dies with `[PARSE_ERROR]` and the app won't boot. Making the gate whitespace-tolerant reproduces exactly this crash; it was tried and reverted, with comments + a contract-lock test in `src/test/webview2-compat.test.ts`. Minified deps that need the fix (xterm) go through the targeted `fix-frozen-intrinsics.mjs` postinstall instead.
>
> `--dangerously-skip-permissions` skips *tool* permissions but NOT Claude Code's first-run *"Do you trust this folder?"* gate. A never-before-opened cwd parks the session at that prompt (rendered as `idle`, not a distinct state), and the trust restart can also break the pre-bound `--session-id` (claude starts a fresh session id post-trust). Real projects the user has opened are already trusted, so this mostly bites brand-new dirs.
- **One PTY session per cwd.** The spawn enforces this to keep hook routing unambiguous (we match hooks by claude_session_id once SessionStart fires, but the bootstrap window uses cwd).
- **Soft kill only (for now).** `fleet_kill_session` drops the writer + master, which makes CC see stdin EOF. Phase 6's cancellation token will land a hard kill path that does `Child::kill()`.
- **Port mismatch ŌĆö now self-healing (critical).** local_http binds the first free port in 17400-17415, so a restart can land a *different* port than the installed Claude Code hooks target. When that happens every hook POSTs into the void ŌĆö no SessionStart/Stop/Notification reaches Fleet, so spawned sessions silently lose all hook-driven state and fall back to transcript-growth alone (which reads "Stale" where the session is really idle/awaiting). This was a real, hard-to-spot break. **Startup now self-heals it:** after `local_http::start()` returns the live port, `setup()` calls `check_hooks(port)` and, if hooks are installed but `!port_matches`, immediately `install_hooks(port)` to re-point them. The Settings banner + manual re-install remain as a fallback. Re-install is also now **idempotent across ports**: `is_fleet_tagged` reclaims any entry whose command targets a `/fleet/hooks/` URL (any port, even older untagged ones), so a re-point cleans stale-port stragglers instead of duplicating.
- **No translation of hook payloads.** We only look at the URL path for the event type and `session_id` / `cwd` for routing ŌĆö Claude Code's payload shape varies across versions, so we don't depend on field shapes that may change.
