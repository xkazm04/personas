# Fleet — Claude Code session aggregator

> **Status:** Experimental. DEV builds only. Not yet shipped in any tier.

## What it is

Fleet is a Personas plugin that observes and controls multiple Claude Code (CC) CLI sessions from one Tauri window. It:

- Spawns `claude` in a PTY so we own stdin/stdout (xterm.js renders the live terminal).
- Receives Claude Code's lifecycle **hooks** (`SessionStart`, `Notification`, `Stop`, `PreToolUse`, `SessionEnd`, `UserPromptSubmit`) via an in-app HTTP receiver, turning them into a state machine: *Spawning → Idle (launched) → Running (real work) → AwaitingInput → Idle → Stale → Exited*.
- Watches `~/.claude/projects/*.jsonl` so transcripts produced by *external* `claude` runs (not spawned from Fleet) also flow through the state machine.
- Groups every tracked session by project (cwd-derived label) in a single grid UI, with status badges that update in real time.
- Lets the user **broadcast** a prompt to any subset of sessions at once — selecting "every session currently awaiting input" is the canonical workflow.

The plugin only shows up in `import.meta.env.DEV` builds. The Rust module always compiles so ts-rs bindings and command-name codegen stay stable across build profiles.

## Usage

1. **Open in DEV:** `npm run tauri:dev:lite`. Plugins → Fleet appears in the sidebar with a DEV badge.
2. **Install hooks:** Fleet → Settings → "Install hooks". Patches `~/.claude/settings.json` to POST every lifecycle event to `http://127.0.0.1:<port>/fleet/hooks/*`. Re-install whenever the local_http port changes (Personas detects mismatch on startup and prompts).
3. **Spawn a session:** Fleet → Sessions → "Spawn session" → enter a project directory → click Spawn. `claude` boots inside the Fleet-owned PTY. **Spawn with task** (next to Spawn) opens a small composer instead — the task you type rides as a positional argv (`claude "<task>"`), so the fresh session starts working on it the moment it boots rather than landing at an empty prompt.
4. **Run external sessions:** Run `claude` from any terminal once hooks are installed — it'll register in the grid via hooks alone (no PTY ownership on those rows, but state badges still work).
5. **Broadcast a prompt:** Fleet → Sessions → "Broadcast" → write your prompt → click "Waiting (N)" / "All" to pick targets → Send. Each session receives the bytes via its PTY's stdin (or via the hook stdin path for external sessions in a future enhancement).
6. **Apply a skill to the focused session (skill drawer):** Click **Skills** — above the terminal in the single-pane view, or in the fullscreen grid header. A library drawer slides in from the left over the sidebar region. It lists the **shared library** (`~/.claude/skills`, with a This-project / Global-library toggle), searchable. Click any skill to write its slash command (`/skill-name⏎`) into the **focused** session's terminal, exactly as if you'd typed it. The drawer header shows which session it'll apply to; pick a session first (click its row/tile) if none is focused. The skill must exist in that session's project — install it there first (next).
7. **Install a skill into a repo:** In the skill drawer, hover a skill and click its **Install to repo** (download) icon → pick a target project → Install. Fleet copies the skill's `SKILL.md` + reference files into that project's `.claude/skills/<name>/`. Enable **Overwrite** to replace an existing copy. This is what makes cross-repo apply (step 6) actually work — install once, then apply.
   - **Provenance & drift tracking:** each install stamps a hidden `.personas-skill-meta.json` sidecar recording the source and a content hash. A row then shows a **badge**: *In sync* (the installed copy still matches its source) or *Diverged* (the copy or its source changed since install). Hand-authored skills with no provenance show no badge (local-only) and are never touched. Before an **Overwrite**, the install modal previews a **diff summary** (files changed / added / removed) so you see exactly what a re-install replaces.
8. **Inspect a session's transcript:** In the single-pane view, flip the right-column toggle to **Insights**; in the fullscreen grid, each tile has its own **Terminal/Insights** toggle in its header. Fleet reads that session's `~/.claude/projects/**/<id>.jsonl` and shows a glanceable rollup — token totals (input/output/cache), turns + prompts, tools used (with counts), and files touched. Works for exited sessions too, so it doubles as a "what did this run do" review. Refresh to re-read a live session. The right-column header also carries a **context-size pill** (`ctx`) — the size of the context the focused session re-sends each turn (`last_context_tokens`), colored green→amber→red as it grows — a glanceable efficiency signal.
9. **Search across sessions (Activity tab):** The **Activity** tab lists the most recently-active Claude Code sessions across *all* projects (transcripts modified in the last 7 days, newest first), each with its token/turn/tool/files rollup. Type in the search box to filter across project name, files touched, tool names, and models — e.g. "which sessions touched `auth.rs`?" surfaces every matching run with the matching files shown.
10. **Hibernate & wake a session (F3):** Select a session → **Hibernate** (moon, in the CLI header). The `claude` process is killed to free it; the row stays as `Hibernated` (indigo) so it stays resumable. To bring it back, select the hibernated row and click **Wake** — Fleet runs `claude --resume <id>` in the original cwd and the resumed process restores the conversation. Hibernate needs a bound `claude_session_id` (it's how we resume), so a session must have started before it can sleep.
11. **Auto-hibernate idle sessions (F3/P3.2):** Fleet → Settings → **Auto-hibernate idle sessions** → enable + set the idle threshold (minutes). The always-on staleness ticker then hibernates any Idle/Stale session inactive past the threshold — freeing the process even when Fleet isn't focused (it stays resumable via Wake). `AwaitingInput` sessions are never auto-slept (you may be mid-response).
    - **Limit live sessions (live-slot scheduler, fleet-scale Tier A):** Fleet → Settings → **Limit live sessions** → enable + set the max (1–64, default 10). The fleet becomes *N tracked conversations, ≤max live processes*: the ticker hibernates overflow Idle/Stale sessions (oldest inactivity first; resumable only — a bound `claude_session_id` is required), and a spawn/wake at the cap first hibernates the best idle candidate so the new session starts inside the budget (`stale.rs::free_slot_for_spawn`). It's a **soft cap** — Running/AwaitingInput/Spawning sessions are never evicted, so a burst of genuinely-working sessions can exceed the limit until some go idle. This is what makes a large (40-session) fleet cost what ~10 processes cost: RAM/CPU tracks *active* work, not tracked conversations. Pushed to Rust via `fleet_set_live_slots` on change + on every Fleet refresh (0 = off), like auto-hibernate. External hooks-only rows (no Fleet-owned process) neither count against nor free a slot; the eviction choice is the pure, unit-tested `live_slot_evictions()`.
12. **Spawn a headless (background) session — fleet-scale Tier B:** the *Spawn with task* composer carries a **"Run in background (no terminal)"** toggle. It spawns `claude -p --input-format stream-json --output-format stream-json` with plain piped stdio (`headless.rs`, `fleet_spawn_headless_session`) instead of a PTY — **no ConPTY/conhost and no TUI redraw loop**, so an idle headless session costs ~zero CPU. The stdout reader parses one JSON event per line and drives the state machine directly (`system/init` → alive, `assistant` → Running, `result` → Idle) — structured events instead of vt100 screen-scraping — and feeds *cooked display lines* into the same output ring, so peek/previews/Athena's orchestration context all keep working with cleaner content. Conversation semantics are identical: pinned `--session-id`, persisted transcript, hooks fire, Insights work, and `write_input` wraps any reply (broadcast, quick-reply, Athena's `fleet_send_input`) into a stream-json user message. The UI never mounts an xterm for these (`mode: "headless"` on the session DTO — the tile shows a **Background** badge + status block; the single pane shows Insights). Hibernate works (PID-based killer); **Wake resumes the conversation interactively**. If Personas exits, the child's stdin pipe closes and `claude -p` exits after the in-flight turn — headless sessions can't linger as invisible orphans. This is the recommended lane for Athena-dispatched batch work at scale.
13. **Recover orphaned processes:** Fleet → Settings → **Running Claude processes**. Fleet's session registry is in-memory, so an app restart loses the session list while the underlying `claude` processes can keep running — orphans otherwise reachable only via Task Manager. The panel scans the OS process table (`fleet_detect_processes`) for *interactive* Claude sessions (the app's own transient `claude -p` calls are filtered out so they don't read as orphans), lists each with PID / memory / cwd, and marks which are still Fleet-tracked vs. orphaned. For an orphan you get two actions: **Resume** (`fleet_resume_orphan`) — derives the conversation recorded for that cwd, kills the orphan, and spawns a fresh *Fleet-tracked* `claude --resume <id>` in the same cwd, re-adopting the conversation into Fleet — and **Kill** (`fleet_kill_pid`, confirm-gated, targeted) to just end it. It scans on open + on **Scan**. The **Settings tab carries an orange badge** with the live orphan count (a FleetPage poll keeps it fresh, ~60s), so a restart's orphans are visible without opening the panel.
14. **Footer status cluster (DEV):** a fleet mark sits in the desktop footer, just left of the system-load gauge, followed by a **colored count chip per live lifecycle state** (`FleetFooterIcon`). It is the whole-fleet glance from anywhere in the app — see [Footer status cluster](#footer-status-cluster) below for the counts, the violet needs-you escalation, and the three-way click contract (close grid / raise grid / open page).
15. **Triage hotkeys:** on the Sessions tab, **N** jumps to the next awaiting-input session (same cycle as the banner's skip button), **↑/↓** move session focus through the visible grouped list, **/** focuses the session search, **G** opens/closes the fullscreen terminal grid, and **?** (or the keyboard icon in the page header) opens the shortcuts reference. Hotkeys ignore typing contexts — keys typed into a focused terminal, the search box, or any input never trigger navigation — and pause entirely while a modal or the skills drawer is open. While the grid overlay is up only **G** and **N** stay active (`useFleetHotkeys`).

## State machine

| State | How we enter it | Visual |
|---|---|---|
| `Spawning` | PTY just opened, no PTY output yet. First byte of PTY output (claude's banner/prompt) promotes it to `Idle` via `registry::mark_alive` — so a fresh interactive `claude` sitting at its prompt (no transcript, no hook yet) reads as **alive + ready**, never as never-attached. | spinner |
| `Running` | **Real progress only**: `UserPromptSubmit` hook, a tool firing (`PreToolUse` → `revive_to_running_on_activity`), or transcript **growth** (the stale ticker). NOT `SessionStart` — launching ≠ working. | blue spinner |
| `AwaitingInput` | Notification hook fired (CC is blocked, waiting for the user) | pulsing amber |
| `Idle` | `Stop` hook (turn ended cleanly), **or the session just launched / bound** — `SessionStart` and the transcript-bind fallback both land here: Claude is up and ready, not yet working. A bare spawn correctly sits at `Idle` until real work begins. | emerald check |
| `Stale` | No activity for 6 minutes (ticker) — the cutoff allows for slow console composition before flagging | orange clock |
| `Hibernated` | Operator clicked **Hibernate** (`fleet_hibernate_session`) — the `claude` process was killed to free it, but `claude_session_id` + `cwd` are kept so it can be resumed. NOT terminal. | indigo moon |
| `Exited` | PTY child reaped OR SessionEnd hook | grey ban |

Priority of signals: **process exit > hooks > JSONL mtime > inactivity ticker**. An Exited session never gets re-animated; a Stale session bounces back to Idle on any transcript append.

**Incremental metadata rollup (`transcript_read.rs`, the metadata-abstraction lane).** For scale (Athena managing 10+ parallel CLIs) Fleet must surface per-session *metadata* — tokens, tool counts, message counts, context size, files touched — without each tile re-parsing a multi-MB transcript or holding raw conversation output in memory. The parser is a foldable accumulator (`RollupAcc::fold_line`) shared by the full-file `summarize_lines` and an **incremental `ingest_delta`**: on each transcript append the watcher folds only the newly-written bytes (`[offset, EOF)`, leaving a half-written trailing line for next time) into a compact per-`claude_session_id` rollup, then discards the raw text. `fleet_session_metadata(claudeSessionId)` returns that live rollup (catching up any delta first — full read only on the very first call for a session); the Insights panel prefers it over the whole-file `fleet_read_transcript` and falls back to it on a miss. The raw JSONL stays "on the table" on disk, read in full only on explicit demand. This is the subscription-safe (CLI, not SDK) path to cheap fleet-wide observability; OpenTelemetry (`CLAUDE_CODE_ENABLE_TELEMETRY=1`, every span/metric tagged `session.id`) is the documented complement for aggregate cost/health if a collector is added later.

**Deterministic binding (`pty.rs`, primary for Fleet spawns).** A *fresh* Fleet spawn (not `--resume`) now assigns claude's session id itself — `--session-id <uuid>` on the argv — and pre-binds `claude_session_id = <uuid>` on the registry row at spawn. So the session is bound from the very first tick (no `cc:-` window, no never-attached race), its transcript is the known `<uuid>.jsonl`, hooks match by the known id (not cwd), and **N concurrent sessions in one cwd never cross-bind** — verified with 4 parallel Sonnet sessions in one repo, each tracking its own distinct transcript. `--model <id>` and other flags pass through the same argv (e.g. spawn a fleet on Sonnet for speed). The cwd-reconciliation paths below remain the fallback for externally-launched / resumed sessions that Fleet didn't assign an id to.

**Binding `claude_session_id` — fallback paths (`transcript.rs`).** An externally-launched or resumed session starts `cc:-` (unbound) and needs Claude Code's own session id to read its transcript / resume / show Insights. (1) **Primary:** the `SessionStart` hook POSTs `{session_id, cwd}`; `resolve_session_id` matches an unbound Fleet session by cwd and binds it. (2) **Fallback:** if that hook never lands (observed: a session runs + writes a transcript carrying Fleet's injected `mcp__athena__*` tools, yet stays `cc:-`), the JSONL **watcher** now reconciles by cwd — when a new transcript appears whose recorded `cwd` matches an unbound, non-Exited Fleet session, it binds `claude_session_id` from the transcript filename and lands the session at `Idle` ("Bound to transcript — ready (SessionStart hook missed)") — binding ≠ working, so real progress (a tool, transcript growth) is what drives `Running`. Without this fallback such a session was mislabeled stale/never-attached and Insights had no id to read. cwd matching is separator/case-insensitive (`normalize_cwd`). **Freshness guard:** the watcher only binds a transcript whose **file-creation time ≥ the session's spawn time** (−5 s grace) — otherwise, in a cwd with *other* concurrent claude sessions (the user's own Claude Code session, or a manual `claude` run in the same repo), it would grab a pre-existing/unrelated transcript and the Fleet session would falsely read `Running` off someone else's activity.

**Per-session title (`OscTitleScanner` + LLM naming).** Each session carries a `title` field surfaced on the tile/card as `name ?? title ?? projectLabel`, so sessions read distinctly instead of all showing the same project label. Two sources feed it, both via `registry::set_title` — which **ignores the literal "Claude Code"** (and empties) so the generic value never clobbers a real one:
> - **OSC window-title (`OscTitleScanner`).** Claude Code emits OSC `0`/`1`/`2` window-title escapes; the PTY `reader_loop` parses them (BEL- or ST-terminated, reassembled across reads) into `title`. In practice CC's *headless* OSC title resolves to the generic **"Claude Code"** for every session — which `set_title` drops — so this rarely yields a distinct label on its own; it's kept for sessions/versions that do emit a real summary. Captured in the reader (not xterm), so it works even for autonomous status-block tiles, with no UI subscription.
> - **LLM-assigned name (`naming.rs`, primary in practice).** On a spawn **with a task** (`task_from_args` finds the first positional arg, skipping flags + their values), Fleet fires a **fire-and-forget cheap-model (Haiku) call** — `spawn_claude_and_collect`, subscription auth — that returns a terse 3–5-word Title-Case label and stores it via `set_title`. UI-independent, so it labels the session whether or not anyone's watching; a user rename still wins (`name` precedes `title`); a bare interactive spawn (no task) keeps its project label.

**State provenance in the UI.** Hovering a session row's status dots now shows *why* the row is in its state: the last state-change reason plus the three deriving signals with ages — `hook activity Xs ago · console output Xs ago · transcript growth Xs ago` (`lastActivityMs` / `lastPtyOutputMs` / `lastGrewMs` on `FleetSession`, refreshed by the shared 30s tick). When idle/working ever look mixed again, the misclassifying signal is readable in-app instead of needing `PERSONAS_FLEET_DEBUG` and a dev console.

**Tunable cutoffs (Fleet → Settings → State detection cutoffs).** The stale (default 6 min) and frozen (default 2 min) cutoffs are user-tunable from a Settings card — persisted in the fleet slice and pushed to the Rust ticker on change + on every Fleet refresh (`fleet_set_state_cutoffs`, clamped server-side), the same plumbing as auto-hibernate. Power users running slow models or huge fleets calibrate state accuracy here instead of via env knobs.

### Debug recorder (DEV) — "why did the fleet do that?"

`Record` in the grid overlay header, next to **New session** (`FleetDebugLogButton`, DEV-only). It arms `debug_log.rs`, which appends a high-level, human-readable log to `<app-data>/fleet-debug/fleet-<timestamp>.log` until you stop it. The stop toast carries the path (30 s, because that path is the artifact).

**Stop is always reachable.** Recording is *started* from the grid header, but that header unmounts the moment you leave grid mode — all sessions exit, or you navigate away via the footer section-nav. If stop lived only there, an armed recorder would be stranded with no way to turn it off. So the recorder's status is mirrored into the fleet store (`fleetDebugLog{Active,Events,Path}`) and both surfaces drive it through one hook (`useFleetDebugLog`): the grid button, and an always-mounted **footer stop pill** (`FleetDebugLogFooterPill`) that renders nothing until a recording is active and then shows a red `⏹ Stop N` next to the fleet footer icon. Because the footer is always present (and portaled above the grid), the running recorder can be stopped from anywhere — grid open or not. Starting from the grid and stopping from the footer (or vice-versa) is the same operation and shows the same "saved → path" toast; the footer pill owns the single status poll, so an idle fleet does no background work.

**Why it's Rust-side and not a frontend event tap.** The three things worth debugging are all decided where the operator can't see them: state is decided by four independent signal sources, hibernation by a background ticker that runs whether or not the UI is open, and Athena's orchestration reasons on a path that is *deliberately silent* in the chat (see the orb-only note above). A frontend recorder would only ever see the flattened `{session_id, state, reason}` that crosses IPC — never her confidence, her decision class, or the reason she skipped a session entirely.

**What it records** — one line per event, `[elapsed] KIND  session  headline | detail`:

| Kind | Covers |
|---|---|
| `LIFE` | spawned (interactive/headless, bare vs with-task), killed, exited (with claude's own final line via `mark_exited`) |
| `STATE` | every lifecycle transition, `from → to` plus the deciding reason — tapped at `pty::emit_session_state`, the single emit point all four lanes now funnel through |
| `SLEEP` | hibernations split by cause (manual · auto-idle · live-slot eviction · slot freed for a spawn) and wakes, logged against the new session id with the old one in the detail so a hibernate → wake chain stays followable across the id change |
| `ATHENA` | every orchestration **wake** (situation, screen size, objective), every **skip** with its reason (autonomous mode off · throttled · screen unchanged · already auto-fired on this screen), and every **verdict** (`AUTO_FIRED` / `AUTO_FAILED` / `DEFERRED` + class + confidence + defer reason) with the text she sent and her rationale |

The Athena taps sit at the two natural choke points — `fleet_bridge::orchestrate_session` for wakes/skips and `approvals::record_fleet_decision` for verdicts — so every situation flavour (`awaiting_input` / `idle_needs_next` / `stuck`) and every action (`fleet_send_input`, `fleet_intervene`, `fleet_wake`, `fleet_resume`) is covered by construction.

**What it deliberately does NOT record:** terminal contents. An Athena wake logs the *size* of the screen she read (`screen=1841ch`, or `EMPTY (reasoning blind)` — itself a finding), never the frame. Those carry the user's code and paths, and the file is meant to be shareable. Also absent by design: PTY bytes, per-tool hooks, transcript deltas — this is a causal spine, not a trace.

**Cost when off:** one relaxed atomic load per fleet event (`debug_log::is_armed`). Nothing is opened, allocated or written until armed.

**Sharp edges.** The recorder is global Rust state, so it survives the overlay unmounting and keeps recording after you minimize the grid — the button re-adopts the running recording when the overlay reopens (that's what the `status` poll is for), and re-arming while active is a no-op rather than a second file. Label resolution uses the registry's non-blocking `try_lookup_*` accessors: `record` is called from paths that already hold the registry lock and `std::sync::Mutex` is not reentrant, so a blocking lookup would deadlock the app inside a debugging tool. Keep that rule for any field added later. Each line is flushed immediately — the point is to survive whatever is being debugged, including a hang or a hard exit.

**Doze — light sleep for parked sessions (2026-07-23).** Four parked `claude` processes cost real RAM/CPU, and both "parked" states mean nobody is typing: `AwaitingInput` is waiting on the human, `Stale` is waiting on anything. So the ticker's `doze_pass` frees the process of any session that has sat in either state past **60 s** (`PERSONAS_FLEET_DOZE_SECS`, `0` disables) — but, unlike hibernate, **keeps the displayed state**: the tile still says *Awaiting input* / *Stale* with a small indigo moon indicator (`dozing` on the DTO), because what the session was doing is the information the operator came back for. Mechanics mirror hibernate (kill via the child killer; the reaper's dozing branch keeps the state and never marks `Exited`); timing derives from `idle_since = max(last_grew, last_activity)` — awaiting stamps activity on entry, and `Stale` *means* idle past the stale cutoff, so "stale for a minute" is `cutoff + 60 s`. **Waking is "on return": selecting the session** (tile click, list row) resumes it via the same `fleet_wake_session` path (`claude --resume`, same grid slot via lineage). Only resumable sessions doze (bound `claude_session_id` — never-attached rows are skipped by construction, headless rows cost ~nothing and are exempt). Athena interplay: her orchestration verdict lands within seconds of `AwaitingInput`, well inside the window; a session she auto-fires goes `Running` (never dozed), a deferred one is exactly the waiting-on-the-human case doze exists for. Known limit: approving one of her pending proposals *after* the target dozed fails closed (no PTY writer) — wake the session first. Note the resumed conversation restarts at the prompt; a TUI question that was on screen before the doze is gone (claude doesn't flush it to the transcript), so the wake turn re-orients instead.

**Typed input SUBMITS — the confirmed-submit primitive (2026-07-24, the symbiosis-critical fix).** The second live run caught the fleet's worst failure: Athena `AUTO_FIRED` a full fix instruction into a session (`decision AUTO_FIRED · sent: "Fix all of them — …"`), **no `UserPromptSubmit` ever followed**, and 60 s later the doze pass reaped the session with her message stranded in the composer — to the operator it read as "no reply happens, the terminal got stuck and resumed". Root cause: every programmatic text path shipped `format!("{text}\r")` as **one chunk**, and Claude Code's composer distinguishes a *typed* Enter (a lone `\r` chunk) from a *pasted* one (a `\r` inside a larger chunk) — a paste with a trailing newline inserts a soft line-break and does not submit. (The 2026-06-25 verification never caught it because it exercised the paced multi-select keystroke driver, not this path.) Fix: `registry::write_text_line`, the primitive **all** text-delivery paths now use — Athena's `fleet_send_input` / `fleet_intervene` / `fleet_redirect_op`, the Needs-You quick reply, broadcast, skill apply, `/compact` (the latter four via a shape heuristic in `fleet_write_input`: multi-char text with a trailing newline routes here; single keystrokes and bracketed pastes pass through raw). It writes the text, waits ~350 ms for the TUI to ingest it, sends `\r` as its own chunk, then **confirms the submit** (the session flipping `Running` via hooks), retrying Enter once and logging `input submitted` / `input NOT confirmed` to the debug log. Every PTY write also stamps `last_activity_ms`, so doze backs off anything just typed into.

**Athena assessments resolve ON the session (2026-07-23; protocol hardened 2026-07-24).** The `OK:` / `NEEDS-YOU:` reply protocol closes the second live finding: her "this session is actively progressing — nothing to do" notes were structurally indistinguishable from real defers, so they escalated **healthy** sessions to "Awaiting your input" (once even stomping a session that had already gone back to `Running`). Every no-action orchestration reply must now begin with `OK:` (all fine / simply finished — stays entirely off the operator's screen, clears the "Athena's on it" window, debug-log only) or `NEEDS-YOU:` (a real operator decision — escalates + orb card); untagged replies fail **visible** (treated as NEEDS-YOU), and `escalate_to_awaiting` only acts on parked sessions (`AwaitingInput`/`Stale`/`Idle`) so a session that moved on is never stomped (`classify_fleet_note` / `handle_fleet_defer`, unit-tested). The auto-fire "Athena → project: text" notice also moved from a bespoke fixed-position pill (which floated nowhere near the draggable orb) to a **standard toast**. The first live run exposed the defer path as invisible: she woke on three awaiting sessions, read their screens, and correctly left all three end-of-turn prompts to the user — but the only trace was an orb card (in the wrong language, see below), while the tiles wore the light-blue "Athena's on it" border indefinitely. Two mechanism bugs and one policy gap, all fixed:
> - **The light-blue window never ended.** `athena_active` is a deadline the DTO evaluates lazily, but the frontend only re-snapshots on an event — nothing fired when the window lapsed, so the last snapshot (blue) stuck forever, masking the violet `awaiting_input` underneath. Now every assessment outcome **clears the window explicitly** (`resolve_athena_assessment`): auto-fire/auto-fail clear it in `approvals.rs`; a prose defer clears it via the escalation below; and the ticker sweeps lapsed windows each tick (`sweep_expired_athena`) as the backstop.
> - **A defer now escalates the session.** When her turn ends in a prose defer (no action dispatched — detected in `session::send_turn` via the `fleet_orchestration` turn origin), `note_fleet_defer` stamps the session `AwaitingInput` with reason *"Athena left this to you: \<her note\>"* and emits — lighting the violet border, the Needs-You banner chip, the footer count and the OS notification. Same for a **below-bar / screen-changed consult**: the pending proposal escalates the target session with *"Athena needs your review — she proposes: …"* (`escalate_fleet_consult`). The orb card still carries the full note; the session is just no longer silent about it. Every defer/consult also lands in the debug log (`ATHENA decision DEFERRED` + the note) — the first live run's log had wakes with no verdicts, which is what made this path look like "no reply happens".
> - **Language pinned.** One defer note arrived in Spanish. Every orchestration directive (and the manual Ask-Athena prompt) now ends with an explicit *"write your reply in English; text typed into a session must match its prompt's language"* instruction.

**Fleet at 30 — the 2026-07-24 load test and what it changed.** A 30-session live run (disjoint UI/UX tasks on one repo, autonomous full-auto, recorder + 15s per-pid resource sampling) answered the scale question directly. What held: spawn ramp (2 per 15s) was clean, state tracking stayed accurate, doze reclaimed the ENTIRE fleet footprint autonomously (claude processes 36 → 5; at 12-scale the measured reclaim was ≈6.4 GB RAM), and the grid at full burst — 30 tiles with 25 mounted xterms — cost the app only ~4 MB and ~1% of one core (parked terminals don't stream; note browsers cap WebGL contexts ≈16, beyond which tiles silently fall back to the DOM renderer). What broke: the companion's one-turn-at-a-time `try_lock` **dropped 27 of 57 orchestration wakes** during the completion burst (verdict latency where turns ran: median ~10s), and the 60s doze window swept sessions before the 2-minute re-check could retry — half the fleet was never assessed. Three fixes shipped:
> - **Wake queue** — `fleet_orchestration` turns now QUEUE on the turn lock instead of self-skipping (bounded at 32 waiters; `queued (depth N)` in the debug log). A parked CLI can afford to wait; the queue drains at ~1 verdict per 10s.
> - **Doze waits for the verdict** — the doze pass skips any session with an unresolved wake (`has_pending_assessment`, 6-min expiry so a wedged turn can't pin sessions awake).
> - **Machine-wide hook noise fixed** — installed fleet hooks now end in `|| exit 0`, so an app restart no longer makes every Claude Code session on the machine log hook-error pairs; `check_hooks` reports legacy commands as needing reinstall and the startup self-heal upgrades them.
>
> **Batched assessment (built 2026-07-24, superseding the wake queue as the primary drain).** Orchestration wakes now *enqueue* (`fleet_bridge::enqueue_assessment`) and a single active fleet turn assesses up to **6** parked sessions at once (`BATCH_MAX`): the batch turn re-renders each session's screen fresh, records its decision signature, and the directive requires per-session output — either a `fleet_send_input`/`fleet_intervene` action or a `<id8>: OK:` / `<id8>: NEEDS-YOU:` verdict line. `finish_assessment_turn` parses the verdict lines (with prefix repair against the batch for truncated ids), escalates the NEEDS-YOUs, treats unanswered sessions as unassessed (they re-enqueue), then re-drains the queue. A wedged batch turn releases after 4 min. At the 16-session multistep test this drained completion bursts in one turn instead of six (`assessing in a batch of 6` in the debug log) with no dropped wakes. **A throttled wake is a deferral, not a drop:** a session that re-parks inside the 60s per-session throttle (a multistep flow asking its next question right after Athena answered the last one) joins the queue immediately — arming the `has_pending_assessment` doze guard — and a delayed drain fires at throttle expiry. Without this, the probe's Phase-2 question was throttled at t+24s, doze reaped the process at t+60s, and the 5-min tick later assessed a corpse whose answer `AUTO_FAILED` against a dead PTY. The drain also drops queued sessions that are no longer parked at drain time (`batch drop` in the debug log), so a session that resolved itself never wastes a batch slot or gets a stale-screen decision.

**Self-healing under Claude-side outages (2026-07-24, from a live limit hit mid-16x).** A real usage-limit outage during the 16-session test produced the exact failure the operator should never see: Athena's batch turns died in ~3s with no output, six sessions decayed to `Stale` with questions still on screen, and nothing ever came back for them — the drain had recorded each screen's decision signature *optimistically*, so later wakes hit the "screen unchanged" dedupe and skipped forever. Four mechanisms now make the fleet recover on its own:
> - **Signature rollback + re-enqueue** — an unanswered batch member rolls back its recorded signature, re-arms the doze guard and rejoins the queue; a turn whose *entire* batch is unanswered is treated as the API-failure signature and the re-drain backs off 60s instead of hot-looping failed turns.
> - **The re-check sweep includes `Stale`** — parked-with-question sessions that decayed past the cutoff are re-assessed, not just `AwaitingInput` ones.
> - **Fires at dozed targets convert to wake + re-ask** — a resumed `claude` never re-renders the question TUI, so instead of typing a stale answer into a dead PTY (`AUTO_FAILED`), the executor wakes the session and nudges it to re-ask; the fresh question flows through the normal hook → batch → driver path. Interventions (plain instructions, still meaningful post-resume) wake + deliver directly.
> - **Mechanical limit retry** (`limit_retry_pass`) — a parked session whose screen shows a usage/rate-limit or server-error notice gets a confirmed-submit `continue` typed every ~4 min (max 8 attempts, doze guard stamped between attempts), spending **no** Athena turns — hers fail under the same limit. The batch directive tells her a limit screen is an `OK:` verdict, never a NEEDS-YOU.

**Fleet↔Athena delivery efficiency (2026-07-24).** Athena's orchestration turns spend the same Claude subscription the fleet spends — at 16×multistep scale her turns are a first-order cost and the limit waves are partly self-inflicted. Round-2 measurement: **58 turns served 103 assessments; 40 of the 58 (69%) carried a single session** — wakes mostly arrive alone and the instant drain turned each into a full-priced turn. Two levers shipped:
> - **Coalescing window** — the first queued wake schedules the drain ~12s out (`DRAIN_COALESCE_MS`); a full queue (`BATCH_MAX`) drains immediately. A parked CLI doesn't feel 12 seconds; the subscription feels every turn.
> - **Screen-tail trim** — batch directives carry only the bottom ~1800 chars of each screen (the question/prompt is bottom-anchored; the scrollback above restates the objective). Signature hashing still uses the full render.
>
> Designed, not yet built (each carries a judgment tradeoff worth an explicit decision):
> - **Mechanical prompt answering** — recognized no-judgment screen shapes (ordinary in-project permission prompts under autonomous+bold, scripted confirm gates) answered by the driver directly, no Athena turn. Biggest remaining lever; changes the security posture from "Athena rubber-stamps" to "system auto-approves known shapes" — needs operator sign-off.
> - **Completion-detect skip** — a finished session (Stop hook, summary on screen, idle) still costs an ALL-FINE turn today (27 of round-2's turns were pure completion confirmations). Terminal-state detection could skip these or fold them into one end-of-run digest turn.

**Driving Claude Code's question TUI (the multistep gate, fixed 2026-07-24).** Two independent bugs made every driven multi-select silently fail at 16-scale, with the answers stranded on screen and the sessions dozed:
> - **Runtime lifetime** — the keystroke player ran on `tokio::spawn` inside a proactive turn's *throwaway current-thread runtime*, so the task died mid-plan the moment the turn returned. It now spawns on `tauri::async_runtime` (the app's long-lived runtime), like `write_text_line` always did.
> - **Tabbed AskUserQuestion layout** — current Claude Code renders questions as tabs (`←  ☐ Question  ✔ Submit  →`) above the options. Submission is the **Submit tab** — `→` then Enter on "Submit answers" — not a Submit *row* below the list; the legacy plan's trailing `↓…↓ + Enter Enter` landed on `Type something`/`Chat about this` and never submitted. `multiselect_keystrokes` detects the ☐/☒ tab markers and ends tabbed plans with `→`+Enter (legacy flat menus keep the row walk); both confirm-retry paths (the multi-select driver and `write_text_line`, which carries single-select digit answers) now recover with `→`+Enter — in a plain composer `→` is just a caret move, so the retry stays safe for ordinary text. Diagnosed by driving a stuck session key-by-key over the bridge; every drive logs its plan + the screen it saw (`fleet_multiselect` tracing target) and confirms the submit against the session flipping `Running`.

**Test knobs (prod-safe, env-gated):**

**Test knobs (prod-safe, env-gated):** `PERSONAS_FLEET_STALE_SECS` / `PERSONAS_FLEET_NEVER_ATTACHED_SECS` shorten the cutoffs for fast observation (default 6 min / 2 min when unset; env knobs take precedence over the Settings-tuned values); `PERSONAS_FLEET_DEBUG=1` makes the ticker log each session's decision inputs (`state · cc · idle · grewAgo · size`) every tick to the dev console — the fastest way to see *why* a session is (mis)classified. Drive the whole loop via the test-automation server (`npm run tauri:dev:test`, HTTP `:17320`; `fleet-spawn` / `footer-fleet-toggle` / `fleet-session-row-*` test-ids).

**Growth-hardened staleness (`stale.rs`).** Hooks fire at discrete points, so a session's state could drift — a hung session stuck `Running` ("in progress" when it's not), or a working session marked `Stale` mid-long-operation. The ticker polls each session's transcript **size** every tick. Freshness is the **most recent of (a) real transcript growth and (b) hook-driven `last_activity`** (`idle_since = max(last_grew, last_activity)`): a session whose log **grew** is bounced `Stale`/`Idle` → `Running`; a `Running`/`Idle`/`Spawning` session that's had **neither growth nor a hook past the cutoff** is marked `Stale`. (Growth alone marked a working session `Stale` during a long tool op — hooks firing, transcript not yet flushed; hooks alone let a hung session look alive. The max of both keeps active sessions fresh without reviving genuinely-hung ones, which have neither.) `AwaitingInput` is never staled by flat logs (it's correctly waiting on the user). The decision is a pure `staleness_transition()` fn with unit tests.

**False-`AwaitingInput` correction (two revive paths).** Claude Code fires its idle *"waiting for input"* `Notification` during long tool waits / model-latency gaps, which wrongly parks an in-progress session at `AwaitingInput`. Two correctors pull it back to `Running` — but only on *real* evidence of work, so a session genuinely blocked on a question (e.g. AskUserQuestion) is left alone: (1) **Immediate — tool activity.** Tool hooks (`PreToolUse`/`PostToolUse`) no longer flow through the lifecycle state machine (they feed operative memory), so a spurious `AwaitingInput` used to get stuck. `receive_hook` now calls `registry::revive_to_running_on_activity` on every tool event — a running tool is proof the session is working, so it flips `AwaitingInput`/`Idle`/`Stale` → `Running` (emitting only on the actual transition, to keep per-tool volume low). (2) **Backstop — growth after the await.** For no-tool sessions (pure text streaming), the ticker revives `AwaitingInput` on transcript growth that happens *strictly after* the await began: the first tick that sees `AwaitingInput` snapshots a size baseline (`AWAITING_BASELINE`) — by then the assistant message that may justify a *legitimate* await is already flushed and folded into the baseline — and any later tick whose transcript exceeds it means the session kept working, so it's revived. A genuinely-waiting session never grows past the baseline, so it stays `AwaitingInput`.

The flat-log cutoff is `STALE_AFTER_SECS` = **6 min** — deliberately generous so a session doesn't flip to `Stale` while the user is still composing a long prompt in the console (the transcript only grows on submit).

**Frozen-process fast path (`is_frozen_mid_run`).** A hung session used to wear the blue `Running` spinner for the full 6-minute cutoff. The PTY reader now stamps `last_pty_output_ms` (throttled to 1/s) on every chunk — and since claude redraws its status line continuously *even when idle*, **total PTY silence is a high-confidence "the process is frozen" signal** (while output *presence* proves nothing about work, which is why this field never feeds freshness). A `Running` session with zero PTY bytes AND no transcript growth/hook activity for `STALLED_AFTER_SECS` = **2 min** is flagged `Stale` with a distinct verdict ("No console output for 2 min — claude looks frozen mid-run. Safe to kill, or wake it with a prompt."). Sessions that never produced a byte are the never-attached check's case, not this one. Test knob: `PERSONAS_FLEET_STALLED_SECS`; the debug ticker line now includes `outAgo`.

**Never-attached spawns (`is_never_attached`).** A session that *never produces a byte of PTY output* — so it's still `Spawning` (never promoted by `mark_alive`) with no bound `claude_session_id` and no activity for `NEVER_ATTACHED_SECS` (2 min) — is flagged `Stale` with the distinct reason *"Claude never attached — the folder may need trust approval, or claude failed to start. Safe to kill."* This catches the case where `claude` is spawned into a folder it hasn't been trusted in (the first-run trust prompt blocks startup) or otherwise fails to come up — previously these were mislabeled as generic "stale" three minutes later, hiding the real problem. It's safe because the transcript watcher bumps `last_activity_ms` (matched by cwd, even before a cc id binds) for any session that's actually running, so a frozen timestamp here genuinely means nothing started. If the same project keeps producing never-attached sessions, set its folder up for Claude Code (trust it once / add `.claude`) or stop spawning there. `Hibernated` is operator-driven (not a signal-derived state): the reaper records the hibernation-triggered child exit as `Hibernated`, not `Exited`, and the staleness ticker leaves it alone. **Wake** (`fleet_wake_session`) spawns a fresh PTY running `claude --resume <claude_session_id>` in the original cwd and drops the placeholder — the resumed process restores the conversation itself.

## Terminal experience

The session terminals are rendered with **xterm.js**, but the xterm instances are no longer owned by the React pane. They live in a singleton **terminal manager** (`fleetTerminalManager.ts`, parked on `globalThis` so HMR doesn't reset them), keyed by session id. `FleetTerminalPane` is a thin **mount point** that *attaches* a session's terminal into its container on mount and *detaches* (never disposes) on unmount.

### Output capture is decoupled from rendering (scaling to 16 CLIs)

The single biggest cost at scale is streaming + parsing PTY output for sessions nobody is looking at. So **capture and rendering are decoupled**:

- **Backend ring buffer + subscription gate** (`pty.rs` reader_loop + `registry.rs::OutputRing`). The per-session reader thread ALWAYS drains its PTY into a bounded ring (`OUTPUT_RING_CAP`, 512 KiB) — so the pipe never fills and `claude` never blocks, and the app's memory per session stays flat no matter how much a 1M-token run prints. It only emits `FLEET_SESSION_OUTPUT` over IPC **while the session is subscribed**. The reader holds the ring `Arc` directly, so the per-read hot path never touches the registry-map lock.
- **Persistent per-session screen model (fleet-scale Tier C).** `OutputRing::render_screen` no longer re-parses up to 512 KiB through a fresh `vt100::Parser` per call. The ring holds a **persistent parser**, lazily materialized on the first screen read (one catch-up feed of the ring) and then fed **incrementally by every `push`** — so steady-state screen reads (orchestration wakes, screen-hash dedupe, execution-time re-checks, previews, exit-reason tails) are O(screen) instead of O(ring). A PTY resize rebuilds it once at the new dims. Memory: one rows×cols cell grid per session.
- **Subscribe on attach, unsubscribe on detach** (`fleet_subscribe_terminal` / `fleet_unsubscribe_terminal`). Attaching a pane subscribes (and gets the ring snapshot back to hydrate the terminal); detaching unsubscribes. So **only the watched session(s) cross IPC and pay the xterm parse/render cost** — a 16-CLI fleet with one focused pane streams one session, not sixteen. This is what lets Fleet out-perform sixteen `cmd.exe`/conhost windows each rendering in realtime: the app's work tracks *watched* sessions, not *running* ones.
- **Hydration ordering.** Between issuing the subscribe and writing its snapshot, live chunks are queued (`hydrating` / `pendingLive` in the manager) and flushed right after the snapshot, so a re-focus never interleaves or duplicates. The flip-subscribed-and-snapshot happens under the ring lock, so each byte is either in the snapshot or live-emitted, never both, never neither.
- **Switch-back replays the recent tail, not full history.** Because a parked terminal unsubscribes, switching back replays the ring (≈ last 512 KiB of scrollback) rather than a fully live-updated buffer. This is the deliberate trade — *we don't render background rows in realtime* — and it's what keeps a large fleet cheap. The terminal is `reset()` before the snapshot is written so the replay can't duplicate the prior buffer.

- **Attention-driven grid rendering (watch what needs *you*).** The fullscreen grid renders neither a live terminal nor a cooked preview per tile. A tile mounts a real (subscribed) xterm **only** when the session needs the operator (`needsLiveAttention` in `fleetAttention.ts` — today `awaiting_input`) or it's the focused tile (the manual peek). Every other session is autonomous (`running`/`spawning`/`idle`/`stale`) and renders a lightweight `FleetTileStatusBlock` — state affordance + heartbeat (`formatAgo`) + the conversation-size pill — with no xterm, subscription, or ring poll. This is safe because Athena's decision access is **fully decoupled from the UI**: she reads each session via operative memory + the always-populated ring + the transcript (see [Active fleet orchestration](#) below), so dropping a tile's terminal blinds nobody. She triages the autonomous/stale tiles in the background and *escalates by raising the tile* — an on-tile proposal, or the session flipping to `awaiting_input`. The live-terminal count therefore tracks how many sessions need you at once (usually 0–3), so a 9-tile (3×3) grid stays calm and extends cleanly. *(The earlier cooked-preview tier — `FleetTilePreview` / `useFleetTilePreviews` / `fleet_terminal_previews` + `cook_lines`, change-gated by a per-ring `rev` and visibility-gated via `useDocumentVisibility` — is retained as infra for a possible future large-grid mode but is currently unused by the grid.)* This is **Increment 1** of the Athena-triage redesign; later increments surface the live `current_tool` and a per-tile Athena watching/reasoning/escalation signal.

Consequences:

- **Near-instant session switching.** Switching the active session — or leaving and returning to the Fleet page — re-subscribes and replays the ring snapshot in a single write; the durable xterm instance (renderer, keyboard, addons) is reused, never disposed.
- **Fullscreen grid overlay — an app-wide LAYER, not a page surface.** The overlay is mounted at app root (`FleetGridLayer`, rendered in `App.tsx`'s global-overlays island, DEV-only), **not** by `FleetGridPage`. That's what lets the footer raise grid mode over Overview, an agent editor, or anything else without a navigation: check the fleet, reply to whatever needs you, minimize, and the page underneath never moved. Two consequences worth knowing:
  - `FleetGridLayer` itself is deliberately **cheap** — no xterm in its import graph. It owns the fleet's app-wide *bootstrap* (attaching the session listeners + the first `fleet_list_sessions` snapshot at launch, which is also what keeps the footer counts live and what finally pushes the persisted auto-hibernate / live-slot / state-cutoff policy to the Rust ticker at startup instead of on first Fleet visit) and lazy-loads `FleetGridOverlayHost` — the xterm-bearing half — only on first open. `@xterm/*` is heavy and WebView2-sensitive (see the frozen-`Object.prototype` note below); it must not sit in the app-boot chunk.
  - **`FleetTerminalOverlay` must never write `fleetGridOpen`.** It used to set the flag on mount and clear it on unmount (so the Athena orb could lift above it). Now that the layer *renders from* that flag, the write is a cycle — under StrictMode's mount → cleanup → mount effect double-invoke the cleanup's `setGridOpen(false)` re-enters the layer's render mid-flush and leaves the overlay **wedged open**: mounted, but no longer closeable from the footer. Observed live, then removed; `onClose` is the single close path and the orb reads the same store flag, so nothing was lost. Pinned by `__tests__/FleetTerminalOverlay.test.tsx`.

  - **It leaves the footer alone — and the footer becomes the navigation.** The overlay stops at `DESKTOP_FOOTER_HEIGHT_PX` above the bottom rather than running to `bottom-0`, so the desktop footer stays fully visible and clickable in grid mode. That gap is *reserved in layout*, not left to z-order, because z-order provably could not solve it: the footer lives under `PersonasPage`'s `contain: layout style`, and layout containment makes that element both a stacking context and the containing block for fixed descendants — so the footer's `z-[210]` was scoped to a subtree that itself sits at `z-auto`, and this `z-[200]` body portal painted straight over it regardless. `DesktopFooter` is therefore **portaled to `document.body`** too, which is what finally makes its z-index mean what it says (and fixes its upward-opening popovers — theme / account / project picker / fleet — which open *into* the overlay's region). Since the grid also covers the sidebar, the footer's centre slot swaps radio for **`FooterSectionNav`**: a compact rail of every section the navigation registry declares reachable, gated by the same `passesGates` tier/dev filter the sidebar uses, where picking one minimizes the grid and navigates in a single click. Outside grid mode the rail is hidden — the sidebar owns navigation then, and two permanent nav systems would just disagree about which is authoritative.

  A **Grid** button in the Sessions action row (left of Spawn, disabled when no live sessions) opens the same overlay by setting the same flag. It is portaled to `document.body` so it sits above the framer-motion transform ancestors, and starts at `top-12` (below the 48px titlebar) rather than `inset-0` so the always-on-top titlebar — its window controls **and the global Back button** — stay visible and usable above it. The grid is **density-adaptive** — columns = `min(4, ceil(√N))`, i.e. 1×1 → 2×2 → 3×3 → 4×4 as sessions spawn (rows auto-fill and scroll past 16). While the overlay is open the single pane is unmounted so the two don't contend for the same managed terminal's holder; every tile attaches a durable terminal, so maximizing/minimizing is lossless. Terminals for removed sessions are reaped via `gcTerminals` keyed on the live session list. The overlay is self-sufficient: its header carries **Spawn** (new session in the active project) + **Skills** (drawer), and each tile (`FleetOverlayTile`) has a Terminal/Insights toggle and a **Kill** control — so you can grow and prune the fleet without leaving the grid.

  - **Tiles never move (stable positioning, 2026-07-23).** The first live 4-terminal run showed tiles shifting under the operator — any membership change compacts a CSS grid, so "session X is the top-left tile" muscle memory broke every time a session left. Three rules now make positions stable *by construction* (`fleetSessionScope.ts`): (1) order is locked to spawn time with the session **id as a total tie-break** (the Rust snapshot iterates a HashMap, so equal timestamps would otherwise re-shuffle per fetch); (2) **every session keeps its tile** — `exited` and `hibernated` rows render as in-place **tombstones** (status block + exit code / wake hint) and leave the grid exactly one way: the operator's trash control, which is state-aware (live session → **Kill**, tombstone → **Dismiss** = `fleet_remove_session`); (3) **Wake keeps the slot** — `fleet_wake_session` reads the old row's `created_at_ms` + name (`registry::lineage_of`) and stamps them onto the resumed row (`adopt_lineage`), so a resumed conversation lands where its predecessor sat instead of jumping to the end. The grid auto-minimizes only when the last tombstone is dismissed (zero tracked sessions).
  - **Click focuses the terminal.** Clicking a tile frame used to reveal the xterm but still demand a second click inside it before typing. The tile now calls `focusTerminal` whenever it is the active tile showing a terminal (parent effects run after the child pane's attach effect, so the terminal is attached by then) — one click to select *and* type.
- **Four ways back.** The footer's fleet toggle, the overlay's own header Back button, **Escape**, and the **titlebar Back button** all minimize to the single pane (showing the last-selected session). The titlebar wiring uses a generic `backInterceptor` on `uiSlice.navigateBack`: a fullscreen surface registers a Back handler on mount (and the titlebar surfaces its Back button while one is set), so Back dismisses the overlay instead of navigating the underlying page out from under it. If every session exits while the grid is open it auto-minimizes.
- **Density-scaled font.** While the grid is open the terminal font is scaled to the grid density via a transient override (`setFleetFontOverride`): 1×1→15px, 2×2→14px, 3×3→13px, 4×4→12px (floored at 12px for legibility). The override never touches the persisted single-view `fleetTerminalFontSize`; it's cleared on minimize.
- **Renderer + addons.** WebGL rendering (`@xterm/addon-webgl`) is loaded **per attach** and disposed on detach, so N background terminals don't each pin a GL context (it falls back to the DOM renderer if WebGL is unavailable). `@xterm/addon-unicode11` fixes emoji/CJK/box-drawing widths; `@xterm/addon-web-links` makes URLs clickable, opened via `openExternalUrl` after `sanitizeExternalUrl`. The panes are otherwise chrome-free — paste is right-click / Ctrl+Shift+V / Cmd+V; font size, copy-on-select and theme live in Fleet Settings (no per-pane buttons).

### Athena copilot on the grid (experimental)

A UI-fusion layer (`fleetAttention.ts`, `FleetTileAthenaBar.tsx`) that surfaces Athena's existing fleet reasoning *on the terminal tiles*, rather than only in the Needs-You banner. Pure frontend over existing backend (`companion_send_message`, the approval pipeline, `fleet_send_input` / `fleet_intervene`).

- **Attention borders.** Tiles (and the single pane) get a pulsing border by state — violet for `awaiting_input`, light-blue while Athena's working it (`athenaActive`), amber for `stale`, red for a non-zero `exited` — via the `fleet-attn-*` classes in `globals.css`, chosen by `sessionAttention()` / `attentionClass()`. Healthy tiles stay plain. The grid overlay header carries an always-visible **`FleetAttentionLegend`** (swatch colours mirror the `.fleet-attn-*` rgba so they can't drift) decoding those border colours at a glance — it's now the *only* header legend (the dot-decoding `FleetStatusLegend` was retired from the header on 2026-06-25; the per-row status dots keep their own hover tooltips).
- **On-tile suggestions.** When a pending companion approval is a `fleet_send_input` / `fleet_intervene` targeting a tile's session (matched by parsing `paramsJson.session_id`), the tile shows Athena's proposed text with **Approve** (→ `companion_approve_action` → writes it into the PTY) / **Dismiss**. The existing approval pipeline is the write gate — nothing auto-types.
- **Ask Athena.** A *genuinely* stale tile (it attached, then went quiet) with nothing pending shows an "Ask Athena" button that fires a session-scoped turn (`companionSendMessage(craftStalePrompt(session))`) asking her to decide the next step and, if there's a clear winner, propose writing it — which returns as an on-tile suggestion. The tile shows a "thinking" affordance until the turn resolves. **Never-attached sessions don't get the button** — instead the strip shows a non-actionable *"Claude never attached — kill & retry"* note, because there's no live agent for Athena to nudge. This stops the wasteful loop where every inert spawn invites a futile investigation. `isNeverAttached` keys on the **Rust ticker's `state_reason` verdict** (the confident 2-min check), not a broad "stale without a cc id" guess — a genuinely-working session whose cc id simply hasn't bound yet is normal `stale` and still shows real state + the Ask-Athena button. The tile stays amber `stale` either way (never painted red `failed` — that misread as a crash and hid the real state). As a backstop, `craftStalePrompt` also returns a never-attached-specific prompt (telling Athena to confirm the kill, not propose `fleet_send_input`) if she's asked anyway. **Provenance:** the Ask-Athena prompt is *synthetic* (Fleet's words, not the user's), so it's forwarded with `companionSendMessage(prompt, …, 'Fleet')` → the backend records it as a `TurnOrigin::External { source: "Fleet" }` turn (persists as `EpisodeRole::System`, the model gets an explicit "automated request from Fleet — not the user" prefix). In the chat it renders as a slim **Fleet divider** (`companion-fleet-marker`, grid icon), never an impersonated user bubble — so user / system-Fleet / Athena turns read apart at a glance.

- **Athena visible in the grid.** The fullscreen overlay is `z-[200]`, above the orb's normal `z-50` — so while it's open the orb would be hidden. The overlay sets `fleetGridOpen` (fleet slice) on mount and `AthenaOrbLayer` raises itself to `z-[210]` while that's true, so Athena's orb (her thinking/speaking state + decision bubbles) floats above the grid and you can see + react to her there. **The chat panel rides the same lift:** tapping the orb opens the full chat at `z-[220]` (above the overlay) — otherwise the panel opened *behind* the grid and read as "orb tapped, then nothing" (the chat, her messages, and any decision/approval UI were all buried under the overlay). Ask-Athena → propose → approve-on-tile → `write_input` is wired end-to-end (verified at `approvals.rs::execute_fleet_send_input`). Note: if a target session never attached (see [State machine](#state-machine) → never-attached), Athena correctly declines to propose `fleet_send_input` — there's no live agent to type into; the fix is to kill/recover that session, not nudge it.

**Active fleet orchestration (autonomous mode).** When a session enters `AwaitingInput` — it finished its turn / is paused for the next instruction — and autonomous mode is on, Fleet wakes Athena to manage the fleet: `companion_record_fleet_event(state_changed)` → `maybe_orchestrate` → `spawn_proactive_turn` with the live **operative-memory digest** AND the session's **reconstructed screen** as context. The screen view (`registry.render_screen_for` → `OutputRing::render_screen`) feeds the always-on PTY ring's raw bytes through a **`vt100` emulator** and snapshots the rendered grid — UI-independent — which is what lets her actually *read the prompt/decision a session is blocked on* — a single- or multiple-select question, a permission, free-text — instead of reasoning blind from the fleet digest. **Why a full VT emulator, not the line-cooker:** an `AskUserQuestion` (and most permission prompts) is an **alt-screen, cursor-addressed TUI**; the old `cook_lines` tail reset on the alt-screen toggle (`?1049h`) and ignored cursor positioning, so it collapsed the menu to empty/fragments and Athena correctly refused to "blind-answer." Reconstructing the screen renders the menu exactly as the operator sees it (verified live 2026-06-25: she went from "nothing visible" to naming the actual options + a reasoned lean). Note the transcript is **not** an alternative source here — claude doesn't flush the JSONL while parked at a question, so the rendered live screen is the only place the pending prompt exists. She then either (a) **answers a clear decision** with a `fleet_send_input` carrying the chosen option (number/text) — a single-select / free-text answer is typed verbatim, a **multi-select is *driven* by keystrokes** (see [Driving multi-select menus](#driving-multi-select-menus) below), auto-applied via the autonomous allowlist (see [Auto-apply](#) below), or surfaced as an orb decision with her recommended text when she's less sure — or (b) **defers**: if it's genuinely the user's call (a preference, a risk she shouldn't take on their behalf, or the work looks finished) she does NOT send input and instead surfaces a concise orb decision saying she's leaving it to them, naming her recommended option if she has a lean. Throttled per session (≤1 wake/min) so it can't spam or loop. This is the endgame loop — Athena keeping 10+ CLIs moving and only escalating key decisions. **Fires Rust-direct:** `apply_hook` calls `orchestrate_on_awaiting` the moment a hook sets `AwaitingInput`, so it works headlessly — no dependency on the frontend store or the JS bridge (the frontend bridge also calls it; the per-session throttle dedups the two paths). Verified live: an invoke-spawned session not in the frontend store still auto-woke Athena on task completion.

**Timer re-check + no blind nudge (2026-07-03, Phase 1).** Orchestration used to be *event-only* — it fired on the `AwaitingInput` hook, then never re-looked — so a session whose first screen render came back empty (the PTY ring hadn't captured the alt-screen TUI yet) or whose screen changed without a fresh `Notification` hook silently stalled. And in autonomous mode a *separate* blind proactive nudge (`fleet_attention` → *"…awaiting your input for ~N min. Want me to peek at what it needs?"*) fired in parallel — pure contradictory noise, since orchestration already owns `AwaitingInput` sessions in exactly that mode, and the blind card discarded the screen-reading capability to wait for a human Engage click. Both are fixed: the blind `fleet_awaiting` arm is removed from `fleet_triggers.rs`, and `fleet_bridge::reassess_stale_awaiting` re-runs `orchestrate_on_awaiting` on the 5-min proactive tick for sessions parked in `AwaitingInput` ≥ 2 min. Re-running every tick is safe because orchestration's 60s throttle + screen-hash dedupe skip any session whose *current* screen was already assessed — a genuinely-deferred session is never re-nagged; only a changed or newly-rendered screen is re-reasoned. This is Phase 1 of the fleet-orchestration arc; the confidence-gate widening, Idle/stuck triggers, autonomous recovery, and durable decision memory are Phases 2–5 in [`docs/tests/athena/fleet-orchestration-direction.md`](../../tests/athena/fleet-orchestration-direction.md).

**More-than-AwaitingInput triggers + recovery + durable memory (2026-07-04, Phases 3–5).** The event-driven `AwaitingInput` path is now one of three orchestration flavors — `orchestrate_on_awaiting` was generalized to `orchestrate_session(situation)`, sharing the gate/throttle/screen-dedupe/spawn machinery, with a per-situation directive:
> - **Stuck (Phase 3b).** `reassess_stuck_sessions` (proactive tick) detects a `dispatched_by_athena` session that failed and stalled (a `recent_failure` stamped + event-silent ≥ 4 min + not yet intervened) and wakes Athena on its real screen + failure to propose a **`fleet_intervene`** (a one-line unblock typed into the session). This replaces the old ask-only *"…looks stuck — want me to propose a `fleet_intervene`?"* nudge. `fleet_intervene` is now on the same **confidence × class × boldness** gate as `fleet_send_input`; the operative-memory **cap of one intervention per session** is the structural backstop, so an auto-fired intervene can't loop.
> - **Idle-needs-next (Phase 3a).** A session that finished its turn and idles at the prompt (`Stop → Idle`) had no trigger — orchestration only fired on `AwaitingInput`. `reassess_idle_needs_next` (tick) wakes Athena on a `dispatched_by_athena` Idle session (**never** the user's own ad-hoc CLIs) to judge **done-vs-needs-next** against the session's objective and send the next step via a gated `fleet_send_input`, or leave a finished session alone. The 60 s throttle + screen-hash dedupe bound the loop to *real progress* (a done session is woken at most once; it only re-triggers after work moves its screen).
> - **Per-session objective (Phase 5b).** Every orchestration directive now carries the session's objective — its self-reported `intent` (via the MCP `athena.report_intent` path) or `role`, paired with the owning operation's `user_intent` — pulled from operative memory, so she judges the next step against what the session is *for*, not just its screen.
> - **Recovery actions (Phase 4).** Two new confidence-gated dispatcher actions — **`fleet_wake`** (revive a hibernated session via `--resume`) and **`fleet_resume`** (adopt an orphaned CLI process after a restart) — wrap the existing `fleet_wake_session` / `fleet_resume_orphan` commands, wired through all five gate lists (same boldness bar, no screen re-check since the target is asleep/gone). Taught in the constitution's action menu (v42) for chat-driven revival ("wake the X session I slept"). Deliberately **no timer auto-reviver**: with idle-needs-next driving live sessions to completion, a hibernated dispatched session is one she already judged done, and orphan recovery needs a process scan — a timer would fight the resource-saving auto-hibernator.
> - **Durable decision ledger (Phase 5a).** A `fleet_decisions` table persists every fleet verdict — each defer (with a reason: `below_confidence_bar` / `screen_changed`) and each `auto_fired` / `auto_failed` — so you can see **why** she stopped or acted on a session. `orchestrate_session` also skips re-waking on a screen she already **auto-fired** on, keyed on the STABLE `claude_session_id` + `screen_hash` (the in-memory dedupe is per-run; the registry id is regenerated each launch), so the suppression survives a restart.

**Orb UX during orchestration.** The orb is the operator's window into a running fleet — it reuses the existing communication surfaces: the **thinking** avatar + **perimeter status dots** (`taskDots`, one per in-flight job) show Athena working, and a glanceable **"Managing the fleet…" caption** appears beside the orb whenever she's working with the grid open. Key decisions she escalates surface in the existing **`OrbDecisionBubble`** (numbered options, `0` = explain/recommend, answerable by click or the `;`-leader key). Both the orb and the decision bubble are **lifted to `z-[210]`/`z-[220]` while the grid overlay (`z-[200]`) is open**, so the cue + the decision render *over* the grid instead of being buried behind it. The bubble's render gate was also widened (`70f55b90f`): it previously only appeared when the orb was *minimized*, so in Grid mode with the chat open an escalated decision rendered nowhere — now `companionState !== 'minimized' && !fleetGridOpen` is the skip condition, i.e. **Grid mode forces the bubble to show regardless of chat state**. (Note: the bridge `/query` test endpoint reports `visible:false` for the `position:fixed` bubble — an `offsetParent`-based false negative — so verify its presence with a screenshot, not `/query`.)

> **Orb-only, never a chat essay.** A `fleet_orchestration` turn is a background assessment, not a conversation — its only user-facing output is on the orb. The companion turn for it runs with `suppress_chat` set (`session::send_turn`), which skips **every** chat side-effect (the `[proactive: fleet_orchestration]` opening marker, the `Started`/`Finished` stream events, all progress/interim/final episode persists, the turn-summary chip) so nothing lands in the transcript. Her surfacing still happens, just on the orb: an **ACT** (`fleet_send_input`) rides the approval pipeline (`APPROVALS_EVENT`, a global emit) as before; a **DEFER** prose note (no approval to carry it) is routed to a **proactive orb card** via `fleet_bridge::surface_fleet_orb_note` (the same `enqueue_external` → `companion://proactive` path the op-wrap-up reconciler uses), keyed on the unique `turn_id` so sequential decisions never dedupe-collide. A "progressing fine" no-op stays fully silent (empty reply → no card). And because the whole orchestration path is gated on autonomous mode, **turning autonomous mode off silences her entirely** — no wake, no card, no chat.

**Auto-apply under autonomous mode.** By default Athena's `fleet_send_input` stays approval-gated (a tile click). When **autonomous mode** is on, `fleet_send_input` is on the `auto_resolve_if_allowed` allowlist (`approvals.rs`), so the autonomous chain (`session.rs`) auto-approves + executes it — Ask Athena → she types into the session, no click. This is a deliberate higher-blast-radius exception (gated on the toggle); it matters that Fleet spawns run with `--dangerously-skip-permissions`, so an auto-applied write executes its tools unprompted. The on-tile suggestion strip shows on the grid tiles; the single pane relies on the Needs-You banner.

> **Confidence gate — the boldness dial (Phase 2, 2026-07-04).** Auto-apply is gated on a **confidence × decision_class × boldness** matrix, not a flat high-only rule. The orchestration directive requires every `fleet_send_input` to carry a `confidence` (`"high"`/`"medium"`/`"low"`), a `decision_class` (`"drive_forward"` = a reversible / obvious-next step vs `"choice"` = irreversible, a real fork, or the user's preference), the exact `text`, and a one-line `rationale`. `auto_resolve_if_allowed` reads the user's **boldness dial** (`COMPANION_FLEET_BOLDNESS`, default `bold`) and calls `fleet_send_input_auto_fires`:
>
> | dial | drive_forward | choice |
> |---|---|---|
> | `cautious` | high | high |
> | `balanced` | high\|medium | high |
> | **`bold`** (default) | **everything** | **everything** |
>
> **`bold` = full-auto (2026-07-24, user policy).** Under the default dial, EVERY fleet proposal auto-fires — low and missing confidence included. The user's explicit call after the first fleet runs: *"loosen the boundaries even if experimental; in autonomous mode she should react in any terminal requiring her attention"* — a wrong keystroke into a CLI is recoverable, a fleet parked on consults is not autonomous. Her stated confidence still lands in the `fleet_decisions` ledger + the debug log, so the policy can be re-tightened from data; `cautious`/`balanced` keep the old matrix for users who want the gate back. Under full-auto a consult only exists when she *declines to act* (a `NEEDS-YOU:` prose defer). Three companion changes shipped with it: **(a) session-id override** — a `fleet_orchestration` turn's actions get `session_id` forcibly rewritten to the session the turn was woken for (`force_fleet_session_ids`, both in-memory and in the stored payload), closing the long-tracked "model omits/hallucinates the id → executor fails closed → silent no-op" gap; **(b) stale-consult GC** — pending `fleet_*` approvals whose target session is gone, or older than 30 min, are auto-rejected each proactive tick (`gc_stale_fleet_approvals`) so consults from a previous app run can't linger in the chat; **(c)** see the re-check below.
>
> **Execution-time re-check → reassess, not consult (Phase 2.4; reworked 2026-07-24).** Even when the gate says fire, a live CLI screen can move between the moment Athena reasoned and the moment her input types. `fleet_bridge::screen_matches_last_decision` still catches that — but instead of parking the proposal as a pending consult (which stranded sessions "awaiting input" under autonomous mode), the stale proposal is **superseded**: auto-rejected with a `screen_changed` ledger entry, and `force_reassess` clears the throttle + screen-dedupe and wakes her again on the **fresh** screen immediately. The loop self-heals instead of asking a human to referee a race.
>
> **Notify-only on auto-decide.** A successful auto-fire emits `athena://fleet/auto-decided`; `useFleetCompanionBridge` surfaces it as a standard **toast** ("Athena → \<project\>: \<text\>", 10 s — the earlier bespoke fixed-position pill floated nowhere near the draggable orb and was removed 2026-07-24). Informational with **no undo** (the user's chosen safety-net level).

> **Athena-owned PTY guard — RELAXED (2026-06-25, user policy).** Auto-apply used to be scoped to sessions **Athena spawned herself** (`registry::is_athena_owned` via the `ATHENA_SESSION_NAME_SENTINEL`), so even a high-confidence answer to one of *your own* CLIs was left pending — which read as "Athena can't do anything autonomously." The user explicitly wants autonomous Athena to **act on their own fleet CLIs** when she's confident, so `auto_resolve_if_allowed` no longer checks ownership. The remaining gates are the real ones: this path **only runs under autonomous mode** (your standing opt-in) and only **`confidence: "high"`** auto-fires (medium/low/absent still surface as an orb consult). A hallucinated/stale `session_id` is harmless — `execute_fleet_send_input` fails closed when the target has no live PTY writer, so it types nothing. (Manual approval is unaffected: an explicit click can drive any live session, as before.)

### Driving multi-select menus

A Claude Code `AskUserQuestion` **multi-select** is a checkbox TUI, not a text prompt. Writing a typed answer like `"1,2,3,4"` into it only lands the string in the first row and toggles nothing — the menu is keyboard-driven (↑/↓ navigate, **space** toggles, **Enter** submits). The early naïve `fleet_send_input` did exactly that, so an approved 4-of-4 recommendation checked one box and never submitted. Detecting and correctly driving this is now done in `approvals.rs::execute_fleet_send_input`:

1. **Detect.** Before sending, it reads the session's **reconstructed screen** (`registry::render_screen_for` → the `vt100` ring snapshot — the same path orchestration uses to read the prompt) and calls `multiselect_keystrokes(lines, text)`. A screen counts as a multi-select menu only if it has checkbox rows (`[ ]`/`[✔]`), a `Submit` row, and a `navigate` hint — **and** the answer names option numbers. Anything else (single-select, free-text, non-menu) returns `None` and falls through to the plain typed `{text}\r`.
2. **Plan.** It parses the options in display order (`N. [state] Label`, skipping the `Type something` / `Submit` pseudo-rows) and builds a keystroke list: for each option top-down, **space** to toggle the requested-and-unchecked ones (already-checked wanted options are left alone — no double-toggle), **↓** between options, then **↓** past the last option (and once more past `Type something` when present) onto `Submit`, then **Enter** (→ "Ready to submit?") and **Enter** again (the confirm screen defaults its cursor to "Submit answers").
3. **Fire timed, not in a burst.** The keys are written one at a time on a spawned task with a **120 ms gap** — the TUI silently drops a rapid burst. The call returns immediately with `"Driving multi-select on `<id>` (N keystrokes)."` while the task plays out.

**Gotcha — the list WRAPS, so never ↑-clamp.** An earlier version prepended an ↑ to "clamp" the cursor to the top before toggling. Claude Code's option list wraps on ↑ at the top, so the clamp jumped the cursor to the *bottom* (Cancel) and the whole plan misfired as "User declined". The menu is always freshly rendered when orchestration fires (cursor already on option 1), so the driver starts from option 1 with no leading ↑ (`3692d130d`). Three unit tests pin the sequences (select-all-four, skip-already-checked, no-op for non-menus). **Verified live 2026-06-25** both manually and end-to-end through the autonomous path (Athena proposed `1,2,3,4` → approve → driver → Claude acknowledged all four selections and proceeded, *not* "declined").

> **Fleet action results don't reach the companion chat.** `fleet_send_input` / `_broadcast` / `_kill` / `_intervene` are operational keystrokes into a CLI the operator is already watching on the grid — their "approved & executed / failed" outcomes are noise in the Athena chat, not conversational turns. `log_action_episode` (`approvals.rs`) now early-returns for any `action.starts_with("fleet_")` (traces for debugging instead of persisting a `System` episode), so approving/rejecting fleet actions no longer floods the transcript with `[Athena action approved & executed] fleet_send_input …` lines (`fc79bba34`, from a user report of an overflowing chat). Non-fleet actions still persist as before.

## Hook installer details

Each entry is tagged `_fleet: true` so uninstall is surgical:

```json
{
  "hooks": {
    "Stop": [
      {
        "matcher": "*",
        "_fleet": true,
        "hooks": [
          {
            "type": "command",
            "command": "curl -s -m 2 --connect-timeout 1 -X POST --data-binary @- -H \"Content-Type: application/json\" http://127.0.0.1:17400/fleet/hooks/stop",
            "_fleet": true
          }
        ]
      }
    ]
  }
}
```

`curl` is the universal HTTP client (Windows 10+ ships it). Payload is streamed via stdin (`--data-binary @-`) so we never have to JSON-escape on the command line.

## Sharp edges to know about

- **`--mcp-config` is VARIADIC — emit it LAST, after the positional prompt.** Claude Code's `--mcp-config <configs...>` greedily consumes every following non-flag token. If it sits before the spawn's positional prompt it eats the prompt as a bogus config file and claude exits 1 (`MCP config file not found: <cwd>/<prompt>`) — the session dies the instant it spawns, which also silently kills Athena's orchestration (a dead session never reaches `AwaitingInput`). `pty.rs::spawn_session` therefore appends `--mcp-config` *after* the caller's args in both the Windows direct-exe and the fallback branch. If you add any new trailing argv, keep `--mcp-config` last. (Found via a live 4-CLI pass: every spawn-with-task exited on spawn.)
- **`claude --resume <id>` needs a prompt or it exits 1.** Claude Code ≥ 2.1 refuses a bare `claude --resume <id>` with no continuation: it hunts for a deferred-tool marker to auto-continue and, finding none or a stale one, exits **code 1** with *"No deferred tool marker found in the resumed session … Provide a prompt to continue the conversation."* This silently killed every **Wake** (`fleet_wake_session`) and orphan **Resume** (`fleet_resume_orphan`) — the session came back and died within seconds ("/resume failed quickly after recover"). Fix (verified live end-to-end 2026-07-02, together with the nesting-strip below: codeword → hibernate → wake → recalled): both resume paths pass `pty::RESUME_CONTINUATION_PROMPT` as the positional after `--resume <id>`, and `spawn_session` **skips LLM naming when `--resume` is present** so the nudge can't become the session title. **Detection half (high value):** `registry::mark_exited` folds the session's final rendered-screen line into the exit reason for any non-zero exit (`last_meaningful_line`), so resume failures self-explain — *"Exited with code 1 — claude said: No conversation found with session ID …"* — instead of a bare, mysterious code 1.
- **Claude Code NESTING env markers disable session persistence — STRIP them (the root cause of "resume fails after recover").** When the Personas app is launched from a Claude Code session (the normal dev workflow), every fleet PTY child inherits the parent's session markers (`CLAUDECODE`, `CLAUDE_CODE_CHILD_SESSION`, `CLAUDE_CODE_SESSION_ID`, …). **claude ≥ 2.1.19x sees them and silently runs as a nested/child session: it never registers in `~/.claude/sessions/` and never persists a `<session-id>.jsonl` transcript** — the session works live but leaves *nothing to resume*, so Wake/orphan-Resume died with *"No conversation found with session ID …"* and Insights had no transcript to read. Root-caused via A/B console spawns (identical argv; markers present → no storage; stripped → registers `kind=interactive` + persists) after per-axis in-app toggles exonerated the `--session-id` pin, `--mcp-config`, `CLAUDE_CODE_DISABLE_NONESSENTIAL_TRAFFIC`, and `--dangerously-skip-permissions`. Fix: `pty.rs::spawn_session` strips `CLAUDE_NESTING_ENV` (7 markers) next to the subscription-auth env strip — fleet sessions are first-class top-level sessions, never children. **Verified end-to-end after the strip:** transcript persists under the pinned id, the session registers a PID file, and hibernate → wake → *"What was the codeword?"* → correct recall. (If a new claude version adds more `CLAUDE_CODE_*` markers, extend the list — symptoms will be the same silent no-transcript behavior.)
- **In-session `/resume` is a known limitation (conflicts with the `--session-id` pin).** Fleet pins `--session-id <uuid>` on every fresh spawn for deterministic binding. When that session runs the **in-session `/resume` slash command** and picks a conversation, claude re-execs *in the same PTY* with `--session-id <uuid> --resume <picked>` and dies instantly: *"Error: --session-id can only be used with --resume if --fork-session is also specified"* — the CLI "drops immediately without loading the content." Adding `--fork-session` to the fresh spawn makes `/resume` **load and stay interactive** (verified live end-to-end: picker → select → conversation loads, tile idle→running→idle, answers normally) — so the crash *is* solvable. It was **not** kept: it forks the fresh session's claude id, diverging it from the pinned `--session-id` (a binding/hook risk not fully characterized). Prefer **Wake/Recover** to resume a conversation — verified working end-to-end since the nesting-strip fix (see the previous bullet); the transcript-persistence bug that once made *both* paths fail with "No conversation found" is resolved. **Kept from this arc (both correct):** `apply_hook` **ignores a SessionEnd hook while the PTY `child_pid` is still alive** (an in-place re-exec such as `/resume`/`/clear` is not a death — the reaper stays authoritative), and `registry::mark_exited` surfaces claude's real final line in the exit reason so these failures self-explain.
- **WebView2 can freeze `Object.prototype` → xterm dies at import (patched dependency).** Since ~2026-06-28 this machine's WebView2 creates every realm with `Object.isFrozen(Object.prototype) === true` (verified: a virgin `about:blank` iframe is born frozen; independent of runtime version 149.0.4022.80/.98, user-data profile, and app code — an Edge machine-level experiment). In strict-mode ESM, assigning a member that shadows a frozen inherited data property throws — `@xterm/xterm` and `@xterm/addon-webgl` both vendor VS Code's `KeyCodeUtils` namespace which does `o.toString = fn` at module scope, so the Fleet page died at lazy-import with *"Cannot assign to read only property 'toString'"*. The old `public/webview2-compat.js` accessor shim can no longer apply (the frozen prototype is also non-configurable). **Fix — two layers:** (1) the `optimizeDeps.rolldownOptions.plugins` "webview2-compat" transform in `vite.config.ts` rewrites `X.toString = fn` → `Object.defineProperty(...)` during dep pre-bundling (covers xterm + decimal.js/d3/es-toolkit in **dev**); (2) `scripts/patches/fix-frozen-intrinsics.mjs` (wired to `postinstall`) mutates node_modules xterm so the two assignments ship patched in **production** too — the optimizeDeps transform is dev-only (there is no production webview2 build plugin), so without the postinstall a `vite build` would ship an unpatched xterm. If a future xterm upgrade reintroduces the pattern (or another dep starts throwing this), the tell is the exact "read only property" TypeError at a dep chunk's module scope.
>
>   **2026-07-03 — the recurrence was a cache trap; and don't "fix" the dev transform:**
>   - *Patching node_modules under a running dev server can't reach the browser.* Vite serves optimized deps `Cache-Control: max-age=31536000,immutable` under `?v=<hash>` derived from lockfile+config, **not** file content — so a content patch keeps the same URL and WebView2's disk HTTP cache serves the pre-patch bundle forever (symptom: `curl http://localhost:1420/node_modules/.vite/deps/@xterm_xterm.js?v=<h>` shows `defineProperty` at the failing line, yet the app still throws there). This is why the committed postinstall fix "un-applied" itself the next morning. To recover: kill the app, then delete `node_modules/.vite` **and** `%LOCALAPPDATA%\com.personas.desktop\EBWebView\Default\{Cache,Code Cache,GPUCache}`, then relaunch. Clearing `.vite` alone is not enough (re-optimize is deterministic → identical hash → browser still cache-hits the stale immutable copy).
>   - *The `optimizeDeps` "webview2-compat" transform is intentionally inert on minified deps — leave it that way.* Its `needsTransform` gate requires a space before `=` and its dedup guard is newline-bounded, which together stop it ever touching single-line minified bundles. That looks like a bug (it's why the plugin never patched xterm — the postinstall does) but it's load-bearing: the transform's value-extent parser assumes `;`/function-body termination, so a minified sequence-expression assignment like `p.constructor=A,p.render=q$1)` (Preact inside `@sentry-internal/feedback`) makes it swallow past the comma and emit invalid JS → the optimizer dies with `[PARSE_ERROR]` and the app won't boot. Making the gate whitespace-tolerant reproduces exactly this crash; it was tried and reverted, with comments + a contract-lock test in `src/test/webview2-compat.test.ts`. Minified deps that need the fix (xterm) go through the targeted `fix-frozen-intrinsics.mjs` postinstall instead.
>
> `--dangerously-skip-permissions` skips *tool* permissions but NOT Claude Code's first-run *"Do you trust this folder?"* gate. A never-before-opened cwd parks the session at that prompt (rendered as `idle`, not a distinct state), and the trust restart can also break the pre-bound `--session-id` (claude starts a fresh session id post-trust). Real projects the user has opened are already trusted, so this mostly bites brand-new dirs.
- **One PTY session per cwd.** The spawn enforces this to keep hook routing unambiguous (we match hooks by claude_session_id once SessionStart fires, but the bootstrap window uses cwd).
- **Soft kill only (for now).** `fleet_kill_session` drops the writer + master, which makes CC see stdin EOF. Phase 6's cancellation token will land a hard kill path that does `Child::kill()`.
- **Port mismatch — now self-healing (critical).** local_http binds the first free port in 17400-17415, so a restart can land a *different* port than the installed Claude Code hooks target. When that happens every hook POSTs into the void — no SessionStart/Stop/Notification reaches Fleet, so spawned sessions silently lose all hook-driven state and fall back to transcript-growth alone (which reads "Stale" where the session is really idle/awaiting). This was a real, hard-to-spot break. **Startup now self-heals it:** after `local_http::start()` returns the live port, `setup()` calls `check_hooks(port)` and, if hooks are installed but `!port_matches`, immediately `install_hooks(port)` to re-point them. The Settings banner + manual re-install remain as a fallback. Re-install is also now **idempotent across ports**: `is_fleet_tagged` reclaims any entry whose command targets a `/fleet/hooks/` URL (any port, even older untagged ones), so a re-point cleans stale-port stragglers instead of duplicating.
- **No translation of hook payloads.** We only look at the URL path for the event type and `session_id` / `cwd` for routing — Claude Code's payload shape varies across versions, so we don't depend on field shapes that may change.
