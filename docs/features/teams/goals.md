# Goals

Goals give **high-level direction** to both development and teams, and are the cross-cutting surface Athena tracks. A goal is a project-scoped objective with progress, status, an optional target date, a dependency graph, and a composed checklist; team and dev activity feed its progress, and Athena can read, propose updates to, and proactively flag goals.

> Design rationale + decisions: [`docs/concepts/goals-direction-hub.md`](../../concepts/goals-direction-hub.md).

## Where it lives

- **Inside the top-level `Teams` section** — Teams was promoted to a 1st-level sidebar section (2026-06-05) and Goals consolidated under it as an L2 item (next to the team Workspace), in addition to the legacy Dev Tools › Goals tab (a contextual shortcut). Both render the same surface (`src/features/teams/sub_goals/`).
- **View switcher** — two surfaces: **Board · Timeline**, switched via the Goals submenu nested under the Teams L2 sidebar (`TeamsSidebarNav` `GOAL_VIEWS`). The submenu is **always expanded** (not gated on the Goals tab being active), and clicking a view navigates into the Goals hub from anywhere in the Teams section. (The pan/zoom React Flow **Map** view was retired 2026-06-15 — the dependency graph is still authored in the detail drawer, just no longer visualised as a canvas. **Acceptance is not a sidebar view** — it's a TitleBar-summoned full-screen overlay; see *Accept goals* below.)
- **Header** — the title/subtitle sit on the top row; all controls (scope switch, project picker, **+ New goal**) live on a thin **toolbar bar** below them (`ContentHeader`'s `toolbar` slot), so a long project path and the control cluster never collide.
- **Scope switch (Board + Timeline)** — a header `SegmentedTabs` toggles **All projects** (cross-project, the default) vs **This project**. In cross-project scope the page loads every project's goals into the same store array (`fetchAllGoals` → `dev_tools_list_all_goals`), so drag-to-move and progress edits keep working unchanged, and each card/row carries a `GoalProjectBadge` naming its origin project. The `LifecycleProjectPicker` stays in the header in both scopes — it remains the new-goal / Obsidian-sync target, and selects the single project when narrowed. (This folds the cross-project overview the retired Portfolio surface once provided directly into Board + Timeline.)
- **Persistence** — the active project is persisted (`systemStore` partialize), so goals re-fetch after a hard refresh.

## Canonical status model

All status handling funnels through `goalStatus.ts` — a `GoalStatus` type (`open | in-progress | awaiting_acceptance | blocked | done`), a tolerant `normalizeGoalStatus` (maps the hyphen/underscore/team-step/alias variants onto one set), predicates (`isComplete` / `isOngoing` / …), and `GOAL_STATUS_META` (icon + lane + chip + tint + map colours). `GoalStatusBadge` renders status everywhere from it. The Rust side mirrors this in `normalize_goal_status` so cross-project rollups bucket exactly like the UI. (v1 compared `in-progress` vs `in_progress` inconsistently and silently mis-laned in-progress goals — v2 makes that class of bug impossible.)

## Surfaces

### Board (active project)
- **Board** — a `your turn → agent's turn → done` kanban; lane membership derives from `GOAL_STATUS_META.lane`. The **Done lane is hidden by default** — "Your turn" / "Agent's turn" split the full content width — and toggles via a CheckCircle2 icon button next to *Sync to Obsidian* (board view only; preference persists in localStorage, and hidden done goals leave the item set so the fallback column can't re-bucket them). Each card surfaces its **checklist inline**: the first few to-dos render as checkboxes you toggle in place — ordered **open-first, newest-first** so the actionable items are on top — and the rest fold into a "+N more" link to the detail drawer. **When a goal has to-dos, they drive its completeness** — `done / total` is written back to the goal's `progress`; a goal with no to-dos keeps the manual ± progress nudge. Items are fetched in one batch query per project the visible goals span (`dev_tools_list_goal_items_for_project`) — one query in single-project scope, a small fan-out across projects in cross-project scope. Cards are **title-first**: the full title wraps (never truncated) and the description is deliberately absent — it lives in the detail drawer (open-details affordance / "+N more"). Lane backgrounds are transparent; colour lives on the cards and the lane border/icon. Status is carried entirely by the **status-tinted left edge** (no redundant status badge on the card), and cards also carry a date that turns red when an ongoing goal is overdue, plus (in cross-project scope) a project-origin chip.

### Timeline (active project or cross-project)
`GoalsTimeline` — ongoing goals on a vertical target-date rail, bucketed **Overdue → This week → This month → Later → No date**, each row showing the relative due date, status, and progress (plus a project-origin chip in cross-project scope). Opens the goal on click. Honors the same Board/Timeline scope switch.

### Accept goals (TitleBar overlay)
The **human-acceptance queue** for agent-completed goals — a **full-screen overlay** summoned from the TitleBar acceptance badge (the Persona-Monitor pattern: it covers the app over a shared `FullScreenOverlay` shell, dismissable by the corner ✕ or Escape, and does **not** change your underlying navigation). It is **not** a sidebar view — acceptance is also fully reachable from the **Board** (the *Your turn* lane) and the detail drawer, so the dedicated sidebar item was dropped.

Agent/team completion no longer flips a goal straight to `done`: the close-loop lands it in **`awaiting_acceptance`** (a canonical status laned to *Your turn*, teal), so a person decides whether the work is actually acceptable. `GoalAcceptanceOverlay` wraps `GoalAcceptanceView` (`dev_tools_list_pending_acceptance`, enriched server-side with project + owning team + served KPI) and groups pending goals **by project → KPI thin-divider sub-headers → goal rows**; each row shows the team that shipped it, a readable summary, and **Accept** / **Send back** (reject-with-comment) controls, plus a per-project *Accept all*. **Accept** (`dev_tools_resolve_goal_acceptance` decision=`accept`) sets `done` (off-board) + a `goal_accepted` signal; **Send back** (decision=`reject`) returns the goal to `in-progress` (Agent's turn), clears the completion stamp, and writes a `goal_rejected` signal carrying the comment (the feedback the team reworks against). The **TitleBar badge** (`dev_tools_count_pending_acceptance`, polled) shows the pending count and toggles the overlay. (The TitleBar dock now uses one pattern: full-screen overlays for Schedules + Goal acceptance, partial overlays for Notifications + Quick Answer.)

Clicking a goal in Board / Timeline (or in the acceptance overlay) opens the **detail drawer** — which leads with the same Accept / Send-back panel while the goal is `awaiting_acceptance` (replacing the hand-off panel until it's resolved).

### Authoring
`+ New goal` (header and empty state) opens `GoalEditorModal` — create/edit/delete with title, description, status, and target date. (Goals can also still be imported from GitHub issues.)

### Detail drawer
`GoalDetailDrawer` is the focused view for one goal. It is laid out for non-technical users: the human, actionable content leads (status, the progress nudge, the checklist, the hand-off, and what the team is doing right now), and the power-user surfaces fold away under a collapsible **More details** section (a shared `SectionCard`, collapsed by default, state persisted).

- **KPI cross-reference** — when a goal carries a `kpi_id` (it was derived from, or serves, a [KPI](./kpis.md)), the drawer leads with a `GoalKpiLink` panel: the steering KPI's name, current → target, and the shared plain-language pace sentence (`paceSentence` from `sub_kpis/kpiMeta.ts`), so a non-technical user sees **why** an autonomously-derived goal exists and what moving the needle accomplishes. It states the honesty rule directly — *a goal finishing is not success; the next measurement decides* — which is exactly why the panel shows the KPI, not just the goal. Read-only projection; it stays silent if the soft `kpi_id` link dangles (KPI archived/removed). Off-track colour comes from the same `kpiTrack` rule the derivation loop obeys. **When a goal has NO `kpi_id`**, the drawer instead shows an honest *"Activity, not a measured outcome"* note (UAT P9 F-GOALS-PLACEBO): goal progress is checklist completion and can read done even if the work never moved the real number, so the note points the user to set up a [KPI](./kpis.md) as the grounded outcome layer — the manual/checklist progress is never dressed up as a measured result.

- **Description** — rendered as **markdown** (`MarkdownRenderer`, GFM + code-span/heading/list/table support), so autonomously-generated goal descriptions (which carry code-span file paths, structure, and a backlog-provenance footer) read cleanly instead of as a flat blob. This is the only place the description renders — Board cards are title-only.
- **Hybrid progress nudge** — `dev_tools_resolve_goal_progress` composes the goal's checklist items, sub-goals, and linked team-assignment steps into a *suggested* progress %. When it differs from the stored value, the drawer offers **Accept / edit** — progress is never written silently; a manual override always wins.
- **Tasks table** (`GoalTaskTable`) — ad-hoc to-dos (`dev_goal_items`) and team-assignment steps (`team_assignment_steps`) used to render as two separate lists that visually overlapped, because the backend mirrors a decomposed goal's steps into `dev_goal_items` by **exact title** (goal_advance.rs). They are now **one table**, de-duped by title (the richer team-step row wins — see `partitionGoalTasks`). Each row carries: a **state** cell (a toggleable checkbox for ad-hoc to-dos; a read-only status icon for steps — done/running/awaiting/failed), the **task title** (expandable to the step's markdown `outputSummary`), the **Owner** (the responsible persona, resolved from `assignedPersonaId` via `useAgentStore`; "You" for ad-hoc to-dos), and inline **skip / abort** intervention on `awaiting_review` steps. Ad-hoc to-dos remain addable/toggleable/deletable inline.
- **Hand to your AI team / Stop the team** — the plain-language hand-off control (`GoalHandoffPanel`): one button plus an inline confirm explaining what starting the team will do. Shown when the project has an owning team and the goal isn't complete; once a team is on it the panel flips to a live "already working" state **with a Stop control**. Stop aborts the goal's active assignment(s) (`abort_team_assignment`) — a **gate-close**: the orchestrator stops launching new steps and an in-flight step finishes on its own, the abort `reason` is recorded, and a `team_aborted` goal signal is logged. The **goal is kept** so it can be handed back later. This matters because **deleting a goal does NOT stop a running team** — `team_assignments.goal_id` is a soft link with no FK, so a delete leaves the team executing orphaned; Stop is the safe halt path.
- *Under **More details*** — **Linked teams** (assignments advancing this goal, title + status, with **unlink**); **Dependencies & follow-ups** (author cross-goal links — the **Product Strategist also writes these autonomously**, its backlog-triage pass relating open goals with `depends`/`follows` edges where real sequences exist, so promoted goals stop being islands: **Depends on** = a `blocks` edge, must finish first, backend cycle-checks; **Follows** = a `follows` edge, sequence; pick from the project's goals, add/remove via `add_goal_dependency` / `remove_goal_dependency`); and the **Activity feed** (recent `dev_goal_signals`, rendered readably — new `team_step` signals store the agent's own outcome summary extracted from `outcome_assessment.summary` / `task_completed` at write time, and the drawer's `readableSignal()` applies the same extraction to legacy raw-protocol rows — including the `team_*` and `athena_update` signals described below).

## How progress flows in

- **Dev tasks** — a task linked to a goal (`dev_tasks.goal_id`) writes a `dev_goal_signal` on completion/failure (`task_executor`).
- **Teams** — an assignment linked to a goal (`team_assignments.goal_id`, set in the assignment composer's *Advance goal* picker) writes `team_step` / `team_done` / `team_failed` / `team_awaiting_review` signals as the orchestrator runs it. Step status also feeds the hybrid progress resolver.
- **Athena** — proposed updates write an `athena_update` signal.

Progress is **hybrid**: signals + step/checklist completion compute a suggestion; the user (or Athena, gated) accepts it.

## How goals reach team executions

Goals are not just a planning view — since 2026-05-29 they **steer runtime team behavior**. Every member of a team executes with a compact `## Team Alignment` block in its prompt (`engine/runner/team_context.rs`) that lists the team's **active goals** alongside the teammate roster, and instructs the persona to judge — from its own capabilities — whether and how its work advances them (align where it relates, don't force-fit). See [team-orchestration.md › Shared state](./team-orchestration.md#shared-state-reaching-a-running-persona).

The block resolves a team's goals by walking **team → project → goals** via the canonical durable link **`dev_projects.team_id`** ("this team owns this project"; goals belong to the project). This makes `dev_projects.team_id` the team-mission link and `team_assignments.goal_id` the granular per-task "advancing" link — both pointing at the **same `dev_goals` spine**, so a goal authored anywhere (Goals UI, an assignment's *Advance goal* picker) flows into executions. Resolution order: the persona's pinned project (`design_context.dev_project_id`) → `dev_projects.team_id` → goals the team is directly advancing.

## Advancing a goal (teams work it)

Linking a goal to a team makes it *visible* to executions; **advancing** is the team actually working it. An advance turns a goal into a running, goal-linked `team_assignment`:

- **Initiator** — `engine/goal_advance.rs::advance_goal` (command `advance_team_goal`) builds an assignment **with `goal_id` set** and runs it on the orchestrator, behind a one-active-assignment-per-goal guard. **Hybrid step source**: if the goal has open to-dos (`dev_goal_items`), one step per to-do (title verbatim); otherwise the goal is LLM-decomposed.
- **Triggers** — the **Hand to your AI team** control in the goal detail drawer (`GoalHandoffPanel` — a plain-language relabel of the former "Advance with team", gated behind an inline confirm; shown when the goal's project has an owning team and the goal isn't complete); Athena (team path); and the autonomous tick below.
- **Progress closes automatically — and moves live per step.** When a goal-linked *step* completes, the orchestrator immediately checks off its matching to-do (step title ↔ to-do title) and recomputes progress via `apply_resolved_goal_progress`; when the whole assignment reaches `done` the same close-loop finalizes it. A decomposed goal also gets its steps **mirrored into `dev_goal_items`** at advance time, so the Board card shows a live checklist that ticks as the team works (and a future re-advance takes the open-items path verbatim). `dev_goal_signals` are observational. Status flips `open → in-progress` the moment the first step runs and, at a composed 100%, **`→ awaiting_acceptance`** (the human-acceptance gate — `apply_resolved_goal_progress` no longer writes `done` directly; the user accepts → `done` or sends back → `in-progress` from the [Accept queue](#accept-acceptance-queue) / detail drawer); progress never regresses a hand-set value, and a QA bounce (rework loop) keeps 100 unreachable until the clean pass.
- **Autonomous advancement** — `GoalAdvanceSubscription`, gated by the **default-OFF** `autonomous_goal_advancement` setting **or a project's `full` [autopilot](./kpis.md#autopilot-per-project) mode** (the per-project switch overrides the global flag in both directions), ticks every 5 min and keeps each goal-linked team's active goal moving unattended. Guardrails: one active assignment per goal, a 30-min per-goal cooldown after any assignment (no failure-retry stampede), eligible-persona check, and a per-tick cap. Nothing spends tokens autonomously until you opt in.
- **Autonomous replenishment (the loop's last link)** — `IdeaReplenishSubscription`, gated by the **default-OFF** `autonomous_idea_scan` setting: when a goal-managed project is FULLY idle (no open goals **and** no pending backlog ideas), it runs an idea scan (architecture-analyst) to refill the backlog — one project per tick, 20h per-project cooldown via `dev_scans`, quota-gated. Together with `autonomous_backlog_to_goal` this closes the self-sustaining cycle: scan → ideas → goals → assignments → PRs → QA-merge → repeat. Replenish scans rotate ROSTER-ALIGNED lenses (architecture, business-strategist, ux-reviewer, security, test, optimizer, accessibility, onboarding, error-handling — LRU per project) and carry the team's shared-ledger digest, so the backlog gets real-life variety grounded in what already shipped.
- **Strategist backlog triage** — `BacklogTriageSubscription` (default-OFF `autonomous_backlog_triage`): when a project has ≥ 6 pending ideas with ≥ 3 unranked, the team's **Product Strategist** runs a triage pass — ranks the next-up queue (≤ 5, theme-balanced across business/UX/technical) into `dev_ideas.priority` and rejects low-value items with reasons that become shared constraint memories; `backlog_to_goal` promotes ranked ideas first. 24h per-project cooldown.
- **Athena channel reactions** — `AthenaChannelReactionSubscription` (default-OFF `autonomous_athena_reactions`): while the loop above runs unattended, Athena watches each goal-managed team's delivery stream and makes a genuine **react/decline decision** at reaction-worthy moments — an assignment parked in `awaiting_review` (the QA fix-loop capped out, so only a human can unblock → she escalates), a repeating QA-bounce pattern (an optional in-context steer to the implementer), a shipped goal (acknowledged only when it's a real milestone). She posts into the team channel (`author_kind='athena'`) with an `⚠️` escalation prefix, a `› rationale` audit footer, and the assignment link — so her orchestration is **visible in the Collab channel** throughout development. Restraint: ≤ 4 teams/tick, deduped against her last post per team, and she usually chooses silence. Decision logic in `companion/athena_reaction.rs`; the certification suite grades her coverage/soundness/auditability/restraint as rubric §8. With **review resolution** also on (`autonomous_athena_review_resolution`, default OFF) she goes one step further on parked awaiting-review cap-outs — approve (one extra QA round with her assessment injected to QA), transform access/credential blockers into Incidents, or escalate — so a parked review can no longer starve the goal loop; rubric §9 (`scripts/test/loop-certify.mjs`) certifies the loop's day-scale liveness.

## Athena integration

- **Reads** — active project goals (id, progress, status, latest signal) are injected into Athena's system prompt, so she's aware of project direction and can reference goals by id.
- **Proposes (gated)** — the `update_dev_goal` op lets Athena propose a status/progress change; it is **approval-gated** (never auto-approved) and writes an `athena_update` signal on approval.
- **Reacts (proactive)** — the `dev_goal_target` / `dev_goal_stalled` proactive triggers surface budget-gated nudges when a goal is target-approaching/overdue or has been stalled (in-progress/blocked, untouched ≥ 7 days). On engage, Athena reasons with the goal context and can propose the gated update.
- **Steers the layer above** — Athena also manages the [KPIs](./kpis.md) that derive goals: she sees each project's active KPIs in her prompt and can propose `calibrate_kpi` / `evaluate_kpi` / `scan_kpis` (all approval-gated). See [companion → Project KPIs](../companion/README.md). Recalibrating a KPI's critical line is how she (or the user) changes *when* a goal gets derived in the first place.

## Data model (main DB)

| Table | Purpose |
|---|---|
| `dev_goals` | the goal: project-scoped, `parent_goal_id` hierarchy, `progress`, `status`, `target_date` |
| `dev_goal_dependencies` | cross-goal blocking edges (Depends-on / Follows, authored in the drawer) |
| `dev_goal_items` | lightweight ad-hoc checklist items |
| `dev_goal_signals` | progress/activity log (dev, team, and athena signals) |
| `team_assignments.goal_id` | the soft link from a team assignment to the goal it advances (granular "advancing" signal) |
| `dev_projects.team_id` | the durable team↔project link — a team owns a project, so the project's goals are the team's mission (drives execution-time goal awareness) |

Key commands — per-goal/project: `dev_tools_{list,create,update,delete,reorder}_goal(s)`, `dev_tools_{list,create,update,delete,reorder}_goal_item(s)`, `dev_tools_list_child_goals`, `dev_tools_resolve_goal_progress`, `set_team_assignment_goal`, `list_team_assignments_for_goal`.

Batch queries (v2): `dev_tools_list_goal_dependencies_for_project` (single-query dependency edges), `dev_tools_list_goal_items_for_project` (single-query Board card to-dos). The cross-project rollup commands (`dev_tools_list_all_goals`, `dev_tools_portfolio_summary`, `dev_tools_attention_queue`) remain in the Rust backend but lost their frontend callers when the Portfolio surface was retired (2026-06-07).
