# Perf-walk Analysis — 2026-05-21 (post data-population)

> Third perf-walk run. App was populated with a realistic data volume
> (~142 personas) since the 2026-05-17 runs. 36 stops.
> Raw JSON: `2026-05-21T11-05-15-185Z.json`. Diff report: `2026-05-21.md`
> (compared against the 2026-05-17 post-fix run `2026-05-17T20-57-02-193Z.json`).

## Headline

The data population exposed **one dominant, scales-with-data bottleneck** and
one secondary render-churn hotspot:

1. **`L1/settings` — `resolve_effective_config` × 142 = 10.0 sec of IPC time**
   (+8278ms / +481% vs the 2026-05-17 run). This is now *by far* the single
   biggest cost in the app. The 2026-05-17 analysis already flagged this loop;
   the data population turned a 1.8 sec annoyance into a 10 sec page-freeze.
2. **`L1/personas` — 80 render commits** (+68 / +567%), DOM 2621 (+2000 / +322%).
   The personas list re-renders ~80 times on cold land and builds 2000 extra
   DOM nodes — an unvirtualized list of 142 persona rows. Pure client-side
   churn (only 1 IPC).

## The #1 bottleneck — `resolve_effective_config` fan-out

`src/features/settings/sub_config/components/ConfigResolutionPanel.tsx:116-118`:

```ts
const results = await Promise.allSettled(
  stale.map((p) => resolveEffectiveConfig(p.id))
);
```

`ConfigResolutionPanel` mounts on the Settings landing. It calls
`listPersonas()` (1 IPC) then fires **one `resolve_effective_config` IPC per
persona**. With ~142 personas that is 142 IPC roundtrips totalling **9965ms**.

Worse, the Rust side re-does global work per persona. Each
`resolve_effective_config` call:
- `repo::get_by_id` — 1 DB query
- `group_repo::get_by_id` — 1 DB query
- `resolve_global_model_profile` — 1 `app_settings` query **(global — identical for every persona)**
- `resolve_global_auth_token` — up to 1 more `app_settings` query **(also global)**

So 142 personas = 142 IPC roundtrips + ~560 DB queries, of which ~280 are
the *same global settings re-fetched 142 times*.

The 5-min in-memory `configCache` (added 2026-05-17, commit `f8df06833`)
elides this on *revisits* — that is why `settings/account` and
`revisit/settings-2nd` show only 1 IPC. But the **cold mount still pays the
full 10 seconds**, every fresh app launch.

### Fix (this run) — bulk resolution, client + server

**Server:** new `config_merge::GlobalConfigContext` (loads the global tier
once — 3 DB reads), `resolve_effective_config_with_globals(persona, ws, ctx)`
(merge against a shared context, zero DB reads), and a new bulk Tauri command
`resolve_effective_config_bulk(persona_ids)` that fetches all personas + all
groups + the global context **once** and resolves in-memory.

**Client:** `resolveEffectiveConfigBulk(ids)` API wrapper; `ConfigResolutionPanel`
replaces the `stale.map(resolveEffectiveConfig)` fan-out with a single
`resolveEffectiveConfigBulk(staleIds)` call.

Net: **142 IPC roundtrips → 1**, **~560 DB queries → ~4** (`get_all` personas,
`get_all` groups, 3 settings reads). Expected `L1/settings` cold-mount IPC
time: ~10000ms → a few hundred ms.

## Verified result — fix landed (run `2026-05-21T11-35-40-510Z.json`)

The bulk-resolve fix was merged to master and the app rebuilt; the perf walk
was re-run. Diff report: `2026-05-21-after-fix.md`.

| `L1/settings` | Before fix | After fix | Δ |
|---|---:|---:|---:|
| IPC calls | 143 | **3** | **-98%** |
| IPC time | 10000ms | **102ms** | **-9899ms / -99%** |
| Top command | `resolve_effective_config`×142 | `resolve_effective_config_bulk`×2 | — |
| Render commits | 11 | 8 | -27% |

The 10-second IPC fan-out on the Settings landing is **eliminated** — the
single biggest cost in the app is gone. `resolve_effective_config_bulk`
appears twice (once for the initial mount, once because the walk re-lands on
Settings); each is a single roundtrip.

### Secondary finding resolved by concurrent work

`L1/personas` render commits dropped **80 → 17** (-79%) and DOM **2621 → 1047**
(-60%) between the two runs — the in-flight `/prototype` PersonaOverview
rework (part of the large uncommitted working tree at rebuild time) already
addresses the personas-list churn. No separate fix needed.

### New regression surfaced — NOT from this fix

The rebuilt app pulled in the full in-flight working tree (~813 uncommitted
files from concurrent sessions). Three settings tabs regressed sharply:

| Stop | IPC before | IPC after | Storm |
|---|---:|---:|---|
| `settings/config` | 11 | 62 | `set_app_setting`×34, `get_app_settings_bulk`×24 |
| `settings/byom` | 11 | 60 | `set_app_setting`×32, `get_app_settings_bulk`×19 |
| `settings/portability` | 9 | 55 | `set_app_setting`×32, `get_app_settings_bulk`×18 |

~32 **write** IPCs (`set_app_setting`) fire on each settings tab *mount*. This
is unrelated to the `resolve_effective_config` fix (which touches no settings
write path) — it is uncommitted work in another session's tree (note the
`get_ambient_context_snapshot` calls; cf. the 2026-05-17 audit's
`AmbientContextPanel` polling finding). Flagged for whoever owns that branch:
a tab mount should not write 32 settings rows. Re-run the walk once that work
commits to confirm.
