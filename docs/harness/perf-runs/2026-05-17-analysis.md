## Findings ranked by cost

### P0 — Vault `healthcheck_credential` fan-out (Pipeline-B audit was right)

- **One stop, 7.7 seconds of IPC time, 308ms per call × 25 calls** in a single page-mount.
- Audit finding `credential-vault-crud.md #1` ("35+ scattered `fetchCredentials()` callers with zero in-flight dedup") is **fully corroborated** — the issue isn't dedup of `fetchCredentials` though; it's the bulk healthcheck pattern: every credential's health is checked on mount, serially or near-serially.
- This is the **single biggest cost in the entire app** by IPC time. Wins here are user-visible (vault page is slow to interactive).
- **Fix shape:** lazy-check (only when user expands/hovers a row), OR batch into a single `healthcheck_credentials_bulk` IPC, OR rate-limit to N concurrent + retry stale rows in the background.

### P0 — `resolve_effective_config` loop (audit missed this)

- **104 calls across 2 stops, 3.6 sec** of IPC time.
- 52 calls per stop strongly suggests **one IPC per setting key** rather than one bulk fetch — the Settings page is iterating its UI controls and resolving each individually.
- The codebase already has `get_app_settings_bulk` (seen on settings/engine, settings/notifications, etc.) — so the pattern exists; it just isn't used by the Settings index / Account tab.
- **Fix shape:** identify the loop, replace per-key `resolve_effective_config` with one bulk fetch + frontend Map lookup. Estimated 1-2 commits.

### P1 — `get_rotation_status` × 25 on Overview (audit missed this)

- 25 IPC calls of the same command, 2.9 sec IPC time, on **Overview landing**.
- Pattern matches "one call per credential" — Overview probably shows credential rotation status per item in a list/badge.
- **Fix shape:** batch into a single `get_rotation_status_bulk` IPC, or accept that the page polls and reduce polling frequency.

### P1 — `L1/overview` 74 render commits (highest in the app)

- 74 renders on landing is **2-7× any other stop**. Even after the i18n + realtime fixes, Overview is still a render hotspot.
- 1960 DOM nodes — large but not the biggest.
- Possibly **chart-driven** (Recharts re-renders on tooltip / data updates) or **cascade from the 42 IPCs** each triggering downstream selector updates.
- **Fix shape:** add per-component Profilers temporarily on Overview to identify which subtree is committing most; then memo / stabilize selectors.

### P1 — `L1/design-reviews` DOM = 4001 nodes (audit had this for Recipes; it's the design-reviews list)

- 4001 DOM nodes is **3-8× any other stop**. Render count and IPC count are moderate (18 / 12) — the DOM is built once but is huge.
- The audit's `recipes #1` finding ("RecipeList renders all ~291 cards without virtualization") talked about Recipes; this measurement points to **Design Reviews** as an equally large unvirtualized list.
- **Fix shape:** adopt `@tanstack/react-virtual` (already a project dep) for design-review rows. Same pattern as the deferred LiveStreamTab virt.

### P2 — `langfuse_get_config` slow

- 2 calls × ~700ms each on `twin/profiles`. Other commands on the same page average <40ms.
- Either external network or a heavy backend serialize/deserialize.
- **Fix shape:** cache the config aggressively (only invalidate on save); confirm it's not blocking first-paint.

## Verified call sites (grep)

All three Tier-1 fix-shape hypotheses point to the same anti-pattern:
**frontend `Array.map → invoke(per-id)` instead of a bulk Rust endpoint**.

### `resolve_effective_config` — 104 calls
```ts
// src/features/settings/sub_config/components/ConfigResolutionPanel.tsx:85-87
const results = await Promise.allSettled(
  personas.map((p) => resolveEffectiveConfig(p.id))
);
```
Pattern: parallel-but-N-roundtrips. With 52 personas in this test DB → 52 IPCs per panel mount. Repeated on `L1/settings` *and* `settings/account` (likely the panel mounts in multiple settings shells, or settings landing opens this tab by default).

### `healthcheckCredential` — 25 calls × 308ms
```ts
// src/features/vault/shared/hooks/health/useBulkHealthcheck.ts:134
const hcResult = await credApi.healthcheckCredential(cred.id);
```
Inside a `for...of credentials` loop with `await` — **sequential, not parallel**. Hook is called "bulk" but is sequential per-credential. 7.7s wall time = 25 × 308ms.

Even just parallelizing with `Promise.all` would cut this to ~600ms (assuming sub-resources don't contend). A real bulk Rust command would be best.

### `getRotationStatus` — 25 calls
```ts
// src/features/vault/shared/hooks/health/useRemediationEvaluator.ts:87
rotationStatus = await getRotationStatus(cred.id);
// src/features/vault/shared/playground/CredentialPlaygroundModal.tsx:53
const status = await getRotationStatus(credential.id);
```
Same shape — per-id call from inside a per-credential evaluation loop in `useRemediationEvaluator`. On Overview, mounting the remediation surface fires 25 of these.

### Why this matters across the codebase

The pattern **`Array.map(item => invoke('per_id_command', { id }))`** appears to be the dominant cost driver. It's syntactically clean on the frontend but is an architectural mismatch: every map iteration is a serialize → IPC → deserialize roundtrip with Tauri overhead (~5-15ms minimum) on top of the actual backend work.

The recommended unification:
1. **Bulk Rust commands**: `resolve_effective_config_bulk(persona_ids)`, `healthcheck_credentials_bulk(ids)`, `get_rotation_statuses_bulk(ids)` — one IPC, backend handles parallelism + DB batching.
2. **Or** at minimum: parallelize the frontend `for/await` patterns into `Promise.all` (immediate ~25× wall-time win even without backend changes).

A grep for `\.map\(.*invoke\(` or `for.*await invoke` across `src/api/` would surface every instance of this anti-pattern — likely a one-evening cleanup.

## Tier-1 sketches

### 1. Vault healthcheck fan-out

```
Current:   onMount(CredentialList) → for each cred → invoke('healthcheck_credential', { id })
           25 calls × 308ms = 7.7 sec on the IPC bus

Proposed:  Add Rust `healthcheck_credentials_bulk(ids: Vec<String>) -> Vec<HealthcheckResult>`
           Frontend: useEffect(() => { invoke('healthcheck_credentials_bulk', { ids: vault.creds.map(c => c.id) }) }, [...])
           1 call × ~3-4 sec parallel work on backend = same user-visible latency, ~1/25 the IPC overhead
           OR: lazy-check — only the rows visible in the viewport get checked,
           rest defer until scrolled into view
```

### 2. `resolve_effective_config` loop

```
Find the loop:
   grep -rn "resolve_effective_config" src/features/settings/

Expected pattern:
   useEffect(() => {
     SETTINGS_KEYS.forEach(key => {
       invoke('resolve_effective_config', { key }).then(...)
     })
   }, [])

Replace with:
   useEffect(() => {
     invoke('resolve_effective_config_bulk', { keys: SETTINGS_KEYS })
       .then(values => setConfig(new Map(values)))
   }, [])

If `resolve_effective_config_bulk` doesn't exist Rust-side, add it.
```

### 3. `get_rotation_status` × 25 on Overview

```
Find the caller:
   grep -rn "get_rotation_status" src/features/overview/

If it's a per-credential map in a useEffect, same fix as Vault: bulk command.
If it's a polling loop, reduce interval and/or move out of the render path.
```
