## Architect findings — what the walk shows

- `L1/overview`: **64 → 39 render commits (−43%)**, render −19%, IPC −68%.

## NEW issues the walk surfaced (NOT architect work — next-pass candidates)

These are exactly the "page-load freezes" the user is chasing, newly introduced by
concurrent work since 2026-05-21:

1. **`L1/personas` IPC time 69ms → 2405ms (+3391%)** — 10 IPCs but one is ~2s. A
   multi-second backend call now blocks the personas landing (`obsidian_available`,
   `fetch_roadmap`, or `list_personas_using_connector`). Top suspect for a real freeze.
2. **`settings/account` IPC time 56ms → 2000ms** — `radio_get_state` / `radio_get_now_playing`
   / `radio_list_stations` (×2 each) now cost ~2s on the Account tab mount.
3. **`L1/schedules` DOM 494 → 5141 (~10×)** — an unvirtualized list scaling with
   schedule/cron data; render 142ms. Same class as architect finding [2].
4. **`twin/profiles` 1426ms IPC**, **`settings/engine` 880ms IPC (health_check_local)** —
   slow per-mount backend calls.
