## Fixed in this pass (verified live on :17320)

### 1. Settings write storm — `NotificationSettings` auto-save loop ✅ FIXED
`set_app_setting` fired **48×** across the walk (×14 on portability, ×10 limits,
×8 api-keys/config). Root cause: both auto-save effects in
`NotificationSettings.tsx` depended on the whole `setting` object returned by
`useAppSetting`, which is **a new object every render** — so every render
scheduled a `save()`, and `save()` flips the `saved` flag causing more renders.
Self-perpetuating write loop the entire time the tab was mounted **plus 30s
after leaving it** (SettingsPage keeps idle tabs mounted for 30s). Each write
also lands a settings-history row.
Fix: scalar deps (`setting.loaded`, `setting.value`).
**Verified: 8s idle on the tab now fires 0 writes (was ~14/4s).**

### 2. `marathon-default` duplicate React keys ✅ FIXED
The user's DB carries 42 marathon-adopted personas whose configs hold
**duplicate connector entries literally named `marathon-default`** (559
occurrences; test-seeded adoption answers). Six components keyed connector
chips/cards by bare `name`, colliding on every render:
`ConnectorsTabSections`, `PersonaConnectorsTab`, `CommandPanelToolsRow`,
`PersonaOverviewCardList`, `PersonaOverviewVariantGrid`, `PersonaOverviewCells`,
`PersonaOverviewVariantConstellation`. The warning frequency in the console
(18–21×/2s) was itself evidence those components re-render constantly on the
Agents page. Fix: composite `${name}-${i}` keys.
**Verified: personas list → select → connectors tab now logs 0 warnings.**
Follow-up candidate: dedupe/clean the marathon-polluted persona configs
themselves (data hygiene, not a UI bug).

### 3. Home landing double event fetch ✅ FIXED
`useNavCardStatus` (Welcome nav-card status, added in the recent corner-numbers
redesign) fetched two separate 24h `list_events_in_range` windows (~156ms each
backend-side), ×2 under dev StrictMode = **4 calls / ~620ms per Home landing**
— and every `revisit/*` path goes through Home. Fix: one 48h fetch partitioned
client-side. **Verified: 2 calls in dev (1 in prod), ~278ms total.**

### 4. `langfuse_stack_get_info` docker probe on unrelated surfaces ✅ FIXED (follow-up commit)
The original report attributed `settings/account` 2739ms to radio commands —
**wrong**: per-command timing shows radio calls cost ~12ms each; the entire
2.7s was `langfuse_stack_get_info` (×2 @ ~1.3s, spawns 4-5 docker.exe
subprocesses per call). It leaked onto other stops two ways: (a) slow resolves
from the Langfuse page land in the *next* page's measurement window, and
(b) `navigate('plugins')` re-mounts the last-active plugin page — if that was
Langfuse, every Plugins landing re-probed docker before the sub-tab switch.
Fix: module-level TTL cache (30s) + in-flight dedupe in `useLangfuseStack`;
mount reads tolerate the cached snapshot, user refresh / job polling /
post-job reads force a re-probe.
**Verified: settings/account window 2739ms → 56ms; twin/profiles 1504ms → 91ms.**

### 5. `twin_list_profiles` ×3 per twin visit ✅ FIXED (follow-up commit)
TwinPage + ProfilesAtelier each fetch on mount (×2 StrictMode) → 3-4 identical
IPCs. Fix: in-flight dedupe + 10s freshness window in
`twinSlice.fetchTwinProfiles`; mutation paths (`setActiveTwin`,
brain bind/unbind) pass `{ force: true }`.
**Verified: 1 call per twin-tab visit.**
