## 1. Hand-rewritten copies of ts-rs-generated bindings in `api/obsidianBrain/index.ts`

- **Severity**: high
- **Category**: duplication
- **File**: `src/api/obsidianBrain/index.ts:8-88` and `:160-258`
- **Scenario**: The file declares its own `interface` versions of 16 types that all have authoritative `ts-rs`-generated counterparts under `src/lib/bindings/`: `DetectedVault`, `VaultConnectionResult`, `FolderMapping`, `ObsidianVaultConfig`, `SyncLogEntry`, `SyncConflict`, `PushSyncResult`, `PullSyncResult`, `VaultTreeNode`, `DriveSyncResult`, `DriveStatus`, `VaultSearchHit`, `VaultLinkRef`, `VaultMocEntry`, `VaultStats`, `DailyNoteRef`. The auto-generated header in each binding even says `// Do not edit this file manually` — yet the hand-written copies diverge (e.g. `DriveStatus.storageUsedBytes` is `bigint | null` in the binding but `number | null` here, which silently truncates large drives over 2^53 bytes / 9 PB).
- **Root cause**: The file pre-dates the `ts-rs` migration for those structs and was never trimmed when the bindings landed.
- **Impact**: ~80 LOC of dead hand-typed shapes. When the Rust struct changes, only the bindings file regenerates; this file drifts silently — already does (see `bigint` divergence). Frontend keeps consuming the local shape, masking the contract break until runtime.
- **Fix sketch**: Replace each `export interface X { ... }` with `export type { X } from '@/lib/bindings/X'` (or just delete and let consumers import from bindings). The remaining types without bindings (`DriveSyncResult` actually has a binding too — `DriveSyncResult.ts`) can be either generated or kept inline. Same pattern lives in `api/drive.ts:60-82` (DriveEntry, DriveEntryKind, DriveTreeNode, DriveStorageInfo, DriveSearchHit — all have bindings) and `api/artist/index.ts:7-30,89-104,197-227` (ArtistAsset, BlenderMcpStatus, FfmpegStatus, MediaProbeResult, LoudnessStats — all have bindings). Total drift across the four API modules: ~25 manually-restated DTOs.

## 2. Seven near-identical `TwinVariantTabs` wrapper page files

- **Severity**: high
- **Category**: duplication
- **File**: `src/features/plugins/twin/sub_brain/BrainPage.tsx:1-22`, `sub_channels/ChannelsPage.tsx:1-22`, `sub_identity/IdentityPage.tsx:1-22`, `sub_knowledge/KnowledgePage.tsx`, `sub_profiles/ProfilesPage.tsx`, `sub_tone/TonePage.tsx`, `sub_training/TrainingPage.tsx`
- **Scenario**: Every twin sub-area has a 21-line `*Page.tsx` that does nothing but lazy-import three sibling components named `<Sub>Atelier|Console|Baseline` and render them inside `<TwinVariantTabs storageKey="<sub>">`. The bodies differ only in (a) the sub-name string in the imports and (b) the storage key.
- **Root cause**: Each variant was scaffolded by hand. `TwinVariantTabs.tsx:23-30` itself comments "*throwaway scaffolding — once a winner is picked the wrapper is collapsed*" but the choice hasn't been made and meanwhile the same 21-line lazy-bag has been retyped 7 times.
- **Impact**: 147 LOC of pure copy-paste; bug-fix-once-fix-seven; new sub-area requires another copy.
- **Fix sketch**: Add a `createTwinVariantPage(storageKey, { atelier, console, baseline })` factory in `_variants/`, or accept a `loaders: { atelier: () => Promise<...> }` prop on `TwinVariantTabs` and call the page from a single `*Page.tsx` via a registry. Each sub-area drops to a 5-line file or vanishes entirely into a route table.

## 3. CRUD-action boilerplate triplicated across twin slice (15+ try/catch shapes)

- **Severity**: high
- **Category**: duplication
- **File**: `src/stores/slices/system/twinSlice.ts:155-442`
- **Scenario**: The slice has 15+ async actions that all follow one of two identical shapes:
  - **fetch X**: `set({xLoading: true}); try { const x = await api.list...; set({x, xLoading: false, error: null}); } catch (err) { reportError(err, "Failed to ...", set, { stateUpdates: { xLoading: false } }); }`
  - **delete X**: `try { await api.delete(id); set(state => ({ x: state.x.filter(...), error: null })); } catch (err) { reportError(err, "Failed to ...", set); }`
  Repeated for tones, pending memories, communications, voice profile, channels, plus the create/update variants.
- **Root cause**: Slice grew organically per phase (P0 → P4); no helper was introduced.
- **Impact**: ~280 LOC of structural duplication; every new entity adds ~30 LOC of identical try/catch/reportError; one fix to the error-handling contract touches 15+ sites.
- **Fix sketch**: Extract two helpers in `stores/util/` (already a folder): `createListAction<T>(loadingKey, listKey, apiCall, label)` and `createCrudAction(...)`. The pattern is already partially used in `devToolsProjectSlice.ts` (per the slice's own header comment) — adopt the same factory there and reuse here.

## 4. `ToneForm`/`ChannelMeta` and `DraftForm` declared identically in each twin variant trio

- **Severity**: high
- **Category**: duplication
- **File**: `src/features/plugins/twin/sub_tone/ToneAtelier.tsx:36-40`, `ToneBaseline.tsx:24-28`, `ToneConsole.tsx:20-24`; `sub_channels/ChannelsAtelier.tsx:23-40`, `ChannelsBaseline.tsx:19-34`, `ChannelsConsole.tsx:20-35`; `sub_profiles/ProfilesAtelier.tsx:29-30`, `ProfilesBaseline.tsx:17-22`, `ProfilesConsole.tsx:24-25`
- **Scenario**: In each twin sub-area, the same internal `interface ToneForm { voiceDirectives; examplesJson; constraintsJson; lengthHint }`, `const EMPTY/EMPTY_FORM`, and `function toneToForm(tn): ToneForm` are declared three times (once per Atelier/Console/Baseline variant). Channels duplicates `CHANNEL_TYPES = DEPLOYMENT_CHANNELS.map(...)`, `getChannelMeta`, and `channelOptions` triplicated identically across its three variants. Profiles triplicates `interface DraftForm { name: string; role: string }` + `EMPTY/EMPTY_DRAFT`. The store-selector pull (`useSystemStore((s) => s.twinTones)` × 6 selectors) is also identical across the three tone variants.
- **Root cause**: Variants were forked for A/B prototyping (per `TwinVariantTabs.tsx:23-30`). The shared form schema was never lifted.
- **Impact**: ~120 LOC of form-shape duplication across tone/channels/profiles; a backend schema change to `examples_json` requires touching 3+3+3 = 9 form files. This is intra-plugin duplication (within the twin plugin's own A/B variants), so the brief's "plugins pin shapes for independence" carve-out does not apply.
- **Fix sketch**: Each sub-area's three variants share a `useToneForm()` / `useDraftForm()` / `useChannelDerived()` hook in `sub_tone/`, `sub_profiles/`, `sub_channels/` that returns `{ forms, getForm, setForm, save, hasTone }` and the form interface. Variants render whatever UI they like over the shared state.

## 5. Three byte-formatter implementations across the plugins

- **Severity**: medium
- **Category**: duplication
- **File**: `src/api/drive.ts:245` (`driveFormatBytes`), `src/features/plugins/artist/utils/format.ts:1` (`formatFileSize`), `src/features/plugins/obsidian-brain/sub_cloud/CloudSyncPanel.tsx:19` (`formatBytes`)
- **Scenario**: Each plugin has its own `bytes → "1.5 MB"` formatter. Drive's lives in the api wrapper (and thus is exported app-wide), artist's lives in a utils file, obsidian's is a private function in a single component. Outputs differ subtly (drive uses `value.toFixed(value >= 100 || i === 0 ? 0 : 1)`, artist/obsidian use 1 decimal always for KB/MB, 2 for GB).
- **Root cause**: Each surface grew its own helper before any shared util landed. There are 7 total byte-formatters in `src/` (see also `vault/.../documentTabHelpers.ts`, `sharing/.../NetworkDashboard.tsx`, `BundleExportDialog.tsx`, `templates/.../n8nUploadTypes.ts`) — but inside the plugins scope the three above are the live ones.
- **Impact**: ~30 LOC; users see inconsistent display between Drive (`1 KB`) and Obsidian's Drive-storage row (`1.0 KB`); locale formatting can't be added uniformly.
- **Fix sketch**: Move `driveFormatBytes` from `api/drive.ts` (where it sits oddly mixed with IPC wrappers) into `src/lib/utils/formatters.ts` as `formatBytes(bytes, { precision? })`. Have all three call sites import from there.

## 6. Two divergent `atomic_write` implementations in plugins

- **Severity**: medium
- **Category**: duplication
- **File**: `src-tauri/src/commands/obsidian_brain/mod.rs:53-65` (sync, sibling-suffix `.tmp`), `src-tauri/src/commands/artist/persistence.rs:296-312` (async, `<ext>.<uuid>.tmp`)
- **Scenario**: Both implement write-tmp-then-rename for crash safety, but the obsidian variant is sync `std::fs` + plain `.tmp` suffix (collides under concurrent writers, per artist's own header comment at `persistence.rs:288-295` calling out exactly that bug), while artist's is async `tokio::fs` + uuid-suffixed staging.
- **Root cause**: Same idea grown twice independently.
- **Impact**: The obsidian variant is the inferior of the two — two concurrent vault writes to the same file path can collide on the `.tmp` sibling. Since prior wave already shipped atomic writes in some places (per task note), the obsidian copy probably needs to converge.
- **Fix sketch**: Move artist's uuid-suffixed `atomic_write` to a shared module (e.g. `src-tauri/src/db/utils.rs` or a new `src-tauri/src/io.rs`). Expose both a sync and an async flavour that share the uuid-staging logic. Have obsidian use the sync flavour. Re-audit drive.rs too — it does plain `std::fs::rename` in `drive_rename`/`drive_move`/`drive_copy` at lines 1028, 1097, 1137 without staging, but those are user-initiated single-target renames so the concurrency story is different.

## 7. Wholesale-restated knowledge-base shapes in `api/twin/twin.ts`

- **Severity**: medium
- **Category**: structure
- **File**: `src/api/twin/twin.ts:155-183`
- **Scenario**: The file embeds a 30-line `validateKeyFactsJson` (UTF-8 byte cap + JSON.parse round-trip) inside the api wrapper module. This kind of trust-boundary validator is exactly the shape Drive's path validators take at `api/drive.ts:10-56`, but no shared module exists. So we get N copies of "validate-this-string-meets-a-budget-and-shape" hand-rolled per plugin.
- **Root cause**: First-pass cargo-cult of the pattern without extracting it.
- **Impact**: ~50 LOC of similar shape that will keep getting forked for each new IPC-trust-boundary string field; one fix to the byte-counting strategy has to be hand-applied each time.
- **Fix sketch**: Add `src/lib/validation/ipcGuards.ts` with `assertJsonStringBudget(value, label, maxBytes)`, `assertRelativePath(path, label)`, `assertSimpleFileName(name, label)`. Twin and Drive both call those. Future plugin wrappers will too.

## 8. Twin variant baselines/consoles re-pull the same 6 store selectors

- **Severity**: medium
- **Category**: duplication
- **File**: `src/features/plugins/twin/sub_tone/ToneAtelier.tsx:65-71`, `ToneBaseline.tsx:32-37`, `ToneConsole.tsx:36-41` (and the matching trios in `sub_channels/`, `sub_brain/`)
- **Scenario**: Each variant component issues the same 6+ store-selector calls: `useSystemStore((s) => s.activeTwinId)`, `s.twinTones`, `s.twinTonesLoading`, `s.fetchTwinTones`, `s.upsertTwinTone`, `s.deleteTwinTone`. Tone has three variants × 6 selectors = 18 nearly identical hook calls. Multiply across channels/brain/etc.
- **Root cause**: No domain-level hook abstraction in `_shared/`.
- **Impact**: 50+ duplicated selector lines; every store-shape rename touches all three variant files per sub-area; re-renders happen six times per component on any twin state change.
- **Fix sketch**: Add `useTwinTones()`, `useTwinChannels()`, `useTwinBrain()` hooks in `_shared/` that return `{ activeTwinId, items, loading, fetch, upsert, delete }`. Each variant calls one hook. Bonus: enables consolidating the loading-state logic.

## 9. `obsidian_brain_push_sync` repeats the same memory/persona/connector write loop three times

- **Severity**: medium
- **Category**: duplication
- **File**: `src-tauri/src/commands/obsidian_brain/mod.rs:276-484`
- **Scenario**: The push routine inlines the same 30-line block three times — once for memories (`l.276-364`), once for personas (`l.367-429`), once for connectors (`l.431-483`). Each block: derive relative path → compute_content_hash → fetch existing sync_state → compare hash → skip OR write atomically → upsert sync_state → log_sync. The only real per-entity difference is the dir-derivation and the `*_to_markdown` call.
- **Root cause**: Three sync entities; was easier to copy than abstract.
- **Impact**: ~210 LOC of structural duplication in one of the load-bearing IPC handlers; bug-fix-once-fix-three; e.g. the existing fix for "atomic_write + skipped count + sync_state advance" touched all three blocks separately.
- **Fix sketch**: Lift to a `push_entity_to_vault<T>(state, entity_type, entities, dir_for, markdown_for)` helper with a closure for path derivation and markdown rendering. Reuse for any future synced entity (twin profiles, recipes, etc.).

## 12. `pipelineHelpers.tsx` imports-and-re-wraps with no value added

- **Severity**: low
- **Category**: cruft
- **File**: `src/features/plugins/gitlab/components/pipelineHelpers.tsx:56-60`
- **Scenario**:
  ```
  import { formatDuration as _formatDuration } from '@/lib/utils/formatters';
  export const formatDuration = (seconds: number | null) => _formatDuration(seconds, { unit: 's' });
  import { formatRelativeTime } from '@/lib/utils/formatters';
  export const formatRelative = (iso: string) => formatRelativeTime(iso);
  ```
  Two imports placed at the file's bottom (style nit aside) and two re-exports that add no logic except `formatDuration` baking in `{ unit: 's' }`. `formatRelative` is a literal pass-through. Multiple GitLab components import these aliases instead of the lib helpers.
- **Root cause**: Migrated from a hand-rolled local formatter to the shared lib without finishing the migration.
- **Impact**: 5 LOC of pointless indirection; one more name to maintain. Bottom-of-file imports also bypass the file's nominal "icon/status helpers" purpose.
- **Fix sketch**: Move the imports to the top, delete `formatRelative` (replace consumers with direct `formatRelativeTime`), and either delete `formatDuration` (consumers pass `{ unit: 's' }` themselves — only one or two callers) or move it to `lib/utils/formatters.ts` as `formatDurationSeconds(seconds)` if reused outside gitlab.
