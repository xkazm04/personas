# Bug Hunter — execution-engine-runs
> Total: 6
> Severity: 2 critical, 3 high, 1 medium

## 1. Cancel is silently clobbered back to "completed" by the finalizer (lost-cancel / success theater)
- **Severity**: critical
- **Category**: race-condition
- **File**: src-tauri/src/engine/mod.rs:1041 / src-tauri/src/db/repos/execution/executions.rs:775
- **Scenario**: A run is nearly done. `run_execution` returns a `success` result. The spawned task evaluates `if cancelled.load(Ordering::Acquire)` at mod.rs:1041, reads `false`, and proceeds into the `else` branch toward `handle_execution_result`. *Right after* that load, the user clicks Stop: `cancel_execution` sets the flag (mod.rs:1189), then `persist_status_if_running` writes the row to `cancelled` (mod.rs:1195), then kills the process. The task now reaches `handle_execution_result` → `persist_status_if_not_final(status = Completed)`. Its SQL guard is `WHERE id = ? AND status IN ('running','cancelled')` (executions.rs:775), so it **overwrites the freshly-written `cancelled` row back to `completed`.**
- **Root cause**: Two design assumptions collide. (a) The `cancelled` flag is read once, non-atomically, relative to the DB write, so the cancel can land in the window between "flag check" and "final write". (b) `update_status_if_not_final` deliberately treats `cancelled` as overwritable (to let the cancel branch *enrich* a bare safety-net cancel with metrics) — but it cannot distinguish "enrich my own cancel" from "a completed result is stomping a user cancel". The CAS predicate is too permissive.
- **Impact**: data loss / corruption / UX. The user explicitly stopped the run and killed the process, yet the activity feed, metrics, success-rate, and budget accounting all record `completed`. Any downstream automation gated on `completed` (chain triggers, healing, message delivery) fires for a run the user aborted.
- **Fix sketch**: Make cancellation a true terminal sink. After the safety-net writes `cancelled`, never allow a non-cancel status to overwrite it: split the predicate so the *cancel* branch may write `WHERE status IN ('running','cancelled')` but the *completion/failure* branch writes `WHERE status = 'running'` only. Equivalently, re-check `cancelled.load()` immediately before the `handle_execution_result` write and route to the cancel branch if set. This makes "completed-over-cancel" structurally impossible.

## 2. Fire-and-forget session_id write resurrects terminal rows to "running" (zombie executions)
- **Severity**: critical
- **Category**: state-corruption
- **File**: src-tauri/src/engine/runner/mod.rs:1890 (spawn) → update via executions.rs:530 `update_status`
- **Scenario**: On the first `system/init` stream line, the runner spawns a *detached* task to persist `claude_session_id` with `UpdateExecutionStatus { status: ExecutionState::Running, claude_session_id: Some(...), .. }` (runner/mod.rs:1890-1913). That task retries on failure with backoff. For a very fast run (or a cancel/timeout right after init), the execution can reach a terminal DB status (`completed`/`cancelled`/`failed`) *before* this detached write (or its retry) executes. `update_status` (executions.rs:530) has **no status guard** — its `WHERE id = ?12` matches any row and sets `status = ?1 = 'running'` unconditionally.
- **Root cause**: The session-id persist reuses the full `update_status` (which writes the `status` column) purely to piggy-back one field, and runs on an independent task with its own retry timeline. The assumption "this fires while the row is still running" is not enforced; there is no ordering guarantee between this detached task and the finalizer.
- **Impact**: corruption — a finished execution flips back to `running`. It then never gets a finalizer (the owning task is gone), so it sits as a permanent zombie until `sweep_zombie_executions` (executions.rs:1301, threshold-based) eventually downgrades it to `incomplete` and fires a misleading "execution stalled" notification. Concurrency tracker slot accounting and the UI also desync.
- **Fix sketch**: Persist `claude_session_id` with a column-scoped, status-preserving write (`UPDATE ... SET claude_session_id = COALESCE(?,claude_session_id) WHERE id = ? AND status = 'running'`) — never write the `status` column from this path. More broadly, forbid `status = Running` writes against rows not currently `running` (add the guard to `update_status`, or use a dedicated `set_session_id` repo fn).

## 3. Task handle inserted AFTER spawn — cancel within the startup window can't kill the task (orphaned CLI + double-spend)
- **Severity**: high
- **Category**: race-condition
- **File**: src-tauri/src/engine/mod.rs:1019 (spawn) vs 1148 (`self.tasks.lock().await.insert`)
- **Scenario**: `spawn_execution_task` calls `tokio::spawn(...)` at mod.rs:1019 and only inserts the `JoinHandle` into `self.tasks` at mod.rs:1148 — after the spawn and the surrounding awaits. If the user (or `force_cancel_all_for_persona`) calls `cancel_execution` in that window, step 4 (`self.tasks.lock().await.remove(execution_id)` at mod.rs:1212) finds nothing, so cancel never `await`s/`abort`s the task and skips the grace-period child-kill at mod.rs:1226. The just-spawned CLI keeps running. The same ordering exists for the queue-drain spawn at mod.rs:1667 → insert at 1781.
- **Root cause**: The handle and the task's discoverability are not registered atomically with task creation. Cancel assumes "if it's running, its handle is in `self.tasks`," which is violated during the spawn→insert gap. (The `cancelled` flag may also not be discoverable yet on the drain path if cancel races the flag insert at mod.rs:1639.)
- **Impact**: orphaned CLI subprocess consuming LLM API credits with no kill path; cancel reports `true` but the work continues; possible later write to the DB row the user thinks is dead.
- **Fix sketch**: Register the handle (and cancel flag) *before* the task can finish its first await, or insert the handle synchronously right after `tokio::spawn` returns and before any `.await`. Better: have the task itself self-register/self-deregister under a single lock acquisition keyed on `execution_id`, so "in tasks map" ⇔ "task alive" is invariant.

## 4. App-crash recovery never reaps orphaned child PROCESSES — only the DB row
- **Severity**: high
- **Category**: recovery-gap
- **File**: src-tauri/src/engine/mod.rs:665 `recover_stale_executions`
- **Scenario**: The app is killed (OS crash, force-quit, power loss) while a CLI subprocess is mid-run. On next launch, `recover_stale_executions` marks every `status='running'` row `failed` (mod.rs:675). But on Windows/macOS the spawned Claude CLI child is *not* a transitive child of the new process and `kill_on_drop` never ran (the parent died abnormally), so the previous run's CLI (and its descendant tree) **may still be alive**, continuing to burn API credits and possibly writing to the worktree/exec dir. Recovery updates only the DB; no PID reconciliation occurs (PIDs live only in the in-memory `child_pids` map, which did not survive the crash).
- **Root cause**: The durable state model persists the execution *row* but not the *OS PID*, so post-crash recovery has no handle to kill leaked processes. Recovery conflates "row is orphaned" with "process is dead."
- **Impact**: silent ongoing API spend after a crash; possible corruption of the per-persona scratch dir / worktree by a zombie CLI while a new run for the same persona starts.
- **Fix sketch**: Persist the child PID (and spawn timestamp) alongside the `running` row; on startup, before failing stale rows, best-effort `kill_process` any persisted PID whose start time matches (guard against PID reuse via start-time/creation-time check). Alternatively spawn the CLI in an OS job object / process group that the supervisor can reap on restart.

## 5. `stream_result` discards captured output on timeout — partial output and metrics lost
- **Severity**: high
- **Category**: edge-case
- **File**: src-tauri/src/engine/runner/mod.rs:1768-2232, 2411-2426
- **Scenario**: The stdout loop runs inside `tokio::time::timeout(timeout_duration, async { ... })` (runner/mod.rs:1768). On timeout, `stream_result.is_err()` is true and the run is marked failed/`timed_out`. The closure mutated `assistant_text`, `metrics`, `tool_steps`, `total_stdout_bytes` etc. by capture, so those *are* retained — but the Claude CLI emits its cost/token/`result` summary as the **final** line, which by definition never arrives on a timed-out run, so `metrics.cost_usd / input_tokens / output_tokens` stay 0 even though the run consumed real tokens. Worse, on the engine *safety-ceiling* timeout (mod.rs:383-423) the entire `run_execution` future is dropped, discarding all accumulated `assistant_text`/`tool_steps`/partial trace, and the result is hard-coded to `cost_usd/tokens = 0` (mod.rs:415-421).
- **Root cause**: Cost/usage accounting depends on a single terminal `result` line; any timeout/kill path loses it, and the ceiling path drops the whole future so even the partial in-memory accumulation is gone. Budget enforcement (`get_monthly_spend`, executions.rs / commands) then under-counts spend on exactly the most expensive (longest, timed-out) runs.
- **Impact**: budget under-accounting on stalled runs (a persona can repeatedly time out and never hit its monthly cap); partial output/tool-steps lost on the ceiling path; "success theater" of cost = $0 on runs that actually spent money.
- **Fix sketch**: Estimate tokens/cost from accumulated stream deltas (assistant text length + tool I/O) when the terminal `result` line is absent, and persist whatever partial `assistant_text`/`tool_steps` were captured before the kill. For the ceiling path, capture metrics via a shared `Arc<Mutex<ExecutionMetrics>>` the runner updates as it streams, so dropping the future still leaves the latest snapshot readable.

## 6. Outcome assessment uses English substring heuristics — non-English/quoted output silently mislabels runs
- **Severity**: medium
- **Category**: silent-failure
- **File**: src-tauri/src/engine/runner/mod.rs:2454-2472
- **Scenario**: When no structured `outcome_assessment` is found, the runner lowercases `assistant_text` and scans for English literals — `"error:"`, `"failed to"`, `"unable to"` vs `"successfully"`, `"fixed"`, `"resolved"` (runner/mod.rs:2455-2467) — to decide `Completed` vs `Incomplete`. A persona that ran successfully but *quotes* an error it handled (e.g. an agent summarizing "the API returned `error: rate limited`, so I retried and it worked") with wording the resolution list doesn't cover, or any non-English output, gets mislabeled. Conversely a genuinely failed run that says "completed" in prose is marked `Completed` on exit-code 0.
- **Root cause**: Business outcome is inferred from natural-language pattern-matching on free text rather than a structured signal, with an English-only keyword set. The heuristic is a guess presented as fact and there is no confidence/uncertainty path.
- **Impact**: UX degradation + downstream corruption — `Incomplete`/`Completed` drives the notification bell, success-rate metrics, healing/retry eligibility, and chain advancement. Misclassification silently triggers (or suppresses) retries and notifications.
- **Fix sketch**: Treat the keyword heuristic as low-confidence only: when no structured `outcome_assessment` is present, default to `Completed` on exit-code 0 (already the conservative intent) and record an explicit `unknown`/low-confidence business_outcome rather than flipping to `Incomplete` from prose alone. If automatic assessment is required, prompt the model for a structured `outcome_assessment` block and treat its absence as `unknown`, not as evidence of failure.
