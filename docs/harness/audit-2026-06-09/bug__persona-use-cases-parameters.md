## 1. Lost-update race: backend design_context cascade bypasses the frontend write queue
- **Severity**: critical
- **Category**: state-corruption
- **File**: src-tauri/src/commands/core/use_cases.rs:36 (`cascade_use_case_toggle`), :140 (`patch_generation_settings`); interacting with src/hooks/design/core/useDesignContextMutator.ts:25 (`writeQueue`) and src/features/agents/sub_use_cases/components/recipes-prototype/shared/usePolicyControls.ts:60
- **Scenario**: User edits a use case's prompt/channels in the detail panel (frontend `mutateSingleUseCase` → `applyDesignContextMutation` read-modify-writes the whole `design_context` blob through the in-process `writeQueue`). At the same moment they toggle the capability off, or flip a generation-policy toggle. `set_use_case_enabled` / `set_use_case_generation_settings` read `design_context` from SQLite, patch `enabled`/`generation_settings` in memory, and write the WHOLE blob back — completely outside the frontend's `writeQueue`. The two read-modify-write cycles interleave: whichever writes last clobbers the other's field. The frontend store still holds the pre-toggle blob, so the very next frontend write (the trailing prompt-edit) silently re-persists the OLD `enabled`/`generation_settings`, undoing the backend toggle. The toast says "Paused capability" while the DB ends up enabled.
- **Root cause**: Two independent serialization domains mutate the same JSON blob with last-writer-wins semantics. The frontend queue (useDesignContextMutator.ts:18-23 explicitly notes it exists to prevent exactly this) only serializes frontend writes; the Rust commands are unaware of it and operate on a full-document overwrite, not a field-scoped UPDATE.
- **Impact**: corruption — silently reverted capability toggles / dropped policy edits; a "paused" capability keeps running (real cost, real notifications) because the toggle was clobbered.
- **Fix sketch**: Make backend patches field-scoped and conflict-safe — either use SQLite `json_set(design_context, '$.useCases[i].enabled', ?)` so concurrent edits to other fields can't be lost, or guard the read-modify-write with an `updated_at`/version compare-and-swap and reject on mismatch. Have the frontend route ALL design_context-touching backend commands through the same `writeQueue` and always `fetchDetail` to overwrite the local store after they return.

## 2. Event-rename uses substring REPLACE on trigger config JSON — corrupts unrelated fields and partial names
- **Severity**: critical
- **Category**: state-corruption
- **File**: src-tauri/src/commands/core/use_cases.rs:518-527 (`rename_event_listeners`, Update branch)
- **Scenario**: A user renames event `"alert"` → `"escalation"`. The code builds `from_token = "\"event_type\":\"alert\""` and runs `UPDATE persona_triggers SET config = REPLACE(config, from_token, to_token)`. (a) If any trigger's config contains another field whose serialized text happens to embed that token sequence (e.g. a description, a nested condition string, or a different key whose value is the literal `event_type":"alert`), `REPLACE` rewrites it too. (b) `REPLACE` replaces ALL occurrences in the row, not just the matched event listener's. (c) Worse, the `LIKE` filter (`%"event_type":"alert"%`) matches `"alert"` as a substring of `"alert_high"` if JSON key ordering or whitespace ever differs from the exact token — and conversely the exact-token REPLACE silently no-ops on configs serialized with a space after the colon (`"event_type": "alert"`), so the row passes the `subscriptions_touched`/`triggers_touched` count via the LIKE but the value is NOT rewritten, returning a success count for a trigger that still listens on the old name.
- **Root cause**: String surgery on serialized JSON assumes a single canonical serialization (no whitespace, fixed key order, one occurrence). SQLite REPLACE has no JSON awareness and no per-occurrence scoping.
- **Impact**: corruption — triggers left listening on the stale event (silently dead), or config JSON mangled into invalid/incorrect state; reported `triggers_touched` overcounts vs. actual rewrites.
- **Fix sketch**: Use `json_set(config, '$.event_type', ?)` (SQLite JSON1) scoped to rows where `json_extract(config,'$.event_type') = ?1`, eliminating both the false-LIKE match and the whitespace-sensitive no-op. Never REPLACE serialized JSON as raw text.

## 3. AI icon generation downloads any HTTPS image URL found in provider JSON — SSRF/data-exfil surface
- **Severity**: high
- **Category**: security
- **File**: src-tauri/src/commands/core/persona_icon_gen.rs:271 (`find_image_url`) → :291 (`download_image`)
- **Scenario**: `find_image_url` recursively walks the provider's status JSON and returns the FIRST string under keys `url`/`image_url`/`uri`/`image` that merely `starts_with("https://")` — with NO extension check on the object-key path (only the bare-string fallback at :370 checks for an image extension). A compromised/buggy provider response, a CDN open-redirect, or an attacker who can influence the polled JSON can point `download_image` at an arbitrary `https://` host. The desktop app then issues an authenticated-context GET to that URL from the user's machine and buffers the body. This is a server-side-request-forgery primitive from inside the user's network (can hit internal `https://` endpoints) and an opportunity to fetch attacker-controlled content that is then decoded by the image pipeline.
- **Root cause**: The URL is treated as trusted because it came from "the provider", but the provider response is untrusted input and the URL is unvalidated against an allowlist of provider CDN hosts.
- **Impact**: security — SSRF to internal hosts, exfil of which-internal-hosts-respond, fetching arbitrary content into the decode path.
- **Fix sketch**: Validate the resolved URL against a per-provider host allowlist (`cdn.leonardo.ai`, the documented Higgsfield CDN) before `download_image`; reject non-allowlisted hosts. Require the image extension on the object-key path too, not just the bare-string fallback.

## 4. `verify_promoted_persona` fires a REAL (non-simulated) execution during promote — real connector writes/spend on every build
- **Severity**: high
- **Category**: silent-failure
- **File**: src-tauri/src/commands/core/use_cases.rs:674-686 (`is_simulation = false`)
- **Scenario**: On promote, the build path runs the persona's first manually-invokable capability for real (the comment at :668-673 deliberately sets `is_simulation = false` to exercise connectors). But `build_simulation_input` is still used to construct the input (:667), which injects `_simulation: true` into the payload. So the input claims simulation while the execution flag says real. Any downstream dispatch/connector logic that keys off the payload's `_simulation` field rather than `ctx.is_simulation` will skip delivery; anything keying off `ctx.is_simulation` will deliver for real. With a side-effecting first capability (sends an email, posts a message, writes to an external system), promoting the persona performs that real action every build — and if the two simulation signals disagree, the behavior is non-deterministic across the codebase.
- **Root cause**: Two sources of truth for "is this a simulation" (the `_simulation` payload key vs. the `is_simulation` execution flag) are set to contradictory values on the same run.
- **Impact**: data loss / unwanted side effects — real outbound actions during a build/verify step the user expects to be safe; ambiguous simulation semantics.
- **Fix sketch**: For a real verification run, build the input WITHOUT `_simulation: true` (don't reuse `build_simulation_input`, or add a flag to omit the marker). Make `ctx.is_simulation` the single source of truth and assert the payload marker matches it.

## 5. EventRenameModal reconciles consumers against STALE advisory counts; cancelled-effect leak overwrites fresh rows
- **Severity**: high
- **Category**: race-condition
- **File**: src/features/agents/sub_use_cases/components/core/EventRenameModal.tsx:63-93 (effect) and :127-145 (`handleSave`)
- **Scenario**: The count-fetch effect re-runs on every keystroke (dep is `rows.map(...).join(',')`). It launches `Promise.all` of `countEventListeners` calls and, on resolve, `setRows((prev) => ... merge counts by index)`. The `cancelled` flag guards the resolved branch, but rows are merged BY INDEX into whatever `prev` is at resolution time. If the user added/removed a row (indices shifted) or the requests resolve out of order across rapid edits, stale counts land on the wrong row. Then `handleSave` decides whether to call `renameEventListeners` purely from `r.existingCounts` (:130) — a value that may be stale or attached to the wrong `from` name. A row whose counts haven't loaded yet (`existingCounts === undefined`) is treated as "0 consumers" and skipped entirely (:130), so renaming an event that DOES have consumers silently leaves them orphaned when the user saves before the advisory query returns.
- **Root cause**: Async advisory counts are merged positionally into a mutable list whose shape changes under the user, and the destructive save action depends on those advisory values instead of re-querying authoritatively at save time.
- **Impact**: UX degradation + corruption — consumers silently left on the old event name (no rename/delete applied) despite the user choosing "update"/"delete"; warnings shown for the wrong row.
- **Fix sketch**: Key counts by the `from` string (a stable identity), not array index; cancel in-flight requests via the row identity. At save, do not trust cached counts — let the backend `rename_event_listeners` run unconditionally for every `from→to` pair with the chosen action (it already returns touched counts), so an un-loaded count can't cause a skip.

## 6. Cron visual builder/parser ignore minute/hour bounds and silently round-trip-lose lists
- **Severity**: medium
- **Category**: edge-case
- **File**: src/features/agents/sub_use_cases/libs/scheduleHelpers.ts:66-80 (`buildCronFromVisual`), :83-109 (`parseCronToVisual`)
- **Scenario**: `buildCronFromVisual` interpolates `minute`/`hour` directly into the cron string with no range check — a caller passing `hour = 24` or `minute = 60` (e.g. from an off-by-one in a time picker, or a `Date.getHours()` in a TZ edge) emits `"60 24 * * *"`, which `engine::cron::parse_cron` will reject at `set_persona_curation_schedule` time, surfacing as a confusing "invalid cron" only after the user thinks they saved. `parseCronToVisual` accepts any 5 fields and `parseInt`s the minute/hour without validating 0-59/0-23, so a stored `"75 30 * * *"` round-trips into a visual state that re-serializes to garbage. Day lists like `1-3` (range other than `1-5`) or step syntax `*/2` fall into the final branch where `Number("1-3")` is `NaN` and the day is dropped, silently losing schedule days on edit.
- **Root cause**: The visual model assumes a closed set of cron shapes (all-days / weekdays / weekends / explicit comma list) and never validates numeric bounds, so any input outside that set is silently lossy or produces an invalid expression.
- **Impact**: UX degradation — silently dropped schedule days, save-time "invalid cron" errors for visually-valid selections, schedules that fire at the wrong time after a lossy round trip.
- **Fix sketch**: Clamp/validate `hour ∈ 0..23` and `minute ∈ 0..59` in `buildCronFromVisual`; in `parseCronToVisual` return `null` when minute/hour are out of range or when the DOW field contains range/step syntax the visual model can't represent, so callers fall back to the raw cron editor instead of corrupting the value.
