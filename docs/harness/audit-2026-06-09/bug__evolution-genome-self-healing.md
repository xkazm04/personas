# Bug Hunter — evolution-genome-self-healing
> Total: 6
> Severity: 2 critical, 3 high, 1 medium

## 1. AI healing slot leaked permanently — double-acquire deadlocks healing for the persona forever
- **Severity**: critical
- **Category**: state-corruption / recovery-gap
- **File**: src-tauri/src/commands/execution/healing.rs:127 (acquire) → src-tauri/src/engine/mod.rs:2897 (re-acquire + early return)
- **Scenario**: User clicks "AI heal" on a failed execution. `trigger_ai_healing` calls `state.engine.try_start_healing(persona_id)` (healing.rs:127), which inserts the persona into the `healing_personas` set and returns `true`. The command then calls `start_healing_chain` → `spawn_healing_chain`. Inside the spawned task, line 2897 does `if !healing_personas.lock().await.insert(persona_id) { ... return; }`. The persona is *already* in the set (the command just put it there), so `insert` returns `false`, the task logs "session already in progress, skipping" and returns at line 2903 — **without doing any healing and without ever removing the entry**. The only code that removes the slot (mod.rs:3197 and the early-return arms at 2919/2926/2945/2976) lives *after* the guard, so it never runs.
- **Root cause**: Two layers both treat the `healing_personas` set as the acquisition point. The command acquires it as a "TOCTOU guard", but `spawn_healing_chain` was written as a standalone entry (used by the auto-healing path) that *also* acquires it. When both run, the second acquire is a self-collision and the cleanup path is skipped. The healing lock has no single owner and no RAII/guard tying release to acquisition.
- **Impact**: corruption — the persona is now permanently flagged `is_healing`. Every future `trigger_ai_healing` returns "A healing session is already in progress" (healing.rs:128) and every auto-healing attempt logs "skipping". AI self-healing is silently bricked for that persona until app restart, with no UI signal.
- **Fix sketch**: Pick ONE owner of the slot. Either remove the `try_start_healing` call in the command and let `spawn_healing_chain` own acquire+release exclusively (return a "started/skipped" status from the spawn), or have the command pass an "already-acquired" flag so the spawn skips its own `insert`. Best: wrap the slot in a guard struct whose `Drop` removes the persona id, so every exit path (including early returns and panics) releases it.

## 2. Concurrent evolution cycles overwrite each other and clobber live user edits (lost update)
- **Severity**: critical
- **Category**: race-condition / state-corruption
- **File**: src-tauri/src/commands/execution/evolution.rs:122-157; src-tauri/src/engine/evolution.rs:129-158, 354, 485-516
- **Scenario**: `evolution_trigger_cycle` has no per-persona concurrency guard (grep for `evolving_personas`/`try_start_evolution` returns nothing). Click "Evolve" twice, or trigger manually while auto-evolution fires, and two `run_evolution_cycle` tasks run for the same persona. Each captures its own `persona`/`incumbent_genome` snapshot at start (evolution.rs:140-158), spends minutes running CLI evaluation, then both call `promote_variant` (evolution.rs:354). `promote_variant` is a bare `UPDATE personas SET system_prompt=...` (evolution.rs:492-514) with no compare-and-set and no transaction reading current state — last writer wins. Worse: if the user edits the persona prompt in the UI *during* the cycle, the cycle still writes its stale-snapshot-derived `winner.reassemble_prompt()` over the user's fresh edit.
- **Root cause**: The design assumes a cycle is the sole mutator of a persona between snapshot and promotion. There is no optimistic-concurrency token (e.g. `WHERE updated_at = <snapshot_ts>`), no exclusive evolution lock, and no check that the persona is idle/unedited before overwriting.
- **Impact**: data loss / corruption — a good (or freshly user-authored) persona gets silently replaced by a stale or worse variant; concurrent cycles can interleave-promote producing a frankenpersona.
- **Fix sketch**: Add a per-persona evolution mutex (mirror `healing_personas`) acquired in the command before spawning, released via Drop guard. Make `promote_variant` a transaction that re-reads the persona and aborts/abandons promotion if `updated_at` changed since the snapshot (optimistic lock), so a concurrent edit/cycle can never be silently overwritten.

## 3. Promotion compares two incompatible fitness metrics — promotes variants that are not actually better
- **Severity**: high
- **Category**: silent-failure / state-corruption
- **File**: src-tauri/src/engine/evolution.rs:309-310, 334, 346-349
- **Scenario**: The cycle computes `incumbent_avg` by running the incumbent through the same test scenarios as the variants (line 309-310), and computes each `variant_avg` the same way (line 323-324) — a like-for-like comparison. But the promotion decision at line 348 ignores `incumbent_avg` entirely and instead does `improvement = best_variant_score - incumbent_fitness.overall`, where `incumbent_fitness.overall` is the *historical knowledge-derived* fitness (evolution.rs:178 → genome.rs:490). Scenario composite scores and knowledge fitness are different scales computed from different data; subtracting them is meaningless. `incumbent_avg` is computed (costing real CLI calls) and then never read.
- **Root cause**: Two fitness notions (scenario-eval composite vs. historical-knowledge fitness) were both wired in and the wrong baseline is used in the comparison. The promotion gate is therefore against a noisy, scale-mismatched number.
- **Impact**: corruption — a worse variant can clear the threshold (`best_variant_score` high, `incumbent_fitness.overall` low because the persona has little/no `cost_quality` knowledge → defaults to 0.0, genome.rs:500-507) and get promoted over a perfectly good incumbent; conversely a genuine improvement can be rejected. For a brand-new persona with no knowledge rows, `incumbent_fitness.overall == 0.0`, so *any* variant scoring above the threshold is promoted regardless of whether it beats the incumbent's own scenario score.
- **Fix sketch**: Compare like with like: gate on `best_variant_score - incumbent_avg >= threshold` (both scenario composites), and drop the unused `incumbent_fitness` from the decision (keep it only for the summary). Add a guard refusing promotion when fewer than N scenarios actually executed (see finding 6).

## 4. Breeding pipeline loads parents that may have been deleted/edited after validation; offspring fitness never evaluated
- **Severity**: high
- **Category**: race-condition / silent-failure
- **File**: src-tauri/src/commands/execution/genome.rs:80-118, 124-244
- **Scenario**: `genome_start_breeding` validates that all parents exist and share a project (genome.rs:80-90), then spawns `run_breeding_pipeline` which *re-loads* each parent (genome.rs:148-169). Between the synchronous validation and the background load, a parent can be deleted or have its project changed (no lock held). If a parent is deleted, the whole run is marked Failed (genome.rs:156-167) — acceptable — but if a parent's project changes, the cross-project guard from the command is gone and offspring mix personas across projects. Separately, `parent_fitness` is computed at genome.rs:172-180 and then **never used**; every offspring is persisted with `fitness_json: None, fitness_overall: None` (genome.rs:199-200) and no generation ever evaluates offspring fitness. The run reports "Completed" with a count but zero ranking data.
- **Root cause**: Validation and execution are split across the async boundary with no lock, and the "breed → evaluate → rank" loop was only half-built: breeding and persistence ship, evaluation/ranking does not, yet the run still reports success.
- **Impact**: UX degradation / silent-failure — users adopt offspring (genome.rs:292) with no fitness signal to choose between them; the `parent_fitness` sort (genome.rs:180) is dead code implying ranking that does not exist (success theater). Cross-project leakage is a lower-probability corruption.
- **Fix sketch**: Either evaluate offspring fitness in the pipeline and persist `fitness_overall` (and use `parent_fitness` for elitist selection), or mark the run status explicitly as "bred, unevaluated" so the UI does not imply ranking. Re-validate project consistency on the reloaded parents inside the pipeline.

## 5. Healing retry storm: `run_healing_analysis` re-scans the same recent failures every call with no cooldown
- **Severity**: high
- **Category**: silent-failure / retry-storm
- **File**: src-tauri/src/commands/execution/healing.rs:51-79; src-tauri/src/engine/healing_timeline.rs:142-214
- **Scenario**: `run_healing_analysis` pulls the last 10 failures (healing_timeline.rs:154) and, for each diagnosable one, returns a retry request that the command immediately schedules (healing.rs:62-70). The only brake is per-execution `retry_count >= MAX_RETRY_COUNT` (mod.rs:1420). Issue creation is deduped per persona+execution (healing_timeline.rs:231-239), but *retry scheduling* is driven off the failure list, not off whether a retry is already in flight. If the user (or a polling UI) calls `run_healing_analysis` repeatedly, or several of the 10 recent failures are retries of the same original, multiple delayed retries for the same `original_exec_id` can be spawned before any completes — each retry that fails feeds the next scan, amplifying load.
- **Root cause**: Retry scheduling has no "already scheduled / in-flight for this original execution" guard and no per-persona analysis cooldown; the dedup that exists protects the issues table, not the spawn of `spawn_delayed_retry`.
- **Impact**: UX degradation / resource exhaustion — burst of concurrent CLI retries for the same failure, burning tokens/cost and competing for the persona's concurrency slots; can mask the real fix while looking "busy healing".
- **Fix sketch**: Track in-flight retries per `original_exec_id` (a set, like `healing_personas`) and skip scheduling if one is pending; add a minimum interval between `run_healing_analysis` runs per persona; collapse multiple recent-failure rows that share `retry_of_execution_id` to a single retry decision.

## 6. Evolution treats scenario-execution failures as zero-score, letting an infra outage promote a variant
- **Severity**: medium
- **Category**: edge-case / silent-failure
- **File**: src-tauri/src/engine/evolution.rs:451-477, 334-349
- **Scenario**: `evaluate_persona_on_scenarios` counts a scenario whose `execute_scenario` errored as a 0.0 contribution but still increments `count` (evolution.rs:462-470). If the CLI/provider is down or rate-limited *while scoring the incumbent* but recovers before scoring a variant (or vice versa), the incumbent averages near 0 while a variant gets a real score — making the variant look dramatically better. Combined with finding 3 (incumbent baseline already mis-sourced) and the `best_variant_score` starting at 0.0 with strict `>` (line 334), a single lucky variant during a flaky window clears the threshold and is promoted. There is no check that incumbent and variants were evaluated under comparable success conditions, nor a minimum count of *successful* (non-errored) scenarios.
- **Root cause**: Execution errors are conflated with genuine 0-quality outputs, and the promotion gate has no notion of evaluation validity — it assumes every score reflects persona quality, not transient infra state.
- **Impact**: corruption — a transient provider outage during evaluation can promote an unrelated/worse variant over a healthy incumbent.
- **Fix sketch**: Track executed-vs-errored counts separately; abort the cycle (status Failed, no promotion) if a meaningful fraction of scenarios errored for either incumbent or variants, or if the incumbent could not be scored at all. Require the same set of scenarios to succeed for both sides before comparing.
