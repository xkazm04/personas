## 1. Inbound smee/webhook events bypass the event-source rate limiter entirely
- **Severity**: critical
- **Category**: silent-failure / recovery-gap (DoS / execution storm)
- **File**: src-tauri/src/engine/smee_relay.rs:448 ; src-tauri/src/db/repos/communication/events.rs:166 ; src-tauri/src/commands/communication/events.rs:60-86
- **Scenario**: A user attaches a smee.io channel. smee.io offers no authenticity guarantee — anyone who learns the channel URL POSTs to it (the code itself warns this at smee_relay.rs:633-637). An attacker (or a misbehaving CI hook) floods the channel. If no `allowed_repos` allowlist is configured (the default — `allowed_repos2.is_none()` is the warned-but-permitted path), every payload is relayed straight into `event_repo::publish`.
- **Root cause**: The per-source event rate limiter (`state.rate_limiter.check(&rate_key, event_source_max, EVENT_SOURCE_WINDOW)`) lives ONLY in the Tauri *command* wrappers `publish_event` and `test_event_flow`. Every internal producer — the smee relay (`relay_sse_core`), the scheduler tick, chain cascade (`evaluate_chain_triggers`), and backfill — calls `event_repo::publish()` directly, which performs only `validate_event_input` (type/size checks) with NO rate limiting. The rate limiter is an IPC-boundary guard, not an event-bus guard, so the highest-volume, attacker-reachable producer is the one path with no throttle.
- **Impact**: Unbounded `persona_events` inserts → unbounded event-bus dispatch. The downstream per-persona hourly cap (`schedule_hourly_cap_exceeded`) only applies to `trigger_type == "schedule"`; the smee→event_listener routing path is not covered, and the cascade guard (`running_count > 0`) only serializes per persona, not the insert flood. Result: disk/DB exhaustion, token-spend blowout, execution-queue saturation from a single leaked URL.
- **Fix sketch**: Move event-source rate limiting INTO `event_repo::publish` (or a thin `publish_rate_limited` all internal producers must call), keyed by `source_type` + `source_id`, so the throttle is a property of the bus, not of one entry point. Make smee-origin events fail-closed without an allowlist rather than warn-and-accept.

## 2. Smee relay de-duplication is process-local RAM — app restart replays the whole channel
- **Severity**: high
- **Category**: state-corruption / recovery-gap (duplicate executions)
- **File**: src-tauri/src/engine/smee_relay.rs:202-228, 644, 430-436
- **Scenario**: smee.io replays the channel's recent event history on every new SSE connection. The app processes 50 webhooks, then the user quits and relaunches (or the process crashes). On reconnect, smee replays history; `RelayDedup` was just constructed fresh (`let mut dedup = RelayDedup::default()` at smee_relay.rs:644), so `last_event_id` is `None` and `seen_set` is empty. Every replayed webhook is treated as new and re-published → duplicate persona executions for work already done.
- **Root cause**: Dedup state (`seen_set`, `seen_order`, `last_event_id`) is in-memory and per-spawn. The comment claims it "persists across reconnects" — true for in-process reconnects, but it does NOT persist across process lifetimes, and there is no idempotency key on `persona_events` (publish always generates a fresh UUID). The 512-entry FIFO cap also silently forgets older deliveries within a long-lived session, so a slow replay past 512 distinct ids re-fires too.
- **Impact**: Duplicate downstream executions (duplicate PRs/messages/side-effects) after every restart for any channel with retained history — exactly the class of bug the in-process dedup was added to stop, just shifted to the restart boundary.
- **Fix sketch**: Persist the dedup key (SSE id / `x-github-delivery` / content hash) to a small table with a unique index, or store `last_event_id` per relay in `smee_relays` and always send it as `Last-Event-ID` on the first connect after launch. Make `publish` idempotent on an optional `dedup_key` (INSERT OR IGNORE on a unique index) so duplicates collapse at the DB layer regardless of producer.

## 3. `extract_chain_metadata` panics on a non-char-boundary byte slice of an attacker-influenced payload
- **Severity**: high
- **Category**: panic
- **File**: src-tauri/src/engine/chain.rs:466
- **Scenario**: A chain/event payload that fails `serde_json::from_str` (e.g. a smee webhook body that is technically malformed JSON, or a truncated payload) reaches `extract_chain_metadata`. The error branch logs `payload_prefix = %&payload[..payload.len().min(200)]`. If the payload is ≥200 bytes and byte offset 200 lands in the middle of a multi-byte UTF-8 sequence (one emoji or any non-ASCII char straddling offset 200 — trivial to arrange), slicing a `&str` at a non-char-boundary index panics: "byte index 200 is not a char boundary".
- **Root cause**: Byte-index slicing (`&s[..n]`) on a `&str` assumes `n` is a char boundary. The `.min(200)` bounds the length but not the boundary. The slice is only taken on the failure path, so it is never hit by happy-path tests, and the input is externally influenced (webhook/chain payloads), making it a latent panic landmine.
- **Impact**: Panics the chain-metadata extraction call. Depending on caller context this aborts the dispatching task (the panic boundary in `subscription.rs` counts it as a subscription crash) — a single crafted payload can repeatedly crash the event-bus/dispatch tick, stalling all chain trace propagation.
- **Fix sketch**: Use a char-safe truncation helper (e.g. `payload.char_indices().take_while(|(i,_)| *i < 200).last()` to find a safe end, or `payload.chars().take(200).collect::<String>()`). Add a `truncate_str_chars` util and forbid raw `&s[..n]` on externally-sourced strings via a lint/grep gate.

## 4. Interval schedules drift forward every tick and never backfill missed slots
- **Severity**: medium
- **Category**: edge-case / silent-failure (cron drift / missed ticks)
- **File**: src-tauri/src/engine/scheduler.rs:77-90 ; src-tauri/src/engine/background.rs:1682-1692, 1803-1807
- **Scenario**: A trigger configured with `interval_seconds: 60`. The scheduler subscription does not tick exactly on time (idle-mode cadence, slow ticks, system busy). Each fire computes the next slot as `now + interval_seconds` from the *actual* tick instant (scheduler.rs:81), not from the previous scheduled slot. Over hours, a "every 60s" trigger steadily drifts to 65s, 70s, etc., accumulating skew. Separately, if the app is offline for an hour, the interval trigger fires exactly once on the startup overdue sweep (no catch-up) because the backfill path (`backfill_cap > 1`) is gated on `TriggerConfig::Schedule { max_backfill: Some(n), .. }` — which the cron form sets, but interval schedules realistically never carry.
- **Root cause**: Next-fire time is derived from wall-clock-at-fire rather than from the intended grid (`prev_slot + interval`), so latency compounds into permanent drift. Cron schedules avoid this because `next_fire_time_*` snaps to the cron grid; interval schedules have no grid to snap to and no catch-up.
- **Impact**: "Every N seconds" silently becomes "every N + jitter" and degrades the longer the app runs; missed intervals during downtime are dropped, not replayed. Time-sensitive interval automations fire late and lose events with zero signal.
- **Fix sketch**: Anchor interval next-fire to `last_trigger_at + interval` (advancing in whole `interval` steps until `> now`) instead of `now + interval`, and reuse `compute_missed_backfill_slots` for interval triggers so catch-up is symmetric with cron.

## 5. Client-side trigger rate-limit slice is dead code presented as a live dashboard (success theater)
- **Severity**: medium
- **Category**: silent-failure (success theater / misleading UX)
- **File**: src/stores/slices/pipeline/triggerSlice.ts:185-290 ; src/features/triggers/sub_speed_limits (RateLimitDashboard)
- **Scenario**: A user opens the Rate Limits tab to confirm their throttle/cooldown config is taking effect. `recordTriggerFiring` and `recordTriggerComplete` — the only functions that populate `triggerRateLimits` (firing timestamps, concurrent count, cooldown, queue depth) — are never called anywhere in `src/` (verified by grep: the only references are the definitions and doc comments). So `triggerRateLimits` is permanently `{}`, `getRateLimitSummary` always returns `{ totalQueued: 0, totalThrottled: 0, throttledTriggerIds: [] }`, and the dashboard reports "nothing throttled" no matter what the real Rust-side limiter (`schedule_hourly_cap_exceeded`, `rate_limiter.check`) is doing.
- **Root cause**: The actual rate limiting is enforced server-side in Rust; this client slice was built to mirror it but was never wired to any firing signal (no event listener feeds it). The two sources of truth diverged silently — the same drift the `DeadLetterConfig` ts-rs binding (events.rs:263) was introduced to prevent, recurring in the rate-limit surface.
- **Impact**: Users trust a dashboard that structurally cannot show a throttled trigger; throttling that IS happening in Rust (hourly cap pauses, cooldowns) is invisible, eroding trust and hiding real backpressure during incidents.
- **Fix sketch**: Either drive the slice from a real signal (subscribe to a Rust-emitted firing/throttle event, or fetch the server-side limiter state), or delete the slice and render the dashboard from authoritative backend data. Add a "no code path populates this state" lint or a smoke test asserting the dashboard reflects a real throttle.

## 6. Active-window gate advances the cron schedule but drops the fire — windows silently swallow events
- **Severity**: high
- **Category**: silent-failure (events silently swallowed)
- **File**: src-tauri/src/engine/background.rs:1606-1618 ; src-tauri/src/db/models/trigger.rs:164-199
- **Scenario**: A schedule trigger has an `active_window` of 09:00–18:00 Mon–Fri and a cron of `0 * * * *` (hourly). At 20:00 the scheduler finds it due, sees it is outside the active window, computes the *next* fire time and calls `mark_triggered` (advancing the schedule + version), then `continue`s WITHOUT publishing. The overnight fires are not deferred to window-open — they are advanced past and discarded. Worse, `compute_next_from_config` only snaps to the cron grid; it does not consult `seconds_until_next_open`, so the "next" time it lands on may itself be outside the window, and that slot is dropped too on the next tick — repeatedly burning slots until one happens to fall inside the window.
- **Root cause**: The active-window check treats out-of-window as "skip and advance" rather than "hold until the window opens." `ActiveWindow::seconds_until_next_open` exists (trigger.rs:205) but is never used by the scheduler tick to set `next_trigger_at` to the window boundary. So an active window silently converts a guaranteed-delivery schedule into best-effort, dropping every fire that the cron grid places outside the window.
- **Impact**: Users who set business-hours windows on an hourly/by-minute schedule lose most of their fires with no DLQ entry and no error — the trigger "works" inside the window and silently eats everything outside it. Boundary fires (exactly at 18:00, overnight windows crossing midnight) are especially lossy.
- **Fix sketch**: When outside the active window, set `next_trigger_at` to `max(cron_next, now + seconds_until_next_open)` so the schedule resumes at window-open instead of skipping grid slots; or evaluate the cron grid forward to the first slot that is inside the window. Emit a debug/event breadcrumb counting suppressed-by-window fires so the loss is observable.
