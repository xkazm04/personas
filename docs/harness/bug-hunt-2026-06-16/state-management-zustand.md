## 1. `goals` array clobbered when toggling project / all-projects scope (shared-array-overwrite)
- **Severity**: High
- **Category**: ⚡ Race condition — shared slice array overwritten by sibling consumers
- **File**: `src/stores/slices/system/devToolsProjectSlice.ts:201` (`fetchGoals`) and `:211` (`fetchAllGoals`); consumer `src/features/teams/sub_goals/GoalsPage.tsx:109-111`
- **Scenario**: `GoalsPage` calls `fetchAllGoals()` (all projects) when `crossProject` is on and `fetchGoals(activeProjectId)` (one project) when off — both write the SAME top-level `goals` array. Toggling the scope (or switching `activeProjectId`) fires a new fetch before the previous resolves. Backend latency differs (all-projects scan is heavier), so a slow `fetchAllGoals` can resolve AFTER a fresh `fetchGoals(project)` and overwrite the project-scoped board with every project's goals — or vice versa. The Kanban then shows goals that don't belong to the selected scope.
- **Root cause**: Unlike `overviewSlice.fetchGlobalExecutions`/`fetchPersonaSummaries` (which guard with a module `seq` counter and bail when superseded), neither `fetchGoals` nor `fetchAllGoals` has any sequence/scope guard. The last write wins regardless of which request was issued last. The doc-comment on `fetchAllGoals` ("Writes the same `goals` array, so updateGoal/drag/progress all still work") acknowledges the sharing but not the race. `fetchKpis`/`fetchAllKpis` (`:357`/`:367`) share the identical defect on the `kpis` array (consumers `KPIsPage.tsx:39`, `ContextMapPage.tsx:152`).
- **Impact**: Wrong goals/KPIs displayed for the active scope; drag-reorder and progress edits then operate on rows from another project, and `reorderGoals`/`updateGoal` persist those edits to the backend — silent cross-project data corruption from the user's view.
- **Fix sketch**: Add a `fetchGoalsSeq` module counter (and `fetchKpisSeq`) incremented at call start; capture `const seq = ++fetchGoalsSeq` and `if (seq !== fetchGoalsSeq) return;` immediately after the `await` before `set(...)`. Optionally stamp a `goalsScope: string | null` ('all' | projectId) into state so consumers can ignore data for the wrong scope.

## 2. `contexts` is a single array but `fetchContexts(projectId, groupId)` writes a filtered subset into it
- **Severity**: Medium
- **Category**: 🔮 Latent failure — shared array overwrite (group-filtered vs full list)
- **File**: `src/stores/slices/system/devToolsContextSlice.ts:103` (`fetchContexts`)
- **Scenario**: `fetchContexts(projectId, groupId?)` calls `listContexts(projectId, groupId)` and unconditionally writes the result to the single `contexts` array. Today every caller (`ContextMapPage.tsx:63`, `IdeaScannerPage.tsx:400`, `useDevToolsActions.ts:31`, `ContextKpiDashboard.tsx:49`) passes NO `groupId`, so the bug is dormant. The moment any panel calls it group-scoped (the signature invites it), that panel replaces the whole-project `contexts` array with just one group's contexts — and `IdeaScannerPage` reads `useSystemStore.getState().contexts` immediately after its own `fetchContexts(activeProjectId)`, so a concurrent group-scoped fetch resolving in between hands it a truncated list, silently shrinking the auto-scan's context set.
- **Root cause**: A filter parameter (`groupId`) that narrows the result set writes back to the same unfiltered state key, with no record of which scope produced it. This is the exact "shared slice array overwritten by a sibling panel" pattern flagged in earlier scans, latent here only because the second consumer hasn't been wired yet.
- **Impact**: Latent: context map / KPI dashboards / idea-scanner show a partial context list; scans run against the wrong subset. Activates as soon as a group-filtered consumer ships.
- **Fix sketch**: Either drop the `groupId` param from the slice action and filter in the selector/component (keep `contexts` always whole-project), or store group-filtered results under a separate keyed map (`contextsByGroup: Record<string, DevContext[]>`) rather than overwriting the canonical `contexts`.

## 3. `_pendingReadIds` Set never reset on `fetchMessages(reset)` — permanent no-op guard for recycled IDs
- **Severity**: Medium
- **Category**: 💀 Silent failure — stale in-flight guard survives list replacement
- **File**: `src/stores/slices/overview/messageSlice.ts:58` (`fetchMessages`), guard at `:88`
- **Scenario**: `markMessageAsRead` adds the id to `_pendingReadIds` and removes it only on the backend success/rollback callback. If that IPC never resolves (app backgrounded, Tauri call dropped, navigation tears down before resolution), the id is stranded in the Set. A subsequent `fetchMessages(reset=true)` replaces `messages` wholesale but does NOT clear `_pendingReadIds`. The guard `if (... || _pendingReadIds.has(id)) return;` then silently no-ops every future "mark as read" for that message id — the unread dot never clears no matter how many times the user clicks.
- **Root cause**: `_pendingReadIds` is treated as ephemeral in-flight state but has no reconciliation against the freshly-fetched list. The reset path rebuilds the authoritative `messages` array yet leaves the stale pending-id bookkeeping intact. (It is correctly NOT in `partialize`, so this is in-session only, but the leak persists across every refetch within a session.)
- **Impact**: Messages permanently stuck appearing unread for the affected ids; `unreadMessageCount` can drift positive against the visible state. User-facing "I clicked read and nothing happened."
- **Fix sketch**: On `fetchMessages(reset)`, prune `_pendingReadIds` to ids still present and still unread in the new list (`new Set([..._pendingReadIds].filter(id => next.some(m => m.id === id && !m.is_read)))`), or clear it entirely on reset since the server response is authoritative.

## 4. `fetchExecutions` `finally` unconditionally writes `executionsLoading:false`, racing a sibling persona fetch
- **Severity**: High
- **Category**: ⚡ Race condition — concurrent actions clobbering loading flag + shared list
- **File**: `src/stores/slices/agents/executionSlice.ts:499` (`fetchExecutions`, finally at `:544`)
- **Scenario**: Dedup keys only on a SINGLE in-flight persona (`inflightFetch.personaId === personaId`). Rapidly selecting persona A then B issues two un-deduped fetches. `finishExecution`/`cancelExecution`/`selectPersona` all call `fetchExecutions(selectedPersona.id)`, so overlap is routine. Fetch A resolving last writes `executions: <A's rows>` + `executionsPersonaId: A`, clobbering B's freshly-loaded list. Separately, A's `finally` sets `inflightFetch = null` and `executionsLoading:false` even if B is still loading — the spinner vanishes while B's data is mid-flight, and B's own `finally` then nulls an `inflightFetch` that may already point at a third request.
- **Root cause**: No per-call sequence guard (the slice relies solely on single-slot dedup keyed by personaId, which does not cover A→B→A interleavings). The `finally` block's `inflightFetch = null` and `executionsLoading:false` are unconditional rather than "only if I'm still the active request." `executionsPersonaId` records whose data is shown but nothing rejects a stale write against it.
- **Impact**: Activity list shows the previous persona's executions under the new persona's header; loading spinner flickers off prematurely; cache (`executionsCache[personaId]`) is correct but the visible `executions` is wrong until a manual refresh.
- **Fix sketch**: Add a module `fetchExecSeq`; capture at entry and guard the `set` after `await` (`if (seq !== fetchExecSeq) return`). In `finally`, only clear `inflightFetch` / `executionsLoading` when `inflightFetch?.personaId === personaId` (i.e., you're still the owner), mirroring the `seq !== fetchGlobalSeq` pattern already used in `overviewSlice`.

## 5. `processStarted`/`processQueued` desync `activeProcessCount` when a key flips status family
- **Severity**: Low
- **Category**: 🔮 Latent failure — derived counter drift vs source map
- **File**: `src/stores/slices/processActivitySlice.ts:240` (`processStarted`), `:331` (`processQueued`)
- **Scenario**: `activeProcessCount` is a hand-maintained mirror of `Object.keys(activeProcesses).length`, incremented only when `isNew` (key absent). `processStarted` and `processQueued` both treat an EXISTING key as "not new" and leave the count unchanged — correct for count, but they also overwrite that entry's `toolCallCount`/`costUsd` back to 0 and reset `startedAt` (in `processStarted`). More importantly, the count is only ever reconciled to `Object.keys().length` inside `clearNonActive`/`reapStaleRunning`; the incremental ± paths assume every add goes through `processStarted`/`processQueued` and every removal through `processEnded`. A dropped/duplicated `processEnded` (its own doc-comment at `:177-198` describes the historical key-mismatch leak) decrements once for a row that was never counted, or fails to decrement a row that was — and nothing re-syncs until a manual clear. The titlebar dock subscribes to this primitive specifically to avoid recomputing, so the drift is what the user sees.
- **Root cause**: A derived value maintained by manual ± deltas across many entry points, trusting that adds and removes are perfectly paired. The FSM hardening (`findUniqueProcessKey`) reduces but does not eliminate mismatched-key removals, and `enrichProcess`/`updateProcessStatus` can create a row via no path that the counter knows about if a caller ever mutates the map directly.
- **Impact**: Titlebar "N running" badge drifts from reality (the documented "29 running personas that didn't match reality" class of bug) until a `clearNonActive`/reap resync; cosmetic but erodes trust in the activity dock.
- **Fix sketch**: Make the count authoritative by deriving it inside the same `set` updater that mutates the map (`activeProcessCount: Object.keys(next).length`) in `processStarted`/`processQueued`/`processEnded`, the way `clearNonActive`/`reapStaleRunning` already do — eliminating the ± bookkeeping entirely while keeping the primitive selector for the dock.
