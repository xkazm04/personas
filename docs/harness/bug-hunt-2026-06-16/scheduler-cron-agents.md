## 1. Cron "next fire" is computed from system-LOCAL time but the UI hard-labels it "UTC", and two schedulers disagree on the zone
- **Severity**: High
- **Category**: Clock/timezone bug (UTC vs local mismatch)
- **File**: `src-tauri/src/engine/scheduler.rs:72` (`next_fire_time_local`) + `src/features/overview/sub_cron_agents/components/CronAgentCard.tsx:73`
- **Scenario**: A user authors a trigger with cron `0 9 * * *` and no `timezone` field (the common/legacy case — the comment at scheduler.rs:60-66 confirms most pre-existing triggers have `timezone: None`). On a dev/prod box in `Europe/Prague` (UTC+1/+2) the agent fires at 09:00 *local*, i.e. 07:00–08:00 UTC. But `CronAgentCard.tsx:73` unconditionally renders the badge `UTC` next to every cron expression. Meanwhile the curation scheduler (`curation_scheduler.rs:98`) evaluates the *same kind* of cron with `cron::next_fire_time` — the pure-**UTC** matcher — so a curation cron `0 9 * * *` fires at 09:00 UTC while a trigger cron `0 9 * * *` fires at 09:00 local. Same syntax, two different wall-clock times, plus a UI that lies about which.
- **Root cause**: Three independent zone policies: (a) trigger path → `next_fire_time_local` when `timezone` is None; (b) curation path → `next_fire_time` (UTC, hardcoded at line 98); (c) UI → static "UTC" string. No single source of truth; the "no timezone authored" fallback silently resolves to the host's zone, which is invisible and machine-dependent.
- **Impact**: Schedules fire at the wrong displayed hour, off by the host's UTC offset; "next run" times in the UI are wrong by that offset; behavior is non-reproducible across machines (a laptop that travels timezones changes when its agents fire). Curation vs trigger divergence means identical cron strings behave differently depending on which subsystem reads them.
- **Fix sketch**: Pick one canonical interpretation. Either default unset `timezone` to UTC everywhere (make `compute_next_from_config` call `next_fire_time` not `_local`, and switch curation to honor a stored zone), or render the *actual* resolved zone in the card (`ActiveWindow::resolved_timezone_name()` already exists). At minimum, replace the static `UTC` badge with the trigger's real resolved zone.

## 2. Interval triggers accumulate unbounded drift — next fire is `now + interval`, not `scheduled + interval`
- **Severity**: High
- **Category**: Latent failure (drift accumulation)
- **File**: `src-tauri/src/engine/scheduler.rs:78-83` (`compute_next_from_config`, interval arm)
- **Scenario**: An interval trigger is configured for `interval_seconds: 300` (every 5 min). The scheduler subscription ticks every 2s (active) / 10s (idle). `get_due` returns the trigger once `next_trigger_at <= now`, and the next fire is set to `now + 300s` where `now` is the **tick time**, not the slot it was due for. With idle 10s ticks the trigger is found ~0–10s late each cycle, and that lateness is folded into the new baseline, so each successive fire slips by the tick latency. Over a day an "every 5 minutes" agent fires meaningfully fewer than 288 times, and the phase wanders. Under load (slow ticks / overrun, tracked in `SubscriptionHealth.overrun`) the slip per cycle grows.
- **Root cause**: The interval arm computes `now + Duration::seconds(secs)` (line 81) using wall-clock `now` instead of advancing from the previous scheduled `next_trigger_at`. Cron triggers don't have this problem because `next_fire_time_*` snaps to the matching minute boundary; only interval triggers free-run off `now`.
- **Impact**: Interval cadence is systematically too slow and drifts; SLAs like "poll every 60s" silently degrade to 65–70s effective. Drift is invisible (no metric), so "success theater": the agent appears healthy while firing late and less often than configured.
- **Fix sketch**: Carry the prior scheduled fire time into the computation: `next = prev_next_trigger_at + interval`, catching up in whole intervals if `now` overran by more than one period (`while next <= now { next += interval }`). This keeps phase locked to the original schedule instead of the tick clock.

## 3. DST "spring-forward" gap: a daily cron whose hour does not exist that day silently slips a day (or misfires)
- **Severity**: Medium
- **Category**: Clock/timezone bug (DST transition)
- **File**: `src-tauri/src/engine/cron.rs:468-534` (`next_fire_time_in_zone`)
- **Scenario**: Trigger `30 2 * * *` (02:30 daily) with `timezone: "America/New_York"`. On the spring-forward Sunday, local clocks jump 02:00 → 03:00, so 02:30 *does not exist* that day. The minute-stepping loop checks `matches_in_zone` for each candidate UTC instant; for the non-existent local 02:30 there is no UTC instant that `with_timezone(tz)` maps back to 02:30, so `matches_in_zone` never returns true for that day. The loop advances minute-by-minute (the skip-ahead branches don't apply because hour 2 *is* in the schedule), eventually rolling to the next day's 02:30. The 02:30 run for that day is silently dropped — no log, no backfill marker, no "missed" surfaced.
- **Root cause**: Cron has no DST-gap policy. Standard cron implementations fire the next valid instant (e.g. 03:00) when the target wall-clock time is skipped; here the matcher simply never matches the gap minute and the day is lost. The docstring at lines 463-467 claims DST gaps "fall through to minute-level advancement instead of panicking" — true for not panicking, but the *semantic* result (drop the run) is wrong.
- **Impact**: Once per year a daily/early-morning agent in a DST zone misses a run with zero diagnostics. For backfill (`compute_missed_backfill_slots`), the same gap means the slot is never enumerated, so catch-up won't recover it either. The fall-back (autumn) overlap conversely risks the matcher accepting an ambiguous local hour inconsistently.
- **Fix sketch**: After the loop fails to match an in-schedule hour for a full day, detect a DST gap (local time non-existent) and fire at the first valid instant after the gap (the transition boundary), matching POSIX-cron convention. Add a `tracing::warn!` when a scheduled hour is skipped by a transition so it's auditable.

## 4. Backfill is bounded per-tick but not across ticks — sustained thundering-herd after long downtime
- **Severity**: Medium
- **Category**: Latent failure (backfill flooding / thundering herd)
- **File**: `src-tauri/src/engine/background.rs:1503-1561` (`compute_missed_backfill_slots`) + `1727-1850` (per-tick backfill loop)
- **Scenario**: A trigger with cron `* * * * *` (every minute) and `max_backfill: 100` is offline for 3 days, then the app launches. The startup sweep + first tick emits up to `BACKFILL_HARD_CAP - 1` (99) catch-up events for the most-recent missed window, then `mark_triggered` advances `last_triggered_at` to roughly `now`. But the older ~4200 missed slots aren't discarded — `compute_missed_backfill_slots` only ever returns the **last** `BACKFILL_HARD_CAP` slots before `now` (it caps the vec then `pop()`s the newest). The per-tick hourly ceiling (`SCHEDULE_EXECUTIONS_PER_PERSONA_HOUR_DEFAULT = 60`) then breaks the loop, leaving the trigger advanced but having dumped ~60 events into the bus in one tick. With N such triggers, every one fires its burst on the same startup tick.
- **Root cause**: The cap is per-trigger-per-tick (100) and a separate per-persona-per-hour cap (60), but there is no global rate-shaping or jitter across triggers at startup; the startup overdue sweep (`background.rs:648`) runs them all synchronously. `max_backfill` semantics ("how many to catch up") interact confusingly with the hard cap and hourly ceiling — a user setting `max_backfill: 100` after 3 days down gets at most ~60 events, not 100, with no signal that the rest were abandoned.
- **Impact**: Startup spike floods the event bus (`claim_pending` batches 50/tick) and the execution queue (`queue_rejections` will climb); multiple high-cadence agents amplify it. Catch-up is silently partial and the count delivered doesn't match `max_backfill`, so operators can't reason about what actually ran.
- **Fix sketch**: Spread startup backfill with per-trigger jitter (reuse the `H`/`seed_hash` machinery) and/or a global token bucket across the sweep. Surface "dropped N older missed slots" as a healing issue (the rate-limit path already creates one) so partial catch-up is visible rather than silent.

## 5. Curation scheduler can enqueue duplicate runs when `mark_run_now` fails; relies on a different table's idempotency that doesn't apply
- **Severity**: High
- **Category**: Race condition / silent duplicate (success theater)
- **File**: `src-tauri/src/engine/curation_scheduler.rs:123-156`
- **Scenario**: The curation tick enqueues a `memory_curation_run` job (line 123), then calls `curation_schedule::mark_run_now` to stamp `last_curation_at`. If `mark_run_now` fails (DB busy / lock contention — common under the multi-subscription write load) the code logs a warning and *does not* update `last_curation_at` (lines 136-146). On the **next 60s tick** the same persona is still "due" (its `last_curation_at` is unchanged), so it enqueues a **second** `memory_curation_run` job. The inline comment claims "worker_tick's pop_next_queued atomic UPDATE prevents double-execution at the job level" — but that CAS only prevents two workers grabbing the *same* row; it does nothing about *two distinct queued rows* for the same persona. Both run, both invoke the LLM reviewer, both write `persona_memory_review_proposal` rows.
- **Root cause**: Enqueue and the "mark as fired" stamp are two non-atomic operations, and the de-dup reasoning conflates row-level claim (correct) with logical-job de-dup (absent). There is no uniqueness guard keying on (persona_id, scheduled-minute) before enqueue.
- **Impact**: Duplicate (or worse, unbounded if the DB stays contended) memory-curation runs per persona — wasted LLM spend, duplicate review proposals the user must dispose of, and a memory pipeline that double-processes. The "enqueued += 1" still increments on the failed-stamp path, so metrics report success while the system is mid-loop.
- **Fix sketch**: Do enqueue + `mark_run_now` in one transaction, or stamp `last_curation_at` *before* enqueue (re-fire on enqueue failure is the safe direction). Alternatively add a guard: skip enqueue if a `queued`/`running` `memory_curation_run` already exists for that persona, mirroring the cross-tick idempotency the comment assumes.
