# Bug Hunter — Build Sessions & PersonaMatrix

> Total: 5 findings (1 critical, 2 high, 1 medium, 1 low)
> Context: build-sessions-personamatrix | Group: Templates & Recipes

## 1. `simulate_build_draft` clobbers `personas.design_context` with no restore — abandoned simulations corrupt the row, concurrent sims of one persona race
- **Severity**: Critical
- **Category**: 🔮 Latent failure / ⚡ Race condition / 💀 Silent failure
- **File**: `src-tauri/src/commands/design/build_simulate.rs:248-254`
- **Scenario**: User opens a draft build session, clicks "dry-run a capability". The command writes a *stripped* `design_context` snapshot (only `use_cases` + `summary`, no triggers/channels/policies/credentialLinks) directly onto the persona row, then runs `execute_persona_inner`. The header doc claims "Promote will overwrite the snapshot" — but the user is free to **never promote** (close the wizard, abandon the draft). The snapshot then persists permanently. Worse: a persona can have multiple concurrent draft sessions (multi-draft is a first-class feature — see `matrixBuildSlice` header). Two sessions for the *same* persona simulating in parallel both `UPDATE personas SET design_context = ?` on the single shared column; the second write can land between the first's write and its `execute_persona_inner` read, so simulation A silently runs against simulation B's snapshot.
- **Root cause**: The simulation snapshot is stored on a shared, persistent persona column rather than passed in-memory to `execute_persona_inner`. Recovery depends entirely on a promote that may never happen, and there is no per-session isolation of the snapshot. The phase-machine is deliberately *not* mutated (per the doc), so nothing marks the row as carrying a throwaway snapshot.
- **Impact**: A persona that was previously promoted (real design_context) but is re-simulated from a stale draft session loses its live triggers/channels/policies in `design_context` until the next promote — a downgraded, "broken trigger→use_case UUID" row that other readers (`getPersonaDetail`, matrix display) consume as truth. Concurrent same-persona sims produce wrong dry-run output the user trusts to decide "safe to promote."
- **Fix sketch**: Don't persist the snapshot. Thread an optional in-memory `design_context_override: Option<&str>` into `execute_persona_inner`/`simulate_use_case` for the simulation path. If a row write is unavoidable, capture `prior_design_context` (already read at line 214) and restore it in a `Drop`/`finally` guard after the execution returns, and serialize same-persona simulations behind a per-persona lock.

## 2. `test_build_draft` TOCTOU on phase + per-persona report clobber under concurrent/double test
- **Severity**: High
- **Category**: ⚡ Race condition / 💀 Silent failure
- **File**: `src-tauri/src/commands/design/build_sessions.rs:582-710`
- **Scenario**: `test_build_draft` reads the session (line 582), calls `validate_transition(Testing)` (line 587), then in a *separate* statement flips phase to `Testing` (line 703). Between the read and the write there is no lock and no compare-and-set. Two `test_build_draft` invocations for the same session (rapid double-click on "Test", or a hydration-triggered auto-test racing a user click) both read `draft_ready`/`test_complete`, both pass validation, and both run `run_tool_tests` against real tool APIs. Both then write `personas.last_test_report` (line 728) — the later finisher silently overwrites the earlier report, and the `TestReportModal` shows whichever won the race, not the test the user is looking at. Note `validate_transition` even explicitly permits `Testing -> Testing` (build_session.rs:84), so re-entry is not rejected.
- **Root cause**: Phase is used as a mutual-exclusion token but read-checked and written non-atomically with no `WHERE phase = <expected>` guard on the UPDATE. There is no in-flight test registry equivalent to the build-session handle map.
- **Impact**: Duplicate real API calls (cost + possible side effects on the connected service), and a persona-level `last_test_report` that disagrees with the report returned inline — the user promotes trusting a report from a different test run.
- **Fix sketch**: Make the testing transition a conditional write: `UPDATE build_sessions SET phase='testing' WHERE id=? AND phase IN ('draft_ready','test_complete')` and bail if `rows_affected == 0`. Optionally track an in-flight test set keyed by session_id (mirroring `BuildSessionManager.sessions`) to reject concurrent tests outright.

## 3. Promote/adoption silently swallow normalization + recipe-hydration errors, producing a structurally-broken persona
- **Severity**: High
- **Category**: 💀 Silent failure / 🕳️ Edge case (malformed LLM/template JSON)
- **File**: `src-tauri/src/commands/design/build_sessions.rs:2507-2521` (and `create_adoption_session` 236-255)
- **Scenario**: During promote, `hydrate_recipe_refs` failure is logged at `warn` and the code "proceeds with un-hydrated payload" (line 2507-2513) — a `recipe_ref`-shaped use_case that failed to resolve stays as `{ recipe_ref: {...} }` and is then deserialized into `AgentIr`, silently dropping the capability's real triggers/connectors/events. Separately, in `create_adoption_session` (line 250) the normalized payload is re-serialized with `serde_json::to_string(&payload).unwrap_or(agent_ir_json)` — if serialization fails, it silently falls back to the *original un-normalized* v3-shaped string, which the downstream flat-shape pipeline mis-reads.
- **Root cause**: Trust-boundary errors on untrusted template/LLM JSON are demoted to warnings rather than surfaced. The promote happily continues toward a successful-looking `Promoted` phase even when capability data was lost in normalization.
- **Impact**: Success theater — the build reports "promoted" and the persona row is created, but it is missing triggers/event subscriptions/connectors for any capability whose recipe_ref didn't hydrate. The persona looks built but never fires its scheduled/event work; the user only discovers this much later when nothing happens.
- **Fix sketch**: On promote, treat recipe-ref hydration failure as a hard `AppError::Validation` (the user can retry/repair) rather than proceeding un-hydrated; or at minimum raise a manual-review item (the codebase already has this pattern in `gate_setup_status_on_verification`). Replace the `unwrap_or(agent_ir_json)` fallback with an explicit error.

## 4. Workflow compile aborts the whole team on a single out-of-bounds LLM index — `warnings`/`dropped_connections` recovery path is dead
- **Severity**: Medium
- **Category**: 🕳️ Edge case (malformed LLM output) / 🔮 Latent failure (no partial recovery)
- **File**: `src-tauri/src/engine/workflow_compiler.rs:111-128, 311-319`
- **Scenario**: `WorkflowCompiler::validate` hard-returns `Err` on the *first* connection whose `source_index`/`target_index` is out of bounds or a self-loop (lines 114-126), and `persist_blueprint` maps that straight to `AppError::Validation` (line 163), aborting before the transaction even opens. The LLM-produced `TopologyBlueprint` only needs one bad index (off-by-one, hallucinated member count) to throw away an otherwise-valid N-persona team. Meanwhile `CompiledWorkflow` carries `warnings: Vec<String>` and `dropped_connections: usize` fields (lines 60-65) that are documented as "skipped invalid connections" but are *always* set to `Vec::new()` / `0` (lines 317-318) — the graceful-degradation contract advertised in the type is never honored.
- **Root cause**: Validation is fail-closed (reject everything) while the result type promises fail-soft (drop the bad edges, warn, keep the team). The two disagree, and the LLM is exactly the kind of untrusted source that emits one-off bad indices.
- **Impact**: A user describing a 5-persona workflow gets a hard "Connection[3] has out-of-bounds indices" error and no team at all, instead of a 5-member team with one dropped edge plus a warning. Wasted LLM topology cost and a dead-end UX on a recoverable input.
- **Fix sketch**: Change `persist_blueprint` to filter invalid/self-loop connections into `dropped_connections` + `warnings` and persist the valid remainder (members are unaffected by a bad connection index), reserving the hard error for the case where *zero* valid members exist. Keep `validate` as a strict checker but call it per-connection to partition rather than abort.

## 5. `handleBuildCellUpdate` persona-name write via deferred microtask can clobber a concurrent rename, and "8 vs 9" progress drift
- **Severity**: Low
- **Category**: ⚡ Race condition / 💀 Silent failure (lost progress accuracy)
- **File**: `src/stores/slices/agents/matrixBuildSlice.ts:659-664`; runner `src-tauri/src/engine/build_session/runner.rs:1299,1032`
- **Scenario**: When `agent_ir` arrives, the slice schedules `queueMicrotask(() => { const cur = get(); ... set({ personas: ... }) })` to rename the persona from the IR. The read of `cur.personas` and the subsequent `set` are not atomic across the microtask boundary — if the user renames the persona (or another build session for the same persona resolves) between the schedule and the run, the microtask overwrites that rename with the (possibly stale) IR name. Separately, the runner enters `DraftReady` on `resolved_count >= 8` (runner.rs:1299) and emits status with `total_count = 9`, while the per-cell progress event computes `resolved_cells.len() / 9.0` (runner.rs:1032) and the frontend `completeness` divides by `ALL_CELL_KEYS.length`. The 8-vs-9 mismatch means a "complete" draft can show as ~89% and a session can flip to draft_ready while the progress bar never reaches 100% (or hits 100% a cell early).
- **Root cause**: Deferred cross-slice write reads state outside the committing `set`; and the dimension count is a magic number duplicated as both 8 (gate threshold) and 9 (denominator) with no shared constant.
- **Impact**: Rare silent loss of a user's rename during an active build; persistent minor "success theater"/progress-bar drift that erodes trust in the build UI's completeness signal.
- **Fix sketch**: Do the persona-name update inside the same `set` updater (it already iterates personas elsewhere) instead of a microtask, comparing against the current name only. Extract the dimension count into one `TOTAL_BUILD_DIMENSIONS` constant shared by the gate threshold, the status `total_count`, and the progress denominator.
