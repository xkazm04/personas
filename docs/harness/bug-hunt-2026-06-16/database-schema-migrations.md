# Bug Hunter — Database Schema & Migrations

> Total: 5 findings (1 critical, 2 high, 1 medium, 1 low)
> Context: database-schema-migrations | Group: Data & Persistence

## 1. FK-hygiene rebuild turns off foreign keys *inside* a transaction — a no-op that lets DROP TABLE cascade-wipe child rows on legacy upgrades
- **Severity**: Critical
- **Category**: Latent failure / silent data loss (non-idempotent-path data corruption on upgrade)
- **File**: `src-tauri/src/db/migrations/fk_hygiene.rs:80-85`
- **Scenario**: An existing (pre-FK-hygiene) install upgrades. `recreate_with_fk` runs to retrofit a CASCADE/SET-NULL FK onto e.g. `persona_messages`. It opens `let tx = conn.unchecked_transaction()?;` (line 80) and only THEN executes `tx.execute_batch("PRAGMA foreign_keys = OFF;")` (line 85). Later it does `DROP TABLE persona_messages;` (line 119).
- **Root cause**: SQLite documents `PRAGMA foreign_keys` as **a no-op while a transaction is active** — it can only be changed in autocommit mode. The connection customizer (`mod.rs:136`) sets `foreign_keys = ON` on acquire, so FKs remain ON for the whole rebuild. With FK enforcement live, `DROP TABLE persona_messages` fires the `ON DELETE CASCADE` declared by *other* tables that already reference it (e.g. `persona_message_deliveries.message_id → persona_messages(id)` on a legacy DB that had picked up that FK earlier, or any table whose FK points at the dropped table), implicitly deleting child rows before the `_new` table is renamed in. The contrasting executions rebuild (`incremental.rs:87`) sets the pragma via `FkDisabledGuard` *before* `ddl_step` opens its transaction and is correct — proving the author's own intended pattern.
- **Impact**: On real legacy upgrades the rebuild either aborts mid-way (FK violation on DROP of a table still referenced) or silently cascades child rows away. The `pragma_foreign_key_check` guard at line 133 runs *inside the same transaction* where FKs are still effectively ON, so it can also misbehave. Fresh installs are unaffected (SCHEMA already declares the FKs, so the line-73 count guard early-returns) — which is exactly why every test passes (tests use `init_test_db`, a fresh DB; the legacy rebuild path is explicitly untested per the comment at lines 489-491).
- **Fix sketch**: Move `PRAGMA foreign_keys = OFF` out of the transaction. Either run it in autocommit before `unchecked_transaction()`, or wrap the whole rebuild in `FkDisabledGuard::new(conn)` (as the executions rebuild does) and only open the inner DDL transaction afterward. Add a legacy-shape test: build the old table without the FK, insert parent+child rows, run the sweep, assert rows preserved and FK present.

## 2. Incremental table rebuilds use `INSERT INTO ..._new SELECT *` — a silent corruption trap when column order/count drifted
- **Severity**: High
- **Category**: Silent failure / latent corruption
- **File**: `src-tauri/src/db/migrations/incremental.rs:337` (persona_triggers chain rebuild) and `:931` (event_listener rebuild)
- **Scenario**: The first `persona_triggers` rebuild copies via `INSERT INTO persona_triggers_new SELECT * FROM persona_triggers;` (line 337). If a legacy DB had `persona_triggers` with columns in a different order, or had picked up an extra `ALTER ... ADD COLUMN` (e.g. `use_case_id`) before this rebuild ran, `SELECT *` maps source columns positionally into the `_new` table whose explicit column order is fixed by the CREATE. Column count/order mismatch either errors out or — worse — shifts values into the wrong columns.
- **Root cause**: Positional `SELECT *` copy across two independently-authored column lists. The codebase's own `fk_hygiene.rs::recreate_with_fk` deliberately rejects this pattern (its doc at lines 44-47 mandates an explicit `columns_csv` "so reordering or adding columns in the new shape doesn't silently corrupt the copy"), but the two `persona_triggers` rebuilds in `incremental.rs` still use `SELECT *`. Migration ordering makes this real: `use_case_id` is added to `persona_triggers` at line 473, and the event_listener rebuild at line 931 lists `use_case_id` explicitly — but the chain rebuild at line 337 (older) does not, so the two rebuilds disagree on shape depending on which legacy state a DB is in.
- **Impact**: Trigger configs land in the wrong columns or the migration aborts startup on a subset of upgrade paths (version-skip / out-of-order legacy DBs). Hard to reproduce because fresh installs always have canonical column order.
- **Fix sketch**: Replace both `SELECT *` copies with explicit column lists matching the `_new` shape (same discipline as `recreate_with_fk`). Detect column presence and build the copy list dynamically, or gate the whole rebuild on `pragma_table_info` to confirm the source shape before copying.

## 3. Migrations run statement-by-statement (no outer transaction) — a crash mid-`run_incremental` leaves a permanently half-migrated DB
- **Severity**: High
- **Category**: Latent failure (non-atomic multi-step migration) / brick-on-upgrade
- **File**: `src-tauri/src/db/migrations/incremental.rs:33-38` (`ddl_step`) and `mod.rs:230-234` (orchestration in `init_db`)
- **Scenario**: `init_db` calls `migrations::run` then `migrations::run_incremental` (mod.rs:232-233). `run_incremental` performs ~80 independent `ddl_step` calls, each its own `unchecked_transaction()` + commit. A power loss / panic / disk-full between step 40 and 41 commits the first 40 and leaves the rest unapplied. There is no migration-version row, so on next boot the runner re-evaluates each step from scratch.
- **Root cause**: Idempotency is per-step via `has_column`/`has_table`/FK-count probes — there is no schema_version table and no outer transaction spanning the whole run. Most steps re-probe and skip correctly, BUT several steps are *order-dependent and not individually re-entrant after a partial state*: e.g. the credential field split (`migrate_blob_credentials_to_fields` at incremental.rs:898) → `clear_legacy_credential_blobs` (line 905) → `assert_credential_blob_invariant` (line 906). If the process dies after fields are written but before blobs are cleared, the next boot's `migrate_blob_credentials_to_fields` no-ops (fields already exist) and `clear_legacy_credential_blobs` recovers — but the analogous CHECK-widening rebuilds (n8n at :258, executions in `rebuild_executions_table_with_incomplete_status`) leave a `*_new` staging table behind; only some have `DROP TABLE IF EXISTS *_new` belt-and-braces guards (n8n line 259 does; the FK-hygiene helper does not pre-drop `_new`).
- **Impact**: A crash during the long incremental run can produce a DB that re-runs cleanly on some paths but trips a `CREATE TABLE persona_triggers_new` "table already exists" (no `IF NOT EXISTS`/pre-drop on some rebuilds) on others, aborting every subsequent boot — an effectively bricked upgrade requiring manual DB surgery.
- **Fix sketch**: Add `DROP TABLE IF EXISTS <table>_new;` at the top of every rebuild batch (the executions rebuild does this at incremental.rs:125; make it universal). Longer term, introduce a `schema_migrations(id, applied_at)` ledger and record applied step ids so partial runs resume deterministically rather than re-deriving state from schema shape.

## 4. `QueryBuilder::build_clauses` mis-numbers `OFFSET` when `LIMIT`/`OFFSET` interleave with later params (e.g. `where_raw` after `limit`)
- **Severity**: Medium
- **Category**: Silent failure (wrong SQL from query builder on edge input)
- **File**: `src-tauri/src/db/query_builder.rs:282-288`
- **Scenario**: `build_clauses` assumes LIMIT and OFFSET "occupy the last 1-2 positions in self.params" and emits `LIMIT ?{total-1} OFFSET ?{total}`. This holds only if `.limit()`/`.offset()` are the final params pushed. But the builder exposes `push_param`, `where_raw`, and `set_opt` as public mutators with no ordering guard. A caller that does `qb.limit(50); qb.where_raw(...)` (or appends a param after offset) shifts the real LIMIT/OFFSET bindings, so the generated `?N` indices point at the wrong values.
- **Root cause**: LIMIT/OFFSET bindings are tracked by boolean flags (`has_limit`/`has_offset`) and reconstructed positionally at build time from `params.len()`, instead of capturing each one's actual index at the moment it is pushed. The WHERE/SET/IN builders all capture `next_idx()` at push time and are safe; LIMIT/OFFSET are the lone exception.
- **Impact**: Pagination silently binds the wrong values (e.g. a string filter value used as the LIMIT count → `LIMIT 'abc'` errors, or an int used as OFFSET → wrong page / over-read). Not an injection (still parameterized) but produces incorrect result sets or runtime SQL errors that look like data bugs. No SQL injection found — all values are bound; only column/order/dir are interpolated and are documented as caller-validated.
- **Fix sketch**: Store `limit_idx: Option<usize>` / `offset_idx: Option<usize>` captured when `.limit()`/`.offset()` push their param, and emit those exact indices in `build_clauses`. Or assert/`debug_assert` that no params are pushed after limit/offset.

## 5. `order_by` / `order_by_multiple` interpolate column + direction with zero validation — injection if any caller forwards user input
- **Severity**: Low
- **Category**: Trust boundary / defensive gap (latent injection)
- **File**: `src-tauri/src/db/query_builder.rs:201-214`
- **Scenario**: `order_by(col, dir)` builds `format!("ORDER BY {col} {dir}")` with no escaping or allowlist; same for `order_by_multiple`. The doc comment says "callers must validate," but that contract is unenforced and invisible at the call site. Any repo that passes a frontend-supplied sort column/direction straight through (a common pattern for sortable tables) yields direct SQL injection into the ORDER BY clause (`col = "1; DROP TABLE personas--"` style, or subquery exfiltration via `(SELECT ...)`).
- **Root cause**: Column and direction are the one untrusted-shaped input the builder accepts as raw strings; the safety relies entirely on an out-of-band convention with no enforcement, type wrapper, or runtime guard.
- **Impact**: Low *today* (sampled callers appear to pass internal constants), but it is a foot-gun: a future sortable-list feature wiring `sort_by`/`sort_dir` from IPC into `order_by` opens injection with no compiler or review signal. The internal user-facing `personas_data.db` and management-API surfaces raise the blast radius.
- **Fix sketch**: Add a validated path: `order_by` should reject/escape `dir` to `ASC|DESC` only, and accept columns only via an allowlist enum or `is_valid_ident` check (`[A-Za-z_][A-Za-z0-9_]*`, optionally one `table.col` dot). Provide `order_by_unchecked` for the rare internal raw case so the unsafe path is explicit at the call site.
