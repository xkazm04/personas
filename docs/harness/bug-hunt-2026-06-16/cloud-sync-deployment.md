# Bug Hunter — Cloud Sync & Deployment

> Total: 5 findings (1 critical, 2 high, 1 medium, 1 low)
> Context: cloud-sync-deployment | Group: Onboarding, Home & Settings

## 1. Sync cursor advances to a watermark captured *before* the read snapshot, silently dropping rows committed during the pass
- **Severity**: Critical
- **Category**: 🔮 Latent failure / silent data loss
- **File**: `src-tauri/src/cloud/sync/rows.rs:496` and `src-tauri/src/cloud/sync/mod.rs:239`
- **Scenario**: A pass runs `tick_start = now_rfc3339()` (mod.rs:240), then the `SELECT … WHERE datetime(cursor_col) > datetime(cursor_prev)` (rows.rs:498/505) reads the local table, then on success `set_cursor(pool, cursor_name, &tick_start)` (mod.rs:255). The SELECT acquires a read snapshot at some instant `t_read >= tick_start`. Any row written by the app (a new execution finishing, a persona edit) with `updated_at`/`created_at` in the window `(cursor_prev, tick_start]` but **committed to SQLite after the SELECT's snapshot** is not returned by this pass — yet the cursor is moved to `tick_start`, so the next pass's `> tick_start` filter permanently excludes it.
- **Root cause**: The new watermark is the wall-clock at pass *start*, not the max timestamp actually observed in the result set, and it is not coordinated with the SELECT's transaction snapshot. The fetch runs on a separate pooled connection via `spawn_blocking` (mod.rs:249), so there is real wall-time and a separate read snapshot between `tick_start` capture and the actual query — widening the lost-update window further under load.
- **Impact**: Rows are silently never synced to the cloud — no error, `is_clean()` stays true, `last_sync_at` advances. For config tables (personas, memories, triggers) with `resync=false` and no resync window, the loss is permanent: that persona edit / new memory never reaches the user's other devices or the cloud dashboard. This is the core SYNC data-loss failure mode.
- **Fix sketch**: Set the new cursor to the **maximum `cursor_col` value actually present in the fetched rows** (or `cursor_prev` if none), not `now()`. Alternatively capture `tick_start` *after* opening the read transaction and read on the same connection/transaction that advances the cursor, so the snapshot boundary and the watermark coincide. The append tables are partially shielded by the 24h resync window, but the config tables have none.

## 2. `notify_dirty()` wakes that arrive while a pass is running are coalesced into a single already-in-flight pass and lost
- **Severity**: High
- **Category**: ⚡ Race condition / missed sync
- **File**: `src-tauri/src/cloud/sync/mod.rs:420` (loop) and `mod.rs:139` (`notify_dirty`)
- **Scenario**: The loop does `select! { tick | SYNC_WAKE.notified() }` then `run_sync_once(&state).await`. `run_sync_once` is awaited inline in the loop body, so while a pass is running (network round-trips to Supabase across 11 tables can take seconds) the loop is **not** parked in `notified()`. `Notify::notify_one()` only stores a single permit; multiple `notify_dirty()` calls during the in-flight pass collapse to one permit, and that one permit is consumed by the *next* loop iteration regardless. A mutation that lands mid-pass (after its table was already read) has its wake permit absorbed by an iteration that may find nothing new (because of bug #1's premature cursor advance), so the change waits up to the full 45s periodic tick — or, combined with #1, never syncs.
- **Root cause**: `Notify` is edge-triggered with a single-slot permit and the consumer is busy during the entire pass; there is no "dirty since pass started" flag to force a re-run when wakes were dropped.
- **Impact**: User edits a persona on device A, expects near-real-time propagation (the design promises event-driven wakes "instead of waiting for the next periodic tick"), but the change is invisible on device B / the dashboard for up to 45s or indefinitely. Erodes the core value prop of "live" cross-device continuity.
- **Fix sketch**: Track a `dirty` AtomicBool set by `notify_dirty()` and cleared at the *start* of `run_sync_once`; after a pass, if `dirty` was set again during it, immediately loop without waiting. Or move the wake-drain to after the pass and re-check.

## 3. Last-write-wins conflict resolution trusts client `updated_at` wall clocks, so a device with a skewed clock silently overwrites newer edits
- **Severity**: High
- **Category**: 🕳️ Edge case (clock skew) → 💀 silent wrong resolution
- **File**: `src-tauri/src/engine/workspace_sync/merge.rs:140` (`last_writer_wins`) / `merge.rs:161` (`compare_rfc3339`)
- **Scenario**: Two of a user's devices diverge on the same persona. Resolution is purely "later `updated_at` wins" (snapshot.updated_at, device-local wall clock). Device B's clock is 10 minutes ahead (common: VM, dual-boot, bad NTP). The user makes a *genuinely newer* edit on device A, but A's correct timestamp is still "older" than B's stale-but-future-stamped row. The merge returns `ConflictResolved { winner: Remote }` and the cloud row (B's stale content) overwrites A's newer edit — silently, with no conflict surfaced (the design intentionally suppresses manual conflict UI: "the *same user* on both ends, so a divergent conflict is auto-resolved by last-writer-wins").
- **Root cause**: LWW over untrusted wall clocks with no logical clock / version vector and no skew detection. The tie-break (lexicographic device id, merge.rs:147) is deterministic but does nothing to protect against systematic clock skew, which biases *every* conflict toward the fast-clock device.
- **Impact**: Real data loss of the user's most recent work, with zero feedback. The 4096-byte system prompt the user just rewrote on their laptop is replaced by the version their (clock-skewed) desktop last touched. No tombstone, no toast, no error counter.
- **Fix sketch**: Use a monotonic logical counter or hybrid logical clock per entity instead of (or alongside) wall clock; or record a per-row version that increments on each local mutation and break ties by version, not timestamp. At minimum, detect implausible future timestamps (row `updated_at` > server/now + threshold) and quarantine that resolution for review rather than auto-applying.

## 4. Deployment row actions render success on a stale snapshot — pause/undeploy a deployment another device already removed reports success against a ghost
- **Severity**: Medium
- **Category**: 💀 Silent failure / success theater
- **File**: `src/stores/slices/system/cloudSlice.ts:304` (`cloudPauseDeploy`) and `cloudSlice.ts:334` (`cloudRemoveDeploy`)
- **Scenario**: The dashboard renders `cloudDeployments` fetched once on mount / manual refresh (`UnifiedDeploymentDashboard.tsx:52`). Device B (or a budget cap on the orchestrator) removes/pauses a deployment. On device A the row is still shown. User clicks Pause: `cloudPauseDeployment(id)` is called against a deployment that no longer exists or is in a different state. If the orchestrator returns a (cached/200) body it is mapped into the row as if authoritative; if it 404s, the catch rolls back to `prevDeployments` (the stale list still containing the ghost row) and shows a generic "Failed to pause" — the ghost row stays. There is no polling/subscription to reconcile the table with server truth, so the UI can drive actions against a runtime state that has drifted.
- **Root cause**: Optimistic/last-fetch list with no server reconciliation or per-row staleness check before mutating; the table is the source of truth for what's actionable, but it can be arbitrarily stale (no auto-refresh interval, only mount + manual button).
- **Impact**: User believes they paused/removed an agent that is still running (or already gone), potentially continuing to burn the monthly budget, or repeatedly "fails" to act on a row that the server already deleted. Confusing at best, billing-relevant at worst.
- **Fix sketch**: Re-fetch (or optimistically reconcile from the action's response) the full list after each mutation; treat a 404 on pause/remove as "already gone" and drop the row rather than rolling back to the stale list; add a lightweight poll or realtime subscription while the dashboard is open.

## 5. Desktop runtime executes a plan against `running`/`installed` state that was sampled at discovery time, not validated at execute time
- **Severity**: Low
- **Category**: 🕳️ Edge case (runtime discovered then gone) / deploy to stale runtime
- **File**: `src-tauri/src/engine/desktop_runtime.rs:131` (`execute_plan`) with `desktop_discovery.rs:228` (`discover_apps`)
- **Scenario**: `discover_apps` snapshots `installed`/`running` by reading the binary path and a one-shot `tasklist`/`ps` dump (discovery.rs:230). The UI/agent picks a bridge based on that snapshot. By the time `execute_plan` runs, the user may have quit Docker/Obsidian or uninstalled it. `execute_plan` only pre-validates *security capabilities* (runtime.rs:140-146) — it never re-checks that the target binary still exists or the app is still running. Each step then shells out (`execute_bridge_action`, runtime.rs:296) and fails per-step with a raw spawn error, aborting the plan mid-way (`break` at runtime.rs:227-229) and leaving any prior steps' side effects (files written, containers started) half-applied.
- **Root cause**: Discovery state is a point-in-time sample with no freshness/TTL and no re-validation at the execution trust boundary; `binary_path` captured at discovery isn't even passed back into `BridgeConfig` (execute falls back to `"code"`/`"docker"` on PATH, runtime.rs:307/314), so a runtime found only at a well-known non-PATH install path won't be invokable at all.
- **Impact**: Plans fail partway with a low-level "binary not found"/spawn error instead of a clear "Docker is no longer running" message, and multi-step plans can leave inconsistent partial state (no rollback of completed steps). Lower severity because it surfaces as an error rather than silent data loss, but the partial-application and the discovery→execute path mismatch are real.
- **Fix sketch**: Re-validate the target app's installed/running state (and use the discovered `binary_path`) at the start of `execute_plan`, returning a clear domain error before any step runs; consider recording completed-step side effects so a mid-plan failure can report (or roll back) what was already done.
