# Bug Hunter — Dev Tools & Context Map

> Total: 5 findings (1 critical, 2 high, 1 medium, 1 low)
> Context: dev-tools-context-map | Group: First-Party Plugins

## 1. Concurrent context scans of the same project can wipe the map mid-write
- **Severity**: Critical
- **Category**: Race condition / data loss
- **File**: `src-tauri/src/commands/infrastructure/context_generation.rs:460` (`launch_context_scan`); destructive call at `:916`/`:939`
- **Scenario**: A user clicks "Full re-scan", then a few seconds later clicks it again (or the weekly `system_op_automation` scan fires while a manual scan is mid-flight, or Athena's `register_project` auto-scan overlaps a user scan). Each call mints a brand-new random `scan_id` and registers an independent job — there is no per-project inflight guard (contrast `trigger_automation`, which uses `INFLIGHT_TRIGGERS.guard(&id)`). Both runs are `is_rescan = true` with `needs_lazy_clear = true`.
- **Root cause**: The lazy-clear design assumes a single rescan per project. When two rescans run concurrently, scan A produces its first group and calls `repo::clear_project_context_map(pool, project_id)` — which `DELETE`s ALL `dev_contexts`/`dev_context_groups` for the project, including the rows scan B has already committed. Then scan B emits another group and calls `clear_project_context_map` again, deleting scan A's freshly-written rows. The two streams interleave their `clear` + `create` calls with no transaction or mutex, so the final map is an arbitrary, partial mix of both runs (or nearly empty if a late clear lands after the other run finished). The frontend's `activeScanId` guard only de-dupes UI events; it does not stop the backend from launching the second job.
- **Impact**: A hand-curated context map (groups, KPIs scoped to contexts, goals referencing `context_id`) can be silently destroyed or corrupted. Because `clear_project_context_map` cascades to contexts, any `dev_goals.context_id` / `dev_kpis.context_id` references are orphaned. The scan still reports "success" with whatever counts its own run produced — classic success theater over a corrupted result.
- **Fix sketch**: Add a per-project inflight guard in `launch_context_scan` (reuse `InflightGuard` keyed by `project_id`) that rejects a second scan with a "scan already running for this project" validation error. Additionally, wrap the lazy clear + the entire stream's context creation in a single transaction, or clear-then-insert atomically so a partial interleave is impossible.

## 2. `move_context_to_group` silently succeeds on a non-existent / wrong context
- **Severity**: High
- **Category**: Silent failure
- **File**: `src-tauri/src/db/repos/dev_tools.rs:2378` (`move_context_to_group`)
- **Scenario**: A drag-and-drop in the Context Map re-groups a context whose row was concurrently deleted (e.g. by a rescan from finding #1, or another window), or the frontend sends a stale `id`. The `UPDATE ... WHERE id = ?3` matches zero rows, but the function then calls `get_context_by_id(pool, id)` — and if the row exists at all it returns it, masking that the move didn't happen; if it doesn't exist it returns `NotFound`, but the partial UPDATE state is never surfaced as "move failed."
- **Root cause**: Unlike `update_context` (which calls `get_context_by_id` first to assert existence) and unlike most sibling repo fns that check `rows > 0`, `move_context_to_group` neither pre-validates the id nor inspects the affected-row count from `conn.execute`. It also doesn't validate that `group_id` (when `Some`) is a real group in the same project, so a context can be FK-orphaned into a group that doesn't exist or belongs to another project.
- **Impact**: The UI shows the context as moved (optimistic update), the DB disagrees, and on next `fetchContexts` the card snaps back — or worse, the context is pointed at a foreign/garbage `group_id` and disappears from every group's list (it won't match any group filter and isn't `null`, so it's neither grouped nor in the `__ungrouped__` bucket). Looks like data loss to the user.
- **Fix sketch**: Pre-check `get_context_by_id`, assert `rows == 1` from the `UPDATE`, and when `group_id` is `Some`, validate the group exists and shares the context's `project_id` before writing; return `NotFound`/`Validation` otherwise.

## 3. Patch-release CICD is not idempotent — concurrent/retried runs create duplicate or conflicting releases
- **Severity**: High
- **Category**: Race condition / latent failure
- **File**: `src-tauri/src/engine/platforms/github.rs:473` (`create_patch_release`); commands at `src-tauri/src/commands/tools/github_platform.rs:37`
- **Scenario**: `github_create_patch_release` is invoked twice close together — e.g. a webhook/automation double-fire, a user clicking the button twice, or a retry after a slow network response. Both invocations call `get_latest_release` (sees `v1.2.3`), both call `compare_commits` (sees N > 0 new commits), both compute `bump_patch` → `v1.2.4`, and both POST `create_release`. Unlike `trigger_automation`, this command has NO inflight guard.
- **Root cause**: The "advance → bump → create" sequence is a read-modify-write with no locking and no idempotency key against GitHub. The latest-release read and the create are separate, unguarded API calls. If two runs race, the second either creates a duplicate `v1.2.4` (GitHub allows multiple releases pointing at the same/overlapping commits) or fails with a 422 tag-already-exists that surfaces as a raw `AppError::Execution(... HTTP 422 ...)` with no recovery. The `commands_since == 0` no-op only protects against re-running after the first release is *visible*; within the race window both reads predate either write.
- **Impact**: Duplicate or version-skipped GitHub releases on the user's real repo (a destructive, externally-visible side effect that's hard to undo), or an opaque HTTP-422 error string shown to the user with no idempotent retry path.
- **Fix sketch**: Add a per-(credential,owner,repo) inflight guard around the command. Treat GitHub 422 "tag already exists / release already exists" as a recoverable no-op (look up and return the existing release) rather than `AppError::Execution`. Optionally pass the resolved head SHA as `target_commitish` and verify no release already targets it.

## 4. Backend file-walk skip list diverges from the LLM scan prompt — stale/garbage contexts
- **Severity**: Medium
- **Category**: Latent failure / drift
- **File**: `src-tauri/src/commands/infrastructure/incremental_scan.rs:33` (`SKIP_DIRS`) vs the prompt at `context_generation.rs:217` and the legacy walker `dev_tools.rs:2433`
- **Scenario**: The delta/hash cache (`walk_project_files`) skips a curated 25-entry `SKIP_DIRS` set and only hashes a fixed `SOURCE_EXTENSIONS` allow-list. But the actual context-deriving LLM is a free-roaming Claude CLI told (prose only) to "skip node_modules, target, dist, build, .git" — a 5-entry list it may or may not honor, and which omits `.next`, `coverage`, `out`, `tmp`, `__pycache__`, `.venv`, etc. The legacy `scan_codebase` in `dev_tools.rs` uses yet a third list (`.`-prefixed + node_modules/target/dist/build only).
- **Root cause**: Three independent definitions of "what counts as source." The delta computation, the LLM's exploration, and the legacy walker can each include/exclude different files. The LLM can therefore emit `file_paths` for files that the hash cache never tracks (e.g. a `.cfg` or a file under `out/`). Those paths are then absent from the next delta's `current` set, so `compute_delta` reports them as `deleted` on every subsequent rescan — perpetual phantom churn — while genuinely tracked files the LLM ignored never get their context updated.
- **Impact**: Context `file_paths` drift permanently out of sync with what delta-mode tracks: ghost "deleted" files re-reported each scan, contexts that point at files the cache thinks don't exist, and wasted LLM tokens re-processing the same phantom delta. The map looks fresh but is quietly wrong.
- **Fix sketch**: Make `SKIP_DIRS` + `SOURCE_EXTENSIONS` the single source of truth: inject the exact list into the LLM prompt (and constrain it to only emit paths within the walked set), and replace the legacy `dev_tools.rs::scan_codebase` walker with `walk_project_files`. Validate each emitted `file_path` against the walked snapshot before persisting.

## 5. 3-second fallback timer can double-finalize a scan and emit a bogus "success" notification
- **Severity**: Low
- **Category**: Silent failure / success theater
- **File**: `src/features/plugins/dev-tools/sub_context/ContextMapPage.tsx:252` (`handleScanStatus` setTimeout) vs `:276` (`handleScanComplete`)
- **Scenario**: The status event arrives as `completed`, arming a 3s timer that will call `finalizeContextScan({ outcome: 'success' })` if `activeScanId` is still set. If the real `CONTEXT_GEN_COMPLETE` event (carrying counts and the true `completed_with_warning`/`failed` status) is delayed beyond 3s — slow machine, large `replace_file_hashes` write, harness-doc write — the timer fires first and finalizes as plain `success` with no counts. The status event's `setTimeout` only re-checks `activeScanId === currentScanId`, which is still true, so it does not know the authoritative completion is still pending.
- **Root cause**: Two completion paths (`handleScanStatus` timer + `handleScanComplete`) both call `finalizeContextScan`, and `finalizeContextScan` clears `activeScanId` only inside a further 800ms `setTimeout`. There is a ~3.8s window where both can fire. The guard is "is a scan still active," not "has this scan already been finalized," so a warning/failure that resolves slowly is reported to the user as a clean success.
- **Impact**: User gets a "Context Map Ready" bell notification with no counts for a scan that actually completed-with-warning (partial/timeout) or even failed late; double `fetchContextGroups`/`fetchContexts` and double process-drawer `processEnded`. Cosmetic but actively misleading about scan health.
- **Fix sketch**: Track a `finalizedScanIdRef` and make `finalizeContextScan` a no-op if the scan id was already finalized; clear `activeScanId` synchronously on first finalize rather than after 800ms. Prefer the `CONTEXT_GEN_COMPLETE` payload as authoritative and only let the timer fire if no complete event has been seen.
