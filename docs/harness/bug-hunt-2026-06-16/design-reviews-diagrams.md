# Bug Hunter — Design Reviews & Diagrams

> Total: 5 findings (1 critical, 2 high, 1 medium, 1 low)
> Context: design-reviews-diagrams | Group: Templates & Recipes

## 1. Missing `nodes`/`edges` arrays in LLM-generated flow crash the diagram render
- **Severity**: Critical
- **Category**: Trust boundary / Silent-then-fatal failure
- **File**: `src/features/templates/sub_diagrams/FlowDiagram.tsx:19` (also `:30`, `:73`, `:83`)
- **Scenario**: A user opens the Activity Diagram modal for a review whose `use_case_flows` was generated by Claude. The LLM emitted a flow object that has `name`/`description` but omits `edges` (or `nodes`) — a common shape drift, e.g. `{ "id": "f1", "name": "Happy path", "nodes": [...] }` with no `edges`. `extract_use_case_flows_from_result` (reviews.rs:2438) only verifies the *top* value `is_array()`; it never validates that each element has `nodes` and `edges` arrays. The value is stored verbatim and `parseJsonSafe<UseCaseFlow[]>` (DesignReviewsPage.tsx:98) returns it unchanged because the parse succeeds. `FlowDiagram` then does `for (const edge of flow.edges)` against `undefined`.
- **Root cause**: Trust boundary gap — untrusted LLM JSON is shape-checked one level deep on the backend and not at all on the frontend (TS types assert `edges: FlowEdge[]` but nothing enforces it at runtime). `flow.edges` / `flow.nodes` are dereferenced with no guard.
- **Impact**: `TypeError: flow.edges is not iterable` thrown inside `useMemo` during render → the whole modal (and, since `ActivityDiagramModal` is rendered without an `ErrorBoundary` in DesignReviewsPage, potentially the Templates page subtree) blanks out / React error overlay. Reproducible from any single malformed flow among many; one bad flow takes down all tabs in the modal.
- **Fix sketch**: Normalize on read — coerce `flow.nodes ??= []` and `flow.edges ??= []` at the top of `FlowDiagram` (or filter invalid flows in `ActivityDiagramModal`). Better: tighten `extract_use_case_flows_from_result` to drop flows whose `nodes`/`edges` aren't arrays, and wrap `<ActivityDiagramModal>` in an `ErrorBoundary` like the other tabs.

## 2. Duplicate node IDs from LLM output silently drop nodes and produce React key collisions
- **Severity**: High
- **Category**: Edge case / Silent failure
- **File**: `src/features/templates/sub_diagrams/FlowDiagram.tsx:73` (`nodeMap`), `:139` (`key={node.id}`)
- **Scenario**: Claude generates a flow where two nodes share an `id` (e.g. two `"decision-1"` nodes — frequent when the model copies a template). `nodeMap = new Map(flow.nodes.map(n => [n.id, n]))` keeps only the last node for that id. BFS layering and `interLayerLabels` still reference both, but rendering `nodeMap.get(nodeId)` returns the same object twice, and `<FlowNodeCard key={node.id}>` emits duplicate React keys.
- **Root cause**: Node identity is assumed unique but never validated; `Map` construction is last-writer-wins and the diagram has no dedup/index-disambiguation.
- **Impact**: Nodes silently vanish from the diagram (success theater — diagram renders, looks plausible, but is wrong), edges point at the wrong card, and React logs duplicate-key warnings that can cause incorrect reconciliation (stale popover targeting the wrong node). User reviews an incorrect design as if it were faithful.
- **Fix sketch**: Detect duplicate ids when building `nodeMap` and either disambiguate (`${id}#${index}`) or surface a "diagram has duplicate node ids" warning banner. Use a layer-relative composite key for `FlowNodeCard` instead of `node.id` alone.

## 3. Rebuild reuses a deterministic job key — concurrent/repeat rebuilds of the same review collide
- **Severity**: High
- **Category**: Race condition
- **File**: `src-tauri/src/commands/design/reviews.rs:660` (`rebuild-{id}`), `:667` (`insert_running`)
- **Scenario**: User clicks "Rebuild" on a design review, then clicks it again (slow first run, double-click, or two windows). `rebuild_design_review` always builds `rebuild_id = format!("rebuild-{id}")` and calls `n8n_job_state::manager().insert_running(rebuild_id, …)`. Two rebuilds of the same review map to the *same* job key. Unlike `spawn_design_run` (analysis.rs:63), which calls `registry.begin_run` to atomically cancel the prior run and hand out a fresh per-run cancellation token, the rebuild path has no such supersede logic. Both tokio tasks run, both call `repo::update_review_result(&review.id, …)`, and `get_rebuild_snapshot`/status events for `rebuild-{id}` interleave between the two jobs.
- **Root cause**: Non-unique job identity (deterministic key, not a UUID) combined with no begin/supersede handshake. `insert_running` either rejects the second (so the second click silently does nothing) or overwrites tracking so the first job's terminal status is attributed to the wrong run.
- **Impact**: Last-writer-wins on `update_review_result` — the slower/older Claude output can overwrite the newer one; the UI's "completed — quality X%" milestone may reflect a job the user already abandoned. Wasted Claude CLI credits running two generations. Cancel targets only one.
- **Fix sketch**: Mirror `spawn_design_run`: use the `process_registry` begin_run/supersede pattern keyed on `("rebuild", id)` to cancel+kill the prior rebuild before starting, or make `rebuild_id` a UUID and track active-by-review separately so a second click cancels the first.

## 4. `start_design_review_run` swallows whole-batch failures into per-item DB errors with no run-level guarantee
- **Severity**: Medium
- **Category**: Silent failure / success theater
- **File**: `src-tauri/src/commands/design/reviews.rs:390` / `:428` / `:457` (DB-write error branches) and `:481` (completion emit)
- **Scenario**: During a batch review run, `repo::create_review` fails for every test case (e.g. disk full, DB locked, schema drift). Each iteration logs `tracing::error!` and either `continue`s or falls through, and the loop still reaches the unconditional `"completed"` status emit at the end (line 481) with no error aggregation. The frontend sees a normal `completed` event.
- **Root cause**: The terminal `completed` event is emitted unconditionally regardless of how many items actually persisted; there is no success/failure tally in the completion payload, and the per-item `error` statuses are easy to miss in a fast-scrolling progress list.
- **Impact**: The run reports "completed" while zero reviews were saved — classic success theater. User believes a batch of templates was generated; the reviews list is empty or stale and they have no single signal that the whole run failed.
- **Fix sketch**: Track `persisted`/`failed` counters across the loop and include them in the `DesignReviewStatusEvent` completion payload (and downgrade `status` to `"completed_with_errors"` when `failed > 0`), so the UI can surface "0 of N saved".

## 5. Conversation message ordering relies on caller-supplied JSON array, not server timestamps
- **Severity**: Low
- **Category**: Latent failure / conversation ordering
- **File**: `src-tauri/src/db/repos/core/design_conversations.rs:107` (`append_single_message`), model `design_conversation.rs:8` (`timestamp` is just a string field)
- **Scenario**: Two design refinements for the same conversation race (user submits feedback, then quickly submits again before the first append returns). Both call `append_single_message` which does `json_insert(messages, '$[#]', …)` — appended in DB-execution order, not by the message's own `timestamp`. Ordering is whatever order the two IPC calls happen to hit SQLite, and the 500-cap trim (`json_remove('$[0]')`) drops the *array-front* element, which is only the oldest if insertion order matched chronological order.
- **Root cause**: Conversation order is implicit (array position) and there is no sort-by-`timestamp` on read; `append_single_message` is not serialized per-conversation, so concurrent appends can interleave a user message after the assistant reply it precedes.
- **Impact**: Rare out-of-order or mis-trimmed conversation history feeding the refinement prompt (`conversation_history` in analysis.rs:156), degrading refinement quality; user sees a momentarily scrambled thread. Low because the design refine flow is mostly serial in practice.
- **Fix sketch**: Sort messages by `timestamp` on read (or insert at the correct position), and/or guard `append_single_message` with a per-conversation lock so concurrent appends serialize deterministically.
