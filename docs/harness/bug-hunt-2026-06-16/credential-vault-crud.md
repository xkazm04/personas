# Bug Hunter — Credential Vault CRUD

> Total: 5 findings (1 critical, 2 high, 1 medium, 1 low)
> Context: credential-vault-crud | Group: Credential Vault & Connectors

## 1. `sanitize_secrets` is run over the entire ledger JSON on every metadata write, silently corrupting/erasing legitimate OAuth + custom ledger fields
- **Severity**: Critical
- **Category**: 💀 Silent failure / data corruption
- **File**: `src-tauri/src/db/repos/resources/credentials.rs:829-835` (also `:684-691`, `:757-758`, `:957-964`)
- **Scenario**: Every ledger persist path — `append_healthcheck_metadata`, `update_ledger`, `patch_metadata_on_conn`, `increment_refresh_backoff_atomic` — serializes the `CredentialLedger` to JSON and then runs `sanitize_secrets(&next_meta)` over the *whole string* before the `UPDATE`. `sanitize_secrets` (`src-tauri/src/utils/sanitization.rs:27`) applies the regex `\b(api[-_ ]?key|secret|token|password|...|refresh[-_ ]?token|access[-_ ]?token|dsn|connection[-_ ]?string|cookie|session[-_ ]?id)\b\s*([:= ]|is[: ]?)\s*(\S+)` and rewrites any match to `<key>: [secret]`. The flattened `custom` map (`credential_ledger.rs:111`) explicitly holds keys like `healthcheck_config`, `kb_id`, `auth_type`, `imported_from`, and any connector-defined hint. A custom key or value containing the substrings `token`/`secret`/`key`/`password`/`cookie` (e.g. a `healthcheck_config` JSON value `{"header":"X-Api-Key",...}`, or a custom key named `session_id`) gets the trailing JSON token `(\S+)` replaced with `[secret]`, which destroys JSON structure: `"session_id":"abc"` → `session_id: [secret]` — invalid JSON missing the quotes/braces tail.
- **Root cause**: `sanitize_secrets` is a *log/error-string* redactor (built for "untrusted API responses or error messages"), but it is being applied to *structured JSON metadata that is then parsed back*. It is not JSON-aware and it is not idempotent across the read-modify-write cycle. Next read, `CredentialLedger::parse` (`credential_ledger.rs:123`) does `serde_json::from_str(...).ok().unwrap_or_default()` — invalid JSON silently parses to `Default`, **wiping the entire ledger**: healthcheck history, anomaly tolerance, OAuth expiry/refresh-count/backoff, usage counters, environment.
- **Impact**: A single healthcheck or usage write can silently zero out a credential's entire metadata ledger. OAuth `oauth_token_expires_at` / `needs_reauth` / backoff state is lost → proactive refresh engine stops refreshing → daily-401s; anomaly tolerance resets to the 0.8 default; usage counts reset. Failure is completely silent (swallowed `.ok()`), data is unrecoverable, and it is data-dependent so it passes all tests with plain metadata.
- **Fix sketch**: Do not sanitize structured ledger JSON. Secrets must never reach the metadata column in the first place (they live in `credential_fields`), so the sanitizer is redundant here and actively harmful. Remove `sanitize_secrets` from the four ledger-write paths; if defense-in-depth is wanted, validate that no known secret *field keys* are present and reject, rather than regex-rewriting the serialized object. At minimum, after sanitizing, re-parse and refuse to persist if the result is no longer valid JSON.

## 2. Two metadata write mechanisms race on the same column: raw `json_set` (`record_usage`) vs. parse→reserialize (`append_healthcheck_metadata`/`update_ledger`), causing lost updates
- **Severity**: High
- **Category**: ⚡ Race condition / latent ledger divergence
- **File**: `src-tauri/src/db/repos/resources/credentials.rs:845-869` vs `:776-841` / `:921-970`
- **Scenario**: `record_usage` mutates `metadata` with an in-SQL `json_set(COALESCE(metadata,'{}'), '$.usage_count', ...)` and is called fire-and-forget from `healthcheck_credential` (`crud.rs:299`) and other consumers. In parallel, `append_healthcheck_metadata` and `update_ledger` do a read-modify-write: `SELECT metadata` → `CredentialLedger::parse` → mutate typed struct → `to_json_string` → `UPDATE`. These run on *separate pooled connections* with no shared transaction or row lock. If `record_usage` commits its `usage_count` bump *between* another path's `SELECT` and `UPDATE`, the read-modify-write path overwrites the whole column with its stale snapshot, dropping the usage bump (and vice-versa: a healthcheck append done via raw column overwrite would be clobbered).
- **Root cause**: No single serialization point for the `metadata` column. Some writers treat it as a JSON document (atomic field updates via `json_set`), others as an opaque blob they fully rewrite. SQLite default journal mode gives no row-level locking that would serialize a cross-connection read-modify-write; `append_healthcheck_metadata` wraps its own RMW in a transaction but that transaction does not lock the row against the *other* connection's committed `json_set` write that landed after the SELECT.
- **Impact**: Lost usage counts and occasionally lost healthcheck ring-buffer entries / OAuth-refresh state under concurrent activity (e.g. a healthcheck sweep firing `record_usage` while the OAuth refresh engine updates expiry). Silent — no error, just diverging/under-counted ledger data that feeds anomaly scoring and refresh decisions.
- **Fix sketch**: Route every `metadata` mutation through `update_ledger` (the typed RMW) so there is one code path, and make `record_usage` call `ledger.record_usage()` inside `update_ledger` instead of raw `json_set`. Alternatively run all metadata writes in `IMMEDIATE` transactions on a single dedicated connection, or use `BEGIN IMMEDIATE` + `SELECT ... ` to take a write lock before the read.

## 3. Editing one field of a multi-field credential via `updateCredential` re-encrypts and DELETEs+reinserts every field; a mid-loop encryption failure or a dropped field silently revokes the rest
- **Severity**: High
- **Category**: 🕳️ Edge case / 💀 silent secret loss
- **File**: `src-tauri/src/db/repos/resources/credentials.rs:393-420`; `src-tauri/src/commands/credentials/crud.rs:138-157`
- **Scenario**: `update_credential` treats `Some(field_map)` as the *authoritative full set* and unconditionally `DELETE FROM credential_fields WHERE credential_id = ?1` then re-inserts (`credentials.rs:395-419`). The frontend `updateCredential` slice action (`credentialSlice.ts:151`) sends whatever object the caller passes as `input.data`. Any caller that builds a partial `data` object (only the field the user touched) will, on save, delete *all* the credential's other encrypted fields — the OAuth `refresh_token`, `client_secret`, etc. are gone. There is no merge with existing fields and no `verify_field_roundtrip_on_conn` call on this path (that helper exists at `:1253` but is only wired into the single-field upsert path, not `update_with_fields`).
- **Root cause**: `update_with_fields` has whole-set replacement semantics, but the IPC contract does not enforce that callers pass the *complete* field set, and the slice does no read-merge. The single-field editor correctly uses `update_credential_field`/`upsert_field` (safe ON CONFLICT upsert, `:1235`), but the bulk `updateCredential` path shares no guardrail. Additionally, if `crypto::encrypt_field` fails on field N of the re-insert loop, the transaction rolls back (good) — but if the caller simply *omitted* a field, there is no error at all; the DELETE already removed it and the loop just never re-adds it.
- **Impact**: A partial-edit caller silently strips other secrets from a working credential; the credential row and its name survive, so the UI shows success and the credential looks intact, but it can no longer authenticate (missing refresh_token). This is the classic "success theater on save" + silent secret loss.
- **Fix sketch**: Either (a) make `update_with_fields` merge: upsert the provided keys and leave unlisted keys intact (only delete keys explicitly marked for removal), or (b) require callers to pass the full field set and assert post-write that no previously-present sensitive field disappeared. Reuse `verify_field_roundtrip_on_conn` for each written field before commit on this path too.

## 4. `persona_credentials.name` has no uniqueness constraint and no duplicate guard — duplicate-named credentials of the same service are silently created and become unresolvable
- **Severity**: Medium
- **Category**: 🕳️ Edge case / data integrity
- **File**: `src-tauri/src/db/migrations/schema.rs:160-170`; `src-tauri/src/commands/credentials/crud.rs:32-116`
- **Scenario**: The schema declares `name TEXT NOT NULL` with no `UNIQUE` (contrast `connector_definitions.name TEXT NOT NULL UNIQUE` at `schema.rs:51`). `create_credential` performs zero name validation — empty string, whitespace-only, or an exact duplicate of an existing credential's name all insert successfully. Two credentials named "Production DB" for `service_type=postgres` coexist. The build-session resolver (`get_ambiguous_service_types`, `credentials.rs:179`) already treats "2+ of same service_type" as an unresolvable ambiguity that forces a manual picker, but with identical *names* the picker cannot help the user distinguish them either.
- **Root cause**: No boundary validation on `name` (empty/dupe) at the command layer and no DB constraint. The optimistic frontend append (`credentialSlice.ts:140`) just pushes the returned row, so the UI happily shows two identical entries.
- **Impact**: Confusing vault with indistinguishable entries; persona auto-resolution silently degrades to "ambiguous, ask the user" but the user has no way to tell the duplicates apart; accidental double-submit (see also the lack of a create de-dupe) creates orphan duplicates that each hold a copy of the secret.
- **Fix sketch**: Reject empty/whitespace `name` in `create_credential`/`update_credential` with `AppError::Validation`. Add a `UNIQUE(name, service_type)` index (or a soft uniqueness check returning a clear "a credential with this name already exists" error) so duplicates are caught at the boundary rather than silently stored.

## 5. AES-256-GCM uses 96-bit random nonces with no per-key write counter; the single static master key encrypts unbounded fields, edging toward the birthday-bound nonce-reuse risk with no detection
- **Severity**: Low
- **Category**: 🔮 Latent failure / trust boundary (nonce reuse)
- **File**: `src-tauri/src/engine/crypto.rs:1263-1275` (`encrypt_for_db`), `:1250-1260` (single cached cipher / static key)
- **Scenario**: `encrypt_for_db` generates a fresh 12-byte random nonce per encryption (`OsRng.fill_bytes`) and there is exactly one process-wide master key (`get_cipher` caches a single `Aes256Gcm` from the OnceLock master key; the key "never changes at runtime"). Every credential field, every OAuth token at rest, and the unix fallback-key wrapping all draw random 96-bit nonces under the *same* key. AES-GCM with random 96-bit nonces has a safe limit around 2^32 messages per key before the birthday bound makes a nonce collision (catastrophic for GCM: collision under the same key leaks the XOR of plaintexts and enables forgery). There is no nonce-usage counter, no key-rotation trigger, and no detection if a collision ever occurs.
- **Root cause**: Random-nonce GCM under a long-lived single key with no usage accounting. Realistically a local-first vault will never approach 2^32 field encryptions, so the practical risk is low — but the design has no guardrail, no monitoring, and the master key is permanently static (there is deliberately *no* re-encryption-on-rotation path; `try_upgrade_to_keychain` reuses the same key bytes by design, `crypto.rs:869-895`), so a future bulk re-encrypt or token-churn-heavy deployment has no safety margin.
- **Impact**: Latent only — extremely unlikely to trigger in normal local use, but there is no telemetry that would ever surface a nonce collision, and no key-rotation story if the key is ever suspected compromised (rotating would orphan every existing ciphertext since nothing re-encrypts on key change).
- **Fix sketch**: For long-term keys, prefer a deterministic/counter-based nonce (e.g. monotonic counter persisted per key, or XChaCha20-Poly1305 with 192-bit nonces which is collision-safe with random nonces). At minimum, track cumulative encryptions under the master key and warn/rotate well before 2^32. Document and implement a re-encryption path so master-key rotation is actually possible.
