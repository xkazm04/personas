# Bug Hunter — Settings & BYOM

> Total: 5 findings (1 critical, 2 high, 1 medium, 1 low)
> Context: settings-byom | Group: Onboarding, Home & Settings

## 1. Routing rules with `Simple`/`Critical` complexity never fire — cost-routing contract is silently unfulfilled
- **Severity**: High
- **Category**: 🕳️ Edge case / 💀 Silent failure
- **File**: `src-tauri/src/engine/runner/mod.rs:1259` (call site) + `src-tauri/src/engine/byom.rs:445`
- **Scenario**: An admin opens Settings → BYOM → Routing, adds a rule "format-only → cheap Haiku model" with `task_complexity = Simple`, saves it clean (no warnings, green save button), and expects cheap edits to route to Haiku. Every execution still uses the default/expensive model.
- **Root cause**: The only production caller (`runner/mod.rs:1259`) invokes `byom_policy.evaluate(&[], None)` — it always passes `complexity = None`. `evaluate` resolves `None` to `TaskComplexity::DEFAULT` (`Standard`) at `byom.rs:515`, so the matching loop at `byom.rs:529` only ever compares against `Standard`. There is no task-classification source feeding complexity (acknowledged in the module doc at `byom.rs:93-131`). A `Simple` or `Critical` rule literally cannot match. The UI offers all three complexity levels in `COMPLEXITY_OPTIONS` (`ByomRoutingRules.tsx:99`) with zero indication that two of the three are dead options.
- **Impact**: Users believe they configured cost savings / quality upgrades that never take effect. No warning, no log visible to the user, no UI hint. The whole point of cost routing (the `Simple` and `Critical` branches) is inert. Money is silently overspent (or critical tasks silently routed to a cheap model if `Standard` is the only configured rule).
- **Fix sketch**: At minimum, surface this in the UI: when a rule's complexity is `Simple`/`Critical`, show an Info chip ("complexity-based routing not yet active — this rule will not fire") until a classification source lands. Better: wire a persona-level default complexity (the documented precedence layer 2) so `evaluate` receives a real value, then drop the warning.

## 2. BYOM provider "Test connection" only probes a local CLI binary — it never validates the external model endpoint or API key (success theater)
- **Severity**: High
- **Category**: 💀 Silent failure / 🔮 Latent failure
- **File**: `src-tauri/src/commands/infrastructure/byom.rs:132` (`test_provider_connection`) + `ByomProviderList.tsx:160`
- **Scenario**: Admin allows a provider, the green health dot appears ("reachable", with a version), and they trust that the provider is correctly configured and reachable. The provider then fails at execution time because the API key is missing/invalid/expired, the endpoint is unreachable, or the account is out of quota.
- **Root cause**: `test_provider_connection` only checks that the CLI binary exists in PATH and that `--version` succeeds (`byom.rs:151-181`). It does no network call to the model endpoint, no API-key validation, and no auth handshake. The UI presents this as a generic "Test connection" with a pass/fail dot and even renders a latency value (`ByomProviderList.tsx:373`) that is actually local process spawn time, not endpoint round-trip. The label and green check imply end-to-end reachability that was never tested.
- **Impact**: Classic success theater. The single most reassuring affordance on the BYOM page (green dot + "reachable" + latency) certifies nothing about the external endpoint or credentials — the exact things most likely to be misconfigured for a BYOM provider. Admins ship a broken provider config to production with full confidence.
- **Fix sketch**: Either (a) relabel to "CLI installed" and drop the misleading latency/reachable wording, or (b) actually perform a minimal authenticated probe against the provider endpoint (cheap models-list / health call) and report real reachability + auth status.

## 3. `delete_external_api_key` (hard delete) bypasses the management-API auth that `revoke` preserves — and there is no confirmation safety on the destructive path
- **Severity**: Medium
- **Category**: 🕳️ Edge case / 💀 Silent failure
- **File**: `src-tauri/src/db/repos/resources/external_api_keys.rs:154` (`delete`) + `ApiKeysSettings.tsx:115`
- **Scenario**: A user clicks delete on a key row (two-click confirm with a 3s window, `ApiKeysSettings.tsx:311-319`). If they delete the wrong key, the row is gone permanently — no audit trail. By contrast `revoke` keeps the row with `revoked_at` for audit. Worse: `delete_external_api_key` (`external_api_keys.rs:84`) has no `settings_audit_log::insert` call, unlike `create` and `revoke` which both log. A deleted key leaves zero forensic record.
- **Root cause**: `delete` is a `DELETE FROM` (`external_api_keys.rs:157`) with no audit write in the command wrapper (`commands/credentials/external_api_keys.rs:84-91`), whereas revoke at the same level logs to `settings_audit_log` (`external_api_keys.rs:68-79`). The Settings → History feed therefore shows key creation and revocation but silently loses the fact that a key was hard-deleted — an attacker who steals the privileged session can delete a key (e.g. to cover tracks of which key they used) and leave no trail.
- **Impact**: Incident response gap: no record that a key existed or was removed. Combined with the irreversibility, an accidental or malicious delete is both unrecoverable and invisible in the audit history.
- **Fix sketch**: Add a `settings_audit_log::insert(... "delete" ...)` to `delete_external_api_key` mirroring the revoke path, and ideally fetch the row before deletion so the `before` snapshot (name/prefix/scopes) is recorded. Consider disabling hard-delete in the UI in favor of revoke-only.

## 4. Management-API CORS allows any origin (`allow_origin(Any)`) on a localhost server reachable from every visited web page
- **Severity**: Critical
- **Category**: Trust boundary / 🔮 Latent failure
- **File**: `src-tauri/src/engine/management_api.rs:122-125`
- **Scenario**: The management API binds `127.0.0.1` (per `local_http/mod.rs`) and mounts powerful routes: `/api/execute/{persona_id}`, `/api/build`, `/api/proxy/{credential_id}`, lab runs, version rollback. The router sets `CorsLayer::new().allow_origin(Any).allow_methods([GET, POST, OPTIONS]).allow_headers([CONTENT_TYPE, AUTHORIZATION])`. Any website the user visits can run `fetch('http://127.0.0.1:9420/api/...')` from the browser. The only thing standing between an arbitrary web page and persona execution / the credential proxy is whether the `Bearer` token (the rotating "system" key) can be obtained by that page.
- **Root cause**: `allow_origin(Any)` on a service that hosts state-changing, credential-bearing endpoints is the wrong default for a loopback server. Auth currently rests entirely on the Bearer token in `require_api_key` (`management_api.rs:137`). The system key is process-scoped and fetched via a privileged Tauri command, so a plain web page should not have it — but `allow_origin(Any)` means the browser is *permitted* to send and read cross-origin responses, so any token leak (XSS in the renderer, a logged token, a misconfigured plugin webview) is immediately weaponizable from the open internet, and the CORS layer provides no defense-in-depth. `/api/proxy/{credential_id}` in particular forwards requests through stored decrypted credentials.
- **Impact**: If the system key ever leaks to web-reachable JS, an attacker page can execute personas, drive builds, roll back prompt versions, and pivot through stored credentials via the proxy — all from a normal browser tab, because CORS explicitly allows it. This is a broad SSRF/RCE-adjacent surface gated by a single secret with no origin allow-list backstop.
- **Fix sketch**: Restrict `allow_origin` to the app's own origin(s) / the known MCP client origins instead of `Any`; for non-browser clients (CLI/MCP), CORS is irrelevant so tightening it costs nothing. Add an origin allow-list and reject browser `Origin` headers that aren't explicitly trusted, keeping the Bearer check as the second layer.

## 5. `find_by_token` updates `last_used_at` on a best-effort basis but the auth path returns success even if the touch fails — and lookup is a non-constant-time hash-index probe
- **Severity**: Low
- **Category**: 💀 Silent failure / 🔮 Latent failure
- **File**: `src-tauri/src/db/repos/resources/external_api_keys.rs:124-130`
- **Scenario**: Two minor robustness gaps. (a) The "stale key" detection in the UI (`ApiKeysSettings.tsx:46-55`, `isStaleKey`) relies entirely on `last_used_at`. `find_by_token` touches `last_used_at` only best-effort (`let _ = conn.execute(...)`, `external_api_keys.rs:126`). Under a poisoned/contended pool the touch can silently fail while auth still succeeds, so an actively-used key can be flagged "stale" (amber, "probably forgotten") and an admin may revoke a live production key. (b) Lookup is `WHERE key_hash = ?1` — a SHA-256 of the presented token compared via the DB index. This is not a constant-time comparison; while preimage resistance of SHA-256 largely mitigates token-recovery, it leaves a theoretical timing side channel on hash-prefix index traversal.
- **Root cause**: `last_used_at` write is fire-and-forget by design ("failure here should not block auth") but the staleness UI treats a missing/old `last_used_at` as authoritative evidence of disuse. The two features disagree about how reliable that column is.
- **Impact**: Mostly low: occasional false "stale" labeling could prompt revocation of an in-use key, causing a confusing 401 for an MCP/CLI client that was working seconds earlier. The timing channel is negligible given hashing.
- **Fix sketch**: Either make the `last_used_at` touch reliable (log on failure so staleness logic can be trusted) or have `isStaleKey` treat an unexpectedly-old `last_used_at` more conservatively (e.g. require corroborating signal). Optionally switch token verification to a constant-time compare after fetching by prefix.
