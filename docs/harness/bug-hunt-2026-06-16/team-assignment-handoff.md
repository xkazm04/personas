# Bug Hunter — Team Assignment & Handoff

> Total: 5 findings (1 critical, 2 high, 1 medium, 1 low)
> Context: team-assignment-handoff | Group: Teams & Fleet Orchestration

## 1. Double-started assignment runs two tick loops → same step assigned & executed twice
- **Severity**: Critical
- **Category**: Race condition / double-assignment
- **File**: `src-tauri/src/engine/team_assignment_orchestrator.rs:106` (run_assignment), `:400` (in_flight), `:540-583` (launch); gate doc at `:103-105`
- **Scenario**: `start_team_assignment` (`commands/teams/assignments.rs:125`) accepts an assignment already in `running` status and unconditionally calls `orchestrator::run_assignment`, which `tokio::spawn`s a brand-new `tick_loop` every time. The same happens on `resume_assignment` (used by review-edit/reassign/skip + auto-resume + orphan-recovery, all of which can fire while a loop is already live). The doc comment claims "multiple starts collapse into a single live task… the DB transition `queued → running` is the gate," but there is NO such gate: `update_assignment_status` (`db/repos/orchestration/team_assignments.rs:391`) is an unconditional `UPDATE … WHERE id = ?` with no `WHERE status = 'queued'` compare-and-swap, and `run_assignment` keeps no global registry of live tasks. The `in_flight: HashMap` that prevents relaunch (`:400`) is a *local* variable, private to each loop instance.
- **Root cause**: No single-flight / compare-and-swap guard. Each `tick_loop` independently reads `list_steps`, finds the same `pending` step with satisfied deps, and launches it (`:546-581`). `update_step_status(… "running")` is also a plain UPDATE — the second loop sees the step as `pending` on its own tick before the first loop has flipped it, or right after a transient state, and both create executions.
- **Impact**: The same work item runs twice concurrently — two persona executions, two PRs opened for one step, doubled token spend, and `qa.pr.changes_requested` / goal-progress bookkeeping racing on one step row. A user clicking "Start" twice, or resuming a paused-but-still-ticking assignment, silently forks the pipeline.
- **Fix sketch**: Make the loop genuinely single-flight: keep a `globalThis`-style `Mutex<HashSet<assignment_id>>` (mirroring the executionProcesses/eventBus singleton pattern) and have `run_assignment` no-op if the id is already present, removing it when `tick_loop` returns. Independently, gate the DB transition with a CAS (`UPDATE … SET status='running' WHERE id=? AND status IN ('queued','awaiting_review','paused')` and check `rows_affected == 1`) so only one starter wins.

## 2. Persona running multiple steps concurrently can exceed its `max_concurrent` and collide on one repo
- **Severity**: High
- **Category**: Race condition / concurrent orchestration
- **File**: `src-tauri/src/engine/team_assignment_orchestrator.rs:538-583` (budget + launch loop)
- **Scenario**: The tick loop's only concurrency budget is `max_parallel_steps` minus the count of in-flight *steps* (`:537-538`). When two independent steps (deps satisfied, e.g. an "implement A" and "implement B" with no edge between them) both resolve to the SAME persona — easy under `llm_eval`/`embedding` auto-match, or when a single engineer is the only eligible candidate — the orchestrator launches both at once. The module doc claims "Persona-level `max_concurrent` continues to gate per-persona parallelism because each step routes through `start_execution`" (`:13-15`), but nothing in this file checks the persona's `max_concurrent` before launching, and `start_execution` is awaited inside the spawned task, not gated before spawn.
- **Root cause**: Budget is computed per-assignment-step-count, never per-resolved-persona. Resolution (`resolve_assignee`) happens *inside* the spawned `run_step`, so the tick loop can't know two pending steps will land on the same persona until both are already launched.
- **Impact**: One persona runs two CLI executions against the same working directory / branch simultaneously → git index corruption, interleaved edits, or one execution's worktree clobbering the other. Even if `start_execution` rejects the overflow, the step fails for a spurious "busy" reason and cascade-skips its dependents.
- **Fix sketch**: After resolving the assignee but before `set_step_execution`, count this persona's currently-`running` steps in the assignment (and/or honor `persona.max_concurrent`); if at cap, leave the step `pending` and let a later tick retry instead of launching. Alternatively resolve assignees in the tick loop so the budget can be persona-aware.

## 3. Step failure inside a detached task can be lost, wedging the assignment forever
- **Severity**: High
- **Category**: Silent failure / orphaned assignment
- **File**: `src-tauri/src/engine/team_assignment_orchestrator.rs:562-583` (spawned step task), `:449` (reap)
- **Scenario**: Each step runs in a detached `tokio::spawn` whose only failure handling is the `if let Err(e) = result` arm that writes `update_step_status(… "failed")` (`:564-577`). If that very write fails (DB pool exhausted, transient lock) — or if the task *panics* (e.g. the `best.expect("non-empty candidate list")` in matching, an unwrap deep in `start_execution`) — the `JoinHandle` finishes, `in_flight.retain(!is_finished())` (`:449`) drops it, the step row is left in `running`/`matching`, and no terminal write ever lands. The loop's terminal checks (`:455-478`) require `all_terminal`, so it neither completes nor fails; it just ticks until `ASSIGNMENT_MAX_TICKS` (2h) then errors out — but only if the loop itself is still alive (it dies with the process; orphan-recovery only re-queues `matching`/`running` on next *startup*).
- **Root cause**: Detached tasks with no join/result inspection. A panic or a failed status-write in the child task is invisible to the parent loop, which infers step state purely from the DB row that the child failed to update.
- **Impact**: Assignment hangs in `running` with a step stuck `running` indefinitely (up to 2h of wasted ticking, or forever across a crash until the user notices). The goal linked to it never advances and shows perpetual in-progress. Classic "success theater": the UI shows activity that will never resolve.
- **Fix sketch**: On reap, inspect the `JoinHandle` result (`handle.await`/`is_finished` + stored `Result`); if a task finished but its step is still non-terminal in the DB, force it to `failed` ("worker task ended without writing a result"). Wrap `run_step`'s body so a panic is caught and converted to a `failed` transition. Add a per-step staleness watchdog (step `running` with no execution progress past `STEP_EXECUTION_TIMEOUT_TICKS`) independent of the child task.

## 4. KPI distance/normalization divides by a near-zero current value, producing absurd percentages
- **Severity**: Medium
- **Category**: Edge case / divide-by-(near)-zero, KPI miscount
- **File**: `src/features/teams/sub_kpis/KPIDashboard.tsx:31` (distancePct), `:43` (normValue)
- **Scenario**: For a `direction === 'down'` KPI (e.g. `llm_cost`, `open_errors`), distance is `target / Math.max(cur, 1e-9)`. The `Math.max(…, 1e-9)` guards against a literal zero but not against a *legitimately tiny* current value: when `cur` is small relative to `target` (a down-metric doing great, e.g. target 100 errors, current 1), the ratio explodes (100), then `Math.min(115, …)` clamps it — so a KPI that has overshot its target by 100× is rendered identically to one that just barely hit it. Worse, when `cur` is genuinely `0` (zero open errors — the ideal), `target/1e-9` = a 9-digit ratio, clamped to 115%, indistinguishable from "met." `normValue` (`:43`) has the same pattern feeding the trend chart, so a flawless down-metric draws as a flat 115% line with no signal about how far past target it is.
- **Root cause**: Ratio math with a floor that prevents NaN but not numerical blow-up; no special-case for "down metric at or below target" (which is success, not a 115% bar).
- **Impact**: Off-track classification feeds `paceDescriptor`, but the dashboard's distance bars and trend lines silently misrepresent any down-metric near/at zero — the most important success case (zero errors, near-zero spend) looks like a wild overshoot rather than a clean "met." Misleads the human watching the fleet's KPIs.
- **Fix sketch**: Treat a down-metric where `cur <= target` as 100% (met) explicitly before computing the ratio; for the trend, cap and label overshoot distinctly, or plot raw value with a target reference line for down-metrics rather than a normalized ratio that saturates.

## 5. Handoff wiring silently drops edges and never prunes stale triggers after a roster/graph edit
- **Severity**: Low
- **Category**: Silent failure / latent (orphaned wiring)
- **File**: `src-tauri/src/engine/team_handoff.rs:111-121` (endpoint resolve skip), `:138-156` (create-failure swallow), whole-file (no delete pass)
- **Scenario**: `wire_team_handoff` resolves connection-graph edges to triggers. (a) If an edge endpoint doesn't resolve to a persona it logs a `warn!` and `continue`s (`:114-121`) — the edge is counted in neither `edges_total` nor any error, so a partially-broken graph reports "wiring complete" with a handoff hole the user can't see. (b) Trigger creation failures are caught and logged but execution continues (`:152-156`, `:176-180`); if the EMITTER chain trigger fails but the RECEIVER listener succeeds (or vice-versa), the target gets half its wiring — a listener with no emitter never fires, an emitter with no listener publishes an event nobody consumes (the bus only runs personas with a matching listener, per `:21-24`). (c) The pass is purely additive ("only missing triggers are created"); when a user *removes* a connection edge or swaps a member, the old `chain`/`event_listener` triggers are never deleted, so a re-wire leaves stale handoffs firing the wrong persona.
- **Root cause**: Best-effort, idempotent-add-only design with no reconciliation (no delete of triggers whose edge no longer exists) and no surfaced partial-failure signal in `HandoffWireResult`.
- **Impact**: Handoff chains can silently die mid-graph (broken edge or half-wired target) or silently *keep* firing a connection the user deleted — the team cascades incorrectly with no error. Low severity because it requires a graph edit / endpoint mismatch and the team-assignment orchestrator path (findings 1-3) is the primary runtime; this is the event-driven autonomy path.
- **Fix sketch**: Make wiring transactional per edge (roll back the listener if the chain trigger failed, or vice-versa) and record failures in `HandoffWireResult` (e.g. `edges_failed`, `unresolved_endpoints`) so the UI shows them. Add a reconcile step that deletes team-handoff triggers whose `(source,target)` edge is no longer present in `persona_team_connections`.
