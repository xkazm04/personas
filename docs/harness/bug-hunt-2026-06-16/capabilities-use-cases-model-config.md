## 1. Manual capability run bypasses budget enforcement entirely (real paid CLI spawn)
- **Severity**: critical
- **Category**: validation-gap / budget-enforcement-bypass
- **File**: `src/features/agents/sub_use_cases/libs/useUseCaseDetail.ts:106` (the `executePersona(...)` call inside `handleManualRun`)
- **Scenario**: A persona has blown past its monthly `max_budget_usd` (status `exceeded`) or budget data is `stale`. In the Persona Runner UI this correctly disables the Run button and shows the override card (`isBudgetBlocked` gate in `executionSlice.executePersona:255` and `useScheduleActions.manualExecute:41`). The user instead opens the Use Case detail panel and clicks "Run" (the manual real-execution trigger). It fires immediately — no budget check, no override prompt — spawning a real, paid production CLI run.
- **Root cause**: Budget enforcement is implemented as *frontend-only gating* (see the slice's own doc comment: "the Rust backend doesn't know about budget pause"). The gate lives only in the store action `executePersona`. `handleManualRun` deliberately bypasses the store action and calls the raw IPC binding `executePersona` from `@/api/agents/executions` directly (the comment even brags it triggers "real CLI spawn, real cost"). The trust boundary (budget pause) is enforced at one of two call sites, not at the boundary itself.
- **Impact**: Direct monetary loss — the entire point of budget pause (stop spending once a cap is hit) is silently defeated through a second, equally-prominent button. Also fires real `emit_event` cascades that downstream personas act on, while the user believes the agent is paused.
- **Fix sketch**: Move the budget gate to the only chokepoint both paths share — enforce it server-side in `execute_persona_inner` (read the persona's monthly spend vs `max_budget_usd`, reject with a typed error unless an explicit `override` arg is passed). Frontend gating that any caller can route around is not enforcement. At minimum, `handleManualRun` must call `get().isBudgetBlocked(expectedPersonaId)` before invoking.

## 2. Automation credential requirement checks the wrong column — guaranteed false "unmet"
- **Severity**: high
- **Category**: silent-failure / state-corruption (false design-time diagnostic)
- **File**: `src-tauri/src/engine/capability_contract.rs:210` (`service_type: cred_id.clone()`)
- **Scenario**: A persona has an automation deployed with a `platform_credential_id` (a foreign-key *ID* into `persona_credentials(id)` — confirmed by the schema: `platform_credential_id TEXT REFERENCES persona_credentials(id)`). `validate_persona_contracts` runs at design time / in the runner's validate stage. `collect_automation_requirements` pushes `Requirement::Credential { service_type: <the credential ID> }`. `check_requirement` then calls `cred_repo::get_by_service_type(pool, <credential-id-as-service_type>)`, which never matches because no row's `service_type` equals a UUID.
- **Root cause**: An ID and a `service_type` are conflated. Every other `Requirement::Credential` producer (`collect_tool_requirements`) correctly passes a real service type (`requires_credential_type`). Automations carry the resolved credential *primary key*, not a service type, so the requirement is structurally the wrong variant — there is no `Requirement` variant that resolves a credential by ID.
- **Impact**: Every persona with a credential-backed automation always reports `all_satisfied: false` with a misleading "No credential with service_type '<uuid>' found. Add one in the Vault." This is success-theater in reverse: the contract report — whose entire purpose is to surface real unmet dependencies before runtime — cries wolf on healthy personas, training users to ignore it (so it also hides genuine unmet deps).
- **Fix sketch**: Add a `Requirement::CredentialById { credential_id, needed_by }` variant whose `check_requirement` arm calls `cred_repo::get_by_id`. Make the requirement variants encode *how* the credential is referenced so an ID can never be matched against `service_type`.

## 3. Budget exceeded-ratio derives "ok" when spend is non-finite or maxBudget is non-positive
- **Severity**: high
- **Category**: edge-case / divide-by-zero-class (fail-open budget math)
- **File**: `src/stores/slices/agents/budgetEnforcementSlice.ts:71` (`if (!maxBudget || maxBudget <= 0) return { ratio: 0, status: 'ok' }`)
- **Scenario**: `deriveStatus` returns `status: 'ok'` whenever `maxBudget` is `0`, negative, `null`, `NaN`, or `undefined`. If backend `max_budget_usd` ever arrives as `0` (a plausible "hard cap to zero / disabled" encoding) or as a corrupt `NaN`/negative, the persona is treated as having *no budget cap at all* and runs freely. Symmetrically, if `spend` is `NaN`/`Infinity` (corrupt aggregation), `spend / maxBudget` is `NaN`, and `NaN >= 1.0` is `false`, so it falls through to `ok`.
- **Root cause**: The guard collapses two opposite semantics — "no cap configured" (legitimately unlimited) and "cap is zero / value is garbage" (should be maximally restrictive) — into the single fail-*open* result `ok`. The math also has no `Number.isFinite` defense on `spend`, so any non-finite numerator silently becomes "under budget."
- **Impact**: A persona whose budget is meant to be `0` (fully paused) spends without limit; corrupt spend data fails open rather than fail-closed, directly contradicting the module's stated "fail-closed" design philosophy for stale data.
- **Fix sketch**: Distinguish the cases: `maxBudget == null` → `ok` (no cap intended); `!Number.isFinite(maxBudget) || maxBudget <= 0` → treat as `exceeded`/blocked (garbage or zero cap is restrictive); `!Number.isFinite(spend)` → `exceeded`. Never let a non-finite operand resolve to `ok`.

## 4. `count_event_listeners` mis-escapes the LIKE needle, under/over-counting affected consumers
- **Severity**: medium
- **Category**: edge-case / silent-wrong-result
- **File**: `src-tauri/src/commands/core/use_cases.rs:448` (`event_type.replace('"', "\\\"")`)
- **Scenario**: A user renames an event whose name contains a `"` or a SQL LIKE metacharacter (`%` or `_`) — e.g. an event named `order_paid` or `alert"high`. The "how many consumers will I break" pre-rename count is computed by building a raw LIKE needle: `format!("%\"event_type\":\"{}\"%", event_type.replace('"', "\\\""))`. The `\"` JSON-style escape does not match how the value is actually stored in the trigger `config` JSON (serde stores `"` as `\"` in the byte stream, but this hand-rolled needle assumes a specific escaping and applies no LIKE-wildcard escaping of `%`/`_`). An underscore in the event name (`order_paid`) makes `_` a single-char wildcard, matching `orderXpaid`, `order1paid`, etc.
- **Root cause**: JSON membership is being tested with substring LIKE matching plus ad-hoc escaping, rather than the parsed-JSON approach the *mutation* path (`rename_event_listeners:538`) correctly uses (`json_extract(config, '$.event_type') = ?`). The counting (warning) path and the mutation (action) path use different, incompatible matching strategies.
- **Impact**: The pre-rename warning dialog shows a wrong count — too low (user proceeds thinking only the renaming persona is affected, silently breaks external consumers) or too high (false alarm). The count and the subsequent action disagree, so "Update 3 triggers" can actually touch a different set than "3 listeners will break" reported.
- **Fix sketch**: Make the count use the same `json_valid(config) AND json_extract(config,'$.event_type') = ?` predicate as `rename_event_listeners`. Counting and acting must share one matching function so they can never diverge — eliminating the raw-LIKE escaping class entirely.

## 5. Use-case auto-save reports success while a write is still queued/in-flight (success theater)
- **Severity**: high
- **Category**: silent-failure / recovery-gap
- **File**: `src/features/agents/sub_use_cases/libs/useUseCaseDetail.ts:45` (`setIsDirty(false)` right after `await mutateSingleUseCase`)
- **Scenario**: The user edits two use-case fields in quick succession (e.g. toggles a notification channel, then immediately edits the engine mode). Each `handleUpdate` `await`s `mutateSingleUseCase`, which only *enqueues* onto the module-global `writeQueue` (`applyDesignContextMutation`). On resolve, `handleUpdate` sets `isDirty=false`. The editor-dirty registration uses a no-op save (`saveRef.current = async () => {}` with comment "auto-saved on every change"). If the navigation/close happens after the promise resolves but the underlying `applyPersonaOp` IPC of a *later* queued mutation throws (DB locked, persona deselected mid-flight → `{ applied: false, reason }`), the failure is returned as a *resolved* `MutationResult`, never thrown — so `handleUpdate`'s `try` never sees it, `saveError` stays null, and `isDirty` is already false.
- **Root cause**: `applyDesignContextMutation` converts failures into resolved `{ applied: false }` values rather than rejections, but `handleUpdate` treats any non-throw as success (`setIsDirty(false)` unconditionally in the `try`). The dirty/save contract (`useEditorDirty`) is wired to a no-op, so the "unsaved changes" guard cannot actually flush or detect a lost write. Optimistic success with no rollback.
- **Impact**: Silent data loss — a use-case edit (model override, channel config, engine mode, test fixture) appears saved, the dirty indicator clears, the user navigates away, and the change is gone with no error surfaced. Worst with rapid edits / persona switches where queued writes resolve to `applied:false`.
- **Fix sketch**: Have `handleUpdate` inspect the `MutationResult`: only `setIsDirty(false)` when `applied === true`, and set `saveError` on `{ applied: false }`. Make the class impossible by having `applyDesignContextMutation` reject on failure (or by making `useEditorDirty`'s save actually await the pending write queue and propagate the result), so "no exception" can no longer mean "lost write."
