# Bug Hunter — Persona Templates

> Total: 5 findings (1 critical, 2 high, 1 medium, 1 low)
> Context: persona-templates | Group: Templates & Recipes

## 1. Backend integrity check can never match a real adoption — silently skipped in dev, hard-rejects every adoption in release
- **Severity**: Critical
- **Category**: Silent failure + trust-boundary / latent failure
- **File**: `src-tauri/src/commands/design/template_adopt.rs:28` (and call site `:260`), `src-tauri/src/engine/template_checksums.rs:166`
- **Scenario**: `instant_adopt_template_inner` calls `check_template_integrity(&template_name, &design_result_json)`. `verify_template` looks up `template_name` in `CHECKSUM_MANIFEST` and hashes `design_result_json`. But the manifest is keyed by **full relative file paths** (`"productivity/personal-capture-bot.json"`) and the embedded hash is of the **entire template file** (`{ id, name, payload: {…} }`). Every real caller passes the wrong key AND the wrong bytes: the Dev Clone shortcut passes `templateName = 'Dev Clone'` + `JSON.stringify(template.payload)` (`useDevCloneAdoption.ts:31-32`); the team-preset adopter passes `m.template_id` (bare id `"personal-capture-bot"`, no category prefix) + `design_payload_json()` (only the `payload` sub-object, `team_preset_adopter.rs:336-342`). So `verify_template` returns `is_known_template: false` for 100% of adoptions.
- **Root cause**: The checksum manifest's key/content contract (full path + full file) was never reconciled with the adoption call contract (bare name/label + payload-only JSON). They were generated independently and never round-tripped against each other.
- **Impact**: In **release builds** (`#[cfg(not(debug_assertions))]` at `:30`), the unknown-template branch returns `AppError::Validation("…not in the trusted checksum manifest")`, so **every** instant-adopt and **every** team-preset member adoption fails — the entire Presets feature and Dev Clone are dead on shipped binaries while passing all dev/test runs. In **debug builds** the unknown branch is compiled out, so the check is a pure no-op: it provides zero tamper protection (success theater) — a modified template JSON sails through because it's "unknown" rather than "invalid". Either way the feature is broken: useless in dev, fatal in release.
- **Fix sketch**: Make the caller resolve the canonical relative path (e.g. `category/id.json`) and hash the **full file content** (not `payload`) before calling `check_template_integrity`, OR re-key the manifest by bare id + hash the `payload` to match what callers actually have. Add a release-build integration test that adopts one real preset end-to-end so this divergence can't recur. Until then the release-mode `is_known_template` rejection should at least be guarded behind a feature flag rather than silently bricking adoption.

## 2. Build-session adoption flow performs NO integrity check at all
- **Severity**: High
- **Category**: Trust boundary / silent failure
- **File**: `src-tauri/src/commands/design/build_sessions.rs:204` (`create_adoption_session`), `:2441` (`promote_build_draft_inner`)
- **Scenario**: The primary UI adoption path (questionnaire flow) takes `agent_ir_json` directly from the frontend in `create_adoption_session`, hydrates/normalizes it, and inserts it into `build_sessions` with no call to `check_template_integrity`. Promote then materializes a persona from that stored IR. Only `instant_adopt_template_inner` ever invokes the integrity helper.
- **Root cause**: Integrity verification was bolted onto one of three adoption entry points. The comment at `template_adopt.rs:258-260` claims the backend check "catches tampered templates even if the frontend checksums were bypassed", but that guarantee only holds for the instant path — the build-session path has no equivalent gate.
- **Impact**: The stated defense-in-depth (native binary harder to tamper than JS bundle) is defeated: an attacker who tampers with a template JSON and drives it through the normal questionnaire adoption never hits any checksum verification. The feature advertises integrity protection it doesn't uniformly enforce.
- **Fix sketch**: Run `check_template_integrity` (once finding #1 makes it actually functional) inside `create_adoption_session`/`promote_build_draft_inner` against the originating template, or document explicitly that integrity is only enforced on the instant/preset path and adjust the security comment accordingly.

## 3. Concurrent / double adoption produces duplicate teams and personas with no idempotency guard
- **Severity**: High
- **Category**: Race condition
- **File**: `src-tauri/src/engine/team_preset_adopter.rs:244` (`adopt_preset`), `src/features/templates/sub_presets/usePresetAdoption.ts:113` (`adopt`)
- **Scenario**: `adopt_preset` unconditionally creates a fresh team shell (`team_repo::create`, `:244`) and N personas every invocation — there is no "already adopted this preset" check and no dedupe key. The frontend `adopt()` sets `stage='adopting'` but the "Adopt N" button's only guard is `disabled={a.selectedRoles.size === 0}` (`PresetPreviewModal.tsx:240`); it is **not** disabled while `stage === 'adopting'`. A double-click, or two modals open on the same preset, or a click-then-retry, each spawn a complete duplicate team + duplicate personas. The progress-event filter is only by `preset_id` (`usePresetAdoption.ts:102`), so two concurrent adoptions of the same preset interleave into one row table and corrupt each other's status display.
- **Root cause**: Adoption is a non-idempotent multi-write composition (team + members + connections + handoff wiring) with no in-flight lock, no unique constraint, and a CTA that stays clickable during the async call.
- **Impact**: Users get 2× (or N×) teams named identically with duplicate personas, duplicate event subscriptions, and duplicate handoff triggers that will double-fire at runtime (burning budget). Hard to clean up because the duplicates are valid rows. Unlike the AI-transform adopt jobs (which use `BackgroundJobManager` with a job lock), the preset path has no concurrency primitive.
- **Fix sketch**: Disable the Adopt button while `stage !== 'preview'`. On the backend, gate `adopt_preset` per `preset_id` (in-process mutex or a `pending` marker) and/or short-circuit if a team from this preset already exists for the workspace. Filter progress events by a per-adoption nonce, not just `preset_id`.

## 4. Questionnaire variable substitution rewrites EVERY string in the IR, including structural/id fields, with unsanitized user input
- **Severity**: Medium
- **Category**: Edge case / latent failure (data corruption)
- **File**: `src-tauri/src/engine/adoption_answers.rs:79` (`walk_and_substitute`)
- **Scenario**: `substitute_variables` round-trips the whole `AgentIr` to JSON and blindly replaces `{{param.KEY}}` in **all** string values — identity, instructions, but also any `id`, `event_type`, `trigger_type`, connector name, or `use_case_id` field that happens to contain a placeholder. The replacement is the raw user answer with no escaping. A user answer like an event name `order.created` injected into a field that feeds `event_type` is fine, but an answer containing template-meta tokens (`{{param.other}}`) or characters that collide with downstream parsers (cron expressions, three-level event-name validation, regex in `populate_persona_parameters_from_design`'s `^persona\.parameters\[…\]$`) can silently break that field. Unicode/empty answers pass through unfiltered (only `MAX_JSON_PAYLOAD_BYTES` size is bounded, and only on the instant path).
- **Root cause**: Substitution operates on the serialized tree with no field allow-list — it cannot distinguish "prose the user is meant to customize" from "machine identifiers the runtime keys on". Unresolved placeholders are only logged (`:91-96`), never surfaced to the user.
- **Impact**: A persona can be adopted with a corrupted trigger config, a malformed event type that never matches a subscription, or leftover `{{param.X}}` literals embedded in its live system prompt (degraded silently per the doc-comment at `:54`). The failure manifests at runtime as a persona that "does nothing" with no adoption-time error.
- **Fix sketch**: Restrict substitution to known prose fields (system_prompt, structured_prompt.*) instead of the entire tree; escape/validate answer values against the target field's grammar; promote "unresolved placeholders remain" from a warn log to a returned validation warning the adoption UI can show.

## 5. Number-typed answers accepting `inf`/`nan` silently collapse to a non-default value
- **Severity**: Low
- **Category**: Silent failure / edge case
- **File**: `src-tauri/src/commands/design/template_adopt.rs:1218` (`coerce_answer_to_param_value`), `:644` (preset override stringification)
- **Scenario**: For a `number` question, `coerce_answer_to_param_value` does `trimmed.parse::<f64>()`. Rust's `f64::from_str` accepts `"inf"`, `"-inf"`, `"infinity"`, and `"nan"`. On success it builds `serde_json::Value::from(n)`. `serde_json` cannot represent non-finite floats — `Number::from_f64(NaN/Inf)` yields `None`, so `Value::from` produces `Value::Null`. The parameter's `value` therefore becomes JSON `null` rather than the question's `default`, even though parsing "succeeded". The frontend `NumberControl` (`PresetQuestionnaireForm.tsx:352`) guards finiteness, but a crafted IPC payload or a non-UI caller (test bridge, future API) bypasses that guard, and the HTML form's min/max are advisory only (per the file's own comment, `:66-73`).
- **Root cause**: The coercion treats `parse::<f64>` success as "valid number" without rejecting non-finite values before handing them to `serde_json`, which then lossily maps them to null.
- **Impact**: A persona parameter intended to carry a numeric default lands as `null`, so any runtime code reading it as a number gets a missing value instead of the safe template default the "empty answer → default" branch (`:1213-1216`) was designed to guarantee. Quietly wrong configuration, no error.
- **Fix sketch**: After `parse::<f64>`, reject `!n.is_finite()` and fall back to `default.clone()` (matching the empty-string branch), or clamp to the question's declared `min`/`max`.
