## 1. desktopBridges.ts is entirely dead — zero importers anywhere in src/
- **Severity**: High
- **Lens**: code-refactor
- **Category**: dead-code
- **File**: src/api/system/desktopBridges.ts:1
- **Scenario**: A grep for `desktopBridges` and for every export in the file (`executeDesktopBridge`, `executeDesktopPlan`, `getDesktopRuntimeStatus`, `getDesktopPlanResult`, `BRIDGE_INFO`, `BridgeName`, `VsCodeAction`, `TerminalAction`, `DesktopPlan`, `BridgeConfig`) matches only within the file itself — no other module in src/ imports it. The vault desktop-discovery UI uses `src/api/system/desktop.ts` instead, and `lib/utils/platform/connectors.ts` has its own separate `getDesktopBridgeName`.
- **Root cause**: The bridge/plan-execution frontend API (VS Code / Docker / Terminal / Obsidian actions plus multi-step plans) was superseded or never wired into UI; the wrapper module was left behind. Note `BridgeName` is only `"docker" | "obsidian"` while the action unions still carry VS Code and Terminal shapes — internal inconsistency that confirms abandonment.
- **Impact**: 151 lines of dead API surface, including 4 invoke wrappers and 8 exported types, that future readers will assume is live; the VsCode/Terminal action types actively mislead about what `executeDesktopBridge` accepts.
- **Fix sketch**: Delete `src/api/system/desktopBridges.ts`. Before deleting, verify no dynamic access via the test automation bridge (`src/test/automation/bridge.ts` — it matched only smee terms) and decide separately whether the corresponding Rust commands (`execute_desktop_bridge`, `execute_desktop_plan`, `get_desktop_runtime_status`, `get_desktop_plan_result`) should also be retired or are reachable from the agent runtime.

## 2. cloud.ts hand-writes SmeeRelay and CloudWebhookRelayStatus that duplicate — and have drifted from — the ts-rs bindings
- **Severity**: High
- **Lens**: code-refactor
- **Category**: duplication
- **File**: src/api/system/cloud.ts:165
- **Scenario**: Every other type in this module comes from `@/lib/bindings/*` (generated by ts-rs from the Rust structs), but `SmeeRelay` (cloud.ts:180) and `CloudWebhookRelayStatus` (cloud.ts:165) are hand-written locals. Both already diverge from their generated counterparts: the binding `SmeeRelay` has `eventsRelayed: bigint`, `status: string`, and an `allowedRepos` field (with a security-relevant doc comment) that the local type lacks entirely; the binding `CloudWebhookRelayStatus` uses camelCase (`lastPollAt`, `activeWebhookTriggers`, `totalRelayed`) while cloud.ts declares snake_case (`last_poll_at`, `active_webhook_triggers`, `total_relayed`). A third independent `CloudWebhookRelayStatus` exists in `src/hooks/realtime/useCloudWebhookRelay.ts:5`.
- **Root cause**: Local interfaces were written before (or instead of) regenerating bindings, and consumers (`SmeeRelayTab.tsx` imports `type SmeeRelay` from cloud.ts) bind to the stale local shape, so ts-rs regeneration never forces reconciliation.
- **Impact**: The UI type is blind to `allowedRepos`, so the repo-filter feature the Rust side documents cannot be surfaced without first noticing the drift; if anyone consumes `cloudWebhookRelayStatus()` per its declared type, the snake_case fields would read `undefined` at runtime against a camelCase-serializing command. Three definitions of one payload guarantee future divergence.
- **Fix sketch**: Replace the local `SmeeRelay` interface with `import type { SmeeRelay } from "@/lib/bindings/SmeeRelay"` (adjusting `eventsRelayed` consumers for the bigint typing) and re-export it, as the module does for every other binding. Do the same for `CloudWebhookRelayStatus` in both cloud.ts and `useCloudWebhookRelay.ts`, keeping `@/lib/bindings/CloudWebhookRelayStatus` as the single source. Verify field casing against the actual Rust `#[serde]`/ts-rs attributes when reconciling.

## 3. cloudWebhookRelayStatus() invoke wrapper has no callers
- **Severity**: Low
- **Lens**: code-refactor
- **Category**: dead-code
- **File**: src/api/system/cloud.ts:173
- **Scenario**: Grep for `cloudWebhookRelayStatus` finds only the definition; the live consumer (`useCloudWebhookRelay.ts`) gets relay status via the Tauri event listener (`createSingletonListener` on `CLOUD_WEBHOOK_RELAY_STATUS`), not by polling this command.
- **Root cause**: The pull-based status command was superseded by the push-based event channel, and the wrapper was left in place.
- **Impact**: Small: one dead export plus its (drifted, see finding 2) interface — but it is the only place the wrong snake_case shape is anchored, so removing it also removes the drift hazard.
- **Fix sketch**: Delete `cloudWebhookRelayStatus` and the local `CloudWebhookRelayStatus` interface from cloud.ts (needs a quick cross-check that the Rust `cloud_webhook_relay_status` command isn't required by the event bootstrap). If a pull fallback is intentionally kept for diagnostics, retype it with the generated binding instead.
