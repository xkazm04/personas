## 1. Dead API wrappers: several exports have zero product callers
- **Severity**: Medium
- **Lens**: code-refactor
- **Category**: dead-code
- **File**: src/api/overview/observability.ts:176 (also savedViews.ts:12, reviews.ts:125, observability.ts:63, observability.ts:170, memories.ts:63)
- **Scenario**: Grep across `src/` (excluding `src/api/overview` itself) shows `runPromptAbTest` has zero references anywhere; `listSavedViews` has zero (only `listSavedViewsByType` is used); `importDesignReview` is never invoked — its only reference is the in-file `Parameters<typeof importDesignReview>[0]` type alias feeding `batchImportDesignReviews`; and `getPromptVersions`, `getPromptErrorRate`, `getMemoryStats` are referenced only by `src/api/__tests__/*` tests. All are static named imports, so this is not a dynamic-use false positive.
- **Root cause**: Superseded paths were left behind — `getPromptVersionsBulk`'s own doc comment says it "Replaces N per-persona getPromptVersions calls", and `listMemoriesWithStats` combined `getMemoryStats` into one round-trip — but the old wrappers (and their tests) were never removed.
- **Impact**: Six exported functions plus their tests are maintenance surface that suggests the app still uses these IPC commands, misleading anyone auditing the Rust command surface for pruning. Zero runtime cost, real navigation/audit cost.
- **Fix sketch**: Delete `runPromptAbTest` and `listSavedViews` outright. Replace `Parameters<typeof importDesignReview>[0]` with a named `ImportDesignReviewInput` interface and delete the function. Remove `getPromptVersions`, `getPromptErrorRate`, `getMemoryStats` and their test-only cases. Cross-context check: verify the corresponding Rust commands (`run_prompt_ab_test`, `list_saved_views`, `import_design_review`, …) have no other invokers before pruning them backend-side in a follow-up.

## 2. Hand-written MemoryReviewResult/MemoryReviewDetail duplicate — and drift from — generated ts-rs bindings
- **Severity**: Medium
- **Lens**: code-refactor
- **Category**: duplication
- **File**: src/api/overview/memories.ts:147
- **Scenario**: `src/lib/bindings/MemoryReviewResult.ts` and `MemoryReviewDetail.ts` are generated by ts-rs from the Rust structs in `src-tauri/src/commands/core/memories.rs:248`, yet memories.ts declares its own `MemoryReviewDetail` (lines 147-166) and `MemoryReviewResult` (lines 168-181) and uses those for the invoke generics. The two are already diverging: generated `action: string` vs a hand-maintained 7-member union, and separately-maintained doc comments. `MemoryStats`/`MemoriesWithStats` (lines 56-74) are also hand-written mirrors of Rust structs (`db/repos/core/memories.rs:613/680`) that simply lack `#[derive(TS)]`.
- **Root cause**: The file predates (or ignored) the ts-rs bindings for these types; every other type in this context imports from `@/lib/bindings`.
- **Impact**: When the Rust `MemoryReviewDetail::action` gains a variant or `MemoryReviewResult` gains a field, the TS side silently keeps the stale shape — exactly the drift class the generated-bindings convention exists to prevent. The narrowed `action` union is an unchecked assertion, not a contract.
- **Fix sketch**: Import `MemoryReviewResult`/`MemoryReviewDetail` from `@/lib/bindings` and delete the local declarations; if the UI wants the action union, express it as `type MemoryReviewAction = ...` and narrow at use sites (or add `#[ts(type = ...)]`/an enum on the Rust field so the union is generated). Add `#[derive(TS)] #[ts(export)]` to `MemoryStats`/`MemoriesWithStats` in Rust and switch the manual interfaces to the generated ones.

## 3. overviewBundleCache never evicts expired entries
- **Severity**: Low
- **Lens**: perf-optimizer
- **Category**: leak
- **File**: src/api/overview/observability.ts:82
- **Scenario**: `getOverviewBundle` memoizes each `(days, personaId, tzOffset)` key for 1 s, but expired entries are only ever overwritten on a repeat call with the same key or deleted on rejection. A user browsing per-persona overviews across many personas / day-windows leaves one resolved `OverviewBundle` promise (a full dashboard payload: metrics, charts, rollups) pinned in the Map per key for the life of this long-running desktop app.
- **Root cause**: The de-dupe cache implements TTL on read (`expiresAt > now`) but has no eviction path for entries that are never re-requested.
- **Impact**: Bounded but permanent retention — worst case roughly personas × day-window options bundles held in memory forever. Small in typical use, which keeps this Low, but it is unnecessary retention on a hot dashboard path.
- **Fix sketch**: On each `getOverviewBundle` call, sweep the Map for `expiresAt <= now` entries and delete them (the Map is tiny, a full sweep is O(keys)); or schedule `overviewBundleCache.delete(key)` via `setTimeout(..., OVERVIEW_BUNDLE_CACHE_MS)` when inserting. Either keeps the 1 s de-dupe semantics while dropping payload references promptly.
