## 1. `record_query` takes a global Mutex + 2 heap allocations on every DB query app-wide
- **Severity**: Medium
- **Lens**: perf-optimizer
- **Category**: hot-path-overhead
- **File**: src-tauri/src/db/perf.rs:241
- **Scenario**: Every repo call generated by `crud_get_by_id!`/`crud_get_all!`/`crud_update!`/`timed_query!` funnels into `record_query`, which locks the single global `RING` mutex and does `table.to_string()` + `operation.to_string()` (two allocations) per query. Under concurrent engine ticks + CDC + UI queries, all threads serialize on this one lock for bookkeeping.
- **Root cause**: `QuerySample` stores owned `String`s even though every caller passes `concat!`-produced `&'static str` literals; the ring lives behind one global `Mutex`.
- **Impact**: Constant per-query tax (lock + 2 allocs + drop) on the hottest code path in the backend; contention grows with parallel executions (cap is 20). Not user-visible individually, but it is pure waste multiplied by every query the app ever runs.
- **Fix sketch**: Change `QuerySample.table`/`operation` to `&'static str` and the `record_query` signature to take `&'static str` (all call sites are literals — the macros guarantee it). Optionally replace the global mutex with `parking_lot` or per-thread buffers flushed on snapshot; at minimum the allocation removal is a two-line win.

## 2. CDC drain fetches persona_events rows one-by-one — the serial fetch is what makes the channel drop events
- **Severity**: Medium
- **Lens**: perf-optimizer
- **Category**: n-plus-one
- **File**: src-tauri/src/db/cdc.rs:349
- **Scenario**: A burst of persona_events inserts (e.g. a chain fan-out or replay) produces one `fetch_persona_event_by_rowid` query + one Tauri emit per event, processed strictly sequentially in the async consumer. If the drain falls behind, the bounded sync channel fills and the update hook starts dropping events (`note_cdc_drop`) — the file's own drop counter exists precisely because this happens.
- **Root cause**: The consumer handles `rx.recv()` one event at a time; there is no batching even though consecutive events for the same table could be fetched with a single `WHERE rowid IN (...)` query.
- **Impact**: Under write bursts the frontend silently misses live updates (the documented drop mode); the DB pays N point-queries where one batched query would do. Directly degrades the "reactive" promise of CDC on the hottest table.
- **Fix sketch**: After the first `rx.recv().await`, greedily `try_recv()` up to ~64 more events, dedupe rowids per table, and fetch persona_events rows with one `IN` query (reusing `PERSONA_EVENT_COLUMNS`), then emit in order. Non-persona_events notifications can also be coalesced per (table, action).

## 3. QueryBuilder's entire UPDATE surface is dead code, masked by a blanket `#[allow(dead_code)]`
- **Severity**: Medium
- **Lens**: code-refactor
- **Category**: dead-code
- **File**: src-tauri/src/db/query_builder.rs:34
- **Scenario**: `QueryBuilder` is genuinely used by 12 repo modules — but only its SELECT half. `set`, `set_opt`, `build_update`, `set_clause`, `push_param_boxed`, and `into_params` have zero callers outside this file's own tests (grep across src-tauri confirms; UPDATEs go through the `crud_update!`/`push_field!` macros instead). The struct-level and impl-level `#[allow(dead_code)]` (lines 34/44) silence the compiler for the whole type, so future dead methods will also rot silently.
- **Root cause**: Two parallel dynamic-SQL mechanisms exist (macros.rs vs query_builder.rs); the builder's UPDATE side never won any adopters, and the blanket allow was added to quiet warnings instead of pruning.
- **Impact**: ~80 lines of unmaintained API surface that reads as if it were the sanctioned UPDATE path; the blanket allow disables the compiler's dead-code detection for a module that clearly needs it.
- **Fix sketch**: Delete `set`/`set_opt`/`set_clause`/`build_update`/`push_param_boxed`/`into_params` (and their tests), or move them behind `#[cfg(test)]` if kept as reference. Then remove both `#[allow(dead_code)]` attributes so remaining unused methods surface as warnings. Verify no dynamic use first (none found — these are plain method calls, not reflected).

## 4. `crud_update!` still hand-rolls the mirrored if-let chain that `push_field_param!` was written to eliminate
- **Severity**: Low
- **Lens**: code-refactor
- **Category**: duplication
- **File**: src-tauri/src/db/macros.rs:274
- **Scenario**: `push_field_param!`'s doc comment says it exists to "eliminate the need for a mirrored if-let chain", yet `crud_update!` (same file) does exactly that: `push_field!` builds the SET clause (line 274) and a private `@push` arm (lines 296-311) re-walks the same fields to collect params, with the two kind-dispatch tables (`clone`/`copy`/`bool`) duplicated between `push_field_param!` and `crud_update!(@push ...)`.
- **Root cause**: `crud_update!` predates (or ignored) `push_field_param!`; the param-boxing rules now live in two places that must stay in lockstep (note `push_field_param!` even has an extra `as_str` kind the `@push` arms lack).
- **Impact**: Any new field kind must be added twice; a mismatch between the SET-clause walk and the param walk produces off-by-one parameter binding — a silent data-corruption class of bug the combined macro was designed to prevent.
- **Fix sketch**: Rewrite `crud_update!`'s body to call `push_field_param!(input.$field, stringify!($field), sets, param_idx, params, $kind)` in a single pass, then delete the `@push` internal arms and (if then unused elsewhere) `push_field!`.

## 5. Generated builtin_shared_events repeats an identical `event_schema` JSON literal 123 times
- **Severity**: Low
- **Lens**: code-refactor
- **Category**: duplication
- **File**: src-tauri/src/db/builtin_shared_events.rs:46
- **Scenario**: Every one of the 123 connector entries carries the byte-identical ~430-char `event_schema` string (and a templated description/sample_payload that differ only in connector name), making the generated file ~1500 lines and drowning real diffs when the catalog regenerates.
- **Root cause**: `scripts/events/generate-connector-events.mjs` inlines the schema per entry instead of emitting one shared constant.
- **Impact**: Bounded — rustc dedupes identical string literals so binary cost is minimal — but the file is needlessly huge, review diffs are noisy, and a future schema tweak that misses one entry would silently fork the contract. Fix belongs in the generator, not this file (DO NOT EDIT header).
- **Fix sketch**: Change the generator to emit `const CONNECTOR_EVENT_SCHEMA: &str = r##"..."##;` once and reference it via `event_schema: Some(CONNECTOR_EVENT_SCHEMA)` in each entry; same trick works for the templated description prefix if desired. Regenerate; no runtime behavior change.

## 6. `crud_update!` issues a redundant existence-check SELECT before every UPDATE
- **Severity**: Low
- **Lens**: perf-optimizer
- **Category**: redundant-query
- **File**: src-tauri/src/db/macros.rs:266
- **Scenario**: Every macro-generated `update()` runs three statements: `get_by_id` (existence check), the `UPDATE`, then `get_by_id` again (refresh). The pre-check is redundant — a missing row makes the post-update `get_by_id` return the same `AppError::NotFound`, and the UPDATE on a missing id is a harmless no-op.
- **Root cause**: Belt-and-suspenders existence validation kept even though the trailing refetch already provides the NotFound semantics.
- **Impact**: One wasted point-query (plus a full row map) per update across every repo using the macro. Cheap on local SQLite, but pure overhead on a code path many tables share, and it also widens the non-atomic read-modify window.
- **Fix sketch**: Drop the leading `get_by_id(pool, id)?;` and rely on the final `get_by_id` for both the return value and NotFound mapping. Behavior is identical for callers (verify no caller depends on NotFound being raised before side-effect-free UPDATE — there are no side effects here).
