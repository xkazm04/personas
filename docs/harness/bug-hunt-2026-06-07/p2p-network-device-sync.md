# Bug Hunter — p2p-network-device-sync
> Total: 6
> Critical: 3 · High: 2 · Medium: 1 · Low: 0

The defining structural flaw across this subsystem: **the `peer_id` is treated as an authenticated identity everywhere downstream, but it is never actually proven.** The QUIC layer skips all certificate verification and the freshly-generated TLS cert is not bound to the node's Ed25519 identity key (`transport.rs`), and the application-layer Hello/HelloAck handshake transmits `peer_id` as an unsigned plaintext claim (`connection.rs`, `protocol.rs`). There is no signed-challenge step proving the remote holds the private key for the `peer_id` it announces. The `Verified` trust state documented in `identity.rs` is explicitly "aspirational / not yet wired." Findings 1–3 are different blast radii of this same missing primitive; 4–6 are independent reliability bugs.

---

## 1. Peer identity is fully unauthenticated end-to-end (QUIC accepts any cert + unsigned Hello claim)
- **Severity**: Critical
- **Category**: spoofing / crypto
- **File**: `src-tauri/src/engine/p2p/transport.rs:153` (SkipServerVerification) and `src-tauri/src/engine/p2p/transport.rs:124-131` (cert key is a throwaway, not the identity key); `src-tauri/src/engine/p2p/connection.rs:350-377` (HelloAck `peer_id` accepted verbatim)
- **Scenario**: Alice has trusted Bob (peer_id `Bxx…`) via an imported identity card. Mallory, on the same LAN, advertises an mDNS record claiming `peer_id = Bxx…` (mDNS TXT is unauthenticated; see finding 2) or simply answers when Alice dials Bob's address. Alice's `connect_to_peer` opens a QUIC connection — `SkipServerVerification::verify_server_cert` returns `assertion()` for *any* cert and `verify_tls13_signature` is stubbed to always succeed. Mallory then sends `HelloAck { peer_id: "Bxx…" }`. The only check is the string compare `remote_peer_id != peer_id` (connection.rs:372), which Mallory passes by echoing the requested id. Alice now treats the connection as Bob: `discovered_peers.trust_status` shows `trusted`, manifest sync proceeds, and agent messages route to/from "Bob."
- **Root cause**: The design comment "we verify peer_id in protocol" (transport.rs:5, 153) is false — the protocol verification (connection.rs:372) is a self-consistency check on attacker-supplied data, not a proof of key ownership. No `sign_message`/`verify_signature` (which *exist* in `engine/identity.rs:242,334`) is ever invoked in the handshake. The TLS cert at transport.rs:125 is a brand-new `rcgen` keypair unrelated to the Ed25519 identity, so even pinning the cert would prove nothing.
- **Impact**: Security — full peer impersonation on the LAN. An attacker can masquerade as any trusted peer, harvest the victim's exposed-resource manifest, and inject agent messages attributed to a trusted persona. This is the trust anchor for the entire feature.
- **Fix sketch**: Make the handshake prove key ownership. After Hello/HelloAck, exchange a fresh random nonce and require each side to return `sign_message(nonce)` over the Ed25519 identity key; verify with `verify_signature(public_key_b64, nonce, sig)` and confirm `public_key_to_peer_id(pk) == claimed peer_id`. Reject the connection on mismatch. Better still, bind the QUIC cert to the identity key (use the Ed25519 key as the cert keypair) and pin the cert's SPKI hash to the expected `peer_id` inside `SkipServerVerification` instead of blindly asserting. Make the connection-establishment path *require* a verified identity rather than carry an unverified one.

## 2. mDNS discovery trusts the advertised `peer_id`/address with no proof, enabling trusted-peer impersonation and address poisoning
- **Severity**: Critical
- **Category**: spoofing / validation-gap
- **File**: `src-tauri/src/engine/p2p/mdns.rs:128-181` (`validate_mdns_peer`), `:161` (`is_trusted_peer` keyed on the *advertised* id), `:428-443` (upsert into `discovered_peers`)
- **Scenario**: `validate_mdns_peer` checks only that the announced `peer_id` is well-formed base58/32-byte and that addresses parse as `SocketAddr`. It then sets `trust_status = "trusted"` purely because `is_trusted_peer(pool, raw_peer_id)` finds a row for that *string*. Mallory broadcasts a `_personas._tcp.local.` record with TXT `peer_id = <Bob's trusted id>` and `address = <Mallory's IP:port>`. The browse loop upserts a `discovered_peers` row showing Bob as **trusted** but pointing at Mallory's address. Worse: the upsert (mdns.rs:431-435) overwrites `addresses` for an existing peer_id on every resolve, so Mallory can also *hijack the address of a genuinely trusted, currently-offline Bob* — `resolve_peer_address` (connection.rs:898-915) will then dial Mallory whenever Alice connects to "Bob."
- **Root cause**: mDNS TXT records are unauthenticated broadcast data, but the code uses the self-asserted `peer_id` as a database key and as the trust-lookup key. Trust is bound to a spoofable string, not to a proven key. The "unverified until handshake" comment (mdns.rs:157-165) is undermined because the handshake itself does not verify (finding 1), so nothing ever downgrades a spoofed "trusted" badge.
- **Impact**: Security / silent corruption — a trusted peer's discovery row and routable address can be silently replaced by an attacker; the UI presents the attacker as trusted.
- **Fix sketch**: Treat `discovered_peers.trust_status` as *always* `unverified` until the signed handshake (finding 1) proves the key — never derive `trusted` from the advertised id alone. On address upsert for an existing peer, do not blindly overwrite a previously-verified address; keep last-verified-addr separate from last-advertised-addr, and only dial addresses confirmed by a successful identity-proving handshake.

## 3. Owned-device registry and manifest sync accept unverified peers (spoofed device pairing, resource leak to anyone)
- **Severity**: Critical
- **Category**: spoofing / validation-gap
- **File**: `src-tauri/src/commands/network/owned_devices.rs:33-54` (`register_owned_device` takes a bare `peer_id` string, no key/trust check) and `src-tauri/src/db/repos/resources/owned_devices.rs:74-95`; manifest side: `src-tauri/src/commands/network/discovery.rs:94-105` + `src-tauri/src/engine/p2p/connection.rs:680-683` (`ManifestRequest` → `build_local_manifest` served to any connected peer)
- **Scenario (registry)**: `register_owned_device` validates only that the peer_id isn't the local one (owned_devices.rs:41-47) and isn't empty (repo:80). It never confirms the peer_id corresponds to a peer that proved its key, nor that the `device_group_id` was genuinely shared out-of-band with *that* key. Because the device-ownership registry is the gate the workspace-sync loop uses to decide who it exchanges persona snapshots with (per the model doc, owned_device.rs:6-10), an attacker who learns/guesses a `device_group_id` (or rides finding 1/2) can be registered as one of the user's "own devices" and pull/push persona sync state. **Scenario (manifest)**: `dispatch_inbound_message` answers any peer's `ManifestRequest` with the *complete* local exposure manifest (connection.rs:680-683 → manifest_sync.rs:226), and `requires_auth` on each `ExposedResource` (exposure.rs:108) is never consulted before disclosure. Any peer that completes the (unauthenticated) handshake learns every exposed resource, including ones flagged `requires_auth = true`.
- **Root cause**: Both writers assume `peer_id` is an authenticated principal. The registry command is "a thin wrapper… the registry is the primitive a pairing flow writes into" (owned_devices.rs:1-4) — but no caller enforces that the pairing actually authenticated the peer, and the command itself enforces nothing. Manifest serving has no per-resource `requires_auth` gate.
- **Impact**: Security / data loss — a hostile peer joins the user's private cross-device sync set (persona data exfiltration / injection) and/or reads the full exposure manifest including auth-gated entries.
- **Fix sketch**: Require a verified identity (finding 1) before `register_owned_device` will persist a row; store and check the peer's public key alongside `peer_id` so re-registration of the same id with a different key is rejected. In `dispatch_inbound_message`/`build_local_manifest`, filter out `requires_auth` resources unless the requesting peer is a verified trusted/owned peer; gate `ManifestRequest` and `AgentMessage` handling on trust state rather than mere connectivity.

## 4. Revoked/forgotten peers silently resurrect on the next mDNS tick (tombstone resurrection)
- **Severity**: High
- **Category**: state-corruption
- **File**: `src-tauri/src/db/repos/resources/identity.rs:240-243, 262-265` (revoke/delete only flip the *current* `discovered_peers` row) vs `src-tauri/src/engine/p2p/mdns.rs:428-443` (browse loop re-upserts discovered peers) and `:464-472` (`prune_stale_peers` DELETEs, no tombstone)
- **Scenario**: User revokes trust for Bob. `revoke_peer_trust` sets `trusted_peers.trust_level='revoked'` and updates the *existing* `discovered_peers` row to `unverified`. But `discovered_peers` rows are not tombstoned: `prune_stale_peers` hard-DELETEs stale rows, and the browse loop re-INSERTs any peer that keeps announcing. There is no negative cache. For `delete_trusted_peer`, the trust row is gone entirely, so on Bob's next mDNS resolve `is_trusted_peer` returns false and he reappears as a normal discoverable peer with no record that the user ever revoked him. If the user instead *deletes* a `discovered_peers` row (prune) for a still-trusted peer, it simply comes back. There is no concept of "I deleted this, keep it gone."
- **Root cause**: `discovered_peers` is modeled as a pure cache of live LAN state with no deletion tombstones, while revocation is expressed as an in-place mutation of that cache. Cache regeneration (mDNS upsert) outlives the mutation. Classic deleted-record-syncs-back.
- **Impact**: UX degradation / security-adjacent — revoked peers silently return to the discovery list; user intent ("stop showing/contacting this device") is not durable across discovery ticks or app restarts.
- **Fix sketch**: Add a durable tombstone/deny-list table (`revoked_peers(peer_id, revoked_at)`); have `validate_mdns_peer` consult it and drop or hard-mark any announcement from a tombstoned id, and have `prune_stale_peers` skip resurrecting them. Make revocation a fact about the identity, not about a transient cache row.

## 5. `get_exposure_manifest` (and the sync upsert) embed no per-resource provenance / signature; imported resources can lack provenance entirely
- **Severity**: High
- **Category**: validation-gap / silent-failure
- **File**: `src-tauri/src/commands/network/exposure.rs:94-109` (`get_exposure_manifest`), `src-tauri/src/engine/p2p/manifest_sync.rs:252-297` (`upsert_peer_manifest` stores entries with no source verification), `src-tauri/src/db/models/exposure.rs:156-166` (`ResourceProvenance.signature_verified` exists but is producer-supplied)
- **Scenario**: The exposure manifest is built with a self-asserted `owner_peer_id`/`owner_display_name` (exposure.rs:104-106) and no signature over the resource list. `upsert_peer_manifest` writes every advertised resource into `peer_manifests` keyed only by the (unverified, finding 1) `peer_id`. Provenance is a *separate*, optional table written by the bundle-import path; `ResourceProvenance.signature_verified` is just a boolean field the writer sets (`CreateProvenanceInput`, exposure.rs:168-177) with no enforcement that a resource shown in the UI as "from peer X" was actually signed by X. A peer can therefore advertise resources attributed to a third party, and `get_resource_provenance` may return `None` (the type is `Option`) for a resource the user is viewing — provenance can be entirely absent with no flag.
- **Root cause**: Provenance/`signature_verified` is treated as descriptive metadata supplied alongside the data rather than as a verified cryptographic binding produced by checking a signature against the source peer's public key. The manifest itself is unsigned.
- **Impact**: Silent corruption / security — resources can be displayed/imported with forged or missing provenance; `signature_verified=true` is not trustworthy.
- **Fix sketch**: Sign the `ExposureManifest` with the owner's Ed25519 key and verify it on receipt before `upsert_peer_manifest`; reject manifests whose signature doesn't match the connection's *verified* peer key. Compute `signature_verified` from an actual signature check at import time, never from caller input, and make the UI treat `provenance == None` as "unverified origin," not as benign.

## 6. `try_insert_connection` tie-break can leave an in-memory connection without its inbound dispatch loop (orphaned half-usable connection)
- **Severity**: Medium
- **Category**: race-condition / resource-leak
- **File**: `src-tauri/src/engine/p2p/connection.rs:272-298` (winning new conn closes existing, then inserts) and `:407-414` / `:539-546` (caller spawns dispatch only after a successful DB write) vs `:557` (incoming path spawns dispatch after insert)
- **Scenario**: Two peers connect simultaneously. On the path where the *new* connection wins the tie-break, `try_insert_connection` closes the *existing* `quinn_conn` (connection.rs:278-281) and inserts the new one. The existing connection had a running `spawn_inbound_dispatch` loop bound to the now-closed quinn handle — that loop sees `ApplicationClosed`/`LocallyClosed` and exits cleanly (good). But consider the reverse interleaving on the outgoing path: after `try_insert_connection` returns `true`, `update_connection_status` runs; if the DB write fails, the code removes and closes the just-inserted connection (connection.rs:407-413) — yet for the *incoming* branch the displaced peer that lost the tie-break inside `try_insert_connection` has already been closed while its own caller may still proceed to spawn a dispatch loop on the losing handle (handle_incoming spawns at :557 only when insert returned true, so the loser is fine — but the winner's existing-conn dispatch and the new conn's not-yet-spawned dispatch overlap). The concrete defect: the dispatch loop is spawned by the *caller* after insert, so between `conns.insert` (inside the lock) and `spawn_inbound_dispatch` (outside the lock, after a DB round-trip), the connection is live and reachable by the remote but has **no loop accepting its streams** — a remote Ping/ManifestRequest sent in that window is silently dropped, and if the DB write then fails the connection is torn down without ever having dispatched.
- **Root cause**: Connection insertion and its inbound-dispatch lifetime are not atomic — dispatch spawning is a separate post-insert step gated on a fallible DB write, creating a window where a "Connected" peer cannot be serviced. Tie-break closes are interleaved with this window.
- **Impact**: UX degradation / recoverable — dropped early inbound requests, occasional connections that count as established but never serve a stream until the next ping cycle reconnects.
- **Fix sketch**: Spawn the inbound dispatch loop *inside* the same critical section that inserts the connection (or immediately after insert, before the DB write), and treat the DB write as best-effort/retryable rather than a teardown trigger. Tie the dispatch task's lifetime to the connection entry (store its `JoinHandle`/cancel token in `PeerConnection`) so insert/replace/remove deterministically start and stop exactly one loop per live handle.
