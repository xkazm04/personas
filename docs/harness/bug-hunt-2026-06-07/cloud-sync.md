# Bug Hunter — cloud-sync
> Total: 6
> Critical: 1 · High: 3 · Medium: 2 · Low: 0

This audit covers the desktop → cloud Supabase mirror writer (`src-tauri/src/cloud/sync/**`), the approval-gated remote-command path (`src-tauri/src/cloud/remote_commands.rs`, `src/features/cloud/RemoteApprovalPrompt.tsx`, `src/stores/remoteCommandStore.ts`), the cloud webhook relay (`src-tauri/src/engine/cloud_webhook_relay.rs`), and the cloud connection commands + Zustand slice (`src-tauri/src/commands/infrastructure/cloud.rs`, `src/stores/slices/system/cloudSlice.ts`, `src/api/cloudSync.ts`). 12 files read.

The dominant class of failure here is **wall-clock cursor advancement** in the sync writer: the cursor is set to the time the pass *started reading*, not to the max timestamp of the rows actually pushed. That single design choice produces silent row loss (the Critical) and is the root of the partial-sync inconsistency below.

---

## 1. Sync cursor advances to wall-clock start time, silently dropping rows written during the pass
- **Severity**: Critical
- **Category**: silent-failure / data-loss
- **File**: `src-tauri/src/cloud/sync/mod.rs:239-256` (`sync_table_inner`), with the fetch predicate in `src-tauri/src/cloud/sync/rows.rs:496-509`
- **Scenario**: Sync is enabled. A pass for `synced_executions` starts: `tick_start = now_rfc3339()` is captured at line 240. The `fetch` then runs `WHERE datetime(created_at) > datetime(cursor_prev)` (rows.rs:505). The upsert + network round-trip to Supabase takes, say, 3 seconds (CHUNK batching over a slow link, sync/client.rs:69-90). During those 3 seconds the engine completes another execution and inserts a row with `created_at` = (now, mid-pass). On success the cursor is advanced to `tick_start` (line 255) — a timestamp *earlier* than the row just inserted but *later* than that row will ever be re-selected, because next pass uses `created_at > tick_start`. Wait — the new row's `created_at` is *after* `tick_start`, so it would be caught next pass... **unless** the row's `created_at` was stamped before `tick_start` but the row became visible to the SELECT only after it (read snapshot), OR the row is an *in-place mutation* (status `running`→`completed`) on a row whose `created_at` predates the cursor. For append tables with `resync=true` the 24h window papers over this; for `resync=false` tables (`reviews`, `tool_usage`) and for any mutation older than 24h, the change is lost forever.
- **Root cause**: The watermark is taken from the wall clock at pass start instead of from `max(cursor_col)` over the rows actually fetched-and-confirmed-pushed. Time-based cursors assume "every row with `ts <= tick_start` is already captured," but rows can be inserted/committed concurrently with non-monotonic `created_at`/`updated_at` (another writer, clock skew, transaction-visibility ordering), and in-place edits move no `created_at`.
- **Impact**: Silent permanent data loss in the cloud mirror — rows that exist locally never appear in Supabase, with no error surfaced (`is_clean()` stays true, mod.rs:126-128). The mirror diverges and there is no reconciliation pass to detect or repair it.
- **Fix sketch**: Make the cursor a function of the data, not the clock. Track `max(cursor_col)` across the rows returned by `fetch`, and only advance the cursor to that max after a confirmed upsert (`cursor = max(seen_ts, cursor_prev)`); never advance past data you didn't push. For in-place mutations, drive the cursor off an `updated_at` that is guaranteed to bump on every mutation (and make resync unbounded or watermark on `updated_at` for *all* tables, not a fixed 24h window).

## 2. Resync window is a fixed 24h on `created_at`; in-place mutations older than 24h never re-sync
- **Severity**: High
- **Category**: state-corruption / silent-failure
- **File**: `src-tauri/src/cloud/sync/mod.rs:241-245`, `SYNC_TABLES` at `mod.rs:49-61`, fetch predicate `src-tauri/src/cloud/sync/rows.rs:496-503`
- **Scenario**: An execution from 3 days ago is still `running` in the cloud mirror (its row synced when it started). The user finally cancels it locally, flipping `status` to `failed` and setting `error_message`. The execution table watermarks on `created_at` (rows.rs:540) with a `resync` floor of `now - 24h` (mod.rs:242). Since `created_at` (3 days ago) is below both `cursor_prev` and the 24h floor, the row is never re-selected. The cloud dashboard shows it `running` forever. Same for a manual review (`reviews`, `resync=false` entirely, mod.rs:52) resolved a week after creation, or a healing issue auto-fixed after 24h.
- **Root cause**: Mixing an append-only watermark (`created_at`) with a bounded mutation-catch window assumes mutations happen within 24h of creation. Long-lived rows (`running` executions, open reviews, persisted healing issues) violate that assumption.
- **Impact**: The cloud mirror permanently shows stale status for any row mutated outside its resync window — silent corruption of the "what synced" view the Settings panel claims is accurate.
- **Fix sketch**: Watermark every mutable table on a true `updated_at` that bumps on *every* UPDATE, and use `updated_at > cursor` (drop the separate `created_at` + 24h-floor hack). Tables that genuinely never mutate (`tool_usage`) can keep `created_at`.

## 3. Remote-approval origin is trusted entirely from a cloud row; no nonce/signature binds the request to the user's intent
- **Severity**: High
- **Category**: validation-gap / crypto
- **File**: `src-tauri/src/cloud/remote_commands.rs:216-284` (`remote_command_approve`), surfaced by `src/features/cloud/RemoteApprovalPrompt.tsx:146-155`
- **Scenario**: A `pending_commands` row targeted at this device (`target_device_id=eq.{device}`) is read (remote_commands.rs:201-204), surfaced verbatim to the UI, and on Approve the persona is run locally with the row's `prompt` (remote_commands.rs:249-260). The only integrity checks are: status `== "pending"` (line 233), `command_type == "run_persona"` (line 238), and a 1h freshness window (`is_expired`, line 68). Nothing verifies *who* wrote the row, that the `prompt` wasn't mutated after the user glanced at the prompt-preview poll (the `remote-command-pending` emit at line 160 carries the prompt, but `approve` re-fetches a *fresh* copy at line 226-228 and runs *that* — a TOCTOU: the row can be edited between surface and approve), or that the prompt text the user approved matches what executes. Anyone able to insert/patch a `pending_commands` row for the user's `auth.uid()` (a compromised dashboard session, a second logged-in device, an XSS on the web app) gets arbitrary local persona execution on Approve.
- **Root cause**: The approval model assumes the cloud `pending_commands` table is a trusted, immutable channel and that "the row exists for my device" equals "the user intended this exact prompt." There is no content-binding nonce: the prompt shown in the modal and the prompt re-fetched at approve time are independently sourced, and neither is cryptographically tied to the click.
- **Impact**: Security — silent prompt-substitution / unintended remote code execution against the persona's local credentials and tools. The 450ms arm-delay (RemoteApprovalPrompt.tsx:19) only guards accidental double-clicks, not a substituted prompt.
- **Fix sketch**: Pass the exact prompt+command-hash the user saw from the frontend into `remote_command_approve` and reject if the freshly-fetched row's hash differs (close the TOCTOU). Add a per-request nonce/HMAC the desktop verifies (signed by a key the web dashboard cannot forge), or at minimum require the row to be unchanged since surfacing (compare `updated_at`). Treat any mismatch as a hard rejection, not a silent run.

## 4. Webhook relay swallows the per-firing publish failure but keeps advancing the trigger watermark in memory — silent firing loss
- **Severity**: High
- **Category**: silent-failure / state-corruption
- **File**: `src-tauri/src/engine/cloud_webhook_relay.rs:308-326` and the `max_fired_at`/`last_seen` update logic at `319-331`
- **Scenario**: A trigger has firings F1 (10:00) and F2 (10:01) in one poll batch. F1's `publish_and_upsert_watermark` fails (transient DB lock; the error is only `tracing::debug!`-logged at line 320-324 and the loop `continue`s implicitly to F2). F2 succeeds: it commits an event AND upserts the watermark to **10:01** (the transaction at relay.rs:401-407 writes `max_fired_at`, which for F2 = 10:01). In-memory `s.last_seen` is set to 10:01 (line 317). Next poll, F1 (10:00) is `<= cutoff` (10:01) and is skipped forever (line 255). F1 is silently dropped — the third-party webhook that triggered it never reaches the desktop event bus.
- **Root cause**: The watermark is computed as `max(fired_at over the batch)` (relay.rs:290-306) and persisted per-*successful*-firing, but a failed earlier firing does not hold the watermark back. The design assumes firings within a batch are processed in strictly increasing order *and all succeed*; a single failure between two successes strands the failed one below the advanced watermark.
- **Impact**: Silent loss of inbound webhook deliveries (the whole point of the relay) with only a debug-level log. No retry, no surfaced error — `s.last_error = None` is set at the end of the tick (line 337).
- **Fix sketch**: Do not advance the watermark past the *oldest unpublished* firing. Either (a) process firings oldest-first and stop advancing the watermark at the first failure (carry the highest *contiguous* success), or (b) make the watermark per-firing-id (a seen-set) rather than a timestamp high-water mark, so a gap can't be skipped.

## 5. `cloud_disconnect` clears the in-memory client and keyring but does not stop the sync / remote-command / webhook loops, which keep pushing using the still-valid Supabase JWT
- **Severity**: Medium
- **Category**: resource-leak / state-corruption
- **File**: `src-tauri/src/commands/infrastructure/cloud.rs:242-265` (`cloud_disconnect`); the loops are gated only on `cursor::is_enabled` + leadership (`src-tauri/src/cloud/sync/mod.rs:433`, `src-tauri/src/cloud/remote_commands.rs:173`)
- **Scenario**: User clicks Disconnect. `cloud_disconnect` cancels cloud *executions* and drops the orchestrator `cloud_client` (cloud.rs:255-258), but the **sync writer** and **remote-command poller** are independent loops keyed off `cloud_sync_enabled` and the Supabase auth token (`state.auth.read().await.access_token`, mod.rs:382-387), neither of which `cloud_disconnect` touches. So after "disconnecting," the desktop keeps pushing local data to Supabase every 45s and keeps polling/surfacing remote run-requests. The two notions of "connected" (orchestrator client vs. Supabase sync) are conflated in the UI but decoupled in the backend.
- **Root cause**: "Disconnect from cloud" addresses only the orchestrator client; the local-first Supabase mirror is governed by a separate enable flag the disconnect path never consults. The user's mental model ("I'm offline now") doesn't match the running loops.
- **Impact**: UX/privacy — data continues mirroring to the cloud after the user believes they disconnected; remote-approval prompts can still pop. Recoverable (toggle sync off) but surprising and arguably a data-egress consent violation.
- **Fix sketch**: Make `cloud_disconnect` (or a unified "go offline" action) also disable cloud sync / stop the poll loops, or clearly separate the two toggles in the UI so "disconnected orchestrator" and "sync still mirroring" are never confused. The loops should additionally short-circuit when no orchestrator session exists if the product intent is "one connection."

## 6. OAuth callback `state` is round-tripped but never verified against the pending state — CSRF check is cosmetic
- **Severity**: Medium
- **Category**: crypto / validation-gap
- **File**: `src/stores/slices/system/cloudSlice.ts:231-241` (`cloudCompleteOAuth`), caller `src/features/agents/sub_deployment/components/cloud/CloudDeployPanel.tsx:171-176`, backend `src-tauri/src/commands/infrastructure/cloud.rs:786-795`
- **Scenario**: `cloudStartOAuth` stores `cloudPendingOAuthState` from the authorize response (cloudSlice.ts:212). On completion, `handleCompleteOAuth` passes `pendingOAuthState` (the *locally stored* value) as the `state` arg (CloudDeployPanel.tsx:173) — it is **not** the `state` returned by the OAuth provider/redirect. So the value sent to `cloud_oauth_callback` is whatever the app already had, not what came back from the IdP. The backend (`cloud.rs:788-795`) forwards `code` + `oauth_state` straight to `client.oauth_callback` with no local comparison. The anti-CSRF `state` is therefore never actually compared against the value returned by the authorization redirect — it's echoed back to itself.
- **Root cause**: The code treats the pending `state` as the thing to *send* rather than the thing to *check the redirect against*. The design assumes the manual code-entry flow can't be tampered with, so the state comparison was reduced to a presence check (`if (!pendingOAuthState) return`, cloudSlice.ts:215, CloudDeployPanel.tsx:172).
- **Impact**: Security (lower likelihood given manual code entry) — a forged/substituted authorization `code` paired with the app's own `state` would be accepted; the CSRF protection the `state` parameter exists to provide is not enforced client-side. If a deep-link/redirect handler is ever added that auto-feeds `code`+`state`, this becomes directly exploitable.
- **Fix sketch**: Capture the `state` value that the OAuth redirect actually returns and compare it (constant-time) against `cloudPendingOAuthState`; reject on mismatch *before* calling the callback command. Don't reuse the stored value as the outbound argument.
