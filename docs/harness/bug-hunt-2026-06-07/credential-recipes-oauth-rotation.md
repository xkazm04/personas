# Bug Hunter — credential-recipes-oauth-rotation
> Total: 6
> Critical: 2 · High: 2 · Medium: 2 · Low: 0

Audited the higher-order credential automation surface: OAuth gateway (Google + universal/OIDC), the proactive token-refresh engine, the rotation policy/anomaly engine, and the negotiator/foraging acquisition helpers. The frontend API/store files are thin IPC wrappers; the reliability risk lives almost entirely in the Rust backend, so the findings concentrate there. Token race-protection and CSRF/state validation are largely well-built — the surviving bugs are at the seams: a refresh-token that the provider rotates server-side before the local DB commit lands, a clock landmine that silently rejects every OAuth flow, and a foraging import that trusts the wrong field to decide which service it is writing.

---

## 1. Refresh-token rotation is consumed server-side before the local commit, so any transient encrypt/keyring failure permanently bricks the credential
- **Severity**: Critical
- **Category**: state-corruption
- **File**: `src-tauri/src/engine/oauth_refresh.rs:498-531`
- **Scenario**: The proactive engine (or a keepalive rotation) refreshes a credential whose provider rotates refresh tokens on every exchange (Google, Microsoft, Salesforce, Atlassian all do). `resolve_auth_token` succeeds and returns a brand-new `refresh_token` — at that instant the provider has already **invalidated the old refresh_token server-side**. The code then opens a SQLite transaction and calls `upsert_field_on_conn(... "refresh_token", new_refresh_token ...)` followed by `verify_field_roundtrip_on_conn` (line 520). If that roundtrip verify fails (transient keyring lock on Windows after credential-manager migration, AES seal/unseal hiccup, disk-full on WAL) the whole `tx` rolls back (line 530). The DB keeps the **old** refresh_token, which the provider has already revoked. Every subsequent refresh sends a dead token → `OAuthRevoked` → `mark_needs_reauth`. The credential is permanently broken with no automatic recovery.
- **Root cause**: The design assumes the network exchange and the local persist are part of one atomic unit. They are not: the OAuth provider commits the refresh-token rotation the moment it responds, but the local commit can still fail afterwards. The "atomic persist block" comment (lines 493-497) protects against partial *DB* writes, not against the irreversible *server-side* rotation that already happened.
- **Impact**: Data loss / lockout — silent loss of the credential, requiring a full manual re-authorization, and worse for headless connectors that have no interactive re-auth path.
- **Fix sketch**: Persist the new refresh_token to a durable staging slot *before* (or in the same write as) declaring success, and on commit failure retry the local write with the **already-rotated** token rather than discarding it. Concretely: write `new_refresh_token` to a `pending_refresh_token` field first, retry the commit with backoff, and only fall to `needs_reauth` after the staged token also fails to persist N times. Make the invariant "a refresh_token returned by the provider is never dropped" structurally enforced, not dependent on a single transaction succeeding.

## 2. `now_unix_secs()` returns 0 on any pre-epoch clock, which mints OAuth state tokens that instantly fail their own freshness check — every OAuth flow is rejected
- **Severity**: Critical
- **Category**: edge-case
- **File**: `src-tauri/src/commands/credentials/oauth.rs:330-335` (consumed at 1062-1070 and 1140-1142)
- **Scenario**: A user's machine clock is set before 1970 (dead CMOS battery, manual misconfiguration, a VM that boots with epoch 0, or a timezone/RTC glitch). `SystemTime::now().duration_since(UNIX_EPOCH)` errors, and the `.unwrap_or(0)` silently substitutes timestamp **0**. `generate_oauth_state` then embeds `timestamp = 0` in the HMAC-signed state. On callback, `verify_oauth_state` computes `age = now_unix_secs().saturating_sub(0)` — also 0 if the clock is still bad, OR a huge value once the clock is corrected. With `age > OAUTH_STATE_MAX_AGE_SECS` it returns `Expired`, and the callback handler (oauth.rs:231-239) rejects the flow with "Authorization took too long". Result: **no OAuth connection can ever complete** on that machine, with a misleading "took too long" error that sends the user chasing the wrong cause.
- **Root cause**: `unwrap_or(0)` treats an unrepresentable/invalid clock as a valid timestamp of "the epoch" instead of failing loudly. The whole state-freshness scheme assumes `now_unix_secs()` is monotone-ish and post-epoch; a 0 sentinel violates that silently.
- **Impact**: UX degradation bordering on total feature outage (all OAuth onboarding dead), plus a deceptive error that obscures the real cause (system clock).
- **Fix sketch**: Make clock failure explicit: have `now_unix_secs()` return `Result`/`Option`, and in `start_oauth`/`generate_oauth_state` surface a clear "system clock is invalid — set your clock and retry" error instead of fabricating timestamp 0. At minimum, treat a generated-state timestamp of 0 as a configuration error rather than a stale state.

## 3. Foraging import trusts the user-supplied `service_type` for the DB row but resolves secrets from the `foraged_id`, so the wrong secret can be written under the wrong service
- **Severity**: High
- **Category**: validation-gap
- **File**: `src-tauri/src/commands/credentials/foraging.rs:647-712` and `resolve_real_values` 909-929
- **Scenario**: `import_foraged_credential(foraged_id, credential_name, service_type)` ignores `service_type` when re-reading secrets — `resolve_real_values` discards `_service_type` (line 911) and dispatches purely on `foraged_id`'s prefix. But the credential **row** is created with the caller-supplied `service_type` (lines 668-689), and field encryption sensitivity is chosen by `sensitivity_map_for_connector(pool, &input.service_type)` (credentials.rs:239). A caller can pass `foraged_id = "env:openai"` (which loads `OPENAI_API_KEY`) together with `service_type = "github"`. The vault then stores the OpenAI key as a GitHub credential, encrypted under GitHub's sensitivity map (which may mark different fields as sensitive). The healthcheck/refresh engine later treats it as GitHub → silent auth failures, and the secret may be stored with weaker at-rest handling than its true type warrants.
- **Root cause**: Two independent sources of truth for "what service is this": the `foraged_id` (used to read the secret) and the `service_type` argument (used to write the row). Nothing asserts they agree. The masked-vs-real two-phase design dropped the cross-check.
- **Impact**: Silent corruption (mislabeled credential that never authenticates) and a security weakening (sensitivity classification keyed off the wrong service_type).
- **Fix sketch**: Derive `service_type` authoritatively from the resolved source (return it from `resolve_real_values`) and reject the import if the caller-supplied value disagrees, OR drop the `service_type` argument entirely and use the resolved one. Make the resolved source the single source of truth for both the secret and its classification.

## 4. `rotate_now` does not reschedule `next_rotation_at` on failure, so an event-driven OAuth keepalive that fails leaves a permanently-overdue policy
- **Severity**: High
- **Category**: silent-failure
- **File**: `src-tauri/src/engine/rotation.rs:990-1050` (compare `evaluate_due_rotations` 645-787 which *does* call `schedule_failed_retry`)
- **Scenario**: A credential event (`evaluate_credential_events`, rotation.rs:1239) or a rotation triggered from another path invokes `rotate_now`. On success it advances every enabled policy via `mark_rotated` (lines 1029-1035). On **failure** it records a history row and returns the error — but it never touches the policy's `next_rotation_at`. If that credential also has an enabled scheduled/keepalive policy whose `next_rotation_at` is already in the past, `get_due_policies` keeps returning it, and `evaluate_due_rotations` re-attempts it **every tick**. Because `rotate_now` for OAuth calls `refresh_single_credential` → a real token exchange, this becomes a refresh storm against the provider's token endpoint (rate-limit / temporary IP ban / further breakage). The scheduled path applies exponential backoff; the `rotate_now` path silently does not, so the two interleave and the backoff is effectively defeated whenever a `rotate_now` failure leaves the policy due.
- **Root cause**: Two rotation entry points (`evaluate_due_rotations` and `rotate_now`) with divergent failure handling. Only one of them owns the "reschedule on failure / apply backoff" responsibility; the other can leave the policy in a perpetually-due state.
- **Impact**: Resource exhaustion / retry storm against the OAuth provider, plus accelerated lockout from rate-limit responses being classified as failures.
- **Fix sketch**: On `rotate_now` failure, apply the same `schedule_failed_retry` (with consecutive-failure backoff) that `evaluate_due_rotations` uses, for every enabled policy on the credential. Better: funnel both entry points through one `record_outcome(policy, result)` helper so success/failure rescheduling can never diverge.

## 5. Universal OAuth gateway / OIDC token exchange has no per-grant lock, allowing concurrent `refresh_oauth_token` calls to clobber a rotating refresh token
- **Severity**: Medium
- **Category**: race-condition
- **File**: `src-tauri/src/commands/credentials/oauth.rs:1615-1685`
- **Scenario**: `refresh_oauth_token` is a privileged command that takes a `refresh_token` directly and exchanges it, returning the (possibly rotated) new refresh_token to the frontend. Unlike the engine path (`refresh_single_credential`, which serializes on `oauth_refresh_lock`), this command takes **no lock**. If the UI fires it twice (double-click, re-render, retry-on-slow), or it overlaps with the proactive engine refreshing the same underlying grant, both exchanges run with the same refresh_token. For providers that rotate refresh tokens, the second exchange invalidates the token the first returned; whichever result the frontend persists last may already be dead → lockout.
- **Root cause**: The gateway refresh command is stateless and decoupled from the per-credential lock map, so the "only one refresh in flight per grant" invariant that `oauth_refresh_lock` enforces for the engine does not cover user-initiated/gateway refreshes.
- **Impact**: Recoverable but degrading — intermittent "credential needs re-authorization" after rapid or overlapping manual refreshes, hard to reproduce/diagnose.
- **Fix sketch**: Route gateway refreshes through the same `oauth_refresh_lock::acquire(credential_id)` (or a key derived from provider+client_id when no credential_id is available), and/or debounce the command on the frontend. Make "one in-flight refresh per grant" a single enforced invariant for *all* refresh entry points, not just the engine.

## 6. Foraging YAML/INI parsers trust on-disk content as authoritative, importing attacker-planted secrets into the vault without provenance validation
- **Severity**: Medium
- **Category**: validation-gap
- **File**: `src-tauri/src/commands/credentials/foraging.rs:456-506` (gh hosts.yml), `806-854` (dotenv), `765-804` (aws)
- **Scenario**: The foraging scanners hand-parse `~/.config/gh/hosts.yml`, `.env` in the **current working directory** (`scan_dotenv_files`, lines 327-329), `~/.aws/credentials`, etc., with naive line parsing (e.g. `oauth_token:` substring match, `find('=')` splitting). A `.env` dropped into whatever directory the app happens to launch from (CWD is attacker-influenceable for a desktop app started via file-association, shortcut, or a malicious project folder) is scanned and offered for import as a high-confidence credential. The GitHub-CLI parser keys the host purely off indentation (`current_host` from any line ending in `:`), so a crafted hosts.yml can attribute an arbitrary token to `github.com`. Once imported, the value is decrypted and sent on healthchecks/API calls to the *real* service endpoint — i.e. the app will transmit an attacker-chosen token to GitHub/AWS under the user's identity.
- **Root cause**: The scanner treats "a file on disk that looks like a credential file" as trustworthy provenance. There is no check that the file is in an expected, user-owned location (CWD especially is not), no file-permission/ownership check, and the bespoke parsers accept structurally-loose input that real YAML/INI parsers would reject.
- **Impact**: Security — credential injection / confused-deputy: a planted config file leads the user to import (and then transmit) attacker-controlled secrets. Lower severity only because import requires an explicit user confirmation step.
- **Fix sketch**: Restrict scanning to fixed, home-anchored, user-owned paths (drop CWD `.env`, or clearly label it as untrusted and never high-confidence); verify file ownership/permissions before reading; parse with a real YAML/INI parser; and surface the exact source path in the import confirmation so the user can reject unexpected provenance.

### Notes on areas reviewed and found sound
- OAuth CSRF/state handling is solid: state is HMAC-signed with a per-install keyring secret, the *callback* (untrusted) value is validated for both string-echo and HMAC, and `Expired` is distinguished from `Invalid` (oauth.rs:205-239, 1101-1146).
- The proactive refresh engine correctly takes a per-credential async lock and re-checks freshness after acquiring it (oauth_refresh.rs:332-356), and the keepalive rotation path also funnels through `refresh_single_credential` (connector_strategy.rs:370-373), so engine-vs-keepalive refreshes serialize.
- Rotation history `detail` strings are run through `sanitize_secrets` before storage (rotation.rs:608, 646), and healthcheck ring-buffer messages are sanitized + truncated (rotation.rs:449-455) — no obvious plaintext-secret leak into history.
- `parse_expires_in` correctly handles string-encoded `expires_in` (the silent-daily-401 class), with tests (oauth.rs:347-352, 1774-1808).
- Foraging masks values on scan and only re-reads raw secrets at explicit import time within a single transaction (foraging.rs:158-205, 647-712).
