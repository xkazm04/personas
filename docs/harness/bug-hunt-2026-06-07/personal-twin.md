## 1. UTF-8 byte-slice panics on multi-byte content (memory/reflection generation)
- **Severity**: Critical
- **Category**: edge-case / crash
- **File**: `src-tauri/src/db/repos/twin.rs:587`, with the same class at `src-tauri/src/commands/infrastructure/twin.rs:2138`
- **Scenario**: A user (or persona tool) records any communication whose `content` is ≥500 bytes and has a multi-byte UTF-8 character straddling byte index 500 — e.g. an emoji, an accented name, CJK, or Cyrillic text in a Slack/Discord message. `record_interaction` builds the pending-memory preview with `&content[..content.len().min(500)]`. Slicing a `&str` at a non-char-boundary byte index panics. Because this is a `#[tauri::command]` invoked synchronously, the panic propagates as a poisoned/aborted command and the entire "record interaction" path fails. The twin's owner is highly likely to have a non-ASCII name or to converse in a non-English language (`languages`/`pronouns` are first-class profile fields, and the i18n tree ships ar/bn/hi/ja/ko/ru/vi/zh), so this is a common path, not adversarial. The identical pattern at `twin_reflect` (`&c.content[..240]`) panics whenever any of the last 40 communications crosses byte 240.
- **Root cause**: The code treats `&str` byte offsets as character offsets. `min(500)` only bounds length, never aligns to a `char` boundary. The design assumed ASCII-only content for a feature whose whole purpose is multilingual personal communication.
- **Impact**: Crash / data loss — the communication is inserted (line 576-580) but the pending-memory creation panics afterward, and the command returns an error to the UI even though the row was written, leaving the store and DB inconsistent. Repeatable hard crash of a core write path.
- **Fix sketch**: Never byte-slice user text. Use a char-aware truncation helper (`content.chars().take(500).collect::<String>()`, or `floor_char_boundary`) and centralize it so every snippet site (`record_interaction`, `twin_reflect`, wiki `created_at[..10]`) goes through one safe truncator. Add a unit test feeding an emoji at the boundary.

## 2. Pending-memory "dig deeper" approves before AI succeeds, and approve/reject controls allow cross-item races
- **Severity**: High
- **Category**: race-condition / state-corruption
- **File**: `src/features/plugins/twin/sub_knowledge/KnowledgeAtelier.tsx:115-147` (dig-deeper), `:340-359` (button disable logic)
- **Scenario**: Two distinct races. (a) In `handleDigDeeper`, the memory is approved (`reviewMemory(memId, true, 'dig_deeper')` at line 138) only AFTER the follow-up-question CLI call returns. If the user clicks "Dig deeper", the CLI runs for 10-30s, and meanwhile the user (or a second tab) opens the reject flow on the same card and rejects it, the backend `review_pending_memory` guards on `status='pending'` (repo line 478) so the second write wins; then the dig-deeper's later approve fails silently inside `reviewMemory` (the store swallows it via `reportError`, line 468-470) — the user is routed to Training believing the memory was approved, but it is actually rejected. (b) The per-card buttons disable only on `reviewingId === mem.id || rejectingId !== null || digDeeperId !== null`, but `reviewingId` is set inside `handleReview` (line 107) AFTER the async store call is already dispatched; a fast double-click on Approve fires two `reviewMemory` calls before the first `setReviewingId` paints, so the optimistic list map (store line 462-467) runs twice.
- **Root cause**: The approval gate is an optimistic, fire-and-forget UI guard with no per-item in-flight lock established before dispatch, and dig-deeper inverts the safe order (act on AI result, then commit the irreversible state change) so a long-running AI window leaves the item mutable.
- **Impact**: Silent corruption — memory shown as approved in one surface, rejected in DB; double-apply churn; user trust in the human-review gate is undermined.
- **Fix sketch**: Set the in-flight lock synchronously before any await (or make the store action return a discriminated result the caller checks). Approve the memory FIRST in dig-deeper, then generate questions; if generation fails, the memory is still legitimately approved. Surface the `status='pending'` guard failure (repo returns `NotFound` "already reviewed") to the user instead of swallowing it.

## 3. ReplyOutbox approve-and-log silently records to the wrong channel/recipient (no binding to the drafted context)
- **Severity**: High
- **Category**: silent-failure / validation-gap
- **File**: `src/features/plugins/twin/sub_channels/ReplyOutbox.tsx:78-115`
- **Scenario**: The operator selects channel = "discord", contact = "@alice", generates a draft via `draftReply`. Before clicking Approve, they change the Channel dropdown to "email" or edit the Contact field to "@bob" (both are live `useState` with no lock while a draft exists). `handleApprove` reads the CURRENT `channelType` and `contactHandle` (lines 95, 103-105), not the values the draft was generated against, and records the outbound communication under the new channel/recipient. The drafted prose — written in the Discord tone for Alice — is now logged as a "sent" email to Bob. Because `twin_record_interaction` also auto-creates a pending memory (`create_memory` defaults true, command line 512), this wrong-channel attribution propagates into the memory/recall layer and into `twin_contacts` (auto-upserted from communications, repo line 810-826), permanently skewing per-contact recall and the contact's `message_count`.
- **Root cause**: The draft and its generating context (channel + contact) are decoupled. The staged draft lives in the store while the channel/contact selectors remain freely editable, so "approve" captures a different context than "generate" did. The system assumes the operator never changes the selectors between the two clicks.
- **Impact**: Silent data corruption of the communications/contacts/recall tables plus mis-attributed "sent" record on the user's behalf — exactly the high-impact "authoring on the user's behalf to the wrong recipient" class.
- **Fix sketch**: Capture `{channel, contactHandle, inbound}` into the draft payload at generation time and record against that frozen tuple, not the live form state. Lock or clear the draft whenever the channel/contact selectors change (an effect already exists for defaulting; add the inverse: changing channel while `replyDraft !== null` discards or re-tags the draft). Show the bound channel/recipient in the draft panel header so the operator confirms what they're approving.

## 4. Concurrent profile edits clobber tone/voice/identity via full-field overwrite (lost update)
- **Severity**: High
- **Category**: state-corruption / race-condition
- **File**: `src/features/plugins/twin/sub_identity/IdentityAtelier.tsx:68-81` + `src/stores/slices/system/twinSlice.ts:336-346` + repo `update_profile` `src-tauri/src/db/repos/twin.rs:157-227`
- **Scenario**: The identity form snapshots the profile into local `useState` on mount (lines 42-52). If a second surface mutates the profile in the interim — e.g. `bindKnowledgeBase`/`unbindKnowledgeBase` (which bump `updated_at` and the row), or `twin_generate_bio` flow in another tab, or `twin_set_active_profile` — and then the user clicks Save here, `handleSave` sends the STALE local copy of name/bio/role/pronouns/obsidianSubpath. `update_profile` is a partial update but unconditionally writes every field the caller supplied (no `updated_at`/version precondition), so the stale bio overwrites the newly generated one. The store's `updateTwinProfile` then replaces the cached row with this stale result (slice line 339-342), and the error branch (line 343-345) returns void rather than rethrowing, so a failed update produces no user-visible signal and leaves the optimistic-looking UI untouched.
- **Root cause**: Read-modify-write with no optimistic-concurrency check. The form assumes it holds the only writer; the backend has no `WHERE updated_at = ?expected` guard, so the last writer always wins and silently discards intervening changes.
- **Impact**: Silent loss of profile/bio/role edits — the most user-visible state in the feature — with no conflict surface.
- **Fix sketch**: Add an `updated_at` (or integer `version`) precondition to `update_profile`'s `WHERE` clause; return a distinct `Conflict` error when 0 rows match so the UI can reload and prompt. On the client, re-hydrate form state on profile change instead of only on `id` change, and dirty-merge rather than blind-overwrite.

## 5. Distilled facts have no dedup or contradiction handling — stale/conflicting facts persist and re-enter every prompt
- **Severity**: Medium
- **Category**: state-corruption / validation-gap
- **File**: `src-tauri/src/db/repos/twin.rs:999-1048` (`create_distilled_fact`) and `:1058-1088` (`top_distilled_facts_for_recall`)
- **Scenario**: A fact "I work at Acme" is distilled. Months later the user changes jobs and a new fact "I work at Globex" is created. `create_distilled_fact` does no similarity check, no supersede, no contact-scoped uniqueness — it just inserts. Both rows now coexist with the same/high importance. `top_distilled_facts_for_recall` orders only by `importance DESC, last_seen_at DESC` and returns the top N (12 for drafts/answers). Both contradictory facts can land in the same prompt block labeled "stay consistent — never contradict these" (`build_reply_prompt` line 900, `build_answer_prompt` line 986). The twin then authors replies grounded on mutually contradictory "known facts", with the OLD fact often outranking the new one if its importance was set higher. `last_seen_at` is set once at creation and never refreshed, so "recency" is really "creation time" — a fact is never aged out.
- **Root cause**: The table is a manual append-only log with the doc comment explicitly deferring dedup/contradiction to "Stage 3" (panel comment, `DistilledFactsPanel.tsx:23-26`), but recall already consumes it as authoritative ground truth for content the twin sends on the user's behalf. The assumption that operators will manually delete stale facts does not hold at scale.
- **Impact**: UX degradation / reputational — the twin confidently asserts outdated or contradictory personal facts in drafted communications; no recovery path short of manual pruning.
- **Fix sketch**: Add a `(twin_id, contact_handle, normalized_content)` uniqueness/upsert that refreshes `last_seen_at` and bumps importance on re-observation; introduce an explicit `superseded_by`/`active` flag so a new fact retires its predecessor; have recall exclude superseded rows. Until then, cap recall to the single highest-ranked fact per normalized topic.

## 6. Studio answer-drafting silently drops failed CLI calls and reports the batch "completed"
- **Severity**: Medium
- **Category**: silent-failure
- **File**: `src-tauri/src/commands/infrastructure/twin.rs:1345-1376`
- **Scenario**: `twin_studio_generate_answers` loops N seed questions, each `spawn_claude_with_prompt(prompt)` wrapped in `tokio::select!` with `res = ... => res.ok()` (line 1347). Any CLI failure on an individual question (non-zero exit, spawn failure, transient rate-limit) becomes `None` via `.ok()`, the item is still pushed with `answer: None`, `completed` is incremented regardless (line 1354), and the batch finishes with status `"completed"` plus a "N answers drafted" success notification (line 1371-1375) — even if every single answer failed. The frontend only inspects `studioJustCompleted.status` (slice line 289) which is `"completed"`, so the user is told drafts are ready when some/all are empty.
- **Root cause**: Per-item errors are coerced to a success-shaped result (`None` answer counted as completed) and the batch-level status reflects "loop finished" rather than "work succeeded" — classic success theater. Only whole-batch cancellation maps to `"failed"`.
- **Impact**: UX degradation — the user opens the Studio expecting N reviewed drafts and finds blanks, with no indication which failed or why; wasted CLI spend is invisible.
- **Fix sketch**: Track a `failed` count alongside `completed`; if any item failed, finish with a `"completed_with_errors"` status (or include `failed`/`error` in the complete payload and `TwinStudioBatch`) and word the notification accordingly. Keep the per-item `answer: None` so partial results survive, but never report unqualified success when failures occurred.
