## 1. Autonomous "Stop = type anything" cancel is defeated by the same turn that schedules the next tick
- **Severity**: Critical
- **Category**: race-condition
- **File**: `src-tauri/src/companion/session.rs:832` (and `:149`, `commands/companion/chat.rs:77`)
- **Scenario**: Autonomous mode is on. Athena finishes a turn that emitted `OP: continue_autonomously`, so `schedule_autonomous_tick` is spawned with a 15s delay. ~5s in, the user types "stop please" to break the loop. `companion_send_message` calls `cancel_pending_autonomy()` first (chat.rs:77-79), which sets the single global `AUTONOMOUS_CANCEL = true`. That user turn itself runs under `autonomous_mode = true` (the panel passes the toggle), and if Athena's reply to "stop" *also* contains `continue_autonomously` (very common — the model treats it as conversational, not a control word), `send_turn` reaches `schedule_autonomous_tick` again → `reset_autonomous_cancel()` (session.rs:832) clears the very flag the user just set. The originally-pending tick, still polling, now sees `cancelled == false` and fires. The loop the user tried to stop keeps running.
- **Root cause**: Cancellation is modeled as ONE global `AtomicBool` shared by every pending tick, with no generation/epoch token tying a cancel to the specific tick it should kill. `reset` and `cancel` race on the same cell, and the scheduler resets unconditionally instead of honoring an outstanding cancel.
- **Impact**: The user's primary "stop the autonomous agent" affordance silently fails; an always-on meta-agent keeps spending tokens and taking actions after the user explicitly asked it to halt. Loss of user control over an autonomous system.
- **Fix sketch**: Replace the bool with a monotonic generation counter (`AtomicU64`). Each scheduled tick captures `my_gen = GEN.load()`; `cancel_pending_autonomy()` does `GEN.fetch_add(1)`; the tick aborts whenever `GEN.load() != my_gen`. Never "reset" — only ever advance. A user message advances the generation; a new continuation captures the new generation, so a stale tick can never be revived by a later schedule.

## 2. Proactive daily-budget cap is non-atomic (read-then-increment) — concurrent passes burst past the cap → notification storm
- **Severity**: Critical
- **Category**: race-condition
- **File**: `src-tauri/src/companion/proactive/budget.rs:48` + `:36`; caller `src-tauri/src/companion/proactive/mod.rs:102`-`128`, `commands/companion/proactive.rs:41`-`83`
- **Scenario**: The background scheduler tick and the user clicking the debug "evaluate now" button (or two ticks overlapping on a slow embed turn) both enter `evaluate_with_extra_candidates` near-simultaneously. Each calls `budget::today()` which reads `used = 0` into its own `DailyBudget` struct. Both then loop candidates, each checking its *local* `is_exhausted()` and calling `increment()` (an `INSERT … ON CONFLICT DO UPDATE count = count + 1`). The DB count ends correct, but each in-memory budget independently allowed up to `DAILY_CAP` (3) deliveries before noticing — so the user receives up to 6 "Athena reached out" nudges in one window instead of 3. The same TOCTOU exists between the trigger path and `deliver_due_scheduled` within a single pass for scheduled check-ins.
- **Root cause**: The cap is enforced against a snapshot copy of the counter read once at the top of the pass, not against the authoritative row at the moment of each delivery decision. There is no row lock / transaction spanning the read+increment, and `DailyBudget` is per-call state.
- **Impact**: Notification storm — the exact failure the budget exists to prevent. Defeats the "drip, not spam" design contract for an always-on companion.
- **Fix sketch**: Make increment authoritative and atomic: `UPDATE … SET count = count + 1 WHERE date = ?1 AND count < cap` (insert the row first if absent), and treat `rows_affected == 0` as "exhausted, stop." Decide-by-write instead of decide-by-cached-read. Optionally serialize the whole evaluate pass behind a `tokio::Mutex` so only one proactive evaluation runs at a time.

## 3. Doctrine ingest upsert is read-then-write across separate pooled connections — concurrent ingest duplicates every chunk
- **Severity**: High
- **Category**: state-corruption
- **File**: `src-tauri/src/companion/brain/doctrine.rs:496`-`595` (lookup at `:502`, insert at `:565`)
- **Scenario**: `ingest_all` runs automatically on `companion_init` (app start) and is *also* exposed via `companion_reingest_doctrine` (mod.rs:287, the "refresh doctrine" button). A user clicks Refresh while the startup ingest is still running (or two refreshes double-click). For each chunk, `upsert_chunk` does a `SELECT … WHERE kind='doctrine' AND file_path=?` on one pooled connection, sees `None`, then later does an `INSERT` with a fresh random id on another pooled connection. With no UNIQUE constraint enforced in this path and no transaction wrapping select+insert, both runs see `None` for the same `(doc_path, anchor)` and both INSERT — producing two `companion_node` rows (and two FTS/embedding rows) for one logical chunk.
- **Root cause**: Idempotency is asserted by an application-level "lookup then branch" instead of a DB-level upsert (`INSERT … ON CONFLICT(file_path) DO UPDATE`) or a transaction holding the row. The "Idempotent: safe to call on every app start" guarantee in the module doc assumes single-threaded execution that isn't guaranteed.
- **Impact**: Brain long-term memory grows with duplicate doctrine chunks; retrieval returns near-identical hits, wasting the doctrine budget and skewing what Athena "knows." `prune_orphans` won't clean them (both file_paths are in `seen_keys`). Silent, compounding corruption on every concurrent ingest.
- **Fix sketch**: Add a UNIQUE index on `companion_node(kind, file_path)` and switch the insert to `INSERT … ON CONFLICT DO UPDATE`. Or wrap select+insert in a single transaction (`BEGIN IMMEDIATE`) per chunk. Best: serialize ingest behind a process-level lock so a refresh can't overlap the startup pass.

## 4. Swallowed `mark_delivered` failure causes the proactive panel to re-emit the same nudge every pass (re-notification loop)
- **Severity**: High
- **Category**: silent-failure
- **File**: `src-tauri/src/commands/companion/proactive.rs:60`-`82`
- **Scenario**: A pass produces a new nudge. The loop calls `mark_delivered` (proactive/mod.rs:375, `UPDATE … SET status='delivered' WHERE id=? AND status='queued'`). If that update affects 0 rows or errors transiently (busy lock, the row already moved by a concurrent resolve), the code only `tracing::warn!`s and continues — it does NOT drop the message from `new_msgs`. It then emits `companion://proactive` with that message anyway (line 78). The row stays `status='queued'`, so the dedupe guard in `enqueue_if_new` (which only blocks on `queued`/`delivered`) sees it as still-unresolved and won't re-enqueue — but the *next* evaluate pass's `deliver_due_scheduled`/trigger path can re-surface and re-emit, and the panel keeps popping a card whose backing row never advanced to `delivered`.
- **Root cause**: Delivery is treated as best-effort logging rather than a state machine gate. The emit is decoupled from the success of the `queued → delivered` transition, so an un-transitioned row is announced as delivered.
- **Impact**: Repeated "Athena reached out" cards for the same nudge; a stuck `queued` row that the budget already charged for. Degrades trust in the proactive surface and can become a soft notification loop on a busy DB.
- **Fix sketch**: Have `mark_delivered` return `rows_affected`; only push a message into the emit payload when the transition actually succeeded (== 1). On failure, leave it out of the event and let the next pass retry the transition cleanly. Make the status transition the single source of truth for "this was delivered."

## 5. Approval auto-execute trusts the approval row's `action` but re-loads `params` that another writer could have mutated between propose and approve
- **Severity**: High
- **Category**: auto-action
- **File**: `src-tauri/src/commands/companion/approvals.rs:338`-`403` (auto path), `:407`-`454` (`load_pending`), guard `:2183`
- **Scenario**: Under autonomous mode, `fleet_send_input` is on the autoapprove allowlist (approvals.rs:322). `auto_resolve_if_allowed` checks `fleet_send_input_targets_athena_session(&approval.params_json)` using the `CreatedApproval.params_json` captured at propose time (line 357), then calls `load_pending` which re-reads `payload` from the DB and executes against *that* (line 367, 382). The guard and the executor read two different copies of params. The `companion_approval.payload` column is plain JSON with no integrity check; any code path or manual DB edit that rewrites `payload.params.session_id` between propose and the tick's `load_pending` makes the guard validate the OLD (Athena-owned) session while the executor types `{text}\r` into the NEW session id — including the user's own live terminal. The `is_athena_owned` check is also a pure point-in-time registry lookup (registry.rs:141, name `starts_with("athena")`), so a user who names a session "athena-notes" passes the sentinel guard.
- **Root cause**: Time-of-check (guard reads `CreatedApproval.params_json`) is separated from time-of-use (executor reads freshly-loaded `payload.params`), and ownership is proven by a mutable display-name prefix rather than an immutable spawn-provenance token. The autonomous path executes a PTY write with no second human confirmation.
- **Impact**: Security / unintended action — autonomous Athena can write keystrokes into a terminal she didn't create, driving arbitrary input into the user's session. Highest-blast-radius entry on the allowlist.
- **Fix sketch**: Guard and execute against the SAME loaded params (validate inside `load_pending`'s returned value, not the propose-time copy). Replace the name-prefix sentinel with an immutable provenance flag stamped at spawn (e.g. a `spawned_by_athena` bool keyed on the registry's internal id, not the renameable visible name). Consider keeping `fleet_send_input` off the autoapprove allowlist entirely — a PTY write is not "low-blast-radius reversible."

## 6. STT accepts any base64 as "audio" and feeds it to whisper without WAV-format validation
- **Severity**: Medium
- **Category**: validation-gap
- **File**: `src-tauri/src/commands/companion/stt.rs:36`-`46`; `src-tauri/src/companion/stt/mod.rs:32`-`45`; `src-tauri/src/companion/stt/whisper.rs:142`-`198`
- **Scenario**: `companion_stt_transcribe` base64-decodes the payload and validates only byte *length* (`validate_audio_len`: 76 bytes … 25 MB). Any blob in that size band — a truncated/partial mic capture cut mid-utterance, a non-WAV container, or random bytes — is written verbatim to `in.wav` and handed to the `whisper-cli` subprocess. whisper.cpp on a malformed/headerless WAV produces empty or garbage stdout (or a non-zero exit), which surfaces as a raw `whisper exited with …` Internal error, and a partial utterance silently transcribes to an empty/wrong string with no signal to the caller that the audio was incomplete.
- **Root cause**: The IPC boundary validates size and language but never asserts the bytes are a well-formed 16 kHz mono PCM WAV (RIFF/WAVE magic, fmt chunk, sample rate). The contract "frontend sends a 16 kHz mono PCM WAV" is assumed, not enforced, so partial/adversarial captures reach the engine.
- **Impact**: Voice turns from a clipped capture silently yield wrong/empty transcripts (Athena acts on a half-heard command), and malformed input turns into confusing engine errors rather than a clean "couldn't hear that" message. Degraded, occasionally wrong, voice UX.
- **Fix sketch**: Before spawning, parse the WAV header (verify `RIFF`/`WAVE`, `fmt ` chunk = PCM, channels == 1, sample_rate == 16000, and a non-trivial `data` chunk length); reject with a clear `Validation` error otherwise. Optionally enforce a minimum effective audio duration so sub-200ms partials are rejected up front instead of transcribed to noise.
