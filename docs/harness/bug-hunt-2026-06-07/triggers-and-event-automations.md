## 1. Smee relay re-fires every buffered webhook on each SSE reconnect (no event de-duplication)
- **Severity**: Critical
- **Category**: race-condition / state-corruption
- **File**: `src-tauri/src/engine/smee_relay.rs:258-392` (stream loop), `src-tauri/src/engine/smee_relay.rs:567-630` (reconnect/backoff loop)
- **Scenario**: A persona is wired to a smee.io channel that receives GitHub deploy webhooks. The user's laptop sleeps / drops Wi-Fi for 90 seconds, or smee.io rate-limits and closes the TCP connection. The relay loop catches the disconnect, backs off (`backoff` 1s→30s), then calls `relay_sse_core` again (line 583), which issues a fresh `GET` to the channel URL. smee.io's SSE endpoint replays the channel's recent event history on every new connection. Every one of those already-processed webhooks is parsed and published again via `event_repo::publish` (line 376), each spawning a duplicate persona execution.
- **Root cause**: The relay treats the SSE stream as an exactly-once, never-replayed firehose. It keeps `events_relayed` counters across reconnects (lines 232-237) but keeps **no per-event identity/watermark**. There is no `Last-Event-ID` header sent on reconnect and no dedup set (`grep` for `last_event_id|dedup|seen_ids` returns nothing). The sibling cloud-webhook relay solves exactly this with a persisted `cloud_webhook_watermarks` row (`cloud_webhook_relay.rs:240-317`); the smee path has no equivalent.
- **Impact**: Duplicate/storm executions on every transient network blip — real money (LLM cost), duplicate PRs, duplicate notifications, possible feedback loops. Worst on flaky networks, which is the common case for a desktop app.
- **Fix sketch**: Make replay idempotent at the trust boundary. Capture the SSE `id:` field (or hash the normalized `data` payload + `x-github-delivery` header) and persist a bounded recent-id set / high-water mark per relay; skip publish when the id was already seen. Additionally send the `Last-Event-ID` request header on reconnect so smee resumes instead of replaying. This makes the entire class of "stream replays on reconnect" impossible regardless of how the disconnect happened.

## 2. `seconds_until_next_open` and `is_active_at` mis-handle overnight active windows that cross a day boundary
- **Severity**: High
- **Category**: edge-case
- **File**: `src-tauri/src/db/models/trigger.rs:164-199` (`is_active_at`), `src-tauri/src/db/models/trigger.rs:205-250` (`seconds_until_next_open`)
- **Scenario**: A user configures a trigger active only on `days: [1]` (Monday) with an overnight window `start_hour: 22`, `end_hour: 6` (22:00 → 06:00). At 02:00 Tuesday — which is the back half of the *Monday-night* window — `is_active_at` reads `weekday = Tuesday (2)`, finds `2` is not in `days`, and returns `false` (line 186), suppressing the trigger during a period the user intended to be active. Conversely `seconds_until_next_open` (line 232) only checks `now_minutes < start_minutes` against today's start, so for an overnight window it never reports the trigger as "already open" and computes wake times that disagree with `is_active_at`.
- **Root cause**: The design assumes a window belongs to the calendar day of its *evaluation instant*, but an overnight window logically belongs to the day of its *start*. The `days` membership test (line 186) and the start-time-only "today" branch (line 232) were written for same-day windows (`start <= end`, line 193) and never reconciled with the overnight branch (line 195-197).
- **Impact**: Silent UX corruption — triggers fail to fire (or fire) at the wrong wall-clock times for any day-scoped overnight schedule; the sleep-until-open path and the is-active path can disagree, causing a trigger to wake then immediately decide it is inactive.
- **Fix sketch**: For overnight windows, when `now_minutes < end_minutes` evaluate `days` membership against *yesterday's* weekday (the window's start day), and in `seconds_until_next_open` add an "is the window currently open?" check that mirrors the overnight branch in `is_active_at` so the two functions share one predicate. Unit-test the 02:00-Tuesday-belongs-to-Monday case explicitly.

## 3. Chain cascade marks the trigger fired (CAS-less) before publishing, allowing perpetual suppression or lost fires under concurrency
- **Severity**: High
- **Category**: race-condition / silent-failure
- **File**: `src-tauri/src/engine/chain.rs:282-379`
- **Scenario**: Two execution-completion handlers for the *same* source persona run nearly simultaneously (e.g. a fan-out where persona A completes two queued runs, or a restart sweep racing the live handler). Both load the same chain trigger via `get_chain_triggers_for_source` (line 128), both pass the cycle/predicate checks, and both call `mark_triggered(pool, &trigger.id, None, trigger.trigger_version)` (line 289) with the **same stale `trigger_version`**. Unlike the schedule path (`background.rs:1817`, which branches on `Ok(true)`/`Ok(false)` from the CAS), the chain path treats *any* `Ok(_)` as success (line 290) and never inspects whether 0 rows were affected. Both branches then publish (line 381), double-firing the downstream persona.
- **Root cause**: `PersonaTrigger.trigger_version` exists specifically as a "monotonic version counter for race-safe CAS" (`trigger.rs:440-442`), and `mark_triggered` returns a bool indicating whether the CAS won — but the chain evaluator discards that bool. The design assumes chain evaluation is single-threaded per source, which is not guaranteed across the live handler + startup overdue sweep + concurrent completions.
- **Impact**: Duplicate downstream executions (cost + duplicate side effects), the exact class the surrounding comment (lines 282-285) claims to prevent. Symmetrically, if a real concurrent winner already advanced the version, the loser still publishes against a trigger it did not actually claim.
- **Fix sketch**: Branch on the `mark_triggered` result like the scheduler does: `Ok(true)` → publish; `Ok(false)` → another tick claimed it, `continue` without publishing; `Err` → existing retry/quarantine path. This makes "two evaluators publish for one logical fire" structurally impossible via the existing CAS.

## 4. Cloud-webhook relay duplicates all recent firings after restart because the in-memory relay counters reset but the API returns history within the watermark window
- **Severity**: Medium
- **Category**: edge-case / silent-failure
- **File**: `src-tauri/src/engine/cloud_webhook_relay.rs:240-264`, `src-tauri/src/engine/cloud_webhook_relay.rs:55-74`
- **Scenario**: The watermark is persisted and reloaded correctly (`load_from_db`, line 55), so steady-state dedup works. But the cutoff comparison at line 255 is `f <= c` — strictly-equal timestamps are skipped, which is correct only if `fired_at` has sub-second precision. Cloud firings stored at second granularity that share a `fired_at` with the watermark are skipped; but a *new* firing minted in the same whole second as the watermark (`f == c`) is **silently dropped** (line 255, `(Ok(f), Ok(c)) if f <= c => continue`). Two webhooks landing in the same second → only the first is ever relayed; the second is lost forever once the watermark advances to that second.
- **Root cause**: Using a timestamp as both the dedup cursor and the only ordering key assumes timestamps are unique. The `<=` cutoff conflates "already processed this exact firing" with "another firing happened to share the second."
- **Impact**: Silent event loss for bursty webhooks (CI matrix builds, rapid GitHub pushes) — a persona simply never reacts to the second-in-a-second event, with no error logged.
- **Fix sketch**: Dedup on the firing **id** (`firing.id`), not on `fired_at` alone — keep the timestamp only as a coarse pre-filter (`f < c` strict), then exclude already-seen ids. Persist a small recent-id set alongside the watermark, or rely on a unique constraint on `source_id` (the firing id) at insert and treat the conflict as "already relayed."

## 5. Optimistic store mutations (`createTrigger`/`updateTrigger`/`deleteTrigger`) emit `trigger:changed` only on success, but firing-rate state is never reconciled and SmeeRelayTab/CloudWebhooksTab delete is fire-and-forget on failure
- **Severity**: Medium
- **Category**: state-corruption / silent-failure
- **File**: `src/features/triggers/sub_cloud_webhooks/CloudWebhooksTab.tsx:115-121`, `src/features/triggers/sub_smee_relay/SmeeRelayTab.tsx:108-120`
- **Scenario (CloudWebhooks)**: User clicks delete on a webhook trigger. `handleDelete` (line 115) `await cloudDeleteTrigger`; on success it filters the row out of local state. If the backend delete *fails*, the error is swallowed by `silentCatch` (line 120) and the row stays — fine. But the **smee** delete does the inverse and worse: `handleDelete` (`SmeeRelayTab.tsx:108`) optimistically schedules row removal via `setTimeout(...300ms)` *inside the try after the await*, but if `smeeRelayDelete` throws, `silentCatch` eats it (line 119) and the row remains with **no user feedback at all** — the confirm dialog was already dismissed conceptually and the user believes nothing happened, or retries and double-deletes. There is no rollback contract and no toast.
- **Root cause**: Delete handlers conflate "request issued" with "request succeeded," and route failures to `silentCatch` (a black hole) rather than the `TriggerError`/toast channel the slice was designed for (`triggerSlice.ts:37-42` defines a `fetch`-kind error precisely for this). The mutation success path is the only path that surfaces anything.
- **Impact**: UX degradation and confusing state on backend failure (offline cloud, locked DB): silent no-ops, possible duplicate delete attempts, list that disagrees with the database until manual refresh.
- **Fix sketch**: On delete/create/update failure, surface a toast/inline error instead of `silentCatch`; for optimistic UI, snapshot the row and restore it in the `catch`. Centralize through the slice's `triggerError` channel so every trigger surface presents failures uniformly (the slice already documents this contract).

## 6. `publish_event` / `test_event_flow` rate-limit by `source_type` only, so any caller can starve a legitimate source by flooding under a shared source string
- **Severity**: Medium
- **Category**: validation-gap
- **File**: `src-tauri/src/commands/communication/events.rs:59-86`, `src-tauri/src/commands/communication/events.rs:144-180`
- **Scenario**: The event-source rate limiter keys on `format!("event:{}", input.source_type)` (line 71). `source_type` is fully caller-controlled free text. A buggy or malicious automation that publishes many events under, say, `source_type: "smee_relay"` consumes the shared per-source bucket; the real smee relay (`smee_relay.rs:368`, `source_type: "smee_relay"`) and any other component publishing under that same string then get `RateLimited` (line 77) and silently drop events (relay path logs at `debug` and moves on, `smee_relay.rs:388`). Because the bucket is shared by string, one noisy producer throttles an unrelated trusted producer.
- **Root cause**: The throttle granularity assumes `source_type` is a trusted, low-cardinality identity, but it is an untrusted attribute supplied per-call with no allowlist or per-persona/per-trigger dimension. Distinct logical producers that happen to share a `source_type` share one limit.
- **Impact**: Cross-tenant-style throttling / event loss between unrelated producers; an internal relay's events get dropped because an unrelated source flooded the same label. Recoverable but causes missed automations with only a debug log.
- **Fix sketch**: Key the limiter on a composite of trusted dimensions (e.g. `source_type` + `source_id`/`target_persona_id`, or an internal channel identity) rather than caller-supplied `source_type` alone, and validate `source_type` against the known event-source set. Treat relay-publish `RateLimited` as a surfaced warning/metric, not a debug-level drop, so starvation is observable.
