## 1. Backend safe-mode classifier treats CTE-wrapped writes as read-only (WITH ... DELETE/INSERT/UPDATE bypass)
- **Severity**: Critical
- **Category**: validation-gap / injection (trust-boundary bypass)
- **File**: `src-tauri/src/engine/db_query.rs:221` (`is_mutation`), `:200` (`is_sqlite_read`), enforced at `:311` and `:2070`
- **Scenario**: A persona / user (the SQL explorer is exposed to AI-generated SQL via the chat tab) runs, in safe mode, a query such as `WITH deleted AS (DELETE FROM kb_documents RETURNING *) SELECT * FROM deleted`. `extract_first_keyword` returns `WITH`, which `is_mutation` (line 228) and `is_sqlite_read` (line 206) both classify as read-only. The guard at line 311 (`if !allow_mutation && is_mutation(...)`) passes, and `execute_local_sqlite` (line 2070) routes it into the read branch `conn.prepare(trimmed) → stmt.query([])`. SQLite fully executes the embedded `DELETE`. The user is never shown the mutation-confirmation dialog because the data-modifying intent was hidden behind the `WITH` keyword.
- **Root cause**: Safety classification is keyword-prefix-only and assumes the leading verb determines whether a statement mutates. SQLite (and Postgres/PlanetScale/Neon) support data-modifying CTEs where the leading keyword is `WITH` but the body contains `DELETE`/`INSERT`/`UPDATE`. The frontend (`safeModeUtils.ts:62`) DOES scan the CTE body for mutation verbs — but the frontend is only UX; the Rust layer is the real enforcement boundary and it does not.
- **Impact**: Silent data loss / corruption against the local user DB and against connected external DBs (Neon/PlanetScale execute raw SQL), with the destructive op disguised as a read. Safe mode — the headline guarantee of the SQL explorer — is fully bypassable.
- **Fix sketch**: After extracting the leading keyword, when it is `WITH`, strip string/identifier literals and scan the remaining statement body for `\b(DELETE|INSERT|UPDATE|MERGE|REPLACE|UPSERT|TRUNCATE)\b` (mirror the frontend's `MUTATION_VERBS_RE`). Better: for local SQLite, use `conn.prepare(sql)` then `stmt.readonly()` (rusqlite exposes the authorizer/`sqlite3_stmt_readonly`) as the source of truth instead of keyword heuristics — that makes the whole class of "leading keyword lies about mutation" impossible.

## 2. kb_search never validates query embedding model / dimensions against the KB index
- **Severity**: Critical
- **Category**: silent-failure / state-corruption
- **File**: `src-tauri/src/commands/credentials/vector_kb.rs:691` (`embed_query`) and `:700` (`vector_store.search`); index built at `:54` with `embedder.dimensions()`; `embedding_model`/`embedding_dims` stored at create (`:46`,`:62`) but never re-read on search
- **Scenario**: A KB is created on app version A whose `EmbeddingManager::dimensions()` returns 384 / model `AllMiniLML6V2Q` (embedder.rs:45-52). Later the app ships a different default model (or a user/dev changes the constant, or a future migration adds model choice). On search, `embed_query` (vector_kb.rs:691) embeds the query with the *current* manager, producing an N-dim vector, and passes it straight to `vector_store.search` against a `vec0` table declared `embedding float[384]` (vector_store.rs:61). The stored `knowledge_bases.embedding_model`/`embedding_dims` columns are loaded everywhere EXCEPT the search path — they are never compared.
- **Root cause**: Design assumes the embedding model is globally constant for the process lifetime and across versions, so the per-KB model/dims recorded at creation are treated as cosmetic. Nothing reconciles "what model built this index" with "what model is embedding this query." If dims differ, sqlite-vec errors (search returns `Err`, surfaced as Internal); if dims happen to match but the *model* differs (e.g. another 384-dim model), the search silently returns semantically wrong neighbors with plausible-looking scores.
- **Impact**: Either hard search failure on every query for older KBs, or — worse — silent wrong results (success theater): the persona retrieves and reasons over the wrong chunks with no error. Cross-model contamination is undetectable to the user.
- **Fix sketch**: In `kb_search`, load the KB row first and assert `kb.embedding_model == embedder.model_name()` and `kb.embedding_dims as usize == embedder.dimensions()` before embedding; on mismatch return a typed `AppError` instructing the user to re-index. Make dims a checked invariant by also asserting `query_vec.len() == kb.embedding_dims` before calling `vector_store.search`.

## 3. embed_batch length-mismatch silently drops chunks via zip(), creating zombie chunk rows
- **Severity**: High
- **Category**: silent-failure / state-corruption
- **File**: `src-tauri/src/engine/kb_ingest.rs:313` (`.zip(embeddings.into_iter())`); contract source `src-tauri/src/engine/embedder.rs:64-99` (`embed_batch`)
- **Scenario**: `embed_and_store_vectors` zips `batch_ids` with the returned `embeddings` (kb_ingest.rs:313-317). `embed_batch` is an external/ONNX boundary: if it ever returns fewer vectors than inputs (a fastembed quirk on empty/whitespace-only chunks, a truncated batch, or a future model change), `zip` silently stops at the shorter length. The chunk rows were already inserted into `kb_chunks` (kb_ingest.rs:270-278) in a *prior* block, but their vectors are never inserted. No error is raised; ingest reports success and `chunks_created = chunks.len()` (store_chunks_and_vectors returns `Ok(chunks.len())`, line 295) even though some chunks have no vector.
- **Root cause**: Assumes `embed_batch` always returns exactly `texts.len()` vectors in order. The relational chunk insert and the vector insert are decoupled and length-correlated only by an implicit 1:1 contract that is never asserted.
- **Impact**: Permanent desync between `kb_chunks` and the vec0 index — the inverse of the orphan case the code carefully guards elsewhere (kb_delete_document.rs:935 refuses partial deletes for exactly this reason). Affected chunks are unsearchable, and the document counter is inflated. Silent and unrecoverable without re-index.
- **Fix sketch**: Assert `embeddings.len() == batch_texts.len()` immediately after the `embed_batch` call and return `AppError` (triggering the existing `cleanup_orphaned_chunks` path) on mismatch. Defensively, filter/reject empty chunk text before insertion so the embedder is never handed empty strings.

## 4. MCP stdio pool keyed by credential_id ignores env_vars/working_directory/command — stale or cross-config session reuse
- **Severity**: High
- **Category**: state-corruption / race-condition
- **File**: `src-tauri/src/engine/mcp_tools.rs:210` (`take_pooled_session`), `:270` (`return_pooled_session`), pool key is bare `credential_id`
- **Scenario**: The warm-process pool is keyed solely on `credential_id` (line 221 `pool.remove(credential_id)`). MCP credential fields (`command`, `working_directory`, `env_vars` — read fresh in `spawn_stdio_session` lines 278-283) can be edited by the user between calls. After an edit, `list_tools`/`execute_tool` decrypt the *new* fields but `take_pooled_session` happily returns the *old* still-running process spawned with the *previous* command/env (e.g. a now-rotated API key, or an old working directory). The tools/list cache is invalidated by time only (60s TTL, line 156), and there is no `invalidate_pooled_session` call wired to credential updates.
- **Root cause**: Assumes a credential_id uniquely and permanently determines the spawned process configuration. The pool key omits the config fingerprint, so a config change does not evict the now-stale warm process.
- **Impact**: Tool calls run against a process holding stale credentials/cwd for up to the 45s idle timeout (STDIO_POOL_IDLE_TIMEOUT, line 51) after every edit — wrong results, calls authenticated with rotated/revoked keys, or operations in the wrong directory. Hard to reproduce intentionally, easy to hit accidentally right after editing a server.
- **Fix sketch**: Key the pool on `(credential_id, hash(command, working_directory, env_vars))` so any config change forces a cold spawn, or explicitly call `invalidate_pooled_session(credential_id)` + `invalidate_tools_cache` from the credential update command path. Make the config fingerprint part of `PooledStdioSession` and verify it on checkout.

## 5. MCP gateway tool routing breaks when a member display_name contains the `::` separator
- **Severity**: High
- **Category**: edge-case / validation-gap
- **File**: `src-tauri/src/engine/mcp_tools.rs:515` (`parse_gateway_tool_name` uses `split_once`), name built at `:548-553`, matched at `:655` (`m.display_name == member_prefix`)
- **Scenario**: `add_mcp_gateway_member` (mcp_gateways.rs:20) accepts an arbitrary `display_name` with no validation. `list_tools` exposes member tools as `format!("{display_name}::{tool}")` (mcp_tools.rs:548). On execute, `parse_gateway_tool_name` splits on the FIRST `::` (`split_once`, line 516). If a member is named `db::prod` and exposes tool `query`, the persona sees `db::prod::query`; routing splits into member=`db`, real_tool=`prod::query`. The member lookup `m.display_name == "db"` (line 655) fails (real member is `db::prod`) → `AppError::NotFound`, the tool is permanently uncallable. If two members are `db` and `db::x`, a crafted name can route a call to the wrong member entirely.
- **Root cause**: The `::` separator is asserted to be absent from member display names ("rarely contain `::`", comment at line 510) but never enforced at the input boundary. `split_once` (first match) plus exact-equality member match assume the prefix is unambiguous.
- **Impact**: Tools on affected members are silently unreachable (UX degradation), and in the collision case a tool call is dispatched to the wrong underlying credential — wrong server, wrong auth, potential data leakage across gateway members.
- **Fix sketch**: Reject `::` (and leading/trailing whitespace) in `display_name` at `add_mcp_gateway_member`/`set_*` time, or switch the wire format to a length-prefixed / JSON member key instead of string concatenation. Use `rsplit_once` is not enough; enforce the separator invariant at write time so the parse is provably unambiguous.

## 6. introspect_columns identifier "sanitization" silently rewrites table names → wrong-table introspection
- **Severity**: Medium
- **Category**: silent-failure / edge-case
- **File**: `src-tauri/src/engine/db_query.rs:469` and `:2161` (`table_name.replace(|c| !alnum && c!='_', "")`)
- **Scenario**: `introspect_columns` and `introspect_local_sqlite_columns` "sanitize" the table name by *deleting* every non-alphanumeric/underscore character rather than rejecting it. A legitimately quoted table like `"order-items"` or `my.table` becomes `orderitems` / `mytable`. The PRAGMA/`information_schema` query then runs against a different (or nonexistent) table and returns an empty or wrong column set with no error. For local SQLite (line 2162) the result is `PRAGMA table_info('orderitems')` — empty result presented as "this table has no columns."
- **Root cause**: Mutating-sanitize-instead-of-validate. The function assumes stripped-and-different is equivalent to the user's intent; it silently changes the queried object.
- **Impact**: Schema browser shows empty/incorrect columns for tables with hyphens, dots, or other valid-but-quoted characters — confusing, and could mislead AI-driven schema reasoning into thinking a column doesn't exist. Recoverable, low blast radius, but a genuine correctness bug.
- **Fix sketch**: Validate-and-reject: if `table_name` contains characters outside the allowed set, return `AppError::Validation` rather than silently rewriting it. For local SQLite, look the name up against `sqlite_master` and bind it as a parameter where the API allows, instead of formatting it into the PRAGMA string.
