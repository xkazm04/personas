# Brotherhood Phase B1 — Self-Encounters Implementation Design

> Implementation design for Phase B1: personas on the same desktop instance encounter each other, produce improvement proposals, and feed them through the existing Human Review → Lab pipeline.
>
> Status: **DESIGN REVIEW** — awaiting approval before implementation.

---

## Scope & Constraints

**What B1 does:**
- Enroll personas as "brothers" with a Brotherhood Profile
- Run structured encounters between any 2 of the user's own personas
- Knowledge Membrane filters what each brother sees about the other
- Encounter dialogue generated by each persona's own LLM (real executions)
- Proposal Generator extracts improvement signals from completed encounters
- Proposals delivered via PersonaMessage + PersonaManualReview
- Approved proposals promoted to Lab experiments (A/B or Matrix)
- Lab results close the loop (fitness delta → reputation tracking)

**What B1 does NOT do:**
- No P2P networking (all local)
- No circles (no discovery needed when both personas are yours)
- No DHT, mDNS, or any transport-layer involvement
- No vault bridge plugin (no credential exposure at any level)

**Prerequisite:** None. Works on current codebase without P2P.

## End-to-End Flow

### Happy Path: Two Personas Encounter Each Other

```
1. ENROLL
   User opens Brotherhood plugin → selects Persona A ("Code Reviewer")
   → sets disclosure level to Open → enrolls as brother
   Repeats for Persona B ("Security Auditor")

2. INITIATE ENCOUNTER
   User clicks "New Encounter" → selects Persona A and Persona B
   → picks type: Compare → topic auto-derived: "Code Quality Analysis"
   → sets max_rounds: 5, budget: $0.50

3. MEMBRANE FILTER
   Engine reads Persona A's full data → applies membrane at Open level
   → produces BrotherhoodView_A (prompt sections + tool names + tool descriptions)
   Engine does the same for Persona B → BrotherhoodView_B

4. ENCOUNTER DIALOGUE (5 rounds)
   Round 1 (Introduction):
     a. Build prompt for Persona A:
        "You are participating in a Brotherhood encounter. Your peer is: {BrotherhoodView_B}.
         Phase: Introduction. Describe your approach to {topic} using this structure: ..."
     b. Execute Persona A (existing executePersona pipeline) → parse SelfPresentation
     c. Build prompt for Persona B with A's introduction + BrotherhoodView_A
     d. Execute Persona B → parse SelfPresentation

   Rounds 2-4 (Exchange):
     For each round, derive focus_area from overlap in previous messages.
     Each persona gets the full dialogue so far + peer's membrane view.
     Execute alternately. Parse TechniqueShare or ChallengeResponse.

   Round 5 (Synthesis):
     Each persona receives full dialogue → generates Synthesis message
     with top_learnings + suggestions_for_peer.

5. EVALUATE
   Each persona scores the encounter (separate LLM call):
     - relevance_score (0-100): How relevant was the peer's input?
     - novelty_score (0-100): How much did I learn that I didn't know?
     - actionability_score (0-100): Can I act on the suggestions?
     - overall_quality (0-100): Overall encounter quality

6. GENERATE PROPOSALS
   Proposal Generator takes:
     - Full encounter dialogue
     - Both membrane views
     - Both evaluations
   Runs a dedicated LLM call (using default model) that outputs structured
   ImprovementProposal[] for EACH persona.

7. DELIVER
   For each proposal:
     a. Create PersonaMessage (content_type: "brotherhood_proposal")
        → appears in Message Center with encounter thread
     b. Create PersonaManualReview (severity: "info")
        → appears in Human Review queue with context_data containing
          encounter summary + proposed change diff
     c. Emit "brotherhood.proposal.generated" event

8. USER REVIEWS
   User opens Human Review → sees proposal:
     "Suggestion from encounter with Security Auditor:
      Add OWASP top-10 checklist to your error handling section"
   User clicks "Approve & Run Lab Test"

9. LAB EXPERIMENT
   System creates a new PersonaVersion with the proposed prompt patch applied.
   Launches Lab A/B run: current version vs. proposed version.
   (Or Lab Matrix if proposal is a TechniqueAdoption type.)

10. CLOSE THE LOOP
    Lab completes → fitness delta measured.
    If positive: version tagged "experimental", user can promote to "production".
    Encounter reputation updated: source persona gets credit for improvement.
    ImprovementProposal status → Adopted (or Rejected if negative delta).
```

## Rust Backend Modules

### Module Structure

```
src-tauri/src/engine/brotherhood/
├── mod.rs                    # Public API: start_encounter(), get_profile()
├── membrane.rs               # Knowledge Membrane: filter_persona_for_brotherhood()
├── encounter.rs              # Encounter engine: run rounds, manage state machine
├── prompt_builder.rs         # Build encounter prompts for each round/phase
├── proposal.rs               # Proposal generator: extract improvements from dialogue
├── reputation.rs             # Local reputation tracking
└── types.rs                  # All Brotherhood types (profiles, encounters, proposals, etc.)

src-tauri/src/db/models/brotherhood.rs        # Rust struct definitions (Serialize, Deserialize, TS)
src-tauri/src/db/repos/brotherhood.rs         # CRUD operations for all brotherhood tables
src-tauri/src/commands/brotherhood.rs         # Tauri commands (10-15 commands)
```

### Why Not a Separate Crate

B1 is local-only — no P2P involvement. A separate crate would over-isolate. Instead, `engine/brotherhood/` is a module within the existing engine, with a clear public API surface. When P2P integration arrives in B2, the module boundary is clean enough to extract if needed.

### Credential Isolation (Module-Level)

The `engine/brotherhood/` module has **no `use` statements** importing:
- `db::models::credential` or `db::repos::resources::credentials`
- `engine::crypto` (except for Ed25519 identity signing if needed)
- `keyring` (except for identity)
- Any type containing `CredentialField`, `encrypted_data`, or `iv`

This is enforced by code review convention in B1. In B2+, it becomes a crate boundary.

## Knowledge Membrane

### Implementation

```rust
// src-tauri/src/engine/brotherhood/membrane.rs

use crate::db::models::persona::{Persona, DesignContextData, DesignUseCase};
use crate::db::models::tool::PersonaToolDefinition;
use crate::db::models::brotherhood::BrotherhoodView;

/// The ONLY function through which Brotherhood code accesses persona data.
/// Receives owned snapshots — never mutable references.
/// Returns a fully owned BrotherhoodView — no credential data at any level.
pub fn filter_for_brotherhood(
    persona: &Persona,
    tools: &[PersonaToolDefinition],
    disclosure: &DisclosureLevel,
) -> BrotherhoodView {
    // Parse structured prompt if available
    let structured = persona.structured_prompt.as_ref()
        .and_then(|s| serde_json::from_str::<StructuredPrompt>(s).ok());

    // Parse design context, stripping credential_links and connector_pipeline
    let design_ctx = persona.design_context.as_ref()
        .and_then(|s| serde_json::from_str::<DesignContextData>(s).ok());

    let use_cases: Vec<DesignUseCase> = design_ctx.as_ref()
        .and_then(|d| d.use_cases.clone())
        .unwrap_or_default();

    BrotherhoodView {
        display_name: persona.name.clone(),
        specialty: persona.description.clone().unwrap_or_default(),
        domain: "general".into(), // derived from profile, not persona

        // Graduated disclosure
        prompt_section_names: match disclosure {
            DisclosureLevel::Silhouette => None,
            _ => structured.as_ref().map(|s| s.section_names()),
        },
        prompt_section_contents: match disclosure {
            DisclosureLevel::Open | DisclosureLevel::Transparent =>
                structured.as_ref().map(|s| s.section_contents()),
            _ => None,
        },
        tool_names: match disclosure {
            DisclosureLevel::Silhouette => None,
            _ => Some(tools.iter().map(|t| t.name.clone()).collect()),
        },
        tool_descriptions: match disclosure {
            DisclosureLevel::Open | DisclosureLevel::Transparent =>
                Some(tools.iter().map(|t| t.description.clone().unwrap_or_default()).collect()),
            _ => None,
        },
        use_case_titles: match disclosure {
            DisclosureLevel::Open | DisclosureLevel::Transparent =>
                Some(use_cases.iter().map(|u| u.title.clone()).collect()),
            _ => None,
        },
        use_case_descriptions: match disclosure {
            DisclosureLevel::Transparent =>
                Some(use_cases.iter().map(|u| u.description.clone()).collect()),
            _ => None,
        },
        design_summary: match disclosure {
            DisclosureLevel::Transparent =>
                design_ctx.as_ref().and_then(|d| d.summary.clone()),
            _ => None,
        },
        model_profile: match disclosure {
            DisclosureLevel::Open | DisclosureLevel::Transparent =>
                persona.model_profile.clone(),
            _ => None,
        },

        // Never exposed (not even at Transparent):
        // - credential_links (stripped from design_ctx parse)
        // - connector_pipeline (stripped from design_ctx parse)
        // - parameter values
        // - sample inputs from use cases
        // - tool configs
    }
}
```

### What's Stripped (All Levels)

These fields are **never** present in `BrotherhoodView`, regardless of disclosure level:

| Field | Reason |
|-------|--------|
| `design_context.credential_links` | Maps connector names to credential IDs |
| `design_context.connector_pipeline` | Contains service endpoints |
| `use_case.sample_input` | May contain real data |
| `use_case.input_schema` | May reveal data structure of connected systems |
| `PersonaParameter.default_value` | May contain secrets (API base URLs, etc.) |
| `PersonaToolDefinition.config` | May contain credential-adjacent configuration |
| `PersonaToolDefinition.implementation_guide` | Contains curl commands with auth patterns |

## Encounter Engine

### State Machine

```
                  ┌──────────┐
           create │ Proposed │
                  └────┬─────┘
                       │ start_encounter()
                  ┌────▼─────┐
                  │Accepted  │  (B1: auto-accepted since both are local)
                  └────┬─────┘
                       │ run_round()
                  ┌────▼──────┐
                  │InDialogue │◄────────────┐
                  └────┬──────┘             │
                       │               more rounds?
                  ┌────▼──────┐        yes  │
                  │ (round N) ├─────────────┘
                  └────┬──────┘
                       │ all rounds done
                  ┌────▼──────┐
                  │Evaluating │
                  └────┬──────┘
                       │ evaluations complete
                  ┌────▼──────┐
                  │Completed  │
                  └───────────┘

  At any point: budget exceeded → auto-Synthesis → Completed
  At any point: user cancels → Abandoned
  At any point: error → Abandoned (with error stored)
```

### Round Execution

Each round produces 2 persona executions (one per brother). The encounter engine orchestrates them sequentially:

```rust
// src-tauri/src/engine/brotherhood/encounter.rs

pub async fn run_round(
    app: &AppHandle,
    pool: &DbPool,
    encounter: &mut Encounter,
    round_number: u8,
) -> Result<EncounterRound, AppError> {
    let phase = determine_phase(round_number, encounter.max_rounds);
    let focus_area = if phase == RoundPhase::Exchange {
        Some(derive_focus_area(&encounter.rounds, &encounter.initiator_view, &encounter.responder_view))
    } else {
        None
    };

    // 1. Build prompt for initiator
    let initiator_prompt = prompt_builder::build_round_prompt(
        &encounter.topic,
        &encounter.encounter_type,
        &phase,
        &focus_area,
        &encounter.responder_view,    // what initiator sees about responder
        &encounter.rounds,            // dialogue so far
        BrotherRole::Initiator,
    );

    // 2. Execute initiator's persona
    let initiator_exec = execute_brotherhood_turn(
        app, pool,
        &encounter.initiator_persona_id,
        &initiator_prompt,
        encounter.id.clone(),
    ).await?;

    // 3. Parse structured output
    let initiator_message = parse_round_content(&initiator_exec.output, &phase)?;

    // 4. Accumulate cost
    encounter.cost_usd += initiator_exec.cost_usd;

    // Budget check before responder turn
    if encounter.cost_usd >= encounter.budget_limit_usd {
        // Force synthesis round next, or conclude if already in synthesis
        // ...
    }

    // 5. Build prompt for responder (includes initiator's message from this round)
    let responder_prompt = prompt_builder::build_round_prompt(
        &encounter.topic,
        &encounter.encounter_type,
        &phase,
        &focus_area,
        &encounter.initiator_view,    // what responder sees about initiator
        &encounter.rounds,            // dialogue so far (now includes initiator's message)
        BrotherRole::Responder,
    );

    // 6. Execute responder's persona
    let responder_exec = execute_brotherhood_turn(
        app, pool,
        &encounter.responder_persona_id,
        &responder_prompt,
        encounter.id.clone(),
    ).await?;

    // 7. Parse structured output
    let responder_message = parse_round_content(&responder_exec.output, &phase)?;

    // 8. Accumulate cost
    encounter.cost_usd += responder_exec.cost_usd;

    // 9. Store round
    let round = EncounterRound {
        round_number,
        phase,
        focus_area,
        initiator_message: Some(initiator_message),
        responder_message: Some(responder_message),
        initiator_execution_id: Some(initiator_exec.id),
        responder_execution_id: Some(responder_exec.id),
    };

    Ok(round)
}
```

### Persona Execution for Brotherhood Turns

Each dialogue turn is a real persona execution through the existing pipeline. The encounter engine wraps the persona's system prompt with Brotherhood context:

```rust
// src-tauri/src/engine/brotherhood/encounter.rs

async fn execute_brotherhood_turn(
    app: &AppHandle,
    pool: &DbPool,
    persona_id: &str,
    brotherhood_prompt: &str,
    encounter_id: String,
) -> Result<TurnResult, AppError> {
    // Execute using the existing execution pipeline.
    // The brotherhood_prompt is passed as input_data.
    // The persona's own system_prompt + model_profile are used.
    //
    // We use a dedicated input format:
    let input_data = serde_json::json!({
        "mode": "brotherhood_encounter",
        "encounter_id": encounter_id,
        "prompt": brotherhood_prompt,
    });

    let execution_id = crate::engine::pipeline::execute(
        app,
        pool,
        persona_id,
        None,           // no trigger_id
        Some(input_data.to_string()),
        None,           // no use_case_id
        None,           // no continuation
    ).await?;

    // Wait for execution to complete (poll status)
    let result = wait_for_execution(pool, &execution_id, Duration::from_secs(120)).await?;

    Ok(TurnResult {
        id: execution_id,
        output: result.output_data.unwrap_or_default(),
        cost_usd: result.cost_usd,
        tokens_in: result.input_tokens,
        tokens_out: result.output_tokens,
    })
}
```

### Topic Auto-Derivation

When the user doesn't specify a topic, the engine derives one from capability overlap:

```rust
pub fn derive_topic(
    initiator_profile: &BrotherhoodProfile,
    responder_profile: &BrotherhoodProfile,
) -> String {
    // Find overlapping capability tags
    let overlap: Vec<&String> = initiator_profile.capability_tags.iter()
        .filter(|t| responder_profile.capability_tags.contains(t))
        .collect();

    if !overlap.is_empty() {
        format!("Approaches to {}", overlap.join(" and "))
    } else {
        // No overlap — use domain
        format!("Cross-domain exchange: {} meets {}",
            initiator_profile.domain, responder_profile.domain)
    }
}
```

### Focus Area Derivation

For Exchange rounds (2 through N-1), the engine selects a focus area based on what came up in previous rounds:

```rust
pub fn derive_focus_area(
    rounds: &[EncounterRound],
    initiator_view: &BrotherhoodView,
    responder_view: &BrotherhoodView,
) -> String {
    // Strategy: rotate through aspects not yet covered
    // 1. Collect all techniques/tools mentioned so far
    // 2. Find aspects from both views not yet discussed
    // 3. Pick the most promising unexplored area
    // Fallback: "deep dive on {last_round_topic}"
}
```

## Proposal Generator

### How Proposals Are Extracted

After the encounter completes and evaluations are in, a dedicated LLM call analyzes the entire dialogue to extract actionable improvement proposals:

```rust
// src-tauri/src/engine/brotherhood/proposal.rs

pub async fn generate_proposals(
    app: &AppHandle,
    pool: &DbPool,
    encounter: &Encounter,
    evaluations: &[EncounterEvaluation],
) -> Result<Vec<ImprovementProposal>, AppError> {
    // Generate proposals for BOTH personas
    let mut all_proposals = Vec::new();

    for (target_persona_id, target_brother_id, source_brother_id) in [
        (&encounter.initiator_persona_id, &encounter.initiator_id, &encounter.responder_id),
        (&encounter.responder_persona_id, &encounter.responder_id, &encounter.initiator_id),
    ] {
        let prompt = build_proposal_extraction_prompt(
            encounter,
            target_persona_id,
            target_brother_id,
        );

        // Use a lightweight LLM call (not a full persona execution)
        // This uses the default model profile, not the persona's own
        let raw_output = crate::engine::llm::invoke_structured(
            app, pool,
            &prompt,
            "brotherhood_proposal_generator",
            None, // default model
        ).await?;

        let extracted: Vec<RawProposal> = serde_json::from_str(&raw_output)
            .unwrap_or_default();

        // Convert raw proposals to typed ImprovementProposal
        for raw in extracted {
            if let Some(proposal) = raw.into_improvement_proposal(
                target_persona_id,
                &encounter.id,
                source_brother_id,
            ) {
                all_proposals.push(proposal);
            }
        }
    }

    Ok(all_proposals)
}
```

## Lab Integration

### Proposal → Lab Run

When the user approves a proposal in Human Review with "Approve & Run Lab Test":

```rust
// src-tauri/src/engine/brotherhood/proposal.rs

pub async fn promote_to_lab(
    app: &AppHandle,
    pool: &DbPool,
    proposal_id: &str,
) -> Result<String, AppError> {
    let proposal = repos::brotherhood::get_proposal(pool, proposal_id)?;
    let change: ProposedChange = serde_json::from_str(&proposal.proposed_change)?;

    let (lab_run_id, lab_mode) = match change {
        ProposedChange::PromptPatch { section, proposed_fragment, .. } => {
            // 1. Create a new PersonaVersion with the patch applied
            let persona = repos::personas::get(pool, &proposal.persona_id)?;
            let patched_prompt = apply_prompt_patch(
                &persona.structured_prompt,
                &section,
                &proposed_fragment,
            )?;

            // 2. Save as new version
            let current_version = repos::lab::versions::get_latest(pool, &proposal.persona_id)?;
            let new_version = repos::lab::versions::create(pool, CreateVersionInput {
                persona_id: proposal.persona_id.clone(),
                system_prompt: persona.system_prompt.clone(),
                structured_prompt: Some(patched_prompt),
                model_profile: persona.model_profile.clone(),
                change_summary: Some(format!("Brotherhood proposal: {}", proposal.title)),
                tag: "experimental".into(),
                parent_version_id: current_version.as_ref().map(|v| v.id.clone()),
                ..Default::default()
            })?;

            // 3. Start Lab A/B: current vs. patched
            let models = resolve_default_test_models(pool)?;
            let run = crate::commands::execution::lab::start_ab_internal(
                app, pool,
                &proposal.persona_id,
                &current_version.unwrap().id,
                &new_version.id,
                &models,
                None, // no use case filter
            ).await?;

            (run.id, "ab".to_string())
        },

        ProposedChange::TechniqueAdoption { technique_name, example_prompt_addition, .. } => {
            // Use Lab Matrix: let the LLM incorporate the technique
            let instruction = format!(
                "Incorporate this technique into the persona: '{}'. \
                 Suggested addition: '{}'",
                technique_name, example_prompt_addition
            );

            let models = resolve_default_test_models(pool)?;
            let run = crate::commands::execution::lab::start_matrix_internal(
                app, pool,
                &proposal.persona_id,
                &instruction,
                &models,
                None,
            ).await?;

            (run.id, "matrix".to_string())
        },

        ProposedChange::ToolRecommendation { .. } => {
            // Tool suggestions don't go through Lab — they're informational.
            // Mark as approved directly.
            repos::brotherhood::update_proposal_status(
                pool, proposal_id, "approved", None, None,
            )?;
            return Ok("direct_approval".into());
        },
    };

    // Update proposal with lab reference
    repos::brotherhood::update_proposal_status(
        pool, proposal_id, "in_experiment",
        Some(&lab_run_id), Some(&lab_mode),
    )?;

    Ok(lab_run_id)
}
```

### Lab Result → Proposal Completion

A background watcher monitors lab runs linked to Brotherhood proposals:

```rust
// src-tauri/src/engine/brotherhood/proposal.rs

pub fn check_lab_completion(pool: &DbPool) -> Result<(), AppError> {
    let in_experiment = repos::brotherhood::list_proposals_by_status(pool, "in_experiment")?;

    for proposal in in_experiment {
        if let (Some(lab_run_id), Some(lab_mode)) = (&proposal.lab_run_id, &proposal.lab_run_mode) {
            let (status, fitness_delta) = match lab_mode.as_str() {
                "ab" => check_ab_completion(pool, lab_run_id)?,
                "matrix" => check_matrix_completion(pool, lab_run_id)?,
                _ => continue,
            };

            if let Some(delta) = fitness_delta {
                let new_status = if delta > 0.0 { "adopted" } else { "rejected" };
                repos::brotherhood::update_proposal_status(
                    pool,
                    &proposal.id,
                    new_status,
                    None,
                    None,
                )?;
                repos::brotherhood::set_proposal_fitness_delta(pool, &proposal.id, delta)?;

                // Update reputation
                update_reputation_from_proposal(pool, &proposal, delta)?;
            }
        }
    }

    Ok(())
}
```

## Cost & Budget Controls

### Default Limits

```rust
pub struct EncounterDefaults {
    pub max_rounds: u8,                    // 5
    pub budget_limit_usd: f64,             // 0.50
    pub max_tokens_per_turn: u32,          // 2000
    pub evaluation_budget_usd: f64,        // 0.10 (separate from encounter budget)
    pub proposal_generation_budget_usd: f64, // 0.10
    pub cooldown_minutes: u16,             // 60 (same pair)
}
```

### Cost Tracking

Every execution in the encounter creates a `persona_executions` record with `cost_usd`. The encounter engine sums these:
- Encounter dialogue: N rounds x 2 personas = 2N executions
- Evaluations: 2 executions (one per persona)
- Proposal generation: 1 execution (single LLM call)
- **Total**: 2N + 3 executions per encounter

At default 5 rounds: 13 executions, ~$0.50 total.

### Budget Enforcement

```
Before each round:
  if encounter.cost_usd >= encounter.budget_limit_usd * 0.8:
    Force next round to be Synthesis (graceful conclusion)
  if encounter.cost_usd >= encounter.budget_limit_usd:
    Abandon remaining rounds, skip to evaluation

Per-persona monthly budget (optional, user-configurable):
  Settings > Brotherhood > Monthly Budget: [$10.00]
  Prevents runaway encounters across multiple sessions
```

## Testing Strategy

|-------|------|-------------|
| Unit (Rust) | `cargo test` | Membrane filtering at each disclosure level; round phase determination; topic/focus derivation; proposal parsing; budget enforcement; reputation math |
| Unit (TS) | `vitest` | Store slice actions; API type shapes; disclosure level picker logic |

| E2E | Manual | Enroll 2 personas → run encounter → review proposals → approve to Lab → verify Lab run created |

### Critical Test Cases

1. **Membrane at Silhouette**: Verify NO prompt content, NO tool names leak through
2. **Membrane at Transparent**: Verify credential_links and connector_pipeline are still absent
3. **Budget cutoff**: Encounter at $0.49 starts synthesis; at $0.50 stops
4. **Cooldown**: Same pair cannot re-encounter within 60 minutes
5. **Self-encounter guard**: `CHECK(initiator_id != responder_id)` — same brother cannot encounter itself
6. **Proposal → Lab**: Approved proposal creates correct Lab A/B run with correct version IDs
7. **Lab completion → reputation**: Positive fitness delta updates reputation correctly
