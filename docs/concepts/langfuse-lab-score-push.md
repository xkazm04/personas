# Langfuse Lab Score Push — Backlog Item

**Status:** Backlog. Path A closed without this; pick up before Path B work begins, or earlier if Path A+ surfaces it as a need.
**Owners:** TBD
**Last updated:** 2026-05-07
**Parent doc:** [`langfuse-observability.md`](./langfuse-observability.md)

---

## What's missing

Real persona executions flow into Langfuse end-to-end (Phase 1b OTLP exporter). Lab runs **don't**. So Langfuse shows live-execution traces and per-model cost, but the Lab's three rubric scores (`tool_accuracy`, `output_quality`, `protocol_compliance`) — generated by `eval::eval_with_llm` for every scenario × model cell — only live in our local SQLite (`lab_arena_results`, `lab_ab_results`, etc.). They never surface in Langfuse.

This is the visible gap when comparing "live execution observability" vs. "prompt iteration observability" in the Langfuse UI.

## Why it deferred

The infrastructure is there. `src-tauri/src/langfuse/exporter.rs` already has:

```rust
pub fn push_lab_scores(
    trace_id_hex: String,
    tool_accuracy: Option<i32>,
    output_quality: Option<i32>,
    protocol_compliance: Option<i32>,
    rationale: Option<String>,
)
```

It POSTs to `<host>/api/public/scores` with the same Basic auth the OTLP exporter uses. Three POSTs per call (one per rubric), tagged with the supplied `trace_id_hex`.

The blocker is the **call site**. Lab scoring lives in `engine/test_runner.rs::score_result` (definition at line ~915, call sites around lines 312, 1586; also `engine/evolution.rs:454`). Every Lab path runs through `execute_scenario`, which **does not go through `engine::runner::run_execution`** — it spawns the CLI directly via `spawn_cli_and_collect_structured` and returns an `ExecutionOutput` that doesn't carry an `execution_id` or trace ID. There's nothing to anchor scores against.

Picking a `trace_id` requires either:

1. Plumbing the existing run_execution trace through Lab.
2. Synthesizing a Langfuse trace at score time.

## Two implementation paths

### Path 1 — Refactor lab to use `run_execution`

Switch `execute_scenario` so it goes through the real runner. Each scenario produces a proper `ExecutionTrace`, persists to `execution_traces`, gets exported via the existing OTLP path. `score_result` then looks up the trace_id by execution_id (`db::repos::execution::traces::get_by_execution_id`) and calls `push_lab_scores` with the correct ID.

**Pros**
- Lab runs become first-class traces in Langfuse. The same trace inspector users already use for live executions also covers Lab arena/A-B/Matrix runs. No special "lab session" namespace needed.
- One source of truth for trace data — no parallel synthesis path.
- Future Path B (embedded insights) gets Lab visibility for free.

**Cons**
- Big change to a hot path. Lab is heavily used; regressions there are immediately user-visible.
- `execute_scenario` currently has its own trimmed environment (sandbox-section injection, `--max-turns 3`, native Ollama HTTP path that bypasses CLI entirely). Replicating that inside `run_execution` requires either a "lab mode" flag on the runner or new pre/post-processing hooks.
- Cost overhead: each Lab cell creates an `execution_traces` row. Lab can run hundreds of cells per Matrix sweep. Need retention policy work.

**Estimate**: 2–3 days, plus a Lab regression-test pass.

### Path 2 — Synthesize a Langfuse trace per scenario × model

Add a helper alongside `push_lab_scores`:

```rust
// langfuse/exporter.rs (sketch — not yet written)
pub fn push_lab_run(
    persona_id: &str,
    persona_name: &str,
    scenario_id: &str,
    scenario_name: &str,
    model: &str,
    output: &ExecutionOutput,
    scores: &ScoreResult,
) -> String  // returns the synthesized trace_id_hex
```

The helper builds an in-memory `ExecutionTrace` shaped like a real one:

- Synthesized `trace_id` (fresh UUID).
- Two spans:
  - Root `Execution` span tagged `personas.lab.scenario_id`, `personas.lab.scenario_name`, `personas.lab.model`, `langfuse.session.id=lab-<persona_id>` so Lab runs for one persona group as a Langfuse session.
  - `CliSpawn` (generation) span carrying `gen_ai.system`, `gen_ai.request.model`, `gen_ai.usage.input_tokens`, `gen_ai.usage.output_tokens`, `personas.cost_usd`. With `redact_content` off, also `input.value=<scenario.input>` and `output.value=<output.assistant_text>`.

Submits via the existing OTLP MPSC. Then immediately calls `push_lab_scores(trace_id_hex, ...)`. `score_result` only adds two lines: build the input, call the helper, ignore the return value.

**Pros**
- Self-contained — no surgery on `run_execution` or the runner pipeline.
- Lab scenarios appear under their own Langfuse session prefix, easy to filter.
- Scores attach cleanly to traces we control end-to-end.
- Half a day's work; reversible.

**Cons**
- Two trace-emission paths in the codebase (real + synthesized). Slight cognitive load.
- Synthesized traces are missing fine-grained spans the real runner emits (CredentialResolution, ToolCall, ProtocolDispatch, etc.). For most Lab use this doesn't matter — Lab cares about the prompt → output pair plus scores — but anyone debugging via Langfuse will notice Lab traces are flatter than execution traces.
- Lab runs that error before producing output need a separate "failed run" trace shape.

**Estimate**: 4–8 hours including tests.

## Recommendation

**Path 2 (synthesize)** unless Path A+ exploration produces a concrete user-facing reason to need full span coverage on Lab runs. The implementation is small, the failure modes are isolated, and we don't bet a refactor on the assumption that Path B is the right answer. If Path B becomes the chosen direction later, the synthesized traces are still useful — they keep working — and *if* deeper integration is wanted, Path 1's refactor can land then with the user-impact understood.

## Acceptance criteria

- [ ] Score push fires only when the Langfuse exporter is installed (no-op otherwise).
- [ ] Each Lab cell produces three `Score` rows in Langfuse with names `tool_accuracy`, `output_quality`, `protocol_compliance` and a numeric value 0–100.
- [ ] Scores are queryable in Langfuse by `traceId`; trace appears in the Langfuse UI's trace explorer.
- [ ] `redact_content=true` suppresses `input.value` / `output.value` on synthesized spans (Path 2) but keeps tokens, cost, scores intact.
- [ ] Failure paths (eval errored, network unreachable) log a `tracing::warn!` and never disturb the user-facing Lab flow.
- [ ] One end-to-end test in the regression suite: run a Matrix cell, assert three Score rows show up via `GET /api/public/scores?traceId=<id>`.

## Open question for Path A+

If Path A+ exploration shows that Langfuse's prompt CMS / experiments features are how teams want to iterate prompts, this changes shape: Lab and Langfuse experiments become the same thing, and we'd want bidirectional sync rather than push-only scores. Defer that decision until A+ findings land.
