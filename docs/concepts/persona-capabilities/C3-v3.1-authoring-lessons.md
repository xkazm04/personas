# C3 — v3.1 template authoring lessons (pre-mass-authoring)

> Captured 2026-04-20, after the first 6 v3.1 templates were runtime-tested
> and user-reviewed. These rules are the difference between v3.1 templates
> that feel coherent vs. the vague, questionnaire-heavy shapes we shipped
> first. Read this before editing or creating any additional v3.1 template.

---

## 1. Trigger composition lives at the end, not inside questions

**Don't** author `aq_*_time` / `aq_*_cadence` questions. Every v3.1 template
has a dedicated trigger-composition step (WIP prototypes at
`src/features/templates/sub_generated/adoption/TriggerComposition*.tsx`).
That step is where cadence is set — time-of-day, weekday, event-based
chaining.

**Do** put the template's recommended cadence in
`use_cases[].suggested_trigger.config.cron` and describe it as a *default*
in the description — "Default daily 7am — final cadence set at the
trigger-composition step." The user sees the default in the trigger step
and can override it there.

**Do** strip trigger references from UC **titles and descriptions**:

- ❌ `"title": "Daily Morning Briefing"` / `"Sunday Weekly Review"` / `"Hourly Backlog Scan"`
- ✅ `"title": "Personal Briefing"` / `"Weekly Review"` / `"Backlog Scan"`

Cadence belongs in metadata (default cron + description note), not the
human-readable name.

### Phase E status (2026-04-20)

Two variant prototypes exist but are **not wired** into the adoption flow:

- `TriggerCompositionStepChips.tsx` — per-UC chip grid
- `TriggerCompositionStepMaster.tsx` — shared default + per-UC override drawer
- `TriggerCompositionDemo.tsx` — tab-switcher parent (also unwired)

All three carry `@ts-nocheck`. No production code imports them. The
trigger step appears in the authoring mental model but **does not render**
in the adoption flow yet. When picking a winner: drop `@ts-nocheck`, fix
the typed props, delete the rejected variant + the Demo parent, and
insert the chosen step between `UseCasePickerStep` and
`QuestionnaireFormFocus` inside `MatrixAdoptionView.tsx`.

Until that lands, authors should still write templates *as if* the
trigger step exists — it's the contract the templates target.

## 2. Replace "tell me your niche" questions with source-grounded intent

**Don't** ask for `aq_niche_keywords`, `aq_role_business`, or any other
"describe yourself" text that the persona then has to re-interpret on
every run. Keywords drift, fine-grained role descriptions rot.

**Do** use the `source_definition` question type
(`src/features/shared/components/forms/SourceDefinitionInput.tsx`) to
let the user attach a concrete ground-truth source: a local folder, a
registered codebase, a database credential, or a URL set. The persona
reads it at each run. The user's intent lives in one `textarea`
("`aq_*_intent`") that explains how to work with those sources — what
to emphasize, what to suppress, how to rank.

The Daily Personal Briefer v3.1 (2026-04-20) is the canonical example:
- `aq_briefing_sources` — `type: "source_definition"`
- `aq_briefing_intent` — `type: "textarea"` (role, tone, ranking hints)
- `aq_goals` — still a `textarea` (one goal per line with deadlines)

Length preferences, tone, and "what to cut" should live inside the
intent textarea, not as separate enum questions. The intent field is
a real briefing brief — let the persona interpret it.

If the intent is thin at adoption time, the build process can ask
clarifying follow-ups after the questionnaire. Don't over-specify up
front.

## 3. Vault-sourced pickers for "which provider?" questions

Covered in Phase C (commit `70a282cb`). Key rules:

- **Never author `options: [...]`** for provider-picker questions
  (`aq_ad_platform`, `aq_analytics_tool`, `aq_image_model`, etc.). The
  Vault is the source of truth.
- Use `dynamic_source: { service_type: "<category>", operation:
  "list_credentials", source: "vault" }` — the matcher filters the
  user's healthy credentials by connector `categories[]`.
- Tag connector JSON with `categories: string[]` when it participates
  in a category slot. Many connectors need multi-tagging (Leonardo AI
  = `["image_generation", "video_generation"]`).
- Context/help text should explain the category and suggest adding a
  connector when the list is empty — the empty-state vault-redirect
  CTA is still a TODO inside `QuestionnaireFormGrid`.

## 4. Repo selection for source-control templates

**Don't** hardcode `"name": "github"` in a persona.connectors entry or
ask "which GitHub repository" as a plain `text` question. Templates
should support GitHub **and** GitLab (and whatever else lands in the
`source_control` category).

**Do** ask `aq_source_control` — vault-sourced on `service_type:
"source_control"`. The concrete repo is resolved at build time via the
provider's API (GitHub `/user/repos`, GitLab `/projects?membership=true`).

Today `github.json` and `gitlab.json` both declare `categories:
["source_control"]` (2026-04-20 commit following Phase D). The repo-list
discovery IPC (`list_repos` operation) is **not yet implemented** — for
now the vault picker returns credentials, and the build process can ask
the user for the repo string interactively when it's missing. That's
the documented UX seam; a proper chained `dynamic_source` with
`depends_on` + `operation: "list_repos"` is the follow-up.

Also: ensure the picked credential is healthy before persona creation.
The matcher already filters by `healthcheck_last_success`, so this is
mostly enforced; double-check when a template's connector is strictly
required.

## 5. Per-UC scope hint below each answer

UX improvement landed 2026-04-20: `QuestionnaireFormGridParts.tsx` now
renders a muted "Applies to: uc_foo, uc_bar" line below each answer
when `question.use_case_ids` / `question.use_case_id` is populated.

**Do** always set `use_case_ids` on capability-scoped questions, even
single-UC ones. That drives the scope hint and lets the questionnaire
filter questions the user disabled at the UC picker step.

**Don't** use `use_case_ids` as a vague "see also" list — only use the
ids the question is genuinely scoped to. The hint is a disclosure, not
a breadcrumb.

## 6. Known infrastructure gaps

### Codebase connector absent from Credentials Catalog

`scripts/connectors/builtin/codebase.json` has `template_enabled: true`,
`connection_mode: desktop_bridge`, and `always_active: true`. The Rust
seed at `src-tauri/src/db/mod.rs:413` uses `INSERT OR IGNORE` + UPDATE,
so fresh installs should include it. If it's missing in the Catalog
for an existing user, check:

1. Does `connector_definitions` in the app's SQLite have a row where
   `name = 'codebase'` and `is_builtin = 1`? If not, something is
   filtering it out at query time (plugin gating, category filter) —
   look at `filterTemplateConnectors` in
   `src/features/vault/sub_catalog/components/design/CredentialDesignHelpers.ts`.
2. Is the Catalog page filtering by `auth_type`? `codebase.json`
   uses `auth_type: "builtin"` — a filter that only accepts
   `oauth|api_key|password` would exclude it.
3. If the DB is stale from before codebase.json existed, deleting the
   row (or running a migration reset) forces a reseed on next launch.

This is a follow-up to investigate end-to-end; the seeding code looks
correct on paper.

### Empty-state Vault redirect

When a `dynamic_source` with `source: "vault"` produces zero items,
`useDynamicQuestionOptions` returns an error message. The questionnaire
should surface a CTA linking to `Vault → Catalog` prefiltered by the
requested category. Not implemented yet; `QuestionnaireFormGrid` is
where that CTA belongs.

## 7. Authoring checklist — use this before touching any template

- [ ] `persona.goal` — one-line value statement (rendered as subtitle).
- [ ] UC titles have **no** time/cadence words. Descriptions mention
      the default as "managed at the trigger-composition step."
- [ ] No `aq_*_time`, `aq_*_cadence`, or `aq_*_when` questions.
- [ ] No `options: [...]` for provider pickers — use vault-source
      `dynamic_source` with the right category tag.
- [ ] No "describe yourself" free-text questions. Use
      `source_definition` + an `aq_*_intent` textarea instead.
- [ ] Every capability-scoped question sets `use_case_ids` (drives
      the per-UC hint and the disabled-UC filter).
- [ ] Every required connector participates in a documented category
      (e.g. `storage`, `advertising`, `analytics`, `source_control`,
      `image_generation`, `video_generation`).
- [ ] `suggested_trigger.config` has sane defaults; its description
      says "Default X — final cadence set at the trigger-composition
      step."
- [ ] Checksums regenerated:
      `node scripts/generate-template-checksums.mjs`.
- [ ] If connector JSON changed:
      `node scripts/generate-connector-seed.mjs`.
