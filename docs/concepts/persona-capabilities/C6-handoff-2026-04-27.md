### G — Build pipeline: auto-default `source_filter = "*"` for cross-persona inbounds
`src-tauri/src/commands/design/build_sessions.rs`

New helper `collect_persona_emit_event_types(ir, use_cases)` unions the persona-level `ir.events[direction == "emit"]` and per-UC `event_subscriptions[direction == "emit"]` into a `HashSet<String>`. Both `create_triggers_in_tx` (event_listener trigger configs) and `create_event_subscriptions_in_tx` (persona_event_subscriptions rows) consult the set: when a LISTEN target's event_type isn't in the persona's emit set, the event has to come from a different persona — the bus's self-scoping rule (`bus.rs:131-145`) would silently drop it without a `source_filter`, so promote now defaults `source_filter = "*"` (only when the LLM didn't provide one).

**Why this preserves the original safety property** (the rule was added so two unrelated personas with overlapping subscriptions can't trigger each other's runs):

- Intra-persona self-loops → event ∈ emit set → `source_filter` stays null → bus self-scopes correctly.
- Cross-persona chains → event ∉ emit set → `source_filter = "*"` defaulted → bus delivers from any source persona.
- LLM-supplied `source_filter` is always honoured — auto-wiring only fills in when null.

This was discovered during Phase B run #1: persona X emitted `news.draft.captured` 12 times, persona Y had a matching subscription (and trigger), Y had 0 executions because its subscription's `source_filter` was null and the bus filtered out the cross-persona event. Phase B run #2 (against the rebuilt binary with the fix) showed the **X→Y→Z cascade firing automatically with no manual subscription patching**.

**Tests**: 4 new unit tests on `collect_persona_emit_event_types` (per-UC source, persona-level source, union, ignore-non-emit).

## Open architectural items

### 1. ✅ CLOSED 2026-04-27 — Full LLM-judged self-review pass for `auto_triage`
The C6 MVP (fix C above) made `mode: "auto_triage"` stop falling through to `manual_review` by storing the row and auto-resolving with a distinct audit tag. That left `auto_triage` parity with `trust_llm` only in audit-tag space — the row was still trusted unconditionally.

**Now implemented in `src-tauri/src/engine/auto_triage.rs`** (new module, 14 unit tests):
- New `AutoTriageVerdict { Approve, Reject }` + `AutoTriageDecision` types.
- Pure prompt builder (`build_evaluator_prompt`) composes the persona's `decision_principles[]`, `principles[]`, `constraints[]`, and per-UC `review_policy.context` into a single-shot evaluator prompt that asks the LLM for a JSON `{verdict, reasoning}` response.
- Pure verdict parser (`parse_verdict_response`) tolerates surrounding prose, accepts approve/reject synonyms (approved/accept/ok/rejected/deny/denied/...), and rejects unparseable or unknown-verdict responses.
- Pure principles extractors (`extract_principles_from_design_result`, `extract_review_policy_context`) read from `persona.last_design_result` JSON; both v3 nested-under-`persona` shape and legacy top-level shape are honoured.
- Async LLM call (`evaluate`) follows the `genome_critique::run_critique_cli` pattern — single-turn `claude -p -` with `--max-turns 1`, 120s timeout, stdout-only assistant text collection.
- `spawn_evaluator_task` fires a self-contained tokio task with all context cloned/owned, so the dispatcher's stream-processing loop is never blocked on the LLM call.

**Wiring in `dispatch.rs::ProtocolMessage::ManualReview`**: the immediate auto-resolve fast-path is replaced with a spawn. The row is created with `Pending` status, the spawned task runs the LLM, and the row transitions to `Approved` / `Rejected` based on the verdict. On any failure (CLI spawn, timeout, parse) the task falls back to `Resolved` with a `review.auto_triage.fallback` audit tag — preserving the C6 MVP behaviour so a degraded evaluator never blocks a run.

**Audit tag split** (mirrors `policy_event.rs` doc):
- `review.auto_triage` → `review.auto_triage.approved` / `.rejected` / `.fallback`.
- The `trust_llm` path is unchanged — still uses `review.trust_llm` and immediate `Resolved`.

**What's still deferred**: structured confidence scores in the verdict, sophisticated UI for verdict reasoning, rate-limiting if many reviews fire in rapid succession (today each spawns its own CLI subprocess), and the Phase D end-to-end driver that exercises an `auto_triage` persona live.

## C7 — 2026-04-27 EOD increment: dry-run preview

Carries forward from the C6 wishlist of new build-flow features (alongside reference-input questionnaires and webhook triggers, both still pending). The dry-run feature lets a user preview a capability's runtime behaviour from the build wizard *before* promoting — they get the actual agent output (manual reviews + memories) without sending real notifications.

### Backend — `src-tauri/src/commands/design/build_simulate.rs` (new module, 8 unit tests)

- **Pure helper** `build_simulation_design_context(ir: &AgentIr) -> String` — produces a snake-case `{"use_cases": [...], "summary": ..., "_simulation_snapshot": true}` JSON snapshot from an `AgentIr`. Each UC entry preserves `id`, `sample_input`, `time_filter`, `model_override`. When the IR omits an `id`, one is fabricated as `uc_<idx>_<title-slug>` so dry-run can still address the UC.
- **`#[tauri::command] simulate_build_draft(session_id, use_case_id, input_override?)`** — loads the build session, applies adoption_answers (mirrors `test_build_draft`), persists the snapshot to `personas.design_context`, then delegates to `execute_persona_inner` with `is_simulation=true`. The phase machine is *not* mutated — simulation is a side action callable from `draft_ready` / `testing` / `test_complete` / `promoted` (refused from earlier phases with a clear validation error).
- **`#[tauri::command] get_simulation_artefacts(execution_id) -> SimulationArtefacts`** — `{ executionId, reviews, memories }`. Uses existing `get_by_execution` repo accessors. **Messages and events are deliberately deferred** — `persona_messages` has the `execution_id` column but no scoped accessor today, and `persona_events` uses `source_id` for execution linkage which needs a small new query. The Execution Detail tab (post-promote) covers those views; v1 dry-run focuses on the user-visible artefacts.
- **Risk-grep** confirmed it is safe to write `design_context` to a draft (`enabled=false`) persona: schedulers (`background.rs:760`, `director.rs:548`) gate on `enabled=1`; the bus reads `persona_triggers` / `persona_event_subscriptions` rows (empty for drafts); the frontend persona list filters via `isDraft()` so the row stays in the "building" tab; `simulate_use_case` and `execute_persona_inner` already bypass the persona-enabled gate intentionally.
- **Side-discovery (latent bug, NOT fixed in this slice):** `commands/design/build_sessions.rs::build_design_json` writes the design_context as `{"useCases": [...]}` (camelCase), but `commands/execution/executions.rs:174`, `commands/core/use_cases.rs:568`, and `commands/core/use_cases.rs:60` all read `dc.get("use_cases")` (snake_case). This means `simulate_use_case` and `cascade_use_case_toggle` are likely broken on personas built via the matrix builder (the `useCases` array is invisible to those readers). My dry-run snapshot writes snake_case `use_cases` to match the readers — and `promote_build_draft` overwrites it with camelCase `useCases`, restoring the latent bug. Worth a follow-up to make the readers tolerant of both shapes (one-line change with a `or_else`).

### Frontend — `src/features/agents/components/matrix/BuildSimulatePanel.tsx` (new, 12 Vitest tests)

- Modal panel triggered by a new "Dry-run" button next to "Promote" / "Refine" in the test_complete branch of `GlyphFullLayout.tsx` (button only — full panel is `BuildSimulatePanel`).
- Reads `buildSessionId` + `buildDraft` from `useAgentStore`, extracts capabilities via the pure `extractUseCases(draft)` helper (tolerant of both `use_cases` and `useCases` keys, structured + simple-string variants, missing ids fabricated to `uc_idx_<n>`).

- On Run: calls `simulateBuildDraft` → on success calls `getSimulationArtefacts(execution.id)` → renders an artefacts subview with the manual reviews list and the memories list. On failure, surfaces the error message in a `role="alert"` region.
- Test bridge helpers added: `simulateBuildDraft(useCaseId, inputOverride?)` and `getSimulationArtefacts(executionId)` in `src/test/automation/bridge.ts` for future Phase scenarios.

## C7 — 2026-04-28 increment: webhook triggers in the build flow

**Goal:** when the build LLM picks `trigger_type: "webhook"` for a capability, the questionnaire asks the user for a smee.io channel URL up front. At promote time the smee_relays binding is auto-created so the persona starts receiving forwarded events immediately — no manual SmeeRelayTab dance afterwards.

### Backend

#### Trigger config extension — `src-tauri/src/db/models/trigger.rs`

`TriggerConfig::Webhook` gains two optional fields:

```rust
Webhook {
    webhook_secret: Option<String>,
    event_type: Option<String>,
    payload: Option<serde_json::Value>,
    smee_channel_url: Option<String>,    // NEW — set by the LLM from the answer
    smee_event_filter: Option<String>,   // NEW — comma-separated event_type allowlist
}
```

Both fields are `#[serde(default, skip_serializing_if = "Option::is_none")]` — backward-compatible with existing webhook trigger rows. `parse_config` round-trips them; the runtime webhook listener (`engine/webhook.rs`) ignores them (HMAC verification is unchanged).

#### Validator — `src-tauri/src/validation/trigger.rs`

When `smee_channel_url` is present and non-blank, it must start with `https://smee.io/`. Rejects malformed URLs at validate time so a bad answer fails fast rather than silently being skipped at smee-relay-create.

#### `UserAnswer.webhook_source` — `src-tauri/src/db/models/build_session.rs`

```rust
pub struct UserAnswer {
    pub cell_key: String,
    pub answer: String,
    pub reference: Option<UserAnswerReference>,
    pub webhook_source: Option<UserWebhookSource>,  // NEW
}

pub struct UserWebhookSource {
    pub channel_url: String,
    pub event_filter: Option<String>,
}
```

`BuildEvent::{Question, ClarifyingQuestionV3}` both gain `accepts_webhook_source: bool` (defaults false). The parser reads `accepts_webhook_source` from the LLM's clarifying_question JSON.

#### `answer_build_question` IPC — `src-tauri/src/commands/design/build_sessions.rs`

The command now accepts an optional `webhook_source` param. When supplied:
1. Validates `channel_url` starts with `https://smee.io/`.
2. Calls new helper `engine::build_session::reference::append_webhook_source_fence(answer, channel_url, event_filter)` — appends a `--- WEBHOOK SOURCE ---\nchannel_url: ...\nevent_filter: ...\n--- END WEBHOOK SOURCE ---` block to the answer text.
3. Pipes the augmented answer to the CLI via the existing send_answer path.

#### Build prompt rule 24 — `src-tauri/src/engine/build_session/session_prompt.rs`

New rule explains: when the LLM picks webhook trigger, it MUST emit a `clarifying_question` with `accepts_webhook_source: true`. Documents the exact fence shape the user's answer will arrive in on the next `--continue` turn, and gives a concrete IR example showing where to place the URL on the trigger config (`config.smee_channel_url` + `config.smee_event_filter`). Lists when to ASK (default for webhook triggers) and when to SKIP (intent already inlines a smee URL, non-webhook trigger types, user said "I'll set up smee later").

#### Promote-time auto-bind — `src-tauri/src/commands/design/build_sessions.rs::auto_create_smee_relays`

Runs AFTER the promote transaction commits (best-effort — failures here don't unwind the persona promote; the user can attach manually via SmeeRelayTab later). Iterates `ir.triggers`, finds webhook triggers carrying `smee_channel_url`, and:

- **Idempotent on same `(persona_id, channel_url)`** — skips silently when a binding already exists.
- **Refuses to repoint** — if a smee_relay row exists with the same URL pointing at a DIFFERENT persona, logs a warning and skips (the user owns that binding; we don't steal it).
- Calls `smee_repo::create` with `target_persona_id = persona_id`, `event_filter = smee_event_filter`, `label = "Build · trigger <description-or-index>"`.
- After all webhook triggers processed, calls `state.smee_relay_notifier.notify()` so the relay manager picks up the new rows immediately.

Returns the count of relays created; surfaced in the promote response as `smee_relays_created`.

7 unit tests in `commands::design::build_sessions::tests` cover: skip when no webhook triggers, skip when no smee URL, basic create, idempotent same-persona, refuse cross-persona repoint, mixed trigger types, blank URL skipped.

### Frontend — `src/features/agents/components/matrix/WebhookSourcePicker.tsx` (new, 9 Vitest tests)

Self-contained form: smee URL input with format validation (`https://smee.io/...`) + optional comma-separated event-filter input + "+ Don't have one? Create at smee.io/new" button (`openExternalUrl`). Selected source renders as a chip with channel URL + filter brackets + clear button.

### Phase H — webhook trigger + smee auto-bind E2E (`tools/test-mcp/e2e_phase_h.py`)

2. The webhook trigger config's `smee_channel_url` matches what the user attached (proves the LLM correctly placed the URL per build prompt rule 24).
3. A `smee_relays` row exists with `target_persona_id == new persona`, `channel_url == submitted URL`, `event_filter == submitted filter`.

The smee URL is `https://smee.io/phase-h-<random hex>` per-run so re-runs don't collide on the `smee_relays.channel_url` UNIQUE constraint. The relay manager will fail to connect to the bogus channel (smee returns 404 for unknown), which is fine — the driver tests build-time wiring, not runtime forwarding.

### Phase E — preload-error reload (deferred-manual)

## C7 — 2026-04-28 follow-ups: persona block hoist + smee orphan cleanup + Clockify Phase I

Three small follow-ups landed after the C7-2026-04-28 commit:

### #3 — Persona block hoist into `last_design_result`

`AgentIr` had no `persona` field, so serde dropped the entire v3 persona
block (mission / identity / voice / `principles[]` / `constraints[]` /
`decision_principles[]` / verbosity_default / operating_instructions /
tool_guidance / error_handling / connectors / tools / core_memories) at
deserialize time. The v3 normalizer's `compose_structured_prompt` reads
the block once to render `structured_prompt.identity` prose, but the
structured arrays were lost. Net effect: the auto_triage second-pass
evaluator's `extract_principles_from_design_result` got an empty
`decision_principles[]` regardless of what the LLM emitted.

**Fix** — `AgentIr` gains `pub persona: Option<serde_json::Value>` and
`build_design_json` forwards it onto `last_design_result.persona`.
Verified by Phase I run 2: `last_design_result.persona.decision_principles`
now has 2 entries the LLM emitted.

### #7 — Smee relay orphan cleanup

Multiple Phase H runs left `smee_relays` rows pointing at NULL
`target_persona_id` (the FK ON DELETE SET NULL leaves orphans when the
test persona is deleted). Cleanup needed exposing `smee_relay_delete` to
the test bridge, which required relaxing its tier the same way
`smee_relay_list` was relaxed: it's a local-DB-only operation with no
cloud / OAuth dependency. Now `require_auth` only, removed from
`CLOUD_COMMANDS_SET`. Bridge wrapper `smeeRelayDelete(id)` added.

3 orphans nuked; smee_relays table is empty again.
