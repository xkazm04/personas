### G — Build pipeline: auto-default `source_filter = "*"` for cross-persona inbounds
`src-tauri/src/commands/design/build_sessions.rs`

New helper `collect_persona_emit_event_types(ir, use_cases)` unions the persona-level `ir.events[direction == "emit"]` and per-UC `event_subscriptions[direction == "emit"]` into a `HashSet<String>`. Both `create_triggers_in_tx` (event_listener trigger configs) and `create_event_subscriptions_in_tx` (persona_event_subscriptions rows) consult the set: when a LISTEN target's event_type isn't in the persona's emit set, the event has to come from a different persona — the bus's self-scoping rule (`bus.rs:131-145`) would silently drop it without a `source_filter`, so promote now defaults `source_filter = "*"` (only when the LLM didn't provide one).

**Why this preserves the original safety property** (the rule was added so two unrelated personas with overlapping subscriptions can't trigger each other's runs):

- Intra-persona self-loops → event ∈ emit set → `source_filter` stays null → bus self-scopes correctly.
- Cross-persona chains → event ∉ emit set → `source_filter = "*"` defaulted → bus delivers from any source persona.
- LLM-supplied `source_filter` is always honoured — auto-wiring only fills in when null.

This was discovered during Phase B run #1: persona X emitted `news.draft.captured` 12 times, persona Y had a matching subscription (and trigger), Y had 0 executions because its subscription's `source_filter` was null and the bus filtered out the cross-persona event. Phase B run #2 (against the rebuilt binary with the fix) showed the **X→Y→Z cascade firing automatically with no manual subscription patching**.

**Tests**: 4 new unit tests on `collect_persona_emit_event_types` (per-UC source, persona-level source, union, ignore-non-emit).

### H — `notification_channel_array` v3.1 shape support
`src-tauri/src/db/models/agent_ir.rs::AgentIr::notification_channel_array`

Was only handling two of the three IR shapes the build pipeline emits. The v3.1 prompt aliases `suggested_notification_channels` straight into `messages` as a **top-level array**, but the old code looked for `messages.channels` (object) or top-level `notification_channels`. Result: every persona using the v3.1 shape promoted with `NULL` `personas.notification_channels` despite the LLM emitting the channel correctly. Phase C titlebar scenario surfaced this — the IR had `[{"channel":"built-in","target":"titlebar","format":"notification"}]` but the persona row's column was empty.

Resolver now tries all three shapes in priority order (legacy `messages.channels`, v3.1 alias array, top-level field). 5 new unit tests cover all three plus priority order plus the absent case.

### I — `ChannelSpecV2` rename_all reverted (pre-existing master breakage)
`src-tauri/src/db/models/persona.rs`

Removed `#[serde(rename_all = "camelCase")]`. The codebase's wire format is snake_case everywhere — frontend (which doesn't reference the auto-generated TS binding for this struct), validation (`validate_notification_channels` probes `use_case_ids`), the `parse_channels_v2` discriminant, and every test JSON literal. The rename was a copy-paste artefact that broke shape-v2 round-trips — `parse_channels_v2` returned `None` for any real shape-v2 JSON, so the dispatch shape-v2 short-circuit (`dispatch.rs:928`) never triggered.

**Tests**: 6 previously-failing tests now pass — `engine::dispatch::tests::test_resolve_shape_v2_passthrough_pure` and 5 in `notifications::tests`.

`src/lib/bindings/ChannelSpecV2.ts` regenerated to snake_case (verified `npx tsc --noEmit` clean — no frontend code references the old camelCase form).

### J — Build prompt rule 22: verbatim event names
`src-tauri/src/engine/build_session/session_prompt.rs`

> When the user's intent or clarifying answer explicitly names an `event_type` — typically inside backticks — use that exact string verbatim. Do NOT rewrite it into a fresh `<persona>.<task>.<event_type>` token derived from the current persona's name. The three-level-dot rule (rule 5) is a *default* that applies when YOU invent a new event name; it is NOT licence to rename a name the user has hand-wired.

Phase B run #1 found the LLM doing exactly this — produced `hn_scraper.story.draft_captured` instead of the explicitly-requested `news.draft.captured`. The Phase B answer recipe was hardened with stronger event-name lock prose at the time; rule 22 codifies the same expectation in the system prompt itself.

## Open architectural items

### 1. Full LLM-judged self-review pass for `auto_triage`
The current MVP (fix C above) makes `mode: "auto_triage"` stop falling through to `manual_review` — it stores + auto-resolves the review with a distinct audit tag. The original spec calls for a **second LLM round-trip** that evaluates the agent's output against the capability's `decision_principles` and renders an approve/reject verdict. That doesn't exist yet. Real product work; deserves its own design pass.

### 2. WebView2 preload-error recovery — empirical verification
The `vite:preloadError` listener (fix E above) is the canonical Vite pattern, but the actual repro (tauri-cli restarts the Rust binary → WebView's HMR socket goes stale → next lazy import fails) wasn't reproduced in this session. Worth a 5-minute live test next time the dev app is up: deliberately edit a Rust file to force a rebuild, then click around the UI and confirm the WebView reloads cleanly instead of erroring.

### 3. (informational) Phase A.2 expected_use_cases cosmetic fix
`tools/test-mcp/e2e_phase_a.py` still has `expected_use_cases: 3` for the coordinator scenario; rule D (capability-granularity for chained pipelines) makes the LLM legitimately produce 4. The script reports a `fail` on the count check for what is actually correct behaviour. One-line fix; not blocking.
