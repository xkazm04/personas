# E2E Test: Agent Generation → Lab Testing → Execution

Cross-module integration test covering the full agent lifecycle: creation via PersonaMatrix, prompt improvement via Lab, and manual execution with result verification.


## Phase 1: Agent Generation via PersonaMatrix

### Intent
Create an AI news research agent from scratch using PersonaMatrix build flow.

### Steps

#### 1.1 Start Agent Creation
```
POST /click-testid  {"test_id": "create-agent-btn"}
POST /wait          {"selector": "[data-testid='agent-intent-input']", "timeout_ms": 5000}
POST /fill-field    {"test_id": "agent-intent-input", "value": "Research AI news and extract daily 5 most interesting topics"}
POST /click-testid  {"test_id": "agent-launch-btn"}
```

**Expected**: Build session starts. Toast shows "Building persona...". Matrix cells begin populating.

#### 1.2 Wait for First Question
```
POST /wait  {"selector": "[data-testid='freetext-input'], [data-testid='option-button-0']", "timeout_ms": 120000}
```

**Expected**: The build process asks about resources or information type. A question popover appears with either free-text input or option buttons.

**Verify**: `GET /snapshot` should show `isCreatingPersona: true`.

#### 1.3 Answer Build Question (Free-Text)
```
POST /wait       {"selector": "[data-testid='freetext-input']", "timeout_ms": 5000}
POST /fill-field {"test_id": "freetext-input", "value": "any recommended resources, news focused on AI agents"}
POST /click-testid {"test_id": "submit-button"}
```

**Expected**: Build continues. If there's a follow-up question, answer it similarly. The build may ask 1-3 questions total about:
- Preferred news sources (answer: "tech news sites, arxiv, twitter/X")
- Output format preference (answer: "bullet point summaries with links")
- Frequency (answer: "daily")

#### 1.4 Wait for Build Completion
```
# Poll until build completes (all dimensions resolved)
# Check via DB: SELECT build_phase FROM build_sessions ORDER BY created_at DESC LIMIT 1
# Or poll snapshot for isCreatingPersona: false
POST /wait {"selector": "[data-testid='agent-name-input']", "timeout_ms": 180000}
```

**Expected**: All matrix dimensions populated (identity, instructions, toolGuidance, examples, errorHandling). Agent name auto-generated by LLM.

#### 1.5 Verify Dimensions Populated
```
GET /state
```

**Expected state**:
- `buildPhase`: `"draft_ready"` or `"test_complete"`
- `buildCellStates`: All cells should be `"resolved"` or `"filled"` (not `"highlighted"`)
- `personaCount`: incremented by 1

#### 1.6 Verify Agent in Sidebar
```
GET /snapshot
POST /search-agents {"query": "AI"}
GET /agent-cards
```

**Expected**: New agent appears in sidebar with LLM-generated name. Agent has:
- Structured prompt with all sections populated
- web_search tool assigned (for researching news)
- Use cases derived from design context

#### 1.7 (Optional) Test Draft
```
POST /click-testid {"test_id": "agent-test-btn"}
# Wait for test to complete
POST /wait {"selector": "[data-testid='agent-test-btn']", "timeout_ms": 120000}
```

**Expected**: CLI dry-runs the agent. Test results show tool calls (web_search) and formatted output.

### Success Criteria
- [ ] Agent created with all matrix dimensions resolved
- [ ] Agent has a structured_prompt with identity, instructions, toolGuidance, examples, errorHandling
- [ ] Agent has web_search tool assigned
- [ ] Agent name is relevant to AI news research
- [ ] Build took < 3 minutes

## Phase 2: Lab Testing & Persona Improvement

### Intent
Use Lab module to test the agent across models, create a v2 prompt, and compare versions.

### Steps

#### 2.1 Navigate to Lab
```
POST /select-agent     {"name_or_id": "<agent-name-from-phase-1>"}
POST /open-editor-tab  {"tab": "lab"}
POST /click-testid     {"test_id": "lab-mode-arena"}
```

**Expected**: Lab tab opens with Arena mode. Agent's use cases appear in the Focus dropdown.

#### 2.2 Run Arena Test (Haiku + Sonnet)
```
# Verify both Haiku and Sonnet are selected (default)
GET /list-interactive
# Ensure arena-model-haiku and arena-model-sonnet are in the list

POST /click-testid {"test_id": "arena-run-btn"}
```

**Expected**: Toast shows "Generating Test Scenarios...". Progress shows scenario names as they execute.

#### 2.3 Monitor Arena Run (via DB)
```sql
-- Poll every 15s:
SELECT status, scenarios_count FROM lab_arena_runs
WHERE persona_id = '<agent-id>' ORDER BY created_at DESC LIMIT 1;

SELECT COUNT(*) FROM lab_arena_results
WHERE run_id = '<run-id>';
```

**Expected**: Status transitions: `generating` → `running` → `summarizing` → `completed`. Results count reaches `scenarios × models` (e.g., 5 × 2 = 10).

#### 2.4 Verify Arena Results
```sql
SELECT scenario_name, model_id, tool_accuracy_score, output_quality_score,
       protocol_compliance, rationale IS NOT NULL as has_rationale,
       llm_summary IS NOT NULL as has_llm_summary
FROM lab_arena_results ar
JOIN lab_arena_runs r ON ar.run_id = r.id
WHERE r.persona_id = '<agent-id>'
ORDER BY ar.created_at DESC LIMIT 10;
```

**Expected**:
- All results have status `"passed"`
- `has_rationale = 1` (structured rationale JSON)
- Scores reflect actual tool usage and output quality
- `llm_summary` populated on the run record (or null if CLI unavailable)

#### 2.5 Open Arena Results Modal
```
POST /click {"selector": "table tbody tr:first-child"}
POST /wait  {"selector": "[role='dialog']", "timeout_ms": 3000}
```

**Expected**: Modal shows:
- Test Summary (LLM-generated or auto-generated)
- Model Performance cards with score bars
- Scenario Breakdown table
- "Auto-Improve Prompt" button with weakness preview

#### 2.6 Request Persona Change via Matrix
```
POST /click-testid {"test_id": "lab-result-modal-close"}
POST /click-testid {"test_id": "lab-mode-matrix"}
POST /fill-field   {"test_id": "matrix-instruction", "value": "Focus more on general LLM area instead of just AI agents. Cover topics like model releases, benchmarks, fine-tuning techniques, and inference optimization."}
POST /click-testid {"test_id": "matrix-run-btn"}
```

**Expected**: Matrix run starts with phases: `drafting` → `generating` → `running` → `completed`.

#### 2.7 Monitor Matrix Run (via DB)
```sql
SELECT status, draft_change_summary FROM lab_matrix_runs
WHERE persona_id = '<agent-id>' ORDER BY created_at DESC LIMIT 1;
```

**Expected**: Status reaches `completed`. `draft_change_summary` describes the LLM-focused changes.

#### 2.8 Accept Draft (creates v2)
```
-- After matrix completes, open the result modal and accept the draft
POST /click {"selector": "table tbody tr:first-child"}
POST /wait  {"selector": "[role='dialog']", "timeout_ms": 3000}
-- Find and click "Accept Draft" button in the modal
POST /click {"selector": "[role='dialog'] button:has(svg)"}  -- or find by text
```

**Expected**: Draft accepted. New version v2 created. Prompt updated with LLM focus.

#### 2.9 Run Eval (v1 vs v2 across Haiku, Sonnet, Opus)
```
POST /click-testid {"test_id": "lab-mode-eval"}

-- Select v1 and v2
POST /click-testid {"test_id": "eval-version-toggle-1"}
POST /click-testid {"test_id": "eval-version-toggle-2"}

-- Select all 3 models (Haiku default, add Sonnet and Opus)
POST /click-testid {"test_id": "eval-model-toggle-haiku"}
POST /click-testid {"test_id": "eval-model-toggle-sonnet"}
POST /click-testid {"test_id": "eval-model-toggle-opus"}

-- Start eval
POST /click-testid {"test_id": "eval-start-btn"}
```

**Expected**: Eval runs v1 × (haiku, sonnet, opus) + v2 × (haiku, sonnet, opus) = up to 30 results (5 scenarios × 2 versions × 3 models).

#### 2.10 Verify Eval Results
```sql
SELECT r.scenario_name, r.version_number, r.model_id,
       r.tool_accuracy_score, r.output_quality_score, r.protocol_compliance
FROM lab_eval_results r
JOIN lab_eval_runs run ON r.run_id = run.id
WHERE run.persona_id = '<agent-id>'
ORDER BY r.created_at DESC LIMIT 30;
```

**Expected**:
- v2 should show improved scores in output_quality (broader LLM coverage)
- Scores vary by model (Opus typically highest quality, Haiku fastest/cheapest)
- All results have rationale populated

### Success Criteria
- [ ] Arena test completes with 10 results (5 scenarios × 2 models)
- [ ] Arena results have rationale and scores
- [ ] Matrix generates a draft with LLM-focused changes
- [ ] Draft accepted creates v2
- [ ] Eval runs v1 vs v2 across 3 models
- [ ] v2 shows improved or comparable scores to v1
- [ ] Lab runs complete in background (user can navigate away)

## Known Issues & Workarounds

### Build Session: Answer + Continue Pattern
The build session asks questions via a popover. After answering, user must click "Continue Build" (`continue-build-btn`). The answer button's testid is `answer-button-{cell-key}` (e.g., `answer-button-use-cases`).
```
POST /click-testid  {"test_id": "answer-button-use-cases"}
POST /wait          {"selector": "[data-testid='freetext-input']", "timeout_ms": 5000}
POST /fill-field    {"test_id": "freetext-input", "value": "answer text"}
POST /click-testid  {"test_id": "submit-button"}
# Then click Continue Build
POST /click-testid  {"test_id": "continue-build-btn"}
```

### Build Session: Promote Draft
After build completes (`draft_ready` phase), the `structured_prompt` is in the `agent_ir` JSON of the `build_sessions` table but NOT yet applied to the persona. The `promote_build_draft` Tauri command must be called (or "Test Agent" button clicked first, which may trigger it). If promotion fails, extract from DB:
```python
# Extract from build_sessions.agent_ir → personas.structured_prompt
ir = json.loads(agent_ir)
sp = ir['structured_prompt']  # Has: identity, instructions, toolGuidance, examples, errorHandling
```

### Bridge Freeze During Lab Runs
The test automation bridge may freeze during or after Lab runs (15s timeout on snapshot).
**Workaround**: Kill `personas-desktop.exe` and restart tauri dev. Use DB polling instead of bridge for monitoring Lab runs.

### Matrix CLI Timeout
Matrix draft generation may timeout at 120s for complex instructions.
**Workaround**: Use shorter, more focused instructions.

### Use Case Parsing
Use cases stored as string arrays in `design_context` are auto-normalized to `DesignUseCase[]` objects. Verify the Focus dropdown shows them.

### Dropdown Timing
When clicking Listbox dropdowns via bridge, use `/wait` between opening the trigger and clicking the option:
```
POST /click-testid  {"test_id": "trigger-button"}
POST /wait          {"selector": "[data-testid='option-value']", "timeout_ms": 3000}
POST /click-testid  {"test_id": "option-value"}
```
