## The control model — push controls LEFT of CI

A CI gate fires **after** a branch is pushed. By then a leaked secret has already left the machine
and a failed gate is a wasted round-trip. In an LLM-driven workflow the **agent is the first line of
control**, so this skill places work in two layers — and most of it is *not* in CI:

1. **Repo / pre-push (primary).** A checklist the agent self-runs *before* pushing — encoded in
   agent guidance (CLAUDE.md/AGENTS.md), local git hooks, and your verify-before-propose loop. Fast,
   local, and it catches problems before they ever leave the box.
2. **CI / hard pass (thin backstop).** Only the non-negotiables that truly need a remote, clean-room,
   or full-tree environment: SAST over the whole tree, the merge gate itself, scheduled scans. CI
   *confirms* what the agent already enforced; it is not the enforcer.

When you adopt a track, **extend the mechanisms this repo already has** (e.g. an existing lefthook /
husky config, an existing CI workflow) rather than adding a parallel one. Each track below tells you
which layer its controls belong in.
