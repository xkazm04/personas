### Journey 1 — wire-credential-connector → **L1-pass** (trust-legibility minor)

**Rationale.** Secret-at-rest handling is senior-grade and Sam can mostly verify it. `create_credential` (`src-tauri/src/commands/credentials/crud.rs:34`) decrypts the session-wrapped payload, then stores each sensitive field as ciphertext via field-level encryption; the secret column is `credential_fields.encrypted_value` with a base64 12-byte nonce in `iv` (`src-tauri/src/db/migrations/schema.rs:177-188`) — plaintext is never persisted. Crypto is AES-256-GCM (`src-tauri/src/engine/crypto.rs:1254`), per-record random nonce from `OsRng` (`crypto.rs:1266-1268`), nonce length validated on decrypt (`crypto.rs:1284`). Master key lives in the OS keychain (Windows Credential Manager / macOS Keychain) with a **fail-closed** posture — the DPAPI-wrapped local fallback only activates with an explicit `PERSONAS_ALLOW_FALLBACK_KEY=1` opt-in (`crypto.rs:476-524`); the key is `mlock`/`VirtualLock`-pinned off swap (`crypto.rs:319`) and zeroized on drop. OAuth tokens ride the *same* vault path: `EncryptedToken::seal()` → `encrypt_for_db()` (`src-tauri/src/commands/credentials/oauth.rs:402-410`, `crypto.rs:265-286`), debug-redacted as `[ENCRYPTED]`. Credential types reachable from the picker: AI-Guide, MCP, Custom API, Database, Desktop (dev-only) plus workspace-connect and foraging discovery (`src/features/vault/sub_credentials/manager/CredentialAddViews.tsx:28-78`). A wrong/missing field blocks the save with a visible reason rather than failing silently. This satisfies the DoD on both clauses.

**The minor (finding S1).** At the moment Sam pastes an API key, the form shows one emerald 70%-opacity line — "Encrypted with OS Keychain" / "Encrypted at rest" — and only when a `password`-type field is present (`src/features/vault/sub_credentials/components/forms/FormActions.tsx:35-42`). The reviewer-grade reassurance strings he actually wants — `"Credentials never leave this device"`, `"AES-256-GCM … unique random nonce … only this app can decrypt"`, `"Master key in OS Keychain"` — exist (`src/i18n/locales/en.json` `vault.vault_badge.*`, ~lines 4451-4463) but render in a vault status badge, **not in the entry form**. It's a pass because the guarantee is genuinely there and a first-use consent modal discloses it; it's flagged because Sam re-reads the safety story every time he enters a secret, and the entry moment under-tells it.

### Journey 2 — set-trigger-automate → **L1-conditional**

**Rationale.** The mechanics are sound and the armed/off state is *legible* — which for Sam is half the battle. A trigger carries `enabled` + `status` (`active`/`paused`/`disabled`) and a backend-computed `next_trigger_at` (`src-tauri/src/db/models/trigger.rs`). The row renders a clear green/grey toggle (`src/features/triggers/sub_triggers/TriggerRow.tsx:32-58`) and a live countdown/next-fire ring (`src/features/triggers/sub_triggers/TriggerCountdown.tsx:39-56`). The fire path is real and unattended: the scheduler tick (`src-tauri/src/engine/background.rs:1629-1961`) selects due triggers (`status='active' AND next_trigger_at<=now AND persona.enabled=1`), advances the schedule, and publishes an event (`background.rs:1932-1957`); the event-bus tick (`background.rs:800-1241`) matches subscriptions and calls `engine.start_execution()` (`background.rs:1161-1171`). So the DoD — "I configured a trigger, I can tell it's armed and when it next fires, and I believe it'll run" — is structurally met. There's even a one-off `dry_run_trigger` simulation (`src-tauri/src/commands/tools/triggers.rs:1257`).

**What blocks pass — Sam's exact scar (finding B1, major).** Between schedule-fire and a real persona execution there is **no confirmation gate and no per-trigger "require approval" / "always dry-run" option**. The only runtime brakes are: persona-enabled, trigger-enabled, monthly budget cap, active-hours window, and a serial cascade guard. None of those is "should this destructive run happen *this time*." The `PersonaTrigger` model and trigger config carry no `approval_required` / `confirm_before_fire` / `dry_run_only` field. For Sam this is the 3am-auto-restart shape exactly: a cron he armed weeks ago fires a write-capable persona with nobody watching and no per-fire brake. Whether the *action inside* that run is gated is Journey 3's problem — and it isn't reliably gated either. The trigger UI also doesn't surface the run's blast-radius ("this trigger fires a persona that can post to Slack / write to the DB"), so "armed" is legible but "armed to do *what*" is not (finding B2, minor).

### Journey 3 — run-and-review-execution → **L1-conditional**

**Rationale.** The review/learn loop is genuinely well-built and is the strong half. Output legibility is good: execution `status` is `success`/`failed`/`incomplete`/`running` with a human-readable `error_message` and a *separate* `business_outcome` field, rendered as a colored badge + red error line in `src/features/overview/sub_activity/components/ExecutionRow.tsx:43-74`, with a tabbed detail modal (`src/features/overview/ExecutionDetailModal/ExecutionDetailContent.tsx`). The accept/reject loop is NOT a dead-end: `update_manual_review_status` (`src-tauri/src/commands/design/reviews.rs:1074`) writes a `learned` memory atomically with the status transition, emits `MANUAL_REVIEW_RESOLVED`, publishes a `review_decision` event for downstream personas, and calls `react_to_review_decision` (`reviews.rs:1130`) which resumes a blocked team assignment on approve. The Athena-resolved path mirrors it exactly (`src-tauri/src/commands/companion/approvals.rs:750-807`). Severity-correct triage exists via the healing orchestrator (`src-tauri/src/engine/healing_orchestrator.rs`): Credential/Unknown errors *always* route to a human issue; RateLimit/Timeout/External escalate to `CreateIssue` after `MAX_RETRY_COUNT(3)`. And healing **cannot auto-act destructively** — auto-action is limited to retry (`RuleBasedRetry`) and dev-mode AI diagnosis; `CreateIssue`/`CircuitBreakerTripped` take no automatic action (`src-tauri/src/engine/mod.rs:3022-3061`), and AI-healing is explicitly blocked from disabling a persona without human approval (`src-tauri/src/engine/ai_healing.rs:520-537`). That's the right shape.

**What blocks pass — the gate is advisory and opt-in (finding B3, major; the report's sharpest).** "Route low-confidence to a human" is implemented as a protocol message the **persona emits** (`ProtocolMessage::RequestReview` → review row + notification, `src-tauri/src/engine/dispatch.rs:605-707`), governed by a per-capability `review_policy.mode` of `never` / `on_low_confidence` / `always` / `auto_triage` decided **at build/design time by the LLM** (`src-tauri/src/engine/build_session/session_prompt.rs:225,374,421`). The engine does **not intercept or hard-block the external/destructive tool call** — it creates a review record *if the model chose to ask*. The actual MCP tool surface executes writes directly in `call_tool()` with no approval check (`src-tauri/src/mcp_server/tools.rs:1000-1047`; e.g. `drive_write_text`, `obsidian_vault_write_note`). So "destructive actions are gated by default" reduces to three soft assumptions: (1) the persona was *designed* with the publish step split into its own `review_policy=always` capability gated on an `.accepted` event (the producer/publisher-split convention, `session_prompt.rs:206`); (2) the build LLM complied with that convention; (3) the runtime LLM complied with "draft, then emit manual_review and wait" instead of just acting. Worse, the defaults lean *toward* auto-publish: the first option offered is "Never — auto-publish; I can undo/discard myself" (`session_prompt.rs:421`) and a "simple-periodic-report fast-path" resolves `review_policy` to a non-blocking default *without asking* (`session_prompt.rs:581`). `TrustLlm` and `AutoTriage` modes auto-resolve the review with no human queue at all (`dispatch.rs:638-683`). For Sam: there is **no hard, default-on, runtime gate** that stops a persona from sending the email / posting the message / writing the row before a human sees it. The good `requires_approval` *pre-execution* gate — where the action genuinely stays `pending` until a click (`src-tauri/src/companion/dispatcher.rs:1453`, capabilities in `src-tauri/src/companion/connectors.rs`) — exists **only on the companion/Athena path**, not the general persona-execution path. He'd accept the loop quality; he would not accept "the model was told to ask first" as a blast-radius control.

### Journey 4 — trust-and-governance → **L1-conditional**

**Rationale.** Most of Sam's governance asks are answerable in-product, which surprises him pleasantly. **Audit trail:** every run is an append-only `PersonaExecution` record (status/trigger-source/cost/duration), surfaced with filters and counts in `src/features/overview/sub_activity/components/GlobalExecutionList.tsx` over `src-tauri/src/commands/execution/executions.rs:33-98`; there is no `delete_all` for *executions* (the bulk-delete commands only cover messages and memories), so the run log can't be trivially erased. **SLA + incidents:** a real SLA dashboard (`src/features/overview/sub_sla/components/SLADashboard.tsx` over `src-tauri/src/commands/communication/sla.rs:14`) and a cross-source incident inbox with acknowledge/in-progress/resolve workflow (`src/features/overview/sub_incidents/components/IncidentsInbox.tsx` over `src-tauri/src/commands/execution/audit_incidents.rs:80-148`). **Network:** P2P is **off by default** — `NetworkConfig::default()` sets `auto_connect: false` (`src-tauri/src/engine/p2p/types.rs:18-29`), exposure is per-resource opt-in via `ExposureManager.tsx` over `src-tauri/src/commands/network/exposure.rs:16-90`, and every exposure op is `tracing`-logged. **Data locality is surfaced in-product, not just README:** the first-use/onboarding copy states "local-first … all data … stored in a local SQLite database on your machine … No data is sent to Personas servers unless you explicitly use the optional cloud deployment feature" (`src/i18n/locales/en.json` ~10469-10478), credentials "never leave this device" (~4461), and cloud sync is an explicit opt-in read-only copy that keeps connector creds local (`src/features/settings/sub_account/components/CloudSyncCard.tsx`). Export/import lives in `src/features/settings/sub_portability/components/DataPortabilitySettings.tsx`. He can assemble a defensible reviewer story.

**What blocks pass (finding G1, major; finding G2, minor).** G1: **there is no tier gate or team-level RBAC.** No `tier`/`Starter`/`Builder`/`entitlement` enforcement was found in the command layer; the team model (`src-tauri/src/db/models/team.rs:20-78`) has no role/permission fields. Governance is all-or-nothing per device: anyone with the app does anything — including arming write-capable triggers and approving reviews. For a 200-person shop where Sam is supposed to *control access*, "who can create a trigger / approve a destructive action" has no answer in code. Since he's Team tier expecting to administer others, this is the gap that turns his "switch from Zapier/n8n?" verdict cautious. G2: the strongest reviewer numbers — any unencrypted creds, anyone on the legacy weak-IPC path — are computed in `vault_status` but, per the parallel enterprise-admin trace, render nowhere in the UI; and the Admin settings tab is dev-only, so the in-product governance console he'd want isn't there.

## What passed (do not touch)

- **At-rest crypto is senior-grade and fail-closed.** AES-256-GCM, per-record `OsRng` nonce, OS-keychain master key, opt-in-only DPAPI fallback, `mlock`/zeroize. `crypto.rs:476-524,1254-1298`. OAuth tokens use the same path. This is the floor Sam trusts.
- **Healing cannot auto-act destructively.** Auto-action limited to retry + dev-mode diagnosis; `CreateIssue` takes no automatic action; AI-healing is blocked from disabling personas without approval. `mod.rs:3022-3061`, `ai_healing.rs:520-537`.
- **The review→learn→resume loop is real, not a dead-end.** Atomic `learned` memory + event publish + blocked-assignment resume on approve, identical for human and Athena paths. `reviews.rs:1074-1152`, `approvals.rs:750-807`.
- **Audit trail is append-only and complete.** Per-run immutable executions, no bulk-delete for executions, SLA dashboard + incident inbox over real data. `executions.rs:33-98`, `sla.rs:14`, `audit_incidents.rs:80-148`.
- **P2P off by default; data-locality surfaced in onboarding/settings, not just README.** `p2p/types.rs:18-29`; `en.json ~10469-10478`; `CloudSyncCard.tsx`.
- **Armed/off is legible.** Green/grey toggle + live next-fire countdown per trigger. `TriggerRow.tsx`, `TriggerCountdown.tsx`.

## Character voice

Show me the floor and I'll grant it: the crypto is real, the keys are in the keychain, the run log can't be wiped, and P2P is off until I say so. That's more than most "automation" vendors can defend. But I asked the only question that matters — *what happens when it's wrong at 3am* — and the honest answer is that the brake on a destructive action is a sentence in a prompt asking the model to please ask first, plus a hope that whoever designed the persona split the "send" step out from the "draft" step. The hard, click-to-fire gate I'd actually trust exists only on the Athena side, not on the cron-fired persona that's going to write to my production database while I'm asleep — and there's no per-trigger "require my approval," no tier, no role, no one I can stop from arming it. I'd pilot the read-only digest tomorrow. I would not let it touch anything that sends, posts, or deletes until "ask a human first" is a default-on hard gate I can point a security reviewer at — not an instruction the model is free to skip.
