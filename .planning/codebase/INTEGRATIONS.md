# External Integrations

**Analysis Date:** 2026-03-13

## APIs & External Services

**Cloud Execution:**
- Personas Cloud API - Remote job execution, worker management, cost tracking
  - Client: `src-tauri/src/cloud/client.rs` (CloudClient)
  - Auth: Bearer token (cloud_token in database)
  - Endpoints: /health, /api/status, /api/execute, /api/oauth/*, /api/get, /api/post
  - Response types: CloudStatusResponse, CloudExecutionPoll, CloudWorkerCounts

**GitHub Integration:**
- GitHub API (api.github.com)
  - SDK/Client: `src-tauri/src/engine/platforms/github.rs` (GitHubClient)
  - Auth: Personal Access Token (PAT) stored in encrypted credentials
  - Scope: repo, workflow
  - Operations: List repositories, check permissions, trigger workflow dispatch
  - Used for: Automation invocation via GitHub Actions

**n8n Integration:**
- n8n Workflow API
  - SDK/Client: `src-tauri/src/engine/platforms/n8n.rs` (N8nClient)
  - Auth: API key (stored in encrypted credentials)
  - Base URL: User-provided n8n instance URL
  - Operations: List workflows, activate/deactivate workflows, execute workflows, manage tags
  - Used for: Automation deployment and execution

**Zapier Integration:**
- Zapier Platform API (zapier.com)
  - Operations: List Zaps, create Zaps, trigger webhooks
  - Auth: OAuth 2.0 token (stored in encrypted credentials)
  - Used for: Automation creation and webhook invocation

**Generic API Proxy:**
- Dynamic API execution via `api_proxy.rs`
  - Supports arbitrary REST APIs (GitHub, Google Workspace, Google Sheets, Gmail, etc.)
  - Authentication: API key, Bearer token, Basic auth, OAuth flows
  - Route rewriting for common service domains
  - Used by: Automation workflows, tool invocations

**Model Provider APIs (BYOM):**
- Anthropic Claude API (via Claude CLI or direct HTTP)
- OpenAI API
- Other LLM providers via BYOM (Bring Your Own Model)
  - Integration point: `src-tauri/src/engine/byom.rs`
  - Claude CLI: Spawned as subprocess for artifact generation
  - Cloud execution: Supports cloud-provided Claude tokens

## Data Storage

**Databases:**

**Primary (Local SQLite):**
- SQLite 3 (bundled via rusqlite)
  - File: `{app_data_dir}/personas.db`
  - Connection pool: r2d2 with max 8 connections
  - WAL mode enabled for concurrent access
  - Schema: personas, memories, credentials, automations, triggers, tools, executions, webhooks, audit logs, chat history
  - Encryption: Stored credentials are AES-GCM encrypted at rest
  - Client: rusqlite (Rust backend)

**Vector Store (SQLite):**
- sqlite-vec extension
  - In-database vector storage for knowledge base
  - Embeddings: Generated by fastembed (ONNX models)
  - Used for: Semantic search in memory/knowledge base

**User Database (Optional):**
- Separate SQLite instance (`personas_data.db`) for user queries
  - Isolation: Prevents user SQL from affecting app state
  - Pool: UserDbPool (r2d2)
  - Client: rusqlite

**Cloud Storage:**
- None detected (all data remains local)

**Caching:**
- In-memory: Zustand (frontend state)
- No persistent cache layer detected

## Authentication & Identity

**Auth Provider:**
- Custom (local-first with optional cloud integration)
  - Implementation: `src-tauri/src/ipc_auth.rs`, `src-tauri/src/engine/google_oauth.rs`
  - OAuth 2.0 for Google (credential design flows)
  - Supabase integration (optional, enables Google OAuth sign-in)
  - Desktop app OAuth: GCP Console "Desktop" OAuth 2.0 Client

**OAuth Configuration:**
- Google OAuth 2.0
  - Client ID/Secret: From GCP Console
  - Env vars: GCP_CLIENT_ID, GCP_CLIENT_SECRET
  - Flow: Credential design (acquiring API credentials)
  - Scope detection: Automatic from API schema

**Supabase (Optional):**
- Project: https://supabase.com
- Auth method: Google OAuth provider (optional, not required)
- Env vars: SUPABASE_URL, SUPABASE_ANON_KEY
- Row Level Security (RLS) enforced
- Used for: Optional sign-in integration

**API Credentials:**
- Storage: Encrypted in database (AES-GCM, key derived from user password via PBKDF2)
- Types: API keys, OAuth tokens, username/password, custom fields
- Schemas: Connector-specific credential templates
- Rotation: Automatic token refresh for OAuth flows

## Monitoring & Observability

**Error Tracking:**
- Sentry (optional)
  - Env vars: SENTRY_DSN, VITE_SENTRY_DSN
  - Client: sentry 0.34 (Rust), @sentry/react 10.42.0 (React)
  - Configuration: Panic handling, session tracking, source maps uploaded in CI
  - PII scrubbing: User data, request bodies, credentials stripped before sending
  - Release Health: Session tracking with anonymous device IDs
  - Production: Set via GitHub Secrets (SENTRY_DSN, SENTRY_AUTH_TOKEN, SENTRY_ORG, SENTRY_PROJECT)

**Logging:**
- Structured Logging (Rust backend)
  - Framework: tracing + tracing-subscriber
  - Output: JSON format (via tracing-subscriber)
  - Filtering: env-filter support
  - File appending: tracing-appender
  - Integration: sentry-tracing bridges logs to Sentry

- Console Logging (React frontend)
  - Framework: console.log/warn/error
  - Scrubbing: console.log/debug/info stripped in production
  - Sentry: console.error propagates to Sentry

**Metrics/APM:**
- None detected (Sentry traces_sample_rate: 0.0)

## Environment Configuration

**Required env vars:**

Development:
- `GCP_CLIENT_ID` - Google OAuth client ID
- `GCP_CLIENT_SECRET` - Google OAuth client secret
- `SENTRY_DSN` - Sentry error tracking (optional)
- `VITE_SENTRY_DSN` - Frontend Sentry DSN (optional)

Optional:
- `SUPABASE_URL` - Supabase project URL
- `SUPABASE_ANON_KEY` - Supabase public key
- `TAURI_DEV_HOST` - Dev server host override

**Compile-time env vars** (via option_env!()):
- `SENTRY_DSN` - Baked into binary
- `GCP_CLIENT_ID`, `GCP_CLIENT_SECRET` - Baked into binary
- `SUPABASE_URL`, `SUPABASE_ANON_KEY` - Baked into binary
- `CARGO_PKG_VERSION` - Release version

**Secrets location:**
- Development: `.env` file (local, not committed)
- Production: GitHub Secrets (SENTRY_DSN, SENTRY_AUTH_TOKEN, SENTRY_ORG, SENTRY_PROJECT, TAURI_SIGNING_PRIVATE_KEY, TAURI_SIGNING_PRIVATE_KEY_PASSWORD)
- Runtime: Environment variables at process start

**Security:**
- Content Security Policy (CSP): Restricted to self + necessary external domains (Sentry, GitHub, Google)
- CSP allows: cdn.simpleicons.org (icons), GitHub raw/gist (templates), Sentry ingest, Google auth
- CSP disables: iframes, object, plugins, worker-src restricted
- Database: File permissions restricted (0600/0644)
- Credentials: AES-GCM encrypted at rest, PBKDF2 key derivation
- TLS: rustls required (not OpenSSL)

## Webhooks & Callbacks

**Incoming:**
- Webhook server on port 9420 (starts on app launch)
  - Path: `/webhook/{persona_id}` (POST)
  - HMAC validation: SHA-256 with secret
  - Rate limiting: Per-tier limits via RateLimiter
  - Max body: 1MB
  - Response: Creates PersonaEvent, triggers automation
  - Used by: External systems to trigger personas

**Outgoing:**
- Zapier webhook invocation: `zapierTriggerWebhook` API
- n8n webhook support (via N8nClient)
- Generic webhook support via API proxy (automation outputs)

**Cloud Webhooks:**
- Cloud provider webhooks (optional, CloudClient integration)
  - webhook_enabled, webhook_secret stored in CloudConfig
