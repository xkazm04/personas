## Codebase map

This is a Tauri 2 desktop app for authoring and orchestrating AI agent "personas" — the frontend is a React 19 / Vite / Zustand SPA and the backend is a single Rust crate that exposes several hundred Tauri IPC commands, owns a SQLite DB with ~hundreds of migrations, and runs an execution engine plus integration clients for GitHub/GitLab/n8n/OAuth/etc.

Complexity concentrates in three places: (1) the Rust `src-tauri/src/engine/` directory (116 files, single files up to 3k LOC handle compilation, healing, dispatch, runner, prompt templating, and the P2P transport); (2) `src-tauri/src/db/` where `migrations.rs` is a single 4,187-line append-only file and `repos/resources/triggers.rs` / `repos/communication/events.rs` each exceed 2k LOC; and (3) the React template gallery (`src/features/templates/sub_generated/`) which contains ~100 TSX files plus `QuestionnaireFormGrid.tsx` at 882 lines.

Debt has accumulated most visibly in two axes that `CLAUDE.md` explicitly calls out as in-progress migrations: design-token lint warnings (~23k) and i18n extraction (~3,800 hardcoded strings). These are not worth bulk-fixing. The less-flagged debt that *is* in scope — stale ESLint disables, dead Rust symbols, ad-hoc HTTP client construction, a SSRF-safe-DNS gap — is small in LOC but higher-signal.

The IPC boundary is disciplined: every TS callsite goes through `invokeWithTimeout`, and `eslint.config.js` enforces `no-restricted-imports` for raw `invoke`. The Rust side defines two shared HTTP clients (`SHARED_HTTP` and `SSRF_SAFE_HTTP` in `lib.rs:36-47`) with a documented rationale, but 17 other sites still build their own `reqwest::Client`; most have legitimate reasons (custom headers, per-host clients), a few do not.

## Top opportunities (ranked by Score = Impact × Confidence ÷ Risk)

### 1. Close the DNS-rebinding SSRF gap on the healthcheck and trigger-validation paths
- **Area:** Rust
- **Category:** security
- **Evidence:**
  - `src-tauri/src/engine/healthcheck.rs:634` — `execute_healthcheck_request_with_strategy` builds `reqwest::Client::builder().timeout(10s).build()` and sends user-supplied `resolved_endpoint` (plus credential in `Authorization`). Pre-resolution `validate_template_url` (`:622`) and post-resolution `validate_healthcheck_url` (`:628`, implementation `:873`) both inspect the URL *string*; for `url::Host::Domain` the check only blocklists `localhost`, `.local`, `.internal`, `metadata.google.internal` — no DNS resolution. Reqwest does its own DNS lookup at connection time, enabling TOCTOU DNS rebinding (attacker domain resolves to public IP at validate time, to `169.254.169.254` or `127.0.0.1` at connect time).
  - `src-tauri/src/commands/tools/triggers.rs:316` — trigger polling-URL probe builds an identical plain client (timeout 5s, no redirects) for a user-supplied `endpoint`. Same gap.
  - The correct pattern already exists: `src-tauri/src/engine/ssrf_safe_dns.rs:60` defines `build_ssrf_safe_client()`, and `src-tauri/src/commands/credentials/credential_design.rs:177` + `src-tauri/src/engine/api_proxy.rs:591` use it. The healthcheck path was evidently written to the older (string-only) defense and never upgraded.
  - Healthchecks run **on a daily bulk sweep** per the comment at `:630-633`, so this is not an idle code path.
- **Impact:** 4 (security; sensitive creds can be exfiltrated to an attacker-influenced internal endpoint via a credential whose host is under attacker DNS control)
- **Confidence:** 4 (code paths read unambiguously; real-world exploitability depends on whether a threat actor can plant a credential with a hostile hostname — in a multi-user deployment this is plausible)
- **Risk:** 2 (one-line change per call site; behaves like the existing SSRF-safe pattern)
- **Score:** 8
- **Proposed change:** At each of the two sites, add `.dns_resolver(std::sync::Arc::new(crate::engine::ssrf_safe_dns::SsrfSafeDnsResolver))` to the existing `Client::builder()` (preserves the per-call timeout). Optionally, refactor to a helper `ssrf_safe_http_with_timeout(d)` — but that changes shape more than needed.
- **Verification:** `cargo check --features desktop`, `cargo test -p app_lib ssrf` or similar. Add a unit test that builds a resolver and asserts `is_private_ip` rejections are surfaced as errors. Manually verify the 10s / 5s timeouts still hold.

### 5. Fix `coerceArgs` dropping `undefined` inside array arguments (`tauriInvoke.ts:86`)
- **Area:** React / IPC
- **Category:** correctness
- **Evidence:** `src/lib/tauriInvoke.ts:85-98`:
  ```ts
  function coerceArgs(args: InvokeArgs): InvokeArgs {
    if (Array.isArray(args)) return args;            // <-- short-circuit
    const out: Record<string, unknown> = {};
    for (const [k, v] of Object.entries(args as Record<string, unknown>)) {
      if (v === undefined) { out[k] = null; }
      else if (v !== null && typeof v === "object" && !Array.isArray(v) && !(v instanceof Date)) {
        out[k] = coerceArgs(v as InvokeArgs);        // <-- does not recurse into arrays
      } else {
        out[k] = v;
      }
    }
    return out as InvokeArgs;
  }
  ```
  The wrapper was added specifically to survive `JSON.stringify` dropping `undefined`. But the short-circuit on arrays, plus `!Array.isArray(v)` in the object branch, means: **passing an array-of-objects with optional fields silently drops those fields**, and the Rust `serde` side rejects the payload (because `missing key !== null`). This is exactly the class of footgun the wrapper exists to prevent.
- **Impact:** 3 (latent bug; whether it's currently *triggered* depends on whether any live IPC command takes `Vec<RecordWithOptional>` args. Likely at least one does, but not confirmed.)
- **Confidence:** 4 (code reads unambiguously; real-world breakage unverified)
- **Risk:** 2 (behavior change on unknown callsites)
- **Score:** 6
- **Proposed change:** Recurse into arrays: treat arrays like objects — map each element through `coerceArgs` (for object elements) or pass through (for primitives). Add a unit test under `src/lib/__tests__` that passes `{ rows: [{ a: undefined, b: 1 }] }` and asserts the wire payload contains `{ rows: [{ a: null, b: 1 }] }`.
- **Verification:** Unit test; `npm run test` — look for no regressions in IPC-adjacent specs.

### 6. Migrate trivial ad-hoc `reqwest::Client` constructors to `SHARED_HTTP`
- **Area:** Rust
- **Category:** perf / debt
- **Evidence:** `src-tauri/src/lib.rs:36-41` defines `SHARED_HTTP` with the doc comment *"This eliminates the overhead of constructing a fresh TLS connector and connection pool on every request."* Despite this, 17 sites build their own. Many have legitimate reasons (bespoke headers, per-host cookies, redirect policies, GitHub/GitLab user-agent strings, SSRF-safe DNS). But a handful don't:
  - `src-tauri/src/engine/test_runner.rs:683` — `reqwest::Client::new()` with no config at all.
  - `src-tauri/src/engine/mcp_tools.rs` uses `crate::SHARED_HTTP.clone()` already (good precedent).
  The savings per site: one TLS session, one DNS resolution, one connection-pool instance per call. In hot paths (healthcheck sweeps, telemetry pings) this adds up.
- **Impact:** 2 (modest perf; mostly reduces surface area)
- **Confidence:** 4 (matches the stated intent; limited to simple cases)
- **Risk:** 2 (must confirm each candidate site doesn't rely on per-call connection isolation)
- **Score:** 4
- **Proposed change:** Migrate only the sites whose config is *no-op* or *pure timeout*. For each: replace with `crate::SHARED_HTTP.clone()`. Leave clients with `.redirect(...)`, `.cookie_store(...)`, custom `User-Agent`, or `.dns_resolver(...)` alone.
- **Verification:** `cargo check --features desktop`; existing integration tests if any touch the migrated modules.

### Honest self-assessment

**Where I was uncertain:**

- **Wave D-classes codemods.** My regex-based substitution is conservative but not guaranteed visually-faithful. `low-contrast` (5,385 substitutions) IS a visual change — disabled or dimmed text across the app is now full-contrast. This is per the project's design policy (CLAUDE.md explicitly forbids the muted-foreground antipattern), but it has not been eyeballed in the running app. Merging to `master` without visual QA is a real risk I'm flagging plainly.
- **Wave E translations.** I dispatched 13 Sonnet agents to machine-translate ~3,800 missing keys per locale. Placeholder preservation, brand-name / technical-identifier preservation, and register were prompt-specified, but the output ships alongside English fallback and is LIVE until reviewed. Every translated key is marked `// @llm-translated 2026-04-17` to make it filterable. For locales farther from the training distribution (Bengali, Vietnamese, Hindi), quality is lower than for German / French / Spanish / Chinese. The CLAUDE.md rule "translation teams handle localization separately" is being overridden at your explicit direction.
- **C3 migrations split.** The Rust crate compiles and the SQL statement count matches byte-for-byte across all 5 new files, but the migration system here isn't versioned — it's procedural with idempotent `PRAGMA table_info` guards. If a pattern I didn't see depends on line order across the original monolithic file, that regression wouldn't surface until a fresh DB initialization.

**Least-confident changes:**

1. **Wave D-low-contrast** (931 files, visible diff — confidence: design-correct but unreviewed).
2. **Wave E-cs / -bn / -hi translations** (Czech, Bengali, Hindi — low-resource + complex morphology).
3. **Wave B2 coerceArgs array recursion** — correct by construction and tested, but I haven't proven there's a live IPC callsite today that was broken.

**What surprised me vs. Phase 1 predictions:**

1. **21 real `rules-of-hooks` violations** — I predicted Wave C2 would mostly surface `exhaustive-deps` noise. It also surfaced 21 genuine stale-closure / conditional-hook bugs across 7 files. Those are real correctness defects the plugin absence was hiding. They're held at warn pending per-file triage.

That was harmless here (the work was wanted) but it defeats review isolation. For a high-stakes run, a pre-check hook should refuse commits that aren't on `worktree-*`.

Two purpose-built one-shot scripts (`scripts/merge-duplicate-plugins-block.mjs`, `scripts/dedupe-en-keys.mjs`) cleaned it up, but this is a systemic concurrency hazard.
